程序员系统设计与面试指南：原理、工程与面试

第 3 章 MySQL：存储与数据库

MySQL 在系统设计里通常同时承担三类角色：核心事务存储、查询与分析入口、以及其他组件的数据源。理解它的重点，不是死记零散命令，而是串起从建表、索引、事务、日志到扩展治理的整条链路。

存储引擎与整体架构

MySQL 是分层架构。连接层负责认证、连接与线程管理；SQL 层负责解析、优化与执行计划；存储引擎层负责数据组织与读写；文件系统层负责数据文件和日志文件。线上排障时，先分清问题落在哪一层，通常比盲目改 SQL 更有效。

InnoDB 是默认且最重要的存储引擎，适合绝大多数业务库：

能力InnoDBMyISAM
事务支持 ACID不支持
锁粒度行级锁表级锁
崩溃恢复支持弱
典型场景交易、订单、库存读多写少历史场景

InnoDB 的核心内存结构包括 Buffer Pool、Change Buffer、Adaptive Hash Index、Log Buffer；核心磁盘结构包括表空间、redo log、undo log、doublewrite buffer。理解这些组件有助于解释很多线上现象，例如 Buffer Pool 命中率低导致读放大、redo 刷盘策略影响提交延迟、脏页刷新跟不上时出现抖动。

建表阶段要尽量把“后续治理成本”提前考虑进去：

  • 默认使用 utf8mb4，避免字符集升级成本。
  • 主键尽量稳定、短小、单调递增，降低页分裂与二级索引体积。
  • 除 text 等少数类型外，字段尽量明确 NOT NULL 与默认值，减少三值逻辑带来的判断复杂度。
  • 外键约束在分库分表和高并发业务中往往限制扩展，很多团队更倾向于在应用层保证一致性。
  • 表结构变更要把 Online DDL、备份、回滚与发布顺序一起考虑，而不是把 ALTER TABLE 当成普通变更。

一个常见的工程经验是：DDL 先在影子环境验证，再安排低峰执行，最后配合应用发布。大表变更时如果忽略锁表、重建索引和空间放大，很容易把变更做成事故。

索引与查询执行

索引本质上是“用写入成本换查询效率”的数据结构。InnoDB 以 B+ 树作为主流索引结构，是因为它兼顾低树高、顺序访问和范围查询能力。

维度B+ 树红黑树B 树
树高低，千万级数据常见 3 到 4 层高低
磁盘 IO少多较少
范围查询强，叶子节点链表天然有序弱一般
索引页利用率高低中

InnoDB 的聚簇索引把主键与整行数据放在同一棵 B+ 树上；二级索引叶子节点保存的是主键值，所以很多查询会先走二级索引、再回表到聚簇索引读取整行。由此可以推导出几个常用概念：

  • 覆盖索引：查询所需列全部在索引里，避免回表。
  • 联合索引：索引列顺序由建索引时确定，查询是否高效仍然受最左前缀原则约束；优化器可以选择是否使用该索引、选择扫描方式或改写执行计划，但不会自由重排联合索引本身的列定义。
  • 回表：二级索引命中后再去主键索引找整行。
  • 索引失效：隐式类型转换、函数运算、前导模糊匹配、or 条件等都可能让优化器放弃索引。

从 SQL 执行路径看，一条查询通常经历解析、优化器选择索引与 Join 顺序、执行器下推条件到存储引擎、返回结果这几步。排查慢 SQL 时，应该优先看 EXPLAIN：

  • type 是否退化到 ALL 或大范围 range
  • key 是否命中预期索引
  • rows 是否扫描过多
  • extra 里是否出现 Using filesort、Using temporary

深分页是经典性能坑。LIMIT 100000, 10 的真正问题，不只是跳过 10 万行，而是通常要先扫描和排序大量数据。常见优化策略有：

  1. 用合适索引消除全表排序。
  2. 先走子查询只取主键，再回表做延迟关联。
  3. 优先使用游标分页而不是大偏移量分页。

如果查询目标是“稳定翻页”和“低抖动”，游标分页通常比 OFFSET 更适合线上系统。

事务、隔离级别与锁

事务的价值在于把多条操作封装成一个具备原子性、一致性、隔离性、持久性的整体。InnoDB 默认隔离级别是可重复读（RR），理解它要同时看 MVCC 与锁。

隔离级别可能问题
读未提交脏读
读已提交不可重复读
可重复读通过 MVCC 与间隙锁尽量避免不可重复读与幻读
串行化并发最低，但最保守

MVCC 的三个核心部件是隐藏列、undo log 版本链和 ReadView：

  • 隐藏列记录最后修改事务 ID 与回滚指针。
  • undo log 保存旧版本，串成版本链。
  • ReadView 决定当前事务能看到哪个版本。

RC 与 RR 的核心差别，在于 ReadView 创建时机不同：RC 基本是每次读取都创建新视图，RR 则在事务中第一次快照读时固定视图，因此能实现“同一事务内前后两次快照读结果一致”。

锁模型则负责补齐“当前读”和写入冲突控制：

  • 行锁适合精确命中索引的更新。
  • Gap Lock 锁住记录间隙，防止插入。
  • Next-Key Lock = 行锁 + 间隙锁，用来抑制幻读。

悲观锁适合冲突高、必须串行化关键资源的场景，例如账户转账、库存扣减。典型写法是 SELECT ... FOR UPDATE，但它只有在命中索引时才真正是“行锁”；如果条件未命中索引，很容易退化为大范围锁甚至全表锁。

乐观锁适合冲突少的场景，例如商品信息编辑、订单状态流转。工程上最重要的不是加一个 version 字段，而是每次更新后必须检查 affected_rows，否则“更新失败但代码以为成功”会成为隐蔽 bug。

死锁不是异常情况，而是高并发下的常见结果。降低死锁概率的做法包括：

  • 多资源访问统一加锁顺序。
  • 缩短事务时间，避免长事务。
  • 确保条件命中索引，缩小锁范围。
  • 业务上做好重试与幂等。

日志、复制与高可用

MySQL 里至少要分清三类日志：undo log、redo log、binlog。

日志作用关键价值
undo log保存旧版本回滚与 MVCC
redo log记录页修改崩溃恢复与持久性
binlog记录逻辑变更复制、订阅、审计

InnoDB 依赖 WAL 思想：先写 redo log，再异步刷数据页。innodb_flush_log_at_trx_commit 常见取值含义如下：

  • 1：每次提交都刷盘，最安全，也是生产常用配置。
  • 2：每次提交写 OS Buffer，fsync 按秒执行，性能更好但可能丢秒级数据。
  • 0：都按秒刷，风险最高。

主从复制是 MySQL 最常见的高可用基础形态。主库负责写，从库通过 binlog 复制追赶主库，可以实现读写分离、灾备和只读扩展，但也带来三个老问题：

  1. 主从延迟会导致读到旧数据。
  2. 主库故障切换需要明确自动化方案与一致性策略。
  3. 读写分离后，业务要知道哪些读必须回主。

当单主无法满足容量与吞吐时，常见的演进路线是：

  • 单机 InnoDB
  • 主从复制 + 读写分离
  • 垂直拆分不同业务库
  • 水平分片 + 路由

多主复制理论上提升写可用性，但实际会引入冲突解决和一致性复杂度，除非业务非常明确需要多活写入，否则大多数团队更愿意在单主或分片主的模式上建设高可用。

SQL 优化与分库分表

SQL 优化首先应遵循“先把单库单表做到合理，再谈架构升级”的顺序。典型优化路径如下：

  1. 看慢日志与 EXPLAIN，明确瓶颈在扫描、排序、回表还是锁等待。
  2. 优化 SQL 写法，减少不必要列和无效条件。
  3. 补充或调整索引。
  4. 引入缓存、批量化、异步化降低数据库压力。
  5. 仍然不够时，再考虑读写分离、分表分库、归档。

关于“单表多少行要分表”没有绝对阈值。500 万行、2000 万行这类经验值，背后真正决定因素通常是索引能否放入内存、查询模式是否稳定、磁盘 IO 是否已经成为瓶颈。比起迷信固定数字，更重要的是观测实际 QPS、延迟、Buffer Pool 命中率和空间增长速度。

分库分表常见模式包括：

  • 垂直分表：把冷热字段或低频大字段拆出去。
  • 垂直分库：按业务域拆库，例如订单库、商品库、用户库。
  • 水平分表：按用户、订单号、时间等维度打散数据。
  • 历史归档：把长期低频数据迁到冷存储或历史库。

例如订单系统按用户维度分 1000 张表，解决的是热点写入与单表膨胀问题；超过一年再归档到 HBase 或历史库，解决的是“在线表承载全部历史”的存储与查询成本问题。分片之后的核心成本则转移到路由、跨分片查询、扩容再平衡和运维治理。

常见线上问题与排查

线上 MySQL 问题常见于以下几类：

慢查询与深分页。 先查慢日志，再看执行计划，确认是否存在全表扫描、临时表、文件排序、低选择性索引和大偏移分页。深分页优先改写为延迟关联或游标分页。

锁等待与死锁。 可结合 show processlist、performance_schema.data_locks、innodb_lock_waits 等信息定位阻塞链路。排查时重点看：

  • 哪条 SQL 先持锁
  • 是否命中索引
  • 事务是否长时间未提交
  • 是否存在相反顺序加锁

主从延迟。 读写分离后，如果读流量打到从库，延迟会直接转化为数据不一致。常见处理手段是：

  • 对强一致读回主
  • 监控复制延迟并设置熔断阈值
  • 减少大事务和批量阻塞复制线程的操作

DDL 风险。 字段类型变更、加索引、删索引都可能导致锁表、重建表或大量 IO。大表 DDL 应优先考虑 Online DDL、灰度、备份和回滚预案。

连接与资源耗尽。 连接池参数要与服务端 max_connections、wait_timeout 协同设计。粗暴放大连接数往往只是把问题从排队变成雪崩。连接数估算通常要结合 QPS、平均耗时和应用节点数，而不是拍脑袋给极大值。

重点监控项。

类别关注指标
请求量select/insert/update/delete QPS
延迟慢查询数量、P95/P99
连接活跃连接、线程数、超时
InnoDBBuffer Pool 命中率、脏页、redo 刷盘
复制主从延迟、复制中断
容量表空间、索引体积、磁盘水位

本章小结

MySQL 的工程主线可以概括为四句话：

  • 用 InnoDB 的存储模型理解读写路径，而不是只把 MySQL 当 SQL 执行器。
  • 用索引、执行计划和事务模型解释性能与一致性问题。
  • 用 redo、undo、binlog 和复制机制解释可靠性与高可用。
  • 用慢查询治理、归档、分库分表和连接治理支撑系统规模增长。

真正的面试和线上问题，往往都不是孤立考一个概念，而是让你把这些点串成完整决策链。

本章面试题与追问

  1. 为什么 InnoDB 选择 B+ 树而不是红黑树或 B 树？ 追问：一棵三层或四层 B+ 树大致能承载多少数据，为什么？

  2. 聚簇索引、二级索引、覆盖索引和回表分别是什么关系？ 追问：联合索引如何设计字段顺序，为什么区分度高的列常放前面？

  3. MVCC 是如何实现可重复读的？ 追问：RC 和 RR 的 ReadView 创建时机有什么差异？

  4. Next-Key Lock 解决的是什么问题？ 追问：什么时候会退化成更大范围的锁，甚至看起来像“锁表”？

  5. redo log、undo log、binlog 分别解决什么问题？ 追问：innodb_flush_log_at_trx_commit 的不同取值如何权衡性能和数据安全？

  6. 一条 SQL 很慢，你会如何系统排查？ 追问：如果 EXPLAIN 显示 Using filesort 和大量扫描行数，你会优先怎么改？

  7. 悲观锁和乐观锁分别适合什么场景？ 追问：为什么乐观锁一定要检查 affected_rows？

  8. 主从复制为什么会有延迟，业务上如何兜底？ 追问：哪些读请求必须回主，哪些可以接受从库旧数据？

  9. 单表达到多少数据量需要分库分表？ 追问：为什么这个问题不能只靠“500 万行”或“2000 万行”这样的经验值回答？

  10. 订单系统做分库分表和历史归档时，路由层需要解决哪些问题？ 追问：扩容、跨分片查询和冷热数据迁移分别会引入什么复杂度？
