项目经历：Shopee 数字电商 C 端核心购买链路与交易安全
项目定位与规模：端到端参与 Shopee 数字电商 C 端购买主链路建设，支撑多市场，覆盖酒店、机票、电影票、账单缴费、充值等数十种数字商品，承载百万级日订单，核心链路峰值 2,000+ QPS。在供应商履约与平台自营库存并存的复杂场景下，核心负责价格/库存一致性治理与交易模型建设。

1. C 端核心购买链路与交易安全
多品类交易链路标准化

干了什么：针对充值、酒店、机票、电影票、券类等异构商品在导购、锁资源、支付履约和退款规则上的差异，将 C 端链路抽象拆分为 9 大标准阶段（Search / Detail / Cart / Booking / Order / Checkout / Payment / Fulfillment / Refund）。

怎么干的：Search 统一商品召回与导购价；Detail 聚合实时价盘与营销规则；Booking 承接供应商实时查价查库存及超时释放；Order 固化商品/资源快照；Payment / Fulfillment / Refund 通过状态机、幂等机制和补偿任务推进闭款。

效果：统一了异构品类的交易模型，大幅降低了新品类接入和交易一致性治理的开发成本。

计价中心建设与金额一致性治理

干了什么：主导统一计价中心建设，解决列表价、详情价、订单价、收银台支付价因逻辑分散导致的数据口径不一致问题。

怎么干的：按场景拆分计算能力，沉淀 Base Price → Promotion → Fee → Discount 四层价格模型和品类 Calculator 扩展机制；通过 PriceBreakdown、订单/支付双快照、线上下空跑比对、灰度迁移和异常动态拦截保障价格可解释性。

效果：彻底杜绝了价格逻辑不一致问题，项目上线后实现 0 资损事故。

首页入口与导购路径高并发性能治理

干了什么：针对首页配置大/访问集中，以及搜索、详情、试算等高频读路径在“数据新鲜度”与“系统性能”上的冲突进行分层治理。

怎么干的：实施“首页稳、列表快、详情准、创单安全”策略：采用配置瘦身、Redis 热 Key 分散、本地缓存、CDN 降级、多级缓存架构，并配合批量聚合查询与并发控制。

效果：支撑日均 5000 万+ 高频调用，核心接口 P99 延迟从 500ms 优化至 <200ms。

分布式事务与最终一致性保障

干了什么：解决支付、履约、退款回调中由于网络波动导致的重试、乱序和部分失败问题。

怎么干的：抽象支付单、订单、订单项、履约单、退款单多层模型，支持单商品/多商品合并支付、分单履约及部分退款；交易流程中引入严格幂等、防重令牌、乐观锁、延时队列和定时扫表补偿机制。

效果：有效弱化了外部供应商系统不稳定带来的系统冲击，降低了长链路交易的最终一致性风险。

2. 商品供给、库存与运营平台化
供给模型统一与平台化重构

干了什么：建设多品类商品供给与库存平台，解决早期品类模型割裂、供给链路重复、运营效率低的问题。

怎么干的：抽象 item / sku / offer / 属性 / 价格 / 库存 / 审核 / 发布 的通用供给模型，运用“动静分离”、MTI + EAV 扩展模型、策略模式和统一状态机承载品类差异。

效果：供给侧代码复用率从 <10% 提升至 90%+，新品类接入周期从 2 周缩短至 2 天。

多源供给接入与异步发布链路优化

干了什么：统一承接供应商 Push/Pull、运营 Excel、商家 Portal、开放 API 等全渠道供给来源。

怎么干的：构建基于 Kafka + Worker、Outbox 模式、本地消息表、乐观锁和 Saga 编排的异步发布链路，确保上架流程可重试、可追踪、可恢复。

效果：商品上架最终成功率从 85% 提升至 99%+。

多维库存引擎设计

干了什么：针对券码库存、数量库存、时间维度库存（如酒店房态）、供应商实时库存等多维异构场景，抽象构建统一的库存控制引擎。

怎么干的：沉淀“库存管理类型 × 库存单元类型”矩阵模型，支持库存的预占、扣减、释放、补货、对账及活动独立库存划分。

效果：支撑秒杀峰值 20,000 QPS 及日均 200 万订单，数量制预订接口 P99 耗时 <30ms。

流式批量运营框架沉淀

干了什么：解决大批量 SKU 频繁调价、设库、券码导入和批量上下架时的性能瓶颈。

怎么干的：开发统一批量运营框架，复用任务表、流式 Excel 解析、Worker Pool、分批 SQL 及完整审计追踪链路。

效果：支撑万级 SKU 级别的单次批量操作，新增批量功能开发成本降低 7 倍，运营操作耗时从小时级压缩到分钟级。

3. 稳定性与工程治理
API 资产盘点与核心链路分级

干了什么：面向数字商品交易系统内 1000+ 个 API 进行全面资产梳理与风险定级。

怎么干的：沿用户旅程映射至 10 大核心模块，综合用户影响、交易闭环、资损风险、流量特征、供应商依赖及降级可行性，制定 P0/P1 事故级、Critical 核心链路级、业务场景级三级治理口径。

异构错误码治理与精确故障归因

干了什么：治理平台内部 API 以及 200+ 外部供应商的异构错误码，提升复杂链路下的排查效率。

怎么干的：建立系统异常、业务拒绝、供应商限流/超时、状态未知、资损风险 5 大标准分类，并将错误码直接联动重试策略、熔断降级、补偿对账和告警等级；推进日志、指标、Trace 统一携带 error_code、supplier_code、biz_stage。

效果：大幅提升了多方系统交织时的故障定位速度与供应商问题归因效率。

全链路可观测性与业务 SLO 建设

干了什么：将系统稳定性监控目标从“单纯的接口可用率”扩展至“全链路业务成功率”。

怎么干的：围绕用户核心旅程确立业务 SLI/SLO，覆盖交易成功率、价格差异率、支付回调延迟、履约 pending、补偿 backlog 等指标；搭建业务、链路、应用、中间件、基础设施五层看板，支持按市场、品类、供应商、支付渠道多维下钻。

效果：实现了全局风险的分钟级感知，精准定位用户影响面和故障域。

高可用防护、告警治理与预案闭环

干了什么：应对大促流量洪峰、供应商慢依赖、异步状态不一致等系统性风险，落地全面防护体系。

怎么干的：上线入口/出口双向限流、超时预算（Timeout Budget）、指数退避重试、功能开关、自动对账与巡检报表；为 P0/P1 告警补齐明确的 Owner、Runbook、专项看板和恢复触发条件。

效果：整体告警噪音减少 70%，沉淀 20+ 核心大促/故障预案，显著提升了团队的故障响应效率与资损防控能力。
