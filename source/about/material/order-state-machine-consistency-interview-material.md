# 订单状态机与最终一致性面试材料

本文用于支撑简历中的这条经历：

> **状态机与最终一致性**：针对支付、履约、退款回调存在重试、乱序和部分失败的问题，抽象支付单、订单、订单项、履约单、退款单模型，支持单商品、多商品、合并支付、分单履约和部分退款；通过幂等、防重、乐观锁、延时队列和扫表补偿降低最终一致性风险。

面试时不要把这一条讲成“我做了状态机”。更好的叙事是：**外部系统不可控，回调天然至少一次，跨系统不能强事务，所以要把交易流程拆成可持久化的业务状态，并通过幂等、状态机、补偿和对账把不确定性收敛掉。**

---

## 一、项目一句话

在 Shopee Digital Purchase 数字电商交易链路中，支付、供应商履约和退款都依赖外部系统回调，天然存在超时、重试、乱序和部分失败。项目通过抽象支付单、订单、订单项、履约单和退款单模型，并用状态机、幂等键、乐观锁、延时任务、扫表补偿和对账机制，支撑单商品、多商品、合并支付、分单履约和部分退款等复杂交易形态，降低订单、支付、履约、退款之间的最终一致性风险。

---

## 二、30 秒项目介绍

Digital Purchase 覆盖充值、账单、券类、酒店、机票、电影票等多品类。用户下单后，订单系统要和支付系统、供应商履约系统、退款系统协作。这里最大的问题是外部系统不可控：支付回调可能重复或延迟，供应商履约可能异步成功或失败，退款也可能长时间 Pending。如果只靠同步调用和简单字段更新，很容易出现订单已支付但本地未更新、支付成功但履约失败、退款重复发起、部分退款金额不一致等问题。

我的做法是把交易链路拆成几个有自己生命周期的业务对象：支付单、订单、订单项、履约单、退款单。每个对象用状态机约束合法流转；每次外部回调都先做幂等和状态检查，再用乐观锁推进状态；对支付 Pending、履约 Pending、退款 Pending 等中间状态，通过延时队列、定时扫表、主动查询和对账补偿保证最终闭环。

---

## 三、2 分钟项目介绍

这个项目的背景是数字商品交易链路越来越复杂。早期很多品类是一单一商品、一单一支付、一支付一履约，流程相对简单。但随着酒店、机票、电影票、保险、券类等品类接入，交易形态开始变复杂：

- 一个支付单可能对应多个订单，例如主商品 + 附加服务合并支付。
- 一个订单可能包含多个订单项，例如电影票 + 小食套餐。
- 一个订单可能拆成多个履约动作，例如不同供应商或不同数字交付方式。
- 一个订单可能多次部分退款，例如部分履约失败或用户只退其中一个 item。
- 支付、履约、退款都可能通过外部系统异步回调推进。

如果只在订单表上放几个状态字段，很快会遇到问题：支付回调重复时可能重复推进状态；支付成功但本地更新失败时，订单会卡在待支付；订单超时关闭和支付成功回调可能竞态；履约失败后需要自动退款，但退款又可能 Pending；部分退款时可退金额需要扣除历史已退金额，不能靠当前价格重新计算。

解决思路是把“交易事实”和“外部动作”拆开建模。支付单负责资金状态，订单负责交易主状态，订单项负责商品粒度的履约和售后，履约单负责供应商交付，退款单负责逆向资金动作。不同对象之间不追求一个大事务，而是通过领域事件、状态机和补偿任务最终一致。

工程上主要做了五层保护：

1. **模型边界**：支付单、订单、订单项、履约单、退款单职责清晰。
2. **状态机约束**：只允许合法状态流转，终态不可逆，避免乱序回调覆盖正确状态。
3. **幂等防重**：外部流水号、业务单号、回调事件 ID、退款批次号建立唯一约束。
4. **乐观锁推进**：更新状态时带原状态和版本号，避免并发回调重复更新。
5. **补偿与对账**：Pending 状态通过延时任务、扫表查询、主动查单和对账报表修复。

最后这个方案支撑了多品类交易统一，能够处理合并支付、分单履约、部分退款等复杂场景，并让支付、履约、退款异常从“靠人工查日志”变成“有状态、有任务、有对账、有恢复路径”。

## 四、背景：为什么这个问题一定会出现

### 1. 数字商品不是单一订单模型

Digital Purchase 的品类差异很大：

| 品类 | 交易特点 |
| --- | --- |
| 充值 / 账单 | 支付成功后调用供应商充值或缴费，履约通常异步返回。 |
| EVoucher / Gift Card | 需要券码库存、数字交付、可能存在核销和部分退款。 |
| Movie Ticket | 可能包含电影票、小食、选座、出票回调。 |
| Hotel / Flight | 需要 Booking、供应商确认、支付超时释放、出票或锁房状态回写。 |
| 附加服务 / 保险 | 可能和主商品合并支付，但独立履约和退款。 |

如果每个品类都自己做一套状态流转，短期能上线，但长期会导致：

- 状态命名不一致。
- 支付成功后的履约入口不统一。
- 退款规则散落在品类代码里。
- 线上问题难以通过统一监控排查。
- 新品类接入需要重复实现订单、支付、履约、退款链路。

### 2. 外部系统回调是“不可靠但必须相信”的输入

支付、供应商履约、退款通常都依赖外部系统：

- 支付渠道或支付平台回调支付结果。
- 供应商 / Hub 回调履约结果。
- 支付平台回调退款结果。

这些回调具备典型分布式系统特征：

- **至少一次投递**：可能重复回调。
- **延迟到达**：可能几秒、几分钟甚至更久后才到。
- **乱序到达**：查询任务先查到成功，随后失败回调才到。
- **部分失败**：外部成功、本地更新失败；本地成功、事件投递失败。
- **不可事务化**：无法把本地 DB 和外部支付/供应商系统放在一个事务里。

面试表达：

> 这类系统不能假设一次同步调用就能完成交易闭环。我们要接受外部系统不可靠，然后通过状态机、幂等、补偿和对账把不可靠输入收敛成可靠结果。

## 五、核心问题

### 问题一：支付回调重复

支付平台可能因为没有收到成功响应而重试回调。重复回调如果没有幂等，会导致：

- 重复推进订单状态。
- 重复触发履约。
- 重复发券或重复发货。
- 重复发送通知。

### 问题二：回调乱序

典型场景：

```text
T1：支付 Pending
T2：补偿任务主动查单，发现支付成功，更新为 Success
T3：支付平台之前的失败回调延迟到达
```

如果失败回调不做状态检查，可能把已经成功的支付单覆盖成失败。

### 问题三：部分失败

跨系统链路里经常出现局部成功：

```text
支付平台已经扣款成功
  但订单服务更新状态失败

订单已经标记支付成功
  但发送履约消息失败

供应商履约成功
  但本地履约状态更新失败

退款渠道受理成功
  但退款回调丢失
```

这些都不能靠同步重试无限解决，必须有持久化状态和后台补偿。

### 问题四：合并支付和部分退款

简单场景是一笔支付对应一个订单，但复杂场景会变成：

```text
一个 pay_order_id
  → 多个 order_id
      → 多个 order_item
          → 多个 fulfillment / refund
```

挑战：

- 一个子订单履约失败，是否影响整个支付单？
- 一个订单项退款，退款金额如何计算？
- 合并支付用了订单级优惠，部分退款如何分摊？
- 全额退款后支付单状态如何更新？
- 多次部分退款如何防止超过可退金额？

### 问题五：状态字段散落，缺少统一生命周期

如果订单、支付、履约、退款各自更新状态，但没有统一约束，会出现：

- 支付失败的订单进入履约。
- 已取消订单又被支付成功回调拉回。
- 已退款订单继续履约。
- 履约失败但订单仍显示完成。
- 退款成功但订单售后状态未同步。

## 六、设计目标

这个项目不是追求强一致，而是追求**可控的最终一致**。

### 1. 正确性目标

- 外部重复回调不会产生重复副作用。
- 乱序回调不会覆盖更可信的状态。
- 支付成功、履约成功、退款成功等关键事实最终能落到本地。
- 部分失败后有明确补偿路径。
- 退款金额不会超过可退金额。

### 2. 可恢复目标

- Pending 状态可以被后台任务扫描和主动查询。
- 补偿任务可以重复执行且幂等。
- 对账发现差异后可以定位到具体支付单、订单、履约单或退款单。
- 人工介入时有状态、流水、日志和操作入口。

### 3. 可扩展目标

- 支持单商品和多商品。
- 支持合并支付。
- 支持分单履约。
- 支持部分退款和多次退款。
- 新品类接入时复用统一状态机和补偿机制。

---

## 七、核心方案总览

可以把方案分成五层：

```text
业务模型层
  支付单 / 订单 / 订单项 / 履约单 / 退款单

状态机层
  支付状态机 / 订单状态机 / 履约状态机 / 退款状态机

幂等与并发层
  幂等键 / 唯一索引 / 乐观锁 / 状态条件更新

异步补偿层
  延时队列 / 定时扫表 / 主动查单 / 重试 / DLQ

对账观测层
  状态监控 / 差异报表 / 操作日志 / 人工补偿
```

面试表达：

> 我没有把最终一致性理解成“异步就行”，而是把它拆成状态可持久化、动作可幂等、失败可补偿、差异可对账。

## 八、核心设计一：模型拆分

### 1. 支付单 PayOrder

支付单表达一次用户支付行为。

典型字段：

- `pay_order_id`
- `user_id`
- `total_amount`
- `paid_amount`
- `payment_status`
- `payment_channel`
- `channel_trade_no`
- `payment_snapshot_id`
- `version`

职责：

- 管理支付初始化、支付 Pending、支付成功、失败、过期、取消。
- 记录支付渠道、支付金额、支付快照。
- 承接支付回调和主动查单结果。

### 2. 订单 Order

订单表达一笔业务交易。

典型字段：

- `order_id`
- `pay_order_id`
- `user_id`
- `category_id`
- `order_status`
- `order_amount`
- `snapshot_id`
- `version`

职责：

- 管理订单创建、支付后、履约中、已履约、已完成、已取消、已退款。
- 固化商品、价格、履约和售后规则快照。
- 和支付单通过 `pay_order_id` 关联。

### 3. 订单项 OrderItem

订单项表达商品粒度。

职责：

- 支持一个订单多个商品。
- 支持商品级履约。
- 支持商品级退款和退款金额分摊。
- 保存商品快照和价格分摊结果。

### 4. 履约单 Fulfillment

履约单表达供应商或系统交付动作。

职责：

- 记录供应商履约请求和返回。
- 支持同步履约、异步履约、回调履约。
- 支持失败重试、查询补偿、失败退款。

### 5. 退款单 Refund

退款单表达一次逆向资金动作。

职责：

- 每次退款生成独立退款单。
- 记录退款原因、金额、渠道退款单号。
- 支持 Pending、Processing、Success、Failed。
- 多次部分退款时累加已退金额，防止超退。

## 九、核心设计二：状态机

### 1. 支付状态机

```text
INIT
  → PENDING
  → SUCCESS
  → FAILED
  → EXPIRED
  → CANCELED

SUCCESS
  → PARTIAL_REFUNDED
  → REFUNDED
```

关键规则：

- `SUCCESS` 是强事实，不能被后续失败回调覆盖。
- `FAILED / EXPIRED / CANCELED` 一般不能直接转 `SUCCESS`，除非主动查单确认渠道已成功，并进入异常修复流程。
- 支付回调只推进支付单，不在回调线程同步串起所有下游长链路。

### 2. 订单状态机

```text
CREATED
  → EXPIRED
  → PAID
  → CANCELED

PAID
  → FULFILLING
  → FULFILLED
  → COMPLETED
  → REFUNDING
  → REFUNDED
```

关键规则：

- 未支付订单超时可关闭。
- 已支付订单不能被普通超时关单任务关闭。
- 支付成功后进入履约链路。
- 履约失败根据品类策略进入自动退款或人工处理。

### 3. 履约状态机

```text
NOT_YET
  → READY
  → PENDING
  → SUCCESS
  → FAILED
```

关键规则：

- 支付成功后才允许履约。
- 履约请求可以重复发起，但必须有幂等键。
- 供应商回调重复时，只允许从非终态推进到终态。
- `SUCCESS` 和 `FAILED` 是终态，不能互相覆盖，冲突进入人工核查或对账差异。

### 4. 退款状态机

```text
CREATED
  → PROCESSING
  → SUCCESS
  → FAILED
```

部分退款时，支付单可以是：

```text
SUCCESS
  → PARTIAL_REFUNDED
  → REFUNDED
```

关键规则：

- 只有支付成功的订单才能退款。
- 每次退款有独立退款单和幂等键。
- 累计退款金额不能超过实付金额。
- 退款成功后同步订单售后状态。

### 5. 状态机实现要点

不要写成裸字段随便 update，而是集中校验：

```go
func TransitPayment(ctx context.Context, paymentID int64, event PaymentEvent, target PaymentStatus) error {
    p := repo.GetPayment(paymentID)

    if !CanTransit(p.Status, target, event) {
        return ErrInvalidTransition
    }

    affected := repo.UpdatePaymentStatus(ctx, paymentID, p.Status, target, p.Version)
    if affected == 0 {
        return ErrConcurrentUpdate
    }

    return nil
}
```

SQL 侧带状态条件和版本号：

```sql
UPDATE pay_order_tab
SET payment_status = ?, version = version + 1, update_time = NOW()
WHERE pay_order_id = ?
  AND payment_status = ?
  AND version = ?;
```

面试回答重点：

> 状态机不是画图，而是所有状态更新必须经过统一入口和存储层条件更新。这样重复回调、并发任务和人工补偿才不会互相覆盖。

## 十、核心设计三：幂等、防重和并发控制

### 1. 幂等键设计

| 场景 | 幂等键 |
| --- | --- |
| 创建支付单 | `order_id + pay_scene` 或业务生成的 `request_id` |
| 支付回调 | 支付渠道流水号 / 支付平台交易号 |
| 履约请求 | `order_id + fulfillment_type` 或供应商 reference id |
| 履约回调 | 供应商订单号 + 回调事件类型 |
| 创建退款单 | `order_id + refund_batch_no` |
| 退款回调 | 渠道退款单号 |
| 消息消费 | `event_id` 或业务单号 + 事件类型 |

### 2. 幂等处理流程

以支付回调为例：

```text
1. 验签
2. 根据渠道流水号查幂等记录
3. 如果已处理成功，直接返回成功
4. 查询支付单和当前状态
5. 校验状态机是否允许推进
6. 带版本号更新支付单状态
7. 同事务写 Outbox / 事件表
8. 返回渠道成功
```

### 3. 重复请求和参数不一致

幂等不是“同 key 一律成功”。还要处理相同幂等键但参数不同：

```text
同一个 refund_batch_no：
  第一次请求退款 100
  第二次请求退款 200
```

这种情况应该拒绝，而不是复用第一次结果。可以保存请求摘要：

```go
requestHash := sha256(orderID + amount + reason + items)
```

如果同一幂等键的请求摘要不同，返回参数冲突。

### 4. 并发控制

常用组合：

- 唯一索引防重复创建。
- 状态条件更新防重复推进。
- `version` 乐观锁防并发覆盖。
- 退款可用行锁或原子 SQL 防超退。

退款防超退示例：

```sql
UPDATE pay_order_tab
SET refunded_amount = refunded_amount + ?
WHERE pay_order_id = ?
  AND paid_amount - refunded_amount >= ?;
```

## 十一、核心设计四：异步补偿

### 1. 为什么需要补偿

因为跨外部系统无法强事务。即使本地代码完全正确，也会遇到：

- 回调丢失。
- 回调处理超时。
- DB 短暂不可用。
- MQ 投递失败。
- 供应商接口超时。
- 退款渠道长时间 Pending。

所以系统必须主动恢复。

### 2. 补偿任务类型

| 补偿任务 | 触发条件 | 动作 |
| --- | --- | --- |
| 支付 Pending 补偿 | 支付单长时间 PENDING | 主动查支付平台交易状态 |
| 支付过期补偿 | 未支付超过有效期 | 关闭支付单和订单，释放资源 |
| 履约未发起补偿 | 支付成功但履约未开始 | 重新投递履约事件 |
| 履约 Pending 补偿 | 履约单长时间 PENDING | 查询供应商履约结果或重试发货 |
| 履约失败补偿 | 供应商确认失败 | 进入自动退款或人工处理 |
| 退款 Pending 补偿 | 退款单长时间 PROCESSING | 主动查退款渠道结果 |
| 对账补偿 | 支付/退款渠道对账差异 | 生成差异单，自动修复或人工确认 |

### 3. 延时队列

适合处理时间点明确的任务：

- 订单 30 分钟未支付，关闭订单。
- Booking Token 到期释放资源。
- 支付初始化后 5 分钟仍 Pending，主动查单。
- 履约请求后 1 分钟未收到回调，查询供应商。

### 4. 扫表补偿

适合兜底处理：

```sql
SELECT *
FROM pay_order_tab
WHERE payment_status = 'PENDING'
  AND update_time < NOW() - INTERVAL 5 MINUTE
LIMIT 100;
```

注意点：

- 分页扫描，避免大事务。
- 按 user_id / order_id / create_time 分片。
- 加全局锁或分区锁，避免多实例重复扫同一批。
- 每个补偿动作必须幂等。
- 需要最大重试次数和人工介入状态。

### 5. Outbox 保证事件可靠发布

支付成功后需要通知订单或履约，不建议在支付回调线程同步调用多个下游。更稳妥的是：

```text
本地事务：
  1. 更新支付单 SUCCESS
  2. 写 Outbox: PaymentSucceeded

异步 Worker：
  1. 扫描 Outbox
  2. 发布 OrderPaid / StartFulfillment 事件
  3. 消费端幂等处理
```

面试回答重点：

> 支付回调线程要短，核心是验签、幂等、更新支付事实和落可靠事件。履约、通知、积分、报表都应该通过事件异步推进，避免一个下游慢把支付回调拖死。

## 十二、典型异常场景怎么讲

### 场景一：支付成功但订单仍是待支付

可能原因：

- 支付回调处理失败。
- 支付成功事件未投递。
- 订单消费 PaymentSucceeded 事件失败。

解决：

1. 支付 Pending 补偿任务主动查支付平台。
2. 确认支付成功后更新支付单。
3. 检查 Outbox 是否投递。
4. 订单侧消费事件时幂等推进 `CREATED → PAID`。
5. 对账报表发现支付成功但订单未支付的差异。

面试表达：

> 这类问题不能靠人工直接改订单状态，必须从支付事实出发，查渠道、查支付单、查事件投递，再通过状态机推进订单。

### 场景二：订单超时关闭和支付成功回调竞态

时间线：

```text
T1：订单即将过期
T2：关单任务开始执行
T3：用户支付成功
T4：支付成功回调到达
```

处理原则：

- 定义终态优先级。
- 如果支付渠道已经成功，不能简单关闭订单。
- 关单任务执行前主动检查支付状态。
- 支付成功但订单已关，进入异常修复或退款流程。

关键点：

> 关单和支付成功都是状态机事件，必须通过状态条件更新竞争，谁先合法落库谁生效，另一个事件根据当前状态走补偿分支。

### 场景三：支付成功但履约失败

流程：

```text
Payment SUCCESS
  → Order PAID
  → Fulfillment PENDING
  → Fulfillment FAILED
  → Refund CREATED / PROCESSING / SUCCESS
```

处理：

- 履约失败不回滚支付成功事实。
- 根据品类策略判断自动退款、重试履约还是人工处理。
- 退款单独建模，不能直接改支付金额。
- 用户侧展示“履约失败，退款处理中”。

### 场景四：供应商履约回调重复

处理：

- 供应商订单号 + 回调事件类型做幂等键。
- 查询履约单当前状态。
- 如果已经是 SUCCESS，再收到 SUCCESS，直接幂等返回。
- 如果已经是 SUCCESS，再收到 FAILED，不能覆盖，进入冲突告警。

### 场景五：部分退款并发提交

问题：

```text
支付成功金额：100
已退款：0

请求 A 退款：70
请求 B 退款：70
```

如果两个请求都先查已退金额，再更新，就可能超退。

解决：

- 创建退款单使用幂等键。
- 支付单行锁或原子 SQL 校验可退金额。
- 累计退款金额单调递增。
- 全额退款后支付单状态进入 REFUNDED。

## 十三、收益怎么讲

不要只说“保证一致性”，要讲具体收益。

### 1. 支撑复杂交易形态

- 从一单一商品扩展到多商品订单。
- 支持一个支付单对应多个订单。
- 支持合并支付和分单履约。
- 支持部分退款和多次退款。
- 支持供应商异步履约和退款回调。

### 2. 降低重复副作用风险

- 重复支付回调不会重复履约。
- 重复履约回调不会重复交付。
- 重复退款请求不会超退。
- 乱序回调不会覆盖终态。

### 3. 提升故障恢复能力

- Pending 状态可被补偿任务自动扫描。
- 回调丢失可通过主动查询修复。
- MQ 投递失败可通过 Outbox 重试。
- 对账差异可以定位到具体业务单据。

### 4. 提升排查效率

以前排查要看多个系统日志，现在可以沿着：

```text
pay_order_id
  → order_id
  → order_item_id
  → fulfillment_id
  → refund_id
  → event_id / callback_id
```

一路追踪状态、流水、回调和补偿任务。

### 5. 复用到多品类

新品类不需要重新设计订单、支付、履约、退款状态机，只需要定义：

- 是否需要 Booking。
- 履约是同步还是异步。
- 是否支持部分退款。
- 履约失败是否自动退款。
- 退款是否需要人工审核。

## 十四、面试高频追问

### Q1：为什么不能用分布式事务保证强一致？

支付平台、供应商和退款渠道都是外部系统，不可能和本地数据库放在一个事务里。即使用 2PC，也要求参与方支持 Prepare/Commit/Rollback，这在第三方支付和供应商系统里通常不可行。

更现实的方案是 Saga + 状态机 + 补偿 + 对账：

- 本地状态先落库。
- 外部动作通过幂等接口执行。
- 中间状态可恢复。
- 最终通过查询和对账闭环。

### Q2：为什么要拆支付单、订单、履约单、退款单？

因为它们的生命周期不同：

- 支付单关注资金状态。
- 订单关注交易状态。
- 履约单关注供应商交付。
- 退款单关注逆向资金动作。

如果全部塞进订单表，会导致状态字段爆炸，并且无法表达“一笔支付多个订单”“一个订单多个履约”“多次部分退款”等场景。

### Q3：如何处理重复支付回调？

步骤：

1. 验签。
2. 用渠道流水号或支付平台交易号做幂等。
3. 查支付单当前状态。
4. 如果已成功，直接返回成功。
5. 如果是合法状态，带版本号更新。
6. 同事务写支付成功事件。

### Q4：如何处理乱序回调？

核心是状态机和终态保护：

- 成功不能被失败覆盖。
- 已退款不能回到已支付。
- 已履约不能被履约失败覆盖。
- 冲突事件进入差异表或人工核查。

### Q5：补偿任务会不会重复执行？

会，所以补偿任务本身必须幂等。补偿只是重新触发状态推进，不能绕过幂等和状态机。例如重复查单发现支付成功，多次执行都只能把支付单推进到 SUCCESS 一次。

### Q6：退款金额怎么保证不超？

三层保护：

1. 业务层基于订单项实付金额和已退金额计算可退。
2. 数据库层对支付单或退款汇总行加锁。
3. 原子 SQL 保证 `paid_amount - refunded_amount >= refund_amount`。

### Q7：支付成功后履约失败，是不是应该回滚支付？

不能回滚支付成功事实。支付成功是资金事实，履约失败是交易后续失败。正确做法是：

- 记录履约失败。
- 根据品类策略重试履约、自动退款或人工处理。
- 如果退款，创建退款单并推进退款状态机。

### Q8：状态机和幂等是什么关系？

幂等解决“同一个动作执行多次结果一致”，状态机解决“动作能不能在当前状态执行”。两者缺一不可：

- 只有幂等没有状态机，乱序事件可能覆盖正确状态。
- 只有状态机没有幂等，重复事件可能产生重复副作用。

## 十五、回答框架

面试时可以按这个顺序讲：

1. **背景**：多品类交易，支付、履约、退款都依赖外部回调。
2. **问题**：回调重试、乱序、部分失败，简单订单状态字段无法表达复杂交易。
3. **挑战**：跨系统不能强事务，既要保证资金安全，又要支持合并支付、分单履约和部分退款。
4. **方案**：拆模型、建状态机、幂等防重、乐观锁推进、延时队列和扫表补偿、对账闭环。
5. **收益**：复杂交易形态可复用，重复副作用降低，异常可恢复，排查效率提升。
