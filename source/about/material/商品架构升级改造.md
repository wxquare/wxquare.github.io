# 商品架构升级改造面试材料

这份材料用于把“商品系统从单一的 `topup / bill` 商品，逐步演进到酒店 / 电影票 / 机票等第三方供给，再到本地生活券平台化治理”的经历，整理成适合面试表达的 Markdown 面试稿。

建议使用方式：

- 面试开场时讲 `30 秒项目介绍` 或 `2 分钟项目介绍`。
- 简历中优先使用 `简历版本项目描述`。
- 面试官追问架构演进时，展开 `核心问题`、`核心改造方案` 和 `高频追问`。
- 白板环节重点画 `供给平台 -> draft_tab 工作区 -> 审核治理 -> item_tab -> 库存中心` 的边界关系。

---

## 一、项目一句话

我负责推动电商平台商品体系从早期以 `topup` 和 `bill` 为主的单体商品系统，升级为支持酒店、电影票、机票等第三方供给接入，并进一步支撑本地生活券商户建券、QC 审核、库存治理的分布式商品平台，重点解决多源商品接入、异步任务监控、商品模型污染、审核发布解耦和库存资损风险问题。

---

## 二、30 秒项目介绍

平台早期主要卖 `topup` 和 `bill` 这类标准化商品，商品模型比较简单，库存约束弱，系统本质上是一个轻量商品配置后台。后来业务先接入酒店、电影票、机票等第三方供应商商品，需要定时同步商品、价格和库存；再往后又做本地生活券，支持商户建券、Excel 批量导入、系统生码、库存控制和 QC 审核。原来的商品系统很快演变成一个把供给、商品、库存、审核、批处理任务全部揉在一起的大单体。

我主导做了一次商品架构升级，把系统拆成供给接入平台、商户侧 `draft_tab` 工作区、审核治理能力、正式商品中心和独立库存能力，并补齐同步链路与批量任务的可观测性。改造之后，新品类接入更标准化，商品模型更干净，审核链路更清晰，库存和券码风险也更可控。

## 三、2 分钟项目介绍

这个项目最能体现的不是“我设计了一个完美架构”，而是典型的“业务倒逼架构演进”。

最开始平台主要卖 `topup` 和 `bill` 这类标准化商品，商品模型轻，库存约束弱，系统更像一个简单的商品配置后台。后来平台开始接入酒店、电影票、机票等第三方供应商商品，商品数据量、字段复杂度和变更频率都明显上升，系统需要处理定时同步、价格变化、库存变化和上下架变化。再往后，平台又开始做本地生活券，支持商户自主建券和 Excel 批量导入，这时候系统里又新增了生码、控库存、QC 审核和发布治理等能力。

随着这些业务叠加，原有商品系统很快暴露出几个核心问题。

1. 商品、供给、库存、审核、批量任务都耦合在一个系统里，边界非常混乱。
2. 酒店、机票、电影票、本地券等异构品类不断往商品模型里堆字段，代码里充满基于品类的 `if-else`。
3. 第三方同步链路和商户 Excel 批量建券任务分散在各类 background job 中，缺少统一监控，线上排查成本很高。
4. 本地券开始涉及券码生成、锁定、释放和库存扣减，如果继续按商品字段方式处理，资损风险会越来越高。
5. 商户建券后不再是“创建即上架”，必须区分草稿、审核中和正式可售几个阶段。

所以我推动的核心改造思路是：把商品从一个“单体展示对象”升级成一个“分层治理的平台”。

我先把第三方同步和商户批量导入相关能力抽到供给接入平台，统一做协议适配、数据清洗、任务流水线和监控；商户侧工作区在 `draft + staging` 两种建模思路里，我最终选择只保留 `draft_tab`，通过 `DRAFT / UNDER_REVIEW / REJECTED` 等状态管理草稿和提审过程；审核通过后再正式发布到 `item_tab`；最后把券码和库存独立治理，专门处理锁码、扣减、超时释放和补偿。

这次改造的价值不只是“把服务拆了”，而是让系统具备了继续扩展的能力。后续无论接入新的第三方品类，还是新增本地券供给，更多都变成标准化接入和治理问题，而不是继续污染核心商品系统。

## 五、业务演进脉络

### 阶段 1：Topup / Bill 商品阶段

- 平台主要售卖 `topup` 和 `bill` 这类标准化商品。
- 商品大多由平台运营后台配置，库存约束弱。
- 商品系统本质上是一个轻量的商品配置后台。

### 阶段 2：第三方供应商接入阶段

- 接入酒店、电影票、机票等第三方货源。
- 平台需要定时同步商品、价格、库存、上下架信息。
- 外部数据格式不统一，异步同步链路开始变复杂。

### 阶段 3：本地生活券阶段

- 商户开始参与商品供给，支持自助建券和 Excel 批量导入。
- 系统需要支持系统生码、库存控制、审核发布。
- 商品开始同时带有供给、库存、审核和履约属性。

### 阶段 4：商品体系平台化治理阶段

- 需要把供给接入、商品工作区、审核治理、正式商品、库存治理解耦。
- 需要统一标准商品模型，避免每接一个品类就污染一次主模型。
- 需要让新品类接入变成标准接入问题，而不是系统重写问题。

## 六、核心问题

### 1. 大单体越界耦合

早期商品系统只服务 `topup` 和 `bill` 这类简单商品时没有问题，但随着第三方接入和本地券扩张，商品展示、供给同步、商户建券、库存控制、审核发布都堆在一个系统里，职责越来越混乱。

### 2. 商品模型被异构品类持续污染

酒店关注房型、日期和房态，电影票关注场次和座位，本地券关注券码、库存、审核状态。不同品类字段不断叠加，导致商品表和代码不断膨胀，`if-else` 越来越多。

### 3. 异步任务链路黑盒化

第三方同步链路、商户 Excel 批量建券和后台导入任务散落在各类 background job 中，缺乏统一任务编号、状态跟踪和监控告警。一旦数据异常，排查成本非常高。

### 4. 审核边界不清晰

商户建券引入后，商品不再是“创建即售卖”。系统必须区分“草稿中”“审核中”和“正式可售”几个不同阶段，否则错误配置会直接进入前台交易链路。

### 5. 库存和券码开始成为交易安全问题

`topup / bill` 阶段库存不是重点，但本地券一旦做起来，就会出现库存扣减、券码锁定、支付超时释放、补偿回滚等问题。如果继续把库存当成商品字段处理，风险会迅速放大。

### 6. 商品“已创建”不等于“真正可售”

像 `voucher` 这类数字凭证商品，创建商品信息和上传库存往往是两个独立步骤。也就是说，商品可能已经审核通过并发布到 `item_tab`，但如果库存中心还没有注入有效券码或配额，前台依然不应该开放购买。

如果系统没有把“商品状态”和“可售状态”拆开，就很容易出现两个问题：

- 商品已经创建成功，前台却错误亮起“立即购买”，导致下单后无券可发。
- 库存一旦卖完，又试图频繁回写商品主状态，导致商品中心承受不必要的高频写压力。

## 七、我的核心任务

我在这个项目里的任务，不是简单做一个新服务，而是完成一次架构边界重划：

1. 梳理多品类商品体系的业务边界和生命周期。
2. 把外部供给复杂性隔离在供给接入平台。
3. 在 `draft + staging` 的方案权衡中，最终让商户侧草稿和提审流程通过单 `draft_tab` + 状态机承接。
4. 让正式交易链路只读取审核通过后的 `item_tab`，并把“商品状态”和“可售状态”解耦。
5. 把库存和券码从商品属性提升为独立交易安全能力。
6. 补齐第三方同步和批量导入链路的可观测性。

## 八、核心改造方案

### 1. 供给接入平台：隔离外部复杂性

我把酒店、电影票、机票等第三方供应商接入，以及商户 Excel 批量建券等异构输入，统一沉到供给接入平台，负责：

- 供应商协议适配。
- 商户导入任务接收与解析。
- 数据校验、清洗和字段映射。
- 同步任务与批量任务的统一编排。
- 外部异常、脏数据和缺字段的兜底处理。

供给平台的定位是“纯粹的数据接入和转换器”，不直接承载交易主逻辑。

### 2. 正式商品中心：捍卫主数据边界

正式商品中心保留稳定的标准商品主数据，也就是最终对交易、搜索、营销暴露的 `item_tab`，负责：

- 统一商品 ID 和标准视图。
- 类目、基础属性、上下架状态等核心主数据。
- 面向交易、搜索、营销输出稳定模型。
- 正式可售商品状态控制。

我的原则是：让交易系统看到稳定模型，而不是直接感知供应商原始结构或商户编辑中的中间态数据。

### 3. 审核治理：通过单 `draft_tab` + 状态机控制可售性

这是这次材料里一个很关键的边界判断。

在建模上，我们一开始评估过 `draft + staging` 双表方案，也评估过单表方案，最终选择的是“只保留一个 `draft_tab`”，不再单独拆一张 `staging_tab`。也就是商户侧的工作区只有一张草稿 / 待审核表，通过状态机区分不同阶段，例如：

```text
DRAFT -> UNDER_REVIEW -> REJECTED / APPROVED
```

配合正式线上 `item_tab`，整体形成：

```text
draft_tab + qc_flow + item_tab
```

我的落地方式是：

1. 供给平台把数据清洗成标准商品模型。
2. 商户保存时先落 `draft_tab`，允许数据不完整，状态为 `DRAFT`。
3. 商户提交审核时，不做跨表复制，而是直接把 `draft_tab` 状态推进到 `UNDER_REVIEW`。
4. 审核治理能力消费提审事件，触发自动规则校验和人工 QC。
5. 审核通过后，再发布到正式 `item_tab`，并触发后续库存初始化或券码生成。

这样设计的原因有四点：

- 避免供给平台变成又接协议、又做审核、又维护临时商品表的大胖子。
- 避免在供给层再造一套“影子商品中心”。
- 相比 `draft + staging` 双表方案，单表状态流转更轻，少一套 DDL、少一次跨表迁移，业务实现和查询都更简单。
- 通过状态机锁死交易链路，让“草稿中”“审核中”和“正式可售”彻底解耦。

这个方案也有明确边界：它更适合审核周期可控、商户提审后不需要长时间并发二次编辑的场景。对我们当时的本地生活券业务来说，这个权衡是可以接受的，所以在 `draft + staging` 的选择中，我最终保留了 `draft_tab`，没有再引入 `staging_tab`。

### 4. 库存中心：把附属属性升级为交易安全能力

库存能力从“商品字段的一部分”升级为“交易安全能力的一部分”，重点治理：

- 本地券库存扣减。
- 券码锁定与释放。
- 支付超时后的库存回补。
- 稀缺型商品的可售数量控制。
- 后续补偿和状态一致性。

核心判断是：库存不是展示属性，而是高风险写路径。

对于 `voucher` 这类数字凭证商品，我还专门把“商品状态”和“可售状态”拆成两个概念：

- 商品状态：存放在正式商品中心的 `item_tab` 中，表示商品是否已经通过审核、是否合规可发布，例如 `ACTIVE / DISABLED`。
- 可售状态：不是直接落库的单一字段，而是前台按多因子动态聚合出来的结果。只有当商品状态合法，且库存中心判定存在可用券码或配额时，前台才真正展示“立即购买”。

这类设计是因为有些商品存在天然的双阶段流程：

1. 先创建商品主数据。
2. 再异步上传库存、券码或配额。

也就是说，商品即使已经创建成功、审核通过，也不一定立刻可售。对于这类场景，我采用的是“状态与可售解耦”的设计：

```text
Salable = (Item.Status == ACTIVE) && (Stock.Available > 0)
```

这样做的好处是：

- 商品中心只负责“这个商品是否合法、是否允许发布”。
- 库存中心只负责“这个商品当前是否真的有货可卖”。
- 前台详情页和交易链路通过聚合查询动态决定按钮是“立即购买”还是“售罄 / 补货中”。

以 voucher 为例，商户在 `draft_tab` 完成建券并通过 QC 后，商品数据先发布到 `item_tab`，此时商品状态可以是 `ACTIVE`，但如果库存中心还没有上传有效券码，前台依然不可售。只有库存中心的 `available_count > 0`，商品才真正对 C 端开放购买。

这避免了一个常见错误：把库存变更频繁反向写回商品主表。库存可能高频变成 0、再补货、再售罄，如果每次都修改商品主状态，会把高频写压力和交易波动带回商品中心，造成不必要的锁竞争和耦合。

在库存中心内部，voucher 券码通常还有自己的生命周期状态机，例如：

```text
INIT -> AVAILABLE -> LOCKED -> CONSUMED
```

这保证了我们既能支持“先建商品、后补凭证”，又能在下单与履约环节精确锁定具体券码。

### 5. 任务流水线和可观测性建设

针对第三方同步和商户 Excel 批量导入任务，我引入统一的任务流水线思路。

无论是供应商同步包，还是商户上传的 Excel，进入供给平台后都会生成统一 `Task_ID`，并按标准漏斗推进：

```text
接收 -> 解析 -> 校验 -> 清洗 -> 映射 -> 落地
```

每个节点都记录状态、耗时、成功率和失败原因，上报统一监控看板。这样做之后：

- 能快速定位是哪一个供应商、哪一个任务、哪一行数据出错。
- 能对批量任务失败率和积压情况做秒级监控。
- 能把原来散落在 background job 里的黑盒任务变成可跟踪流水线。

### 6. 渐进式演进，而不是一次性重写

这类项目不能停机重构，所以我采用的不是大爆炸式重写，而是：

1. 先围绕新增第三方品类抽离供给接入能力。
2. 再逐步沉淀 `draft_tab` 工作区和审核状态机。
3. 再把正式商品发布和库存治理接进来。
4. 最后逐步把老逻辑收敛到新链路。

这样既能保障线上稳定，也能逐步收敛旧逻辑。

### 7. Excel 批量跑批引擎深化：从串行脚本到四阶段异步流水线

虽然 Excel 批量导入表面上只是供给平台里的一个接入入口，但我们后面发现，它的运行特征和单商品同步完全不是一个量级，所以我又单独对跑批引擎做了一次重构。

#### 为什么要单独重构批量引擎

早期的实现本质上是一条线程从头跑到尾：读取 Excel、逐行解析、逐行校验、逐行调用下游、最后再汇总结果。这个方案在数据量小的时候还能工作，但一旦商家在大促前上传几万行甚至更大的文件，就会非常容易出现几个问题：

- Excel 解析和对象构造把内存顶高，触发 OOM 或长时间 Full GC。
- 网络 RPC 和数据库写入串在同一个线程里，连接池和线程池都容易被拖干。
- 同一个商家、同一个商品的不同规格并发更新时，容易把下游主库打出行锁死锁和乐观锁误杀。
- 一旦任务跑到中间机器猝死，恢复手段粗糙，失败重试容易既慢又脏。

这里我有一个很重要的判断：批量和单品在底座上是对称的，比如都可以共用校验、映射、幂等和任务追踪能力；但在交互和履约上是非对称的。批量操作本质上是“离线大文件、异步非对称、局部容错”的极限场景，不能继续套单品同步链路的设计。

#### 四阶段生命周期设计

所以我把整个批量链路彻底拆成四阶段：

```text
提交 -> 解析 -> 处理 -> 归档
```

- 提交阶段：只负责接收文件、创建任务主表和明细框架，给整个导入动作分配统一 `Task_ID`。
- 解析阶段：只负责流式拆解 Excel、做基础格式校验、把原子明细落盘，不直接调用商品中心发布。
- 处理阶段：只负责按路由消费明细、执行校验补齐和下游 RPC，把高并发写流量治理成可控的顺序消费。
- 归档阶段：只负责汇总任务结果、生成错题本和归档结果文件，不和前面的解析、发布线程混跑。

这样设计的核心收益，是把 CPU / 内存密集的 Excel 解析，与网络 / I/O 密集的下游发布彻底切开，避免二者互相拖垮。

#### 解析阶段的稳定性设计

在解析阶段，我引入了基于 `DB CAS` 状态机的抢占模型。任务会先进入 `PENDING`，抢占成功后推进到 `PARSING`。但我没有只用 `worker_id` 标识属主，而是额外设计了：

- `lease_token`：每次抢占生成新的租约令牌。
- `lease_until`：租约有效期，超时后允许其他机器接管。
- `parse_checkpoint`：解析安全检查点，按批次持久化而不是每行落库。

这里的关键不是“谁抢到了任务”，而是“旧机器假死后苏醒还能不能继续写脏数据”。如果某台 Worker 因 Full GC 或网络毛刺假死，巡检器发现租约过期后，会让新机器重新抢占并刷新 `lease_token`。旧机器苏醒后，所有更新 SQL 都必须校验 `lease_token`，一旦令牌不匹配，受影响行数就是 0，旧 Worker 会立刻自杀退出。这样就能物理阻断脑裂。

`parse_checkpoint` 则用于断点续传，但我不会每解析一行就更新一次数据库，那样写放大太严重。实际做法是按批次持久化，比如每 200 行更新一次检查点。这样机器在第 550 行猝死时，新机器可以从 400 行的安全点继续，接受少量重复解析，再靠下游幂等覆盖，换取整个主库写压力可控。

#### 处理阶段的并发治理设计

解析完成后，真正难的是处理阶段的并发写治理。因为一个 Excel 里往往既有创建，也有更新，而且更新经常会命中同一个商品或同一个 SPU 的不同规格。

我这里没有做固定路由，而是做了“动作特征动态多级路由”：

- `CREATE`：由于这时还没有 `goods_id`，用 `supplier_id` 作为 Sharding Key。
- `UPDATE`：用 `goods_id / spu_id` 作为更精准的资产级路由键。

这样做的目的，是把同一个核心资产的所有增量改动强制顺序化，让它们在同一个 Partition 内串行消费。否则如果统一只按商家维度路由，同一个商品的多条更新仍然可能跨机器并发打到商品中心，造成 `publish_version` 乐观锁误杀，甚至把下游主库打出死锁。

换句话说，动态路由要解决的不是“消息能不能分发出去”，而是“同资产并发写要不要在最外层先被驯化掉”。这一步做对了，后面的脏重试、无效错题本和乐观锁误杀都会明显下降。

#### 收尾阶段的确定性设计

最后一个容易被忽略的点，是结果文件和错题本的收尾确定性。我的做法是双轨制：

- `Redis 原子计数器`：负责快乐路径上的快速收尾。
- `DB 旁路盘点对账`：负责异常场景下的最终真相和接管。

正常情况下，每个 Consumer 处理完自己的明细后，对 Redis 做一次原子递减；谁把计数器减到 0，谁就成为“终结者线程”，负责生成最终结果文件和错题本。这样整个快乐路径的开销非常低。

但我不会把真相建立在 Redis 上。因为如果某台 Consumer 处理到一半猝死，Redis 计数器可能悬空，永远到不了 0。这个时候，后台定时器会去数据库直接盘点 `task_item` 的真实完成情况。如果发现未完成明细已经是 0，就强制接管终结者身份，补做结果生成和归档。也就是说，终结者线程只是执行角色，不是真相来源；最终真相仍然以 DB 明细状态为准。

把“草稿中”“审核中”和“正式可售”拆开，用状态机控制交易可见性，是平台化治理能力的重要标志。

你不是机械追求“大而全”的 `draft + staging` 双表流水线，而是根据审核周期、编辑模式和实现成本，最终选择了只保留 `draft_tab` 的方案。

### 5. 库存从字段升级为能力，体现出你对交易安全的理解

这是从“后台配置系统”走向“交易平台”的分水岭。

### 6. 你把“商品可发布”和“真正可售”拆开了

这说明你不只是把库存抽成一个服务，而是真的理解了 Voucher 这类商品“先有壳，再注入供给”的业务特征。

### 7. 你补齐了可观测性，而不只是做了结构拆分

很多人只会讲拆分服务，但真正落地时，任务监控、告警和排障效率提升才是让系统真正可运维的关键。

### 8. 你把解析算力和发布算力做了资源类型隔离

Excel 解析本质上是 CPU / 内存密集型工作，而调用商品中心发布是网络 / I/O 密集型工作。如果把这两类工作揉在同一个 Worker 线程池里，下游一慢，整个解析算力就会被空耗掉。把解析和处理拆成不同阶段，本质上是在做资源类型隔离。

### 9. 你用轻量组件解决了跑批场景里最棘手的分布式一致性问题

这套方案没有引入很重的中间件堆栈，而是用 `DB CAS + lease_token + checkpoint + 动态路由 + Redis/DB 双轨收尾`，把脑裂、断点续传、乱序并发和结果收尾确定性这些最容易出事故的点逐个钉住了。这会让面试官觉得你不只是会拆服务，而是真的能把复杂异步系统跑稳。

## 十二、高频追问与回答

这一节本质上都是架构决策点。面试时不要只报结论，尽量用“候选方案、比较维度、最终选择”的方式回答，会更像真实做过技术方案评审。

### 决策点 1：为什么不能继续在原商品系统上打补丁，而是要引入 `draft_tab`？

> 短期当然可以继续打补丁，但问题是业务模型已经变了。早期商品主要是 `topup / bill`，商品只是配置对象；后面商品开始承载外部供给、审核、库存和履约属性。如果继续把这些都塞在原系统里，后面每接一个新类型商品，复杂度和风险都会继续上升。所以这次改造本质上是边界重划，而不是简单重构代码。

### 决策点 2：`draft` 应该放在供给平台还是商品中心？QC 流程怎么设计？

> 我们最终没有采用“供给侧 draft -> QC -> 商品正式态”的前置审核链路，而是选择“供给接入 -> 商品主域 draft/staging -> QC -> 本地 merge 发布”。核心原因是，一旦草稿进入待审核、待发布阶段，它本质上就不再是接入层流程数据，而是商品主域的未生效资产版本。若草稿长期留在供给侧，QC 审核的就不是商品主域本地冻结快照，容易出现并发篡改、版本乱序和旧版本回滚覆盖新版本的问题。

> 所以我把 `draft` 收拢到商品主域写侧。供给平台只负责异构输入接入和数据清洗，不承担审核后台、QC 规则和人工流程。供给层把清洗后的数据写入 `draft_tab`，再由审核治理能力通过状态机推进 `DRAFT -> UNDER_REVIEW -> APPROVED/REJECTED`。审核通过后，由商品主域在本地事务内完成 merge、快照生成和 Outbox 投递，交易链路始终只读正式商品表。

### 决策点 3：为什么选择单 `draft_tab` + operation，没有再拆 `staging_tab`？

> 这是一个典型的工程权衡问题。我们最开始也考虑过 `draft + staging` 双表方案，但它更适合审核周期很长、商户在审核期间还要频繁并发二次编辑的场景；同时它会带来额外的跨表迁移、双表 DDL 和查询复杂度。结合当时的业务特点，我最终选择只保留 `draft_tab`，用单表加状态机承接商户工作区，提交审核时只做状态切换，不做数据复制，整体更轻量，开发和运维成本也更低。核心前提是审核期间的编辑规则要清晰，不能让审核中的数据污染正式交易链路。

### 追问 4：既然单商品创建是同步链路，为什么还要先落盘 `product_supply_task` 和 `product_supply_task_item`，不直接透传 RPC？

> 直接透传看似省事，但会让前端和商品中心产生致命的硬耦合。我坚持“无条件先落盘”，第一是为了统一业务口径。B 端商家既有单个创建，也有 Excel 批量导入和 ERP API 同步。如果单品不写 Task，这几条链路在操作日志、发布追踪和审计体系里就会彻底撕裂。单品写轻量 Task，比如 `total_count=1`、`execution_mode=SYNC`，可以让单品和批量共用一套追踪与治理框架。

> 第二是为了构建本地操作沙盒来应对分布式三态问题。网络不可靠，一旦调用商品中心超时或进程假死，没有本地落盘，商家的编辑快照就直接蒸发了。先落盘可以保留 `raw_payload`、任务状态和上下文，为后续的幂等、防重试和状态自愈提供抓手。

### 追问 5：幂等 Key 怎么设计？为什么不能直接把整个 JSON 做 MD5？

> 不能直接对整个请求 JSON 做 MD5，因为前端报文里通常带有随机数、操作人 ID、时间戳之类的非业务决定性字段。商家连续点两次保存，即使业务内容完全相同，也会得到两个不同的 Key，幂等防御等于失效。

> 我的做法是提取商品域真正决定唯一性的业务指纹，例如 `supplier_id + title + spu_category_id + brand_id + bar_codes`。这些字段先做标准化和排序，再拼成稳定字符串，最后计算 MD5 作为业务指纹 Key。在 Tx1 本地事务中，这个 Key 会和 `task_item` 一起落盘，作为整条请求链路的幂等凭证。这样商家高频点击或 ERP 并发重试时，供给平台能识别出这是同一笔业务请求，并带着同一个幂等 Key 再次请求下游商品中心；而下游也基于同一幂等 Key 做幂等处理，已经成功过的请求会直接返回成功，避免重复创建。

### 追问 6：既然上下游都做了幂等，RPC 超时为什么不直接返回失败，而要做旁路自愈？

> 因为超时不等于失败，它代表的是 `Unknown`。如果商品中心在第 3.1 秒其实已经处理成功，但供给平台在第 3.0 秒超时返回失败，商家往往会马上再点一次保存。这个时候第二次请求本质上还是同一笔业务，所以我们不会把它当成一条全新的创建请求，而是识别出相同的幂等 Key，再次带着这个 Key 请求下游。由于下游支持幂等，已经成功处理过的请求会直接返回成功，商家看到的就是一次可恢复的重试，而不是重复提交报错。

> 所以我会把这类请求先挂在 `PROCESSING`。如果用户主动再次点击保存，系统会识别出相同幂等 Key，并带着这个 Key 再次请求下游；如果用户没有点击重试，后台的延迟任务或补偿任务也会拿着同一个幂等 Key 去重试或做结果反查。只要下游已经成功处理过，就会直接返回成功结果，供给平台再把本地任务状态补成 `SUCCESS`；如果确实没成功，再决定是否继续重试或失败收敛。它解决的不是简单重试，而是“提示失败、实际成功”这类分布式不一致问题。

### 追问 7：编辑链路里为什么要引入 `base_publish_version`？

> 这是为了对抗多渠道并发编辑导致的主数据回滚覆盖问题。我们摒弃了无脑覆盖的写模式，在 Tx1 受理时就把商家打开页面时看到的基准发布版本号写进 `task_item`。到了 Tx3，把这个 `base_publish_version` 作为跨服务契约的一部分，下发给商品中心执行基于 `WHERE version = :base_version` 的乐观锁更新。

> 这样做有两个价值。同步链路上，它能精准识别并发冲突，及时返回 `VERSION_CONFLICT`，保护商家表单并提示刷新；异步补偿链路上，由于命令里绑定了基准版本，旁路自愈任务也具备无副作用重放能力。即使 5 分钟后再补偿，只要中台版本已经演进，乐观锁就会在物理层阻断这次过期重放，不会制造脏数据。


### 追问 8：供给平台同时要承接单商品创建编辑、Excel 任务和供应商同步，怎么避免互相拖垮？

> 我的思路是做四象限隔离，本质上就是通过“入口打散、线程池隔离、MQ 异步削峰、资源物理划分”在系统最外层修四道防洪堤。单商品同步链路追求低延迟，就走轻量同步通道；Excel 导入和供应商批量同步天然是大流量异步任务，就走独立队列、独立消费组和独立限流策略。

> 这样即使某个供应商的大批量同步突然放量，也不会把商户后台单商品编辑拖慢；反过来，商户高峰期的人工操作也不会挤占批量任务处理资源。面试里可以强调，这不是简单的“多开几个线程池”，而是把不同业务流量按实时性、体量和容错方式拆开治理。

### 追问 9：你是怎么治理第三方同步和 Excel 批量导入任务监控的？

> 以前这些任务散落在各个 background job 里，失败了只能查日志，属于典型黑盒任务。我引入统一 `Task_ID` 和任务流水线，把所有接入任务都拆成 `接收 -> 解析 -> 校验 -> 清洗 -> 映射 -> 落地` 六个节点。每个节点都上报状态和指标到统一监控看板，这样一旦某个供应商包或某个 Excel 文件出问题，可以快速定位到具体任务甚至具体数据行，排查时效从小时级降到分钟级。

### 追问 10：数据模型被品类差异污染，你是怎么处理的？

> 核心思路是核心与扩展分离。正式商品中心只保留所有品类共有的主字段和状态机，品类特有字段通过扩展属性、类目模板或品类策略承接，避免继续污染主模型。代码层面也要逐步减少基于品类的 `if-else`，把差异行为外包给品类策略或接入层处理。

### 追问 11：库存为什么一定要独立出来？

> 因为商品是读高频的展示模型，而库存是写高频的交易约束。只要进入本地券这类场景，就会出现锁码、释放、回补、补偿等高风险写路径。如果继续把库存留在商品系统里，不但会放大耦合，还会把交易安全问题变成普通字段更新问题，所以必须把它升级成独立安全能力。

### 追问 12：如果 Voucher 类商品先创建商品、后上传库存，怎么保证它在没库存时不可售？

> 这种场景一定要把“商品状态”和“可售状态”拆开。商品中心的 `item_tab` 只表达商品是否已经合规发布，比如 `ACTIVE`；但前台是否真正可售，要由聚合层动态判断 `ACTIVE && available_stock > 0`。这样即使商品已经创建完成，只要库存中心还没有注入可用券码，前台就会显示不可售或补货中，而不会错误地开放购买。

### 追问 13：库存卖完了，要不要把商品状态改成下架？

> 不需要。商品状态代表的是商品准入资格，是低频状态；库存卖完是高频事件，不应该频繁回写商品中心。正确做法是保持商品仍然是 `ACTIVE`，但前台聚合查询库存时发现 `available_count = 0`，自然渲染成“售罄”状态。后续商户补货上传券码后，前台又会自动恢复可售，整个过程商品中心零感知。

### 追问 14：高并发下单时，你们是只扣库存数字，还是锁具体券码？

> 对 Voucher 类商品，我们通常采用两阶段思路。第一阶段先扣减库存资格，保证高并发下用户是否抢到购买资格；第二阶段在支付成功后，再由库存中心从 `AVAILABLE` 的券码池里真正分配一张具体券码，并推进到 `CONSUMED` 或已绑定状态。这样既能扛住高并发，又不会让具体券码在未支付阶段被长时间占死。

### 追问 15：你们是一次性重构吗？

> 不是。一次性重构风险太高，而且线上交易不能停。我们是围绕新增业务场景逐步抽边界，让新接入品类先走新链路，再逐步把老逻辑收敛过去。这种渐进式演进更符合真实业务环境。

### 追问 16：既然已经有 DB CAS 抢占了，为什么还要额外引入 `lease_token`？

> 因为只靠 `worker_id` 只能解决“谁先抢到任务”，解决不了“旧 Worker 假死苏醒后继续写脏数据”的脑裂问题。最典型的场景就是 Worker-A 抢到任务后进入 `PARSING`，随后因为 Full GC 或网络长毛刺假死，巡检器发现租约过期，就把任务放回，让 Worker-B 成功接管。如果没有 `lease_token`，Worker-A 苏醒后会误以为自己还是合法属主，继续更新 `parse_checkpoint` 或写明细，这就会形成双王写脏。

> 引入 `lease_token` 后，每次抢占都会刷新一个新的令牌，所有后续写操作都必须带上 `WHERE lease_token = :my_token`。旧 Worker 苏醒后，由于令牌已经变化，SQL 受影响行数会直接变成 0，进而触发自杀退出。也就是说，`lease_token` 不是为了好看，而是为了在物理层阻断脑裂。

### 追问 17：`parse_checkpoint` 为什么不每处理一行就更新一次？

> 因为那样会把数据库写放大到不可接受的程度。跑 5 万行文件，如果每行都更新一次检查点，本质上是在用主库承接高频无意义小写，慢 SQL 和连接抖动很快就会出来。我的做法是分段持久化，比如每 200 行更新一次 `parse_checkpoint`，并且和当前批次的明细落盘放在同一个事务里。

> 这样机器在第 550 行猝死时，新机器会从 400 行的安全点继续，而不是从 0 重来。虽然会多解析一小段已经处理过的行，但这是故意的工程权衡：用有限的重复解析，换取数据库可承受性和整体吞吐稳定性。重复部分再靠下游幂等或唯一键约束兜住。

### 追问 18：Excel 里创建和更新混编时，为什么路由键不能统一？

> 因为创建和更新的资产身份成熟度不一样。`CREATE` 阶段还没有 `goods_id`，这时你只能用 `supplier_id` 这种商户主权维度做路由，先防止公共配置和接入写链路被打乱；但 `UPDATE` 阶段已经有明确的核心资产标识，这时如果还沿用商家维度路由，就会把同一个商品或 SPU 的多条更新分散到不同消费线程里，下游商品中心的乐观锁和主库行锁压力就会上来。

> 所以我的做法是动态多级路由：创建按 `supplier_id`，更新按 `goods_id / spu_id`。这不是为了追求花哨，而是为了把“同资产并发写”在进入核心域之前就顺序化掉，减少无效重试、错题本污染和版本误杀。

### 追问 19：如果 Redis 计数器因为 Consumer 猝死悬空了，结果文件是不是就永远生成不了了？

> 纯靠 Redis 的确会有这个风险，所以我从一开始就没有把 Redis 当成最终真相，只把它当作快乐路径加速器。正常情况下，Redis 原子递减能以极低成本快速找出最后一个完成的线程，让它生成结果文件；但一旦某个 Consumer 猝死，计数器停在 1，快乐路径就失效了。

> 这时候后台的旁路对账定时器会直接去 `task_item` 表数真实完成情况，比如看还有没有 `INIT / PROCESSING` 状态的明细。如果数据库告诉我未完成数已经是 0，那就说明 Redis 只是漏计了，定时器会直接接管终结者身份，推进任务到结果生成和归档阶段。也就是说，Redis 负责快，DB 负责准，最终不会因为计数器悬空而把错题本卡死。

## 十三、适合背诵的 STAR 版本

### S：背景

平台早期主要卖 `topup` 和 `bill` 类型商品，商品系统比较简单。后来先接入酒店、电影票、机票等第三方供应商商品，需要定时同步商品和库存；再往后逐步扩展到本地生活券，商户可以自己建券、Excel 批量导入，系统又要支持生码、库存和审核，原有商品系统开始变得越来越臃肿。

### T：任务

我负责推动商品体系升级，在不影响现有交易的前提下，重新梳理商品、供给、审核、库存和批处理任务之间的边界，让系统可以继续支撑多品类扩展。

### A：行动

我主要做了五件事：

1. 把第三方接入和商户批量导入能力抽到供给接入平台。
2. 为同步和导入链路设计统一 `Task_ID` 和任务流水线监控。

改造之后，商品体系职责边界更清晰，新品类和新供应商接入更标准化，任务链路更可观测，审核和库存风险更可控，也为后续更多本地生活和第三方商品接入打下了基础。针对 Excel 批量导入链路，吞吐量提升 `400%`，百万级商品主数据导入不再拖垮核心域，批量失败重试误杀率降至 `0`，结果文件生成具备 `100%` 确定性。

## 十四、2 分钟口述稿

面试官好，我过去做过一个很典型的商品架构升级项目。这个项目最有代表性的地方，是它不是一开始就有完整的中台规划，而是业务从简单走向复杂之后，系统被逼着做架构演进。

最开始平台主要卖 `topup` 和 `bill` 这类标准化商品，商品系统比较轻，本质上就是一个配置后台。后来平台先接入了酒店、电影票、机票等第三方供应商商品，需要处理高频定时同步、价格库存变化和异构字段映射；再往后又扩展到本地生活券，支持商户自己建券和 Excel 批量导入，这时候系统里又引入了生码、控库存、QC 审核和发布治理。

业务叠加之后，原来的商品系统很快暴露出几个问题：第一，商品、供给、库存、审核和批处理任务都揉在一个系统里，边界很乱；第二，异构品类字段不断往商品模型里堆，代码里有很多品类 `if-else`；第三，第三方同步和 Excel 批量导入任务散落在 background job 里，缺少统一监控，线上定位问题很慢；第四，本地券开始带来锁码和库存风险，资损问题越来越敏感；第五，商户建券不再是创建即上架，必须把草稿、审核中和正式可售几个阶段拆开治理；第六，Excel 跑批原先是一条线程从头跑到尾，大文件场景下很容易打出 OOM、连接池枯竭和下游主库死锁。

所以我主导做了这次升级。核心思路是把系统拆成几层：供给接入平台负责第三方同步和商户导入，统一做协议适配、数据清洗和任务监控；商户侧工作区在 `draft + staging` 两种思路里，我最终只保留了 `draft_tab`，通过 `DRAFT`、`UNDER_REVIEW`、`REJECTED` 等状态推进审核流，审核通过后再正式发布到 `item_tab`；对于 voucher 这类商品，我进一步把“商品状态”和“可售状态”解耦，哪怕商品已经发布，只要库存中心还没注入有效券码，前台依然不可售；库存和券码则独立出来，专门治理锁定、释放和补偿。与此同时，我把 Excel 批量导入单独重构成 `提交 -> 解析 -> 处理 -> 归档` 四阶段异步流水线，用 `lease_token`、`parse_checkpoint`、动态 Hash 路由和 `Redis + DB` 双轨收尾去解决脑裂、断点续传、乱序并发和结果生成确定性问题。

这次改造的价值是，后续无论接入新的第三方品类，还是扩展新的本地券模式，更多都变成了标准接入和治理问题，而不是继续修改一个越来越脆弱的大单体。尤其在批量导入链路上，吞吐量提升了 `400%`，同时把大文件导入最容易出问题的几个点都提前兜住了。

## 十五、最后的面试建议

这个项目最适合打“架构演进”和“平台化治理”，而不是只讲某个局部技术点。

3. 再讲你怎么重划供给、工作区、审核、正式商品、库存边界。
4. 最后讲结果和你自己的判断。

只要这个顺序讲顺了，面试官通常会认为你是真的经历过一次由业务增长倒逼架构升级的过程。





这份供应商数据同步与库存编排的设计方案非常扎实、惊艳，具备极高的工业级成熟度。方案不仅精准踩中了分布式长任务的痛点（如低QPS、超长耗时、网络抖动、时序错乱、静默丢单），而且在资损防控和边界划分上展现了极其清晰的架构思维（例如“入口归供给，事实归库存”以及“供给侧搬运，商品域Merge”）。为了帮你把这份设计打磨得更加无懈可击，我从整体亮点架构、潜在隐患与边缘场景（Deep Dive）、以及优化建议三个维度为你进行硬核 Review：一、 核心架构亮点点评这份设计最出彩的地方在于它避开了初学者容易犯的“并发冲锋”和“强一致性陷阱”，采用了弹性、可恢复、防资损的异步流水线模式：克制的接入层 vs. 激进的消费层：对老旧/慢源 API 放弃并发切片，选择“单 Worker 串行拉取 + MQ 蓄水”，既保护了供应商接口不被打满，又通过下游并发消峰极大提升了整体吞吐。状态与存储分离（MySQL 台账 + HBase 报文）：HBase 吞吐大原始报文，MySQL 用 CAS 状态机死守并发主权与对账主权。不仅降低了关系型数据库的存储压力，还保留了完整的“证据链”（证据回放能力）。深谙资损防控之道：拉取与消费存在数小时时间差，设计中通过反查商品主域的 publish_version 和乐观锁，构建了完美的写前防线，彻底杜绝了“旧数据覆盖新数据”的幽灵资损。全量 List 下线的“熔断美学”：处理全量列表最怕供应商接口给漏了导致线上商品无故下线。方案引入了时间戳游标打卡（last_seen_at < T_start）+ 比例熔断机制，非常具有实战经验。二、 潜在隐患与边缘场景（Deep Dive）基于你目前透出的表结构和时序，在极端的生产环境下，有以下几个边界细节需要补充钢筋水泥：1. 接入层流水线的“断点续传”与 MQ 重复投递的交织在串行接入循环中，动作顺序是：拉取 Payload -> 存入 HBase -> 写 MySQL台账 -> 发送 MQ -> 推进 Checkpoint潜在风险：如果在“发送 MQ 成功”之后，但在“推进 Checkpoint”之前，Worker 机器突然崩溃或被发布重启，新接管的 Worker 会从旧的 Checkpoint 重新拉取这一页。应对核验：此时同一批商品会再次进入 HBase 并重新写入台账和投递 MQ。由于你下游 Consumer 设计了强大的 CAS 强占（INIT -> PROCESSING），如果状态已经是 SUCCESS 或 FAILED，重发的消息会被优雅丢弃或跳过（Skipped），这证明你的下游防线能完美防御接入层的重试，设计形成闭环。2. INVENTORY_FAILED（半成功）状态下的运营体验与重试盲区在 3.7.4 中，你拆出了一个非常高频的痛点状态：商品建成了，但库存初始化失败（INVENTORY_FAILED）。边缘漏洞：此时商品在主域可能已经是个“无库存”或“默认不可售”的正式商品（或者处于 Staging 状态）。追问设计：商品主域写侧是否应该阻断该商品的线上可见性，直到供给侧的库存补建任务完成？当运营点击“重试补建库存”时，如果调用库存中心再次失败，此时 product_supply_task_item 的重试次数（retry_count）如何限制？建议增设行级死信队列（DLQ）隔离连续失败 5 次以上的顽固商品，防止单商品死循环卡住通道。3. HBase / 对象存储的“历史包袱”与生命周期潜在风险：100w 对象的全量同步每天都在跑，oss://agoda-sync/raw/xxxx.json 这种大文本的积累速度是惊人的。如果缺乏生命周期（Lifecycle）管理，存储成本会飙升。优化动作：建议在设计中明确指出 HBase / 对象存储快照的 TTL（如保留 30~90 天用于审计回放，过期自动转入冷存储或销毁）。三、 补充与微调建议1. 表结构优化：给台账增加“变更痕迹”在 supplier_sync_state_ledger 中，建议增加一个 hash_sign（数据签名/MD5）字段。原因：慢源拉取虽然天天跑，但 80% 的商品可能连续几天字段没有任何变化。Consumer 在拉取到消息、反查 HBase 后，可以先拿本次的 Payload MD5 与 Ledger 表里的 hash_sign 做比对，如果完全一致，直接变更状态为 SKIPPED，不需要再向下游商品主域发起写请求和版本反查。这能帮商品主域写侧至少减负 60% 以上的无效流量。2. 多模式库存的 EXT_POOL（死码池）追加水位预警原因：死码池与数字库存最大的不同在于它是消耗性的，用完就真没了。优化动作：在库存建模的契约中，供给平台作为 B 端入口，除了路由导入命令，还应订阅库存中心的“券码消耗事件”。当死码池剩余行数低于某个阈值（如 $10\%$）时，在供给平台触发product_quality_issue（质量治理问题），生成一个“券码即将耗尽，请商家尽快补货”的问题单，完成运营闭环。总结这是一份可以直接敲代码落地的顶层设计。不仅考虑了快乐路径（Happy Path），更把 80% 的精力放在了处理长任务中断、版本冲突、部分失败等脏活累活上。为了让我们探讨的架构更精确，我想抛出一个细节问题来进一步完善它：当任务处于 INVENTORY_FAILED（商品成功但库存失败）时，你更倾向于让商品主域先上线但展示为“缺货/不可售”，还是在商品主域侧利用某种“发布锁”保持不可见，直到供给这边的库存补偿任务彻底 Done？



这是一个非常经典的系统架构演进决策。在面对话费充值、账单还款、酒店、电影、演出票务、优惠券这类“差异化极大”的业务时，如何有效地组织商品模型，本质上是在做一场“全平台交易底座的通用性”与“垂直业务特性的高内聚”之间的权衡（Trade-off）。

我们可以将这个核心决策点，梳理为一套多维度的思考推导框架，并以结构化决策点的形式呈现。这部分可以直接作为详细设计中分量极重的理论依据。

一、 顶层思考框架：推导模型的四大核心冲突
在决定采用何种模型前，我们需要先看清这些品类在核心域上的本质冲突：

资产属性的冲突（静态实体 vs 时空资源 vs 纯数据服务）

实物/券码： 是静态的物理存在，有固定的库存、规格（SKU）。

酒店/电影： 是“时空资产”。今天买 6 月 6 号的房间和买 6 月 7 号的房间，完全是两个不同的价格和库存。

话费/账单： 本质上没有“自建商品库”，它是纯 API 驱动的数据服务，价格和可用性完全取决于上游渠道商（Carrier）的实时响应。

写频率的冲突（日更/周更 vs 毫秒级变动）

实物或演出票的标题、描述可能几天才改一次，适合走你设计的标准流程（Draft -> 审核 -> 本地 Merge -> 版本自增）。

酒店的日历房态价、话费的渠道进货价，随市场和供给瞬息万变。如果让它们也走“版本演进与发布”的标准写链路，系统会瞬间被无休止的 publish_version++ 撑爆。

状态与风控主权的冲突（整体下架 vs 资源封禁）

如果每一个具体的话费充值项、每一个具体物理酒店都对应一个系统级 item_id，商品中心的状态机就会承载过重的业务治理逻辑，导致核心写模型变得极度臃肿。

交易链路的共用需求（多品类跨品类结算）

尽管供给侧和写模型千差万别，但在读侧和交易侧，购物车需要同时合并结算、营销中心要搞“买充值卡送酒店券”、收银台需要统一支付。因此，交易链路必须对商品有统一的格式认知。

决策结论一：在组织模型上，实行“货架主权”与“资源主权”的彻底分离

动态参数携带： 用户下单时，前端或供给侧将具体的 hotel_id、room_type 或 phone_number 作为 ext_params（动态上下文）伴随交易链路流转。

决策结论三：在生命周期管理上，建立“双层状态防御机制”
针对非标品或单一物理商户的封禁、下架：

商品中心的状态机（status=PUBLISHED）只表达“平台是否整体开启了该业务品类的售卖能力”。

行动：将库存体系划分为实时库存、池化库存和无限库存三大模型。针对 deal 券的高并发锁码场景，设计了基于 Redis LIST 与 MySQL CAS（乐观锁）的双层判码机制：Redis 仅提供不带明文的 code_id 热视图 LPop 弹出，MySQL 运用 CAS 事务判断作为锁码成功的最终权威，避免了复杂的分布式锁开销并实现热队列快速回填。

成果：核心交易链路 SLA 达到 99.95%，支撑了大促期间详情页 25,000 QPS 和下单页 5,000 QPS 的峰值流量，实现零超卖、零丢单。

行动：重新界定读写链路：搜索与导购采用三级缓存架构（L1本地缓存 + L2 Redis + L3 DB）支持弱一致性读；进入结算页则强制触发统一计价服务进行实时价格与营销规则试算。创单时，采用短生命周期的 Saga 模式 串行进行库存预占和优惠锁定。订单生成后对商品、价格、履约规则进行秒级快照冻结，后续履约与售后完全基于快照事实。

成果：消除了“试算价”与“下单价”不一致导致的客诉问题，确保了后交易交易阶段的追溯审计有据可查。

💡 面试加分锦囊（如何用这篇经历惊艳面试官）
面试官问：“你从充值到做机票酒店，最深刻的体会是什么？”

回答话术：以前做充值，系统模型非常简单，近乎无限库存，属于支付后同步履约，失败了事后给用户退款补偿就行；但做机票酒店和本地生活后，系统变成了强实时、强契约交易。机票超卖是零容忍的，要求下单前去航司预占；酒店按日历定价和扣减，库存是二维维度的。所以我深刻体会到，没有一套通用的架构能解决所有品类，必须要用策略模式做抽象，把变与不变隔离开。

面试官问：“你们 deal 券抢购时，券码池怎么保证不卡顿不超卖？”

回答话术：我们没有用重型的分布式锁。我们采用了“接入层与事实层分离”的思路，Redis 的 List 只用来存放券码的 ID 队列，Pop 出来的只是一个信号。真正的主权在 MySQL，用一条 UPDATE ... SET status='BOOKING' WHERE code_id=? AND status='AVAILABLE' 的 CAS 乐观锁来判定。即便 Redis 里的数据陈旧了，MySQL 层的 CAS 也能保证绝对不会超卖，这就把并发压力从数据库成功卸到了缓存层。

### 4. 统一批量操作框架（架构创新 & 效率提升）

**核心痛点**：
- **架构不统一**：商品上架有完整批次框架（task+item+进度+结果），但价格调整、库存设置却是同步接口，缺少跟踪
- **代码重复**：每种批量操作都要实现一遍（任务创建、进度更新、结果生成、错误处理），复用率 < 15%
- **可追溯性差**：批量调价/设库无审计记录，出问题无法回溯"谁改了什么"
- **新功能成本高**：新增一种批量操作需要 550 行代码、2 周开发时间

**技术方案**：
- **统一数据模型**：设计 `operation_batch_task_tab` 和 `operation_batch_item_tab` 两张泛化表，通过 `operation_type` 字段区分不同批量操作（price_adjust、inventory_update、voucher_code_import、item_edit 等），所有批量操作复用同一套表结构
- **统一事件驱动**：单一 Kafka Topic `operation.batch.created`，Worker 通过 `operation_type` 字段动态路由到不同处理器（PriceUpdateWorker、InventoryUpdateWorker 等），避免重复创建 Topic
- **模板化Worker设计**：抽象 `BaseBatchOperationWorker` 提供通用处理模板（流式解析 → 创建明细 → Worker Pool并发 → 结果文件 → 状态更新），新增批量操作仅需实现 50 行业务逻辑
- **完整审计链路**：每条 `operation_batch_item` 记录 `before_value` 和 `after_value` JSON，支持变更前后对比、完整追溯、合规审计
- **流式解析 + 分批处理**：使用 `excelize.NewReader()` 逐行读取 Excel（内存恒定 < 200MB），分批（100条/批）查询+更新，避免 OOM
- **Worker Pool 并发**：20 goroutine 并发处理，配合 Channel 任务分发，充分利用多核 CPU，并发度提升 **30 倍**
- **统一监控告警**：Prometheus 统一指标 `operation_batch_task_total{operation_type, status}`，一套告警规则覆盖所有批量操作

**技术细节**：
```go

   → 记录before/after → 更新进度 → 生成结果文件

4. 新增批量操作接入（仅需3步，50行代码，2天）：
```

## 三、技术 1：流式解析（Streaming Parse）

### 3.1 核心原理

**传统方式 vs 流式解析**：

```go
// ❌ 传统方式：整文件加载
file, _ := excelize.OpenFile("products.xlsx")
rows, _ := file.GetRows("Sheet1")  // 一次性加载所有行到内存
for _, row := range rows {
    processRow(row)
}

// ✅ 流式解析：逐行读取
file, _ := os.Open("products.xlsx")
reader := excelize.NewReader(file)  // 创建流式读取器
for {
    row, err := reader.ReadRow()    // 每次只读一行
    if err == io.EOF {
        break
    }
    processRow(row)  // 立即处理，不在内存中积累
}
```

**内存占用对比**：

| 文件大小 | 行数 | 传统方式内存占用 | 流式解析内存占用 | 节省 |
|---------|------|----------------|----------------|------|
| 10MB | 10000 | 800MB | 150MB | 81% |
| 50MB | 50000 | 4GB（OOM） | 180MB | 95% |
| 100MB | 100000 | 无法加载 | 200MB | ∞ |

```go
// ===== Excel 批量上传：流式解析实现 =====
type ExcelParseWorker struct {
    oss          *OSSClient
    taskRepo     *TaskRepository
    batchItemRepo *BatchItemRepository
}

func (w *ExcelParseWorker) Process(event *BatchCreatedEvent) error {
    // 1. 从 OSS 下载文件（分片下载，节省内存）
    file, err := w.oss.Download(event.FilePath)
    if err != nil {
        return fmt.Errorf("download file failed: %w", err)
    }
    defer file.Close()
    
    // 2. ⭐ 创建流式读取器（关键：不要用 OpenFile）
    reader, err := excelize.NewReader(file)
    if err != nil {
        return fmt.Errorf("create reader failed: %w", err)
    }
    
    // 3. 读取表头（第一行）
    header, err := reader.ReadRow()
    if err != nil {
        return fmt.Errorf("read header failed: %w", err)
    }
    columnMap := buildColumnMap(header)  // {sku_code: 0, title: 1, price: 2, ...}
    
    // 4. ⭐ 逐行读取（内存占用恒定）
    rowNumber := 1  // 从 1 开始（0 是表头）
    successCount := 0
    failedCount := 0
    
    for {
        // 每次只读一行
        row, err := reader.ReadRow()
        if err == io.EOF {
            break  // 文件读完
        }
        if err != nil {
            log.Errorf("Read row %d failed: %v", rowNumber, err)
            failedCount++
            continue
        }
        
        rowNumber++
        
        // 5. 解析行数据（根据列映射）
        item, err := w.parseRowData(row, columnMap)
        if err != nil {
            // 记录失败行（不阻塞后续处理）
            w.recordFailedRow(event.BatchID, rowNumber, err)
            failedCount++
            continue
        }
        
        // 6. 基础校验（必填项、格式）
        if err := w.validateBasicFields(item); err != nil {
            w.recordFailedRow(event.BatchID, rowNumber, err)
            failedCount++
            continue
        }
        
        // 7. ⭐ 立即持久化（不在内存中积累）
        task := &ListingTask{
            TaskCode:    generateTaskCode(item.CategoryID),
            CategoryID:  item.CategoryID,
            SourceType:  "excel_batch",
            ItemData:    item,  // JSON 存储
            Status:      StatusDraft,
        }
        
        // 插入 listing_task_tab
        if err := w.taskRepo.Create(task); err != nil {
            w.recordFailedRow(event.BatchID, rowNumber, err)
            failedCount++
            continue
        }
        
        // 8. 记录批次明细（用于进度跟踪）
        w.batchItemRepo.Create(&BatchItem{
            BatchID:   event.BatchID,
            TaskID:    task.ID,
            RowNumber: rowNumber,
            RowData:   item,
            Status:    "pending",
        })
        
        successCount++
        
        // 9. 每 1000 行打印进度（用户体验）
        if rowNumber%1000 == 0 {
            log.Infof("Batch %s: parsed %d rows (success: %d, failed: %d)", 
                event.BatchID, rowNumber, successCount, failedCount)
            
            // 更新批次进度（前端轮询显示）
            w.updateBatchProgress(event.BatchID, rowNumber, successCount, failedCount)
        }
    }
    
    // 10. 更新批次最终状态
    w.updateBatchStatus(event.BatchID, "parsed", successCount, failedCount)
    
    // 11. 发送下一阶段消息（审核）
    w.publishKafka("listing.batch.parsed", event.BatchID)
    
    log.Infof("Batch %s parse completed: total=%d, success=%d, failed=%d", 
        event.BatchID, rowNumber, successCount, failedCount)
    
    return nil
}

// 解析行数据（根据列映射）
func (w *ExcelParseWorker) parseRowData(row []string, columnMap map[string]int) (*ItemData, error) {
    item := &ItemData{}
    
    // 根据列映射提取数据
    if idx, ok := columnMap["sku_code"]; ok && idx < len(row) {
        item.SKUCode = strings.TrimSpace(row[idx])
    }
    if idx, ok := columnMap["title"]; ok && idx < len(row) {
        item.Title = strings.TrimSpace(row[idx])
    }
    if idx, ok := columnMap["price"]; ok && idx < len(row) {
        price, err := strconv.ParseFloat(row[idx], 64)
        if err != nil {
            return nil, fmt.Errorf("invalid price: %s", row[idx])
        }
        item.Price = price
    }
    if idx, ok := columnMap["category_id"]; ok && idx < len(row) {
        categoryID, err := strconv.ParseInt(row[idx], 10, 64)
        if err != nil {
            return nil, fmt.Errorf("invalid category_id: %s", row[idx])
        }
        item.CategoryID = categoryID
    }
    
    return item, nil
}
```

```go
type VoucherCodeImportWorker struct {

func (w *VoucherCodeImportWorker) ImportCodes(event *CodeImportEvent) error {
    // 1. 从 OSS 下载券码文件（CSV）
    file, err := w.oss.Download(event.FilePath)
    if err != nil {
        return fmt.Errorf("download file failed: %w", err)
    }
    defer file.Close()
    
    // 2. ⭐ 流式解析 CSV
    reader := csv.NewReader(file)
    reader.ReuseRecord = true  // 复用内存，减少 GC 压力
    
    // 3. ⭐ 设置批次大小（根据数据库性能调优）
    batchSize := 1000
    codes := make([]*InventoryCode, 0, batchSize)
    totalCount := 0
    successCount := 0
    failedCount := 0
    
    // 4. 逐行读取并分批处理
    for {
        row, err := reader.Read()
        if err == io.EOF {
            break
        }
        if err != nil {
            log.Warnf("Read CSV row failed: %v", err)
            failedCount++
            continue
        }
        
        // 数据校验
        if err := w.validateCodeRow(row); err != nil {
            log.Warnf("Invalid code row: %v", err)
            failedCount++
            continue
        }
        
        // 5. ⭐ 累积到批次
        codes = append(codes, &InventoryCode{
            ItemID:       event.ItemID,
            SKUID:        event.SKUID,
            BatchID:      event.BatchID,
            Code:         row[0],        // 券码
            SerialNumber: row[1],        // 序列号
            ExpireTime:   parseExpireTime(row[2]),
            Status:       CodeStatusAvailable,
        })
        
        totalCount++
        
        // 6. ⭐ 达到批次大小 → 批量插入
        if len(codes) >= batchSize {
            // 分表存储（按 item_id 取模，避免单表过大）
            tableIdx := event.ItemID % 100
            tableName := fmt.Sprintf("inventory_code_pool_%02d", tableIdx)
            
            // ⭐⭐ 批量插入（关键优化点）
            if err := w.codePoolRepo.BatchInsert(tableName, codes); err != nil {
                log.Errorf("Batch insert failed: %v", err)
                failedCount += len(codes)
            } else {
                successCount += len(codes)
            }
            
            // ⭐ 重置切片（复用底层数组，减少内存分配）
            codes = codes[:0]
            
            // 打印进度
            log.Infof("Imported %d codes so far (success: %d, failed: %d)", 
                totalCount, successCount, failedCount)
            
            // 更新进度（前端轮询）
            w.updateImportProgress(event.BatchID, totalCount, successCount, failedCount)
        }
    }
    
    // 7. 处理剩余券码（不足 batchSize 的部分）
    if len(codes) > 0 {
        tableIdx := event.ItemID % 100
        tableName := fmt.Sprintf("inventory_code_pool_%02d", tableIdx)
        
        if err := w.codePoolRepo.BatchInsert(tableName, codes); err != nil {
            log.Errorf("Batch insert remaining codes failed: %v", err)
            failedCount += len(codes)
        } else {
            successCount += len(codes)
        }
    }
    
    // 8. 更新库存统计
    w.inventoryRepo.UpdateTotalStock(event.ItemID, event.SKUID, successCount)
    
    // 9. 预热到 Redis（券码池缓存）
    w.preloadCodesToRedis(event.ItemID, event.SKUID, event.BatchID)
    
    log.Infof("Code import completed: batch=%s, total=%d, success=%d, failed=%d", 
        event.BatchID, totalCount, successCount, failedCount)
    
    return nil
}

type CodePoolRepository struct {
    db *gorm.DB

func (r *CodePoolRepository) BatchInsert(tableName string, codes []*InventoryCode) error {
    if len(codes) == 0 {
        return nil
    }
    
    // ⭐ 构造批量 INSERT 语句
    // INSERT INTO xxx (col1, col2, ...) VALUES (?, ?, ...), (?, ?, ...), ...
    
    valueStrings := make([]string, 0, len(codes))
    valueArgs := make([]interface{}, 0, len(codes)*7)  // 每行 7 个字段
    
    for _, code := range codes {
        valueStrings = append(valueStrings, "(?, ?, ?, ?, ?, ?, ?)")
        valueArgs = append(valueArgs, 
            code.ItemID, 
            code.SKUID, 
            code.BatchID,
            code.Code, 
            code.SerialNumber, 
            code.ExpireTime,
            code.Status,
        )
    }
    
    // 拼接 SQL
    query := fmt.Sprintf(
        "INSERT INTO %s (item_id, sku_id, batch_id, code, serial_number, expire_time, status) VALUES %s",
        tableName,
        strings.Join(valueStrings, ","),
    )
    
    // ⭐ 单条 SQL 插入 1000 条数据
    result := r.db.Exec(query, valueArgs...)
    if result.Error != nil {
        return fmt.Errorf("batch insert failed: %w", result.Error)
    }
    
    // 监控插入性能
    log.Debugf("Batch insert %d codes to %s, affected rows: %d", 
        len(codes), tableName, result.RowsAffected)
    
    return nil
}
```

## 五、技术 3：Worker Pool 并发处理

### 5.1 核心原理

**串行 vs 并发**：

```go

// ===== Worker Pool 实现 =====
type WorkerPool struct {
    workerCount int
    taskChan    chan func()
    wg          sync.WaitGroup
    ctx         context.Context
    cancel      context.CancelFunc
}

func NewWorkerPool(workerCount int) *WorkerPool {
    ctx, cancel := context.WithCancel(context.Background())
    
    pool := &WorkerPool{
        workerCount: workerCount,
        taskChan:    make(chan func(), 100),  // 缓冲队列，避免阻塞
        ctx:         ctx,
        cancel:      cancel,
    }
    
    // ⭐ 启动 N 个 Worker goroutine
    for i := 0; i < workerCount; i++ {
        go pool.worker(i)
    }
    
    return pool
}

func (p *WorkerPool) worker(id int) {
    log.Debugf("Worker %d started", id)
    
    for {
        select {
        case task, ok := <-p.taskChan:
            if !ok {
                log.Debugf("Worker %d stopped", id)
                return
            }
            
            // 执行任务（带 panic 恢复）
            func() {
                defer func() {
                    if r := recover(); r != nil {
                        log.Errorf("Worker %d panic: %v", id, r)
                    }
                }()
                
                task()  // 执行任务
            }()
            
            p.wg.Done()
            
        case <-p.ctx.Done():
            log.Debugf("Worker %d canceled", id)
            return
        }
    }
}

func (p *WorkerPool) Submit(task func()) {
    p.wg.Add(1)
    p.taskChan <- task  // 任务入队
}

func (p *WorkerPool) WaitAll() {
    p.wg.Wait()         // 等待所有任务完成
    close(p.taskChan)   // 关闭队列
}

func (p *WorkerPool) Stop() {
    p.cancel()
}

type PriceUpdateWorker struct {

func (w *PriceUpdateWorker) updateSinglePrice(update *PriceUpdate) UpdateResult {
    // 1. 读取 SKU（含版本号）
    sku, err := w.skuRepo.GetByID(update.SKUID)
    if err != nil {
        return UpdateResult{
            SKUID:   update.SKUID,
            Success: false,
            Error:   fmt.Sprintf("SKU not found: %v", err),
        }
    }
    
    // 2. ⭐ 乐观锁更新（防并发冲突）
    result := w.skuRepo.DB.Exec(`
        UPDATE sku_tab
        SET price = ?, version = version + 1, updated_at = NOW()
        WHERE id = ? AND version = ?
    `, update.NewPrice, update.SKUID, sku.Version)
    
    if result.Error != nil {
        return UpdateResult{
            SKUID:   update.SKUID,
            Success: false,
            Error:   fmt.Sprintf("Update failed: %v", result.Error),
        }
    }
    
    if result.RowsAffected == 0 {
        // 并发冲突，重试
        return UpdateResult{
            SKUID:   update.SKUID,
            Success: false,
            Error:   "Concurrent modification, please retry",
        }
    }
    
    // 3. 记录价格变更日志
    w.priceLogRepo.Create(&PriceChangeLog{
        SKUID:      update.SKUID,
        OldPrice:   sku.Price,
        NewPrice:   update.NewPrice,
        ChangeType: "batch",
        OperatorID: update.OperatorID,
        Reason:     update.Reason,
        CreatedAt:  time.Now(),
    })
    
    // 4. 清除缓存
    w.redis.Del(context.Background(), 
        fmt.Sprintf("sku:price:%d", update.SKUID),
        fmt.Sprintf("item:detail:%d", sku.ItemID),
    )
    
    return UpdateResult{
        SKUID:    update.SKUID,
        Success:  true,
        OldPrice: sku.Price,
        NewPrice: update.NewPrice,
    }
}
```

### 5.3 并发模型说明

**主线程职责**：

1. 调用 `Submit()` 持续投递任务
2. 将任务写入 `taskChan` 缓冲队列
3. 调用 `WaitAll()` 等待全部任务完成

**Worker Pool 职责**：

1. 启动 20 个 goroutine 持续消费 `taskChan`
2. 每个 Worker 执行 `updateSinglePrice()` 处理单个 SKU
3. 当前任务完成后立即复用 Worker，继续领取下一条任务
4. 所有任务清空后，主线程继续执行后续流程

策略 B（单商品短时间写挂起）虽然稳妥，但在顶级大厂（如阿里、美团、京东）的超大规模高并发场景下，“阻断 C 端写路径”哪怕只有 1 秒，也是非常危险的操作。

在生产环境中，为了不影响 C 端用户的丝滑体验，业界还有两套完全不影响 C 端写入、更加优雅且降维打击的终极解决方案。

方案一：版本位点水位线法（Sequence/Version Watermark）
💡 核心破局点
你之前苦恼于“无法调用 MQ API 查积压”。这套方案换了种思路：不需要查 MQ，让 Redis 和 MySQL 互相把“进度”告诉对方。

1. 核心设计
我们在 Redis 扣减库存时，不仅扣减数量，还要同时维护一个单商品自增的“操作版本号/位点”（Sequence Number）。

Redis 热路径执行（Lua 脚本）：

available_qty = available_qty - 1

current_seq = INCRBY inventory:seq:{key} 1（假设当前变成 105）

MQ 消息投递：消息体中除了包含订单号、扣减数，必须强制带上这个 seq = 105。

MySQL 权威库落地：

更新 inventory_balance 余额。

写入 inventory_ledger 账本，并把 seq = 105 记入账本的独立字段中。

2. 生产环境中如何对账与修复？
对账 Job 启动时，完全不加锁，也不暂停 C 端：

读取 Redis 当前最新的位点：redis_seq = 110。

读取 MySQL 此时此刻已经落库的最大位点：mysql_max_seq = 105。

精妙的裁决逻辑：

因为 redis_seq (110) > mysql_max_seq (105)，说明中间确实有 5 笔交易正在 MQ 里排队，还没落库。

重点来了：对账 Job 发现不相等，立刻放弃本次核对，直接跳过这个 Key，不执行任何修复！

等下一轮对账（比如 5 分钟后），如果这 5 笔消息消费完了，MySQL 的 max_seq 追上了 110。这时候两边 seq 相等，Job 再去比对余额。如果余额也相等，安全；如果两边位点相等但余额不等，说明真的漂移了，直接用差值 Δ 增量冲正。

🎯 面试话术提炼
“在生产中，我们不需要去查 MQ 的排队积压。我们通过在 Redis 和 MySQL 两端引入递增的流水位点（Sequence Number）。对账时只要发现 MySQL 的最大位点落后于 Redis，就证明存在‘飞行中的消息’（In-flight Messages）。系统会选择**‘暂不对账，等待对齐’**，从而实现非阻塞（Non-blocking）的最终一致性校验。”

方案二：库存大闸/令牌桶模型（Token Gatekeeper Pattern）
这是各大电商大促秒杀中最常用的架构降维打击方案。它直接颠覆了“Redis 必须和 MySQL 强一致”的执念。

💡 核心破局点
超卖的底座是 MySQL。只要 MySQL 不超卖，前台 Redis 稍微有点不准，其实根本无所谓。 Redis 在这里不当“绝对精细的账本”，只当一个“拦洪大闸（防穿透过滤器）”。

1. 核心设计
假设商品在 MySQL 里的真实库存有 100 个。

我们在 Redis 里不存 available_qty = 100，而是放入 120 个购买令牌（Token）（故意多放 20% 作为缓冲池，学术上叫超发/预分配）。

C 端用户下单，抢的不是库存数字，抢的是 Redis 里的令牌。

抢到令牌的人，进入 MQ 排队，去 MySQL 里真正扣减库存。

MySQL 才是终审判官：MySQL 执行 UPDATE ... SET qty = qty - 1 WHERE qty >= 1。如果抢到令牌的人到了 MySQL 发现真库存已经变成 0 了，MySQL 拒绝，触发扣减失败，后端订单取消，释放 Redis 令牌。

2. 生产环境中如何对账与修复？
在这种模型下，根本不需要高频对账和增量修复。

定期重置（Reset）代替修复：每隔 10 分钟，由一个异步 Worker 读取 MySQL 的真实剩余可用库存（比如剩下 40 个），直接写一个 Lua 脚本，把 Redis 里的令牌数暴力重置为 40 * 1.2 = 48 个。

因为有 MySQL 最终的 qty >= 1 边界卡死，绝对不可能发生超卖资损。Redis 丢个把数据，只是影响了这 10 分钟内多进去了几个流量，或者少进去了几个流量，在海量大促面前，业务完全可接受。

🎯 面试话术提炼
“我们采用了库存大闸（Gatekeeper）模式，将 Redis 定义为高并发流量过滤器，而非权威记账本。Redis 允许一定比例的令牌超发，通过控制‘进入后端的凭证数量’来保护 MySQL。最终的强一致性防超卖由 MySQL 的行锁和边界检查兜底。这样直接将‘分布式强一致性’问题降维转化为了‘受控的流量漏斗’问题，极大地提升了系统的吞吐量和韧性。”

总结：面试时的降维打击指南
你在面试答辩时，可以把这三个方案组合成一个架构演进故事：

第一阶段（青铜）：直接改 DB 余额，并发一高行锁雪崩，且无幂等拦截，卒。

第二阶段（白银）：引入 Redis 热路径 + MQ 异步落库。但面临对账时“时间差”导致的数值漂移。

第三阶段（黄金）：为了修复漂移，设计了方案 B（单商品写挂起）。该方案实现简单，适合 B 端流量或低频对账。

第四阶段（王者 - 大厂标配）：为了追求 C 端绝对的无损和高可用，演进出了版本位点水位线法（非阻塞对账），或者直接从模型上升级为库存大闸令牌模式（允许不一致，靠底座防超卖）。

这个递进式的回答，能瞬间向面试官展示出你：懂底层原理、懂架构痛点、更有根据不同业务体量选择最合适方案的“架构权衡（Trade-off）能力”。

你觉得这两个新方案中，哪一个更符合你目前准备的面试项目的业务背景和技术体量呢？




那库存场景里，Redis 和 MySQL 不一致是有实质影响的，而且要分方向看。

最适合面试的开场是：

“库存场景里，MySQL 一般是最终权威账本，Redis 是高并发扣减入口。两边不一致不一定立刻资损，但一定会影响卖货结果、流量控制和补偿复杂度。”

然后分 3 种情况讲。

1. Redis 库存小于 MySQL 库存
影响是“少卖”：

明明还有货，但 Redis 先判没货了
用户被提前拦在前面，造成销售损失
大促时会影响转化率
这个问题通常不直接资损，但会损失 GMV。

2. Redis 库存大于 MySQL 库存
影响是“放进来过多请求”：

更多请求穿透到下游
MySQL 会承受额外扣减压力
如果 MySQL 没有 qty >= 1 这种边界保护，就可能超卖
所以这里最关键的是：即使 Redis 不准，MySQL 也必须能兜住最终防超卖。

3. Redis 和 MySQL 长时间不一致
影响是系统会越来越乱：

前台卖货判断不准
对账越来越难
补偿任务会重复触发
库存扣减、回补、取消订单这些链路容易互相打架
这类问题短时间能接受，长时间就会变成稳定性问题。

面试里你可以直接这样答：

“库存场景里，不一致的核心影响有两个。第一，Redis 比 MySQL 少会导致少卖；第二，Redis 比 MySQL 多会导致过量请求进入后端，严重时如果 MySQL 没有边界保护就会超卖。工程上我们一般接受短暂不一致，但不能接受最终不一致，所以必须保证 MySQL 是权威库，并通过边界更新、消息落库、对账补偿来最终收敛。”

如果面试官继续追问“那你们最怕哪种”，你就答：

“最怕的是 Redis 比 MySQL 大，而且 MySQL 没有最终防线，因为那会直接导致超卖资损。Redis 比 MySQL 小通常是少卖，损失收入；Redis 比 MySQL 大但 MySQL 有边界保护，一般是多放流量，性能会受影响，但不会产生最终资损。”

你还可以背一个更短的版本：

“库存不一致的影响本质上是两类：少卖和超卖。Redis 少于 MySQL 会少卖，Redis 多于 MySQL 会多放流量，严重时会超卖。所以库存设计的关键不是让 Redis 永远绝对准确，而是让 MySQL 始终保有最终防超卖能力，同时用对账和补偿把 Redis 拉回正确状态。”
