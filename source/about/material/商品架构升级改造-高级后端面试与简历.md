# 商品架构升级改造高级后端面试与简历材料

这份材料用于高级后端面试，目标是把“商品系统从 `topup / bill` 简单商品，演进到第三方供给接入和本地生活券治理”的经历，讲成一个有业务背景、有技术深度、有结果指标的项目。

核心表达不要只说“我拆了服务”，而是强调：业务复杂度上来后，如何通过异步链路、状态机、模型治理、批量框架、库存中心和可观测性，把一个容易失控的大单体改造成可扩展的平台能力。

## 30 秒项目开场

我主导过一次商品架构升级。平台早期主要售卖 `topup / bill` 这类标准化虚拟商品，商品模型轻、库存约束弱，系统更像一个简单配置后台。后来业务接入酒店、电影票、机票等第三方供应商，又做本地生活券，开始支持商户建券、Excel 批量导入、QC 审核、系统生码和库存控制，原来的商品系统很快变成商品、供给、库存、审核、批处理任务都耦合在一起的大单体。

我的核心工作是把它重构成 `供给接入平台 + draft_tab 工作区 + 审核治理 + item_tab 正式商品 + 库存中心` 的分层架构，并用 `Kafka + Worker`、Outbox、状态机、Saga、乐观锁和统一批量操作框架补齐一致性、可观测性和批量处理能力。改造后，代码复用率提升到 90%+，新品类接入从 2 周缩短到 2 天，上架成功率从 85% 提升到 99%+。

## 2 分钟项目介绍

这个项目的背景是典型的业务倒逼架构演进。

最开始平台只卖 `topup / bill` 这类标准化商品，商品字段少，库存也不是主要矛盾，运营在后台配置后就可以售卖。随着业务发展，我们先接入酒店、电影票、机票等第三方供应商商品，需要处理供应商 Push / Pull、商品定时同步、价格变化、库存变化和上下架变化。再往后，本地生活券开始让商户自助建券和 Excel 批量导入，系统又新增了系统生码、券码池、库存扣减、QC 审核、发布治理等能力。

这些能力叠加后，原来的商品系统暴露出几个问题。

第一，商品、供给、库存、审核和批量任务都混在一起，边界越来越模糊。第二，不同品类字段持续污染商品模型，代码里出现大量按品类分支的 `if-else`。第三，供应商同步和 Excel 批量导入分散在 background job 里，缺少统一任务编号、进度、失败原因和告警。第四，本地券涉及券码锁定、扣减、支付超时释放和库存回补，如果继续把库存当商品字段处理，资损风险会很高。第五，商户建券后不应该创建即上架，必须把草稿、审核中和正式可售隔离开。

我的改造思路是把商品系统从“单体展示对象”升级成“分层治理平台”。

供给侧，我把供应商 Push / Pull、商户 Excel 导入、运营批量创建统一收敛到供给接入平台，用适配层做协议转换、数据校验、清洗映射和任务编排。任务链路通过 `Kafka + Worker` 异步化，并配合 Outbox 保证消息可靠发布。

商品写侧，我选择 `draft_tab + 状态机 + item_tab`，没有再引入更重的 `staging_tab`。商户保存时进入 `draft_tab`，状态可以是 `DRAFT / UNDER_REVIEW / REJECTED / APPROVED`；审核通过后再发布到正式 `item_tab`，交易、搜索、营销只读取正式商品。这样可以让编辑态和交易态彻底隔离，避免半成品商品污染线上。

发布侧，我用 Saga 编排商品、SKU、属性、缓存、ES 事件等多步骤写入；状态推进使用乐观锁控制并发；批量导入和批量调价、设库存、券码导入则抽象成统一批量操作框架，复用任务表、明细表、Worker 模板、进度和结果文件生成。

库存侧，我把库存和券码从商品属性中拆出来，交给库存中心独立治理。商品中心只表达商品是否合规、是否允许发布；前台可售状态由商品状态和库存中心可用数量动态聚合，例如：

```text
Salable = (Item.Status == ACTIVE) && (Stock.Available > 0)
```

这样即使商品审核通过，但库存中心还没有上传有效券码，前台也不会错误开放购买。

最终这个改造带来的价值是：新品类接入从 2 周缩短到 2 天，代码复用率提升到 90%+，上架成功率从 85% 到 99%+，万级 SKU 批量操作从数小时降到分钟级，商品模型、审核链路和库存资损风险也都更可控。

## 5 分钟 STAR 面试话术

### Situation：业务从简单商品变成多源多品类供给

我们平台早期主要做 `topup / bill`，商品结构简单，库存接近弱约束，原系统基本可以理解为一个商品配置后台。后来业务快速扩张，一方面接入酒店、电影票、机票等第三方供应商，供应商的数据格式、价格和库存更新频率都不同；另一方面又做本地生活券，让商户可以自助建券、批量导入券码、走 QC 审核后发布。

这时系统复杂度明显上升：商品创建不再只是写一张商品表，而是涉及多源接入、异步任务、审核流转、正式发布、库存初始化、券码池和下游搜索缓存同步。

### Task：在不影响线上业务的前提下重划边界

我的任务是主导商品供给和商品中心的架构升级，在不中断线上业务的前提下，把原来耦合在一起的供给接入、商品编辑、审核发布、批量处理和库存券码治理拆清楚。

目标主要有四个：

1. 新品类接入不再从头开发，尽量复用统一流程。
2. 商品主模型保持稳定，不被酒店、电影票、本地券等品类字段持续污染。
3. 供应商同步和 Excel 批量导入要任务化、可追踪、可重试、可告警。
4. 本地券库存和券码要独立治理，降低超卖、无券可发和状态不一致的资损风险。

### Action：分层治理、异步化和状态机隔离

第一，我把多源输入收敛到供给接入平台。供应商 Push / Pull、商户 Portal、运营 Excel、API 接入都先进入统一入口，通过 `source_type` 和 `source_user_type` 做来源识别，再走适配、校验、清洗和字段映射。任务创建后立即返回 `task_code`，后续通过 `Kafka + Worker` 异步处理。

第二，我补齐异步链路的一致性和可观测性。任务表记录批次状态和进度，明细表记录每条商品的处理结果；消息发布使用 Outbox 模式，避免本地事务提交成功但消息丢失；Worker 处理失败后支持重试和看门狗兜底；监控上按任务状态、失败原因、供应商同步延迟和批量处理耗时打指标。

第三，我用 `draft_tab + 状态机 + item_tab` 隔离编辑态和交易态。商户保存的数据先在 `draft_tab`，允许不完整；提交审核后只是推进状态到 `UNDER_REVIEW`；审核通过后才发布到正式 `item_tab`。正式交易链路只读取 `item_tab`，不会读到草稿或审核中的中间态。

第四，我在发布阶段使用 Saga 和乐观锁。商品发布会涉及创建 item、SKU、品类属性、库存配置、缓存和搜索索引事件，任何一步失败都要能补偿或转人工。我用编排式 Saga 管理步骤链，每个步骤实现执行和补偿；状态更新使用 `version` 乐观锁，防止并发审核或重复发布。

第五，我抽象统一批量操作框架。原来商品上架、批量调价、设库存、券码导入各自实现一套同步逻辑，容易 OOM，也没有统一审计。我把它们抽象成 `operation_batch_task` 和 `operation_batch_item` 两层模型，通过模板方法封装流式解析、分批查询、Worker Pool 并发、进度更新、结果文件生成和 before / after 审计。新增一种批量操作只需要实现具体业务处理器。

第六，我把库存和券码从商品属性里拆出，交给库存中心独立治理。券码制通过 Redis 热路径和 MySQL 权威账本处理出货、锁定和补偿；数量制通过原子扣减控制可售；商品中心只维护商品合规状态，前台可售由商品状态和库存可用量动态聚合。

### Result：接入效率、稳定性和风险控制都有提升

改造后，商品平台从单品类定制逻辑变成统一供给和运营平台。新品类接入从 2 周缩短到 2 天，代码复用率从低于 10% 提升到 90%+；上架成功率从 85% 提升到 99%+；万级 SKU 批量操作从数小时降到分钟级，10 万券码导入可以在分钟级完成；批量操作和审核链路都有任务进度、失败原因、结果文件和审计日志，问题定位从靠查日志变成按 `task_id` 回溯。

从架构收益看，正式商品模型更干净，供应商脏数据和商户编辑中的半成品不会直接进入交易链路；库存和券码作为交易安全能力独立治理，也避免了“商品已发布但无券可发”这类风险。

## 高频追问回答

### Q1：为什么选择 `draft_tab + item_tab`，而不是 `draft + staging + item` 三层？

A：我们评估过三层方案，但最终选择了单 `draft_tab` 承接草稿和审核中状态，原因是当时业务审核周期可控，商户提审后并没有很复杂的并发二次编辑诉求。如果再引入 `staging_tab`，会多一套 DDL、多一次跨表复制，也会让供给平台变成影子商品中心。

所以我采用的是 `draft_tab + 状态机 + item_tab`：草稿、审核中、驳回都在 `draft_tab` 里通过状态区分，只有审核通过才发布到正式 `item_tab`。这样实现更轻，边界也清楚：编辑态归工作区，交易态归正式商品中心。

### Q2：Outbox 解决了什么问题？

A：它解决的是本地 DB 事务和 Kafka 消息之间的一致性问题。比如创建上架任务时，如果任务表写成功，但发送 Kafka 失败，后续 Worker 就不会处理，任务会卡死。

我的做法是在同一个 DB 事务里写任务表和 `outbox_message` 表。事务提交后由 OutboxPublisher 扫描待发送消息，投递 Kafka，成功后标记已发送；如果发送失败，定时任务会继续重试。这样至少保证任务状态和待发送消息同时存在，不会出现业务数据成功但异步流程丢失。

### Q3：商品发布为什么需要 Saga？

A：商品发布不是单表写入，它会涉及 item、SKU、属性扩展表、库存配置、搜索索引事件、缓存刷新等多个步骤。这里很难用一个大事务包住所有资源，而且部分下游本身就是异步系统。

所以我用编排式 Saga：每个发布步骤实现 `Execute` 和 `Compensate`，按顺序执行；中间失败时，已经完成的步骤逆序补偿；补偿失败则记录 Saga 日志并告警转人工。这样可以把跨表、跨系统发布从“失败后状态未知”变成“可恢复、可补偿、可追踪”。

### Q4：批量导入如何避免 OOM？

A：核心是不要把整个 Excel 一次性读入内存。我们使用流式解析逐行读取，把数据分批落到明细表，再由 Worker Pool 并发处理。每批控制在固定大小，处理完复用缓冲区，并及时更新任务进度。

这样内存占用和文件行数基本解耦，10 万行券码导入不会因为全量加载而 OOM。同时每条明细都有独立状态和错误原因，失败后可以只重试失败行，而不是整批重跑。

### Q5：如何保证并发审核或重复发布不会把状态打乱？

A：状态流转用有限状态机和乐观锁双重控制。业务上只允许合法状态迁移，例如 `DRAFT -> UNDER_REVIEW -> APPROVED / REJECTED -> PUBLISHED`；技术上每次更新都带 `version`：

```sql
UPDATE listing_task
SET status = ?, version = version + 1
WHERE id = ? AND status = ? AND version = ?;
```

如果 `RowsAffected = 0`，说明状态已经被别人推进，当前操作要么重试，要么返回幂等结果。这样可以防止并发审核、重复回调、Worker 重投递导致的状态回退或重复发布。

### Q6：商品状态和可售状态为什么要拆开？

A：因为本地生活券这类商品是双阶段流程：先创建商品主数据并审核通过，再上传库存或生成券码。商品审核通过只能说明它合规、允许发布，不代表现在一定有货可卖。

所以我把商品状态放在 `item_tab`，表示 `ACTIVE / DISABLED` 等合规发布状态；可售状态由前台聚合商品状态和库存中心结果动态计算。只有 `Item.Status == ACTIVE` 且 `Stock.Available > 0` 时，前台才展示立即购买。这样可以避免商品已发布但库存未准备好时误下单。

### Q7：这个项目里你最体现高级后端能力的点是什么？

A：我觉得不是某一个技术点，而是把业务复杂度拆成几个稳定边界：供给平台处理外部复杂性，`draft_tab` 处理编辑和审核工作区，`item_tab` 保护正式商品主数据，库存中心处理交易安全，批量框架处理大规模异步任务。

落地时又把这些边界用工程手段闭环起来，包括 Outbox、Saga、乐观锁、Worker Pool、审计日志、任务监控和失败重试。高级后端不只是写接口，更重要的是能识别系统未来会在哪些地方失控，并提前把边界、一致性和可观测性设计进去。

## 使用建议

- 一面偏编码和工程落地：重点讲 `Kafka + Worker`、Outbox、乐观锁、批量导入 OOM、Worker Pool。
- 二面偏系统设计：重点讲边界拆分、`draft_tab + item_tab`、Saga、库存和可售状态解耦。
- HR 或综合面：重点讲业务演进、个人角色、跨团队推动和结果指标。
