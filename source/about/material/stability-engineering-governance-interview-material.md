# 稳定性与工程治理面试材料

本文用于支撑简历中的这段经历：

> **稳定性与工程治理**
>
> - 建设覆盖首页、详情、下单、支付、履约、退款的业务监控体系，跟踪下单成功率、支付成功率、履约成功率、变价率、缓存命中率、供应商错误率等用户感知指标，并与接口、依赖、中间件、系统监控形成五层可观测体系。
> - 建立 NOC / Critical / 业务三级告警和 20+ 大促/故障预案，配套入口/出口限流、功能开关、熔断降级、支付/履约补偿任务、自动对账与巡检报表，告警噪音减少 70%，提升问题定位、资损防控和线上恢复效率。

这条经历面试时不能讲成“我做了很多监控”。核心表达应该是：我把稳定性从机器指标升级为业务结果治理，围绕用户购买旅程定义 SLI，建立分层监控、分级告警、预案止血、补偿对账和发布治理，让系统在外部依赖不稳定、流量突增、回调乱序和数据不一致时仍然可发现、可隔离、可恢复。

## 一、项目一句话

Shopee Digital Purchase 是一个多市场、多品类数字商品交易平台，核心链路覆盖首页、搜索、详情、加购试算、下单、支付、供应商履约和退款售后。我参与建设稳定性与工程治理体系，把下单成功率、支付成功率、履约成功率、变价率、补偿 backlog、供应商错误率等业务指标纳入监控和告警，并配套限流、熔断、降级、补偿、对账、巡检和大促预案，降低告警噪音，提高故障发现、定位和恢复效率。

---

## 二、30 秒项目介绍

Digital Purchase 的稳定性难点不是单个服务是否存活，而是用户能不能完整完成购买闭环。比如接口都返回 200，但支付成功率下降、履约回调堆积、价格差异率升高、供应商错误率异常，业务上仍然是故障。

我做的事情是围绕首页、详情、下单、支付、履约、退款这条用户旅程建设五层可观测体系：第一层看下单成功率、支付成功率、履约成功率等用户感知指标；第二层看核心接口和阶段耗时；第三层看 SPM、Hub、供应商等外部依赖；第四层看 MySQL、Redis、Kafka、ES 等中间件；第五层看容器、CPU、内存等系统资源。告警上区分 NOC、Critical 和业务告警，并为大促、供应商异常、支付回调异常、履约失败、价格异常等场景沉淀预案、补偿任务和自动对账，最终告警噪音减少约 70%，线上问题定位和恢复效率明显提升。

## 三、2 分钟项目介绍

这个项目的背景是 DP 业务从充值、账单扩展到酒店、机票、电影票、券类、本地生活等多个数字商品品类。系统形态也从单一交易链路变成了多服务、多中间件、多外部依赖协作的复杂平台。用户一次购买可能经过首页入口、商品详情、价格试算、库存确认、订单创建、收银台、支付渠道、供应商履约、退款售后等多个阶段。

早期稳定性治理有几个典型问题。

第一，监控偏接口和机器指标，业务结果指标不够完整。服务 QPS、CPU、P99 都能看到，但下单成功率、支付成功率、履约成功率、变价率、补偿堆积这类用户感知指标没有形成统一链路。

第二，告警噪音比较多。很多告警来自单接口错误率、依赖毛刺、资源水位，没有按业务影响分级，值班同学很容易被噪音淹没，真正的核心链路异常反而需要人工从多个看板里拼。

第三，DP 有大量外部依赖。支付依赖 SPM，履约依赖 Hub 和供应商，部分品类还依赖实时查价、查库存、出票和退款回调。外部依赖慢、失败、回调乱序、部分失败都可能影响订单闭环。

第四，数据一致性和资金安全要求高。支付成功但订单状态没有推进、履约失败但退款补偿没有执行、价格差异未及时发现，都会带来用户投诉和资损风险。

我的解决思路是把稳定性按业务旅程重新组织，而不是按服务孤立治理。核心链路从首页、详情、下单、支付、履约、退款拆成阶段，每个阶段定义成功率、延迟、错误码、依赖耗时、补偿 backlog、对账差异等指标；再把监控分成业务、接口、依赖、中间件、系统五层，让故障能从用户影响一路下钻到具体服务和依赖。

告警上建立 NOC / Critical / 业务三级体系。NOC 关注用户感知的核心链路，比如下单成功率、支付成功率、履约成功率；Critical 关注核心服务或关键依赖退化；业务告警关注单品类、单市场、单供应商、单任务异常。每条高优告警绑定 owner、看板和 SOP，同时通过告警收敛、分桶治理、阈值调整和重复告警合并减少噪音。

恢复能力上，围绕入口/出口限流、功能开关、熔断降级、支付/履约补偿、自动对账和巡检报表沉淀 20+ 大促和故障预案。比如供应商异常时，入口侧可以限制相关品类下单，出口侧可以限制调用并发；支付回调异常时，通过支付查询补偿推进状态；履约失败时，通过重试、状态回查和退款补偿闭环；价格异常时，通过变价率和金额差异监控及时拦截。

最终这套治理体系把稳定性从“服务有没有挂”推进到“核心业务结果是否可信”。它的收益包括：核心链路异常更早发现，故障定位路径更清晰，告警噪音减少约 70%，大促和线上故障恢复更有 SOP，资金、库存、履约相关风险也能通过对账和补偿持续收敛。

## 四、一页速记卡

### 背景

- DP 支撑多市场、多数字商品品类。
- 核心链路覆盖首页、详情、试算、下单、支付、履约、退款。
- 外部依赖多：支付、Hub、供应商、营销、库存、计价。
- 业务要求高：不能资损、不能重复扣款、不能错误计价、不能订单状态长期卡住。

### 核心问题

- 只有接口和机器监控，不足以判断用户购买是否成功。
- 告警噪音多，缺少按业务影响分级。
- 支付、履约、退款依赖回调和补偿，容易出现最终一致性问题。
- 供应商和支付渠道异常具有局部性，需要按市场、品类、供应商、渠道下钻。
- 大促流量和活动流量会放大热 Key、慢依赖、重试和队列堆积。

### 核心方案

- **业务旅程 SLI**：下单成功率、支付成功率、履约成功率、退款成功率、变价率、补偿 backlog。
- **五层可观测**：业务监控、接口监控、依赖监控、中间件监控、系统监控。
- **分级告警**：NOC / Critical / 业务告警，按用户影响和恢复紧急度分级。
- **止血能力**：入口限流、出口限流、功能开关、熔断、降级、暂停销售。
- **恢复能力**：支付查询补偿、履约重试、退款状态回查、自动对账、巡检报表。
- **大促治理**：容量评估、全链路压测、故障演练、预案、变更冻结和发布 checklist。

### 结果表达

- 建设覆盖首页、详情、下单、支付、履约、退款的业务监控。
- 与接口、依赖、中间件、系统监控形成五层可观测体系。
- 沉淀 20+ 大促/故障预案。
- 告警噪音减少约 70%。
- 提升问题定位、资损防控和线上恢复效率。

## 五、为什么稳定性不是“服务不挂”

在交易系统里，接口可用不等于业务可靠。

典型例子：

- 商品详情接口返回 200，但展示了过期价格。
- 下单接口返回成功，但库存预占失败没有回滚。
- 支付回调接口返回 200，但订单状态没有推进。
- 履约服务还活着，但供应商出票失败堆积。
- 退款接口可用，但退款补偿任务长期 pending。

所以我对稳定性的定义分成四层：

| 层次 | 关注问题 | DP 例子 |
| --- | --- | --- |
| 可用性 | 用户能不能访问核心功能 | 首页、详情、下单、支付接口是否可用 |
| 正确性 | 业务结果是否正确 | 不能错误计价、不能重复扣款、不能超卖 |
| 可恢复性 | 失败后能否回到一致状态 | 支付、履约、退款是否能补偿和对账 |
| 可解释性 | 问题是否能定位和审计 | 一笔订单为什么卡住、哪一步失败、谁处理过 |

面试表达：

> 我做稳定性治理时没有只盯 CPU、内存和接口 5xx，而是先定义用户购买旅程的业务结果。对于 DP 来说，下单成功率、支付成功率、履约成功率、变价率、补偿任务 backlog，比单纯的服务存活更接近真实可靠性。

## 六、DP 的故障模型

### 1. 流量冲击

来源：

- 首页入口流量突增。
- 大促或活动带来的搜索、详情、下单峰值。
- 热门商品、热门酒店、热门电影场次。
- 客户端重试或外部调用重试。

风险：

- Redis 热 Key。
- 下游聚合接口 P99 升高。
- 线程池、连接池耗尽。
- 创单或支付请求被排队拖慢。

治理：

- 入口限流：按接口、用户、品类、活动、市场限流。
- 热点保护：本地缓存、Redis Key 分散、批量聚合。
- 大促压测：验证容量水位、P99、降级开关和限流阈值。

### 2. 慢依赖

来源：

- 支付服务 SPM 响应变慢。
- Hub 履约接口变慢。
- 供应商查价、查库存、出票、退款接口变慢。
- MySQL 慢查询、Redis 抖动、Kafka 堆积。

风险：

- 聚合接口被慢依赖拖住。
- 线程池被外部依赖占满。
- 用户下单或支付等待时间变长。
- 上游重试进一步放大流量。

治理：

- 对下游设置超时预算。
- 出口限流和并发控制。
- 熔断和降级。
- 强弱依赖分级，非核心依赖失败不阻断主链路。

### 3. 部分失败

来源：

- 某个市场异常。
- 某个品类异常。
- 某个供应商异常。
- 某个支付渠道异常。
- 某个灰度版本异常。

风险：

- 全站平均指标掩盖局部故障。
- 只有部分用户投诉，监控不敏感。
- 排查时难以快速定位故障域。

治理：

- 按市场、品类、供应商、支付渠道、端分桶监控。
- 告警支持局部维度触发。
- 预案支持按品类、供应商、渠道定向降级。

### 4. 数据不一致

来源：

- 支付回调失败或重复。
- 履约回调乱序。
- 退款回调失败。
- 消息丢失或消费失败。
- 补偿任务堆积。

风险：

- 支付成功但订单未支付。
- 订单已支付但未履约。
- 履约失败但未退款。
- 金额差异未发现。

治理：

- 状态机和幂等。
- 延时队列和扫表补偿。
- 支付/履约/退款对账。
- 巡检报表和人工兜底 SOP。

### 5. 发布和配置事故

来源：

- 代码发布回归。
- 活动配置错误。
- 支付渠道配置错误。
- 价格规则或营销规则变更。
- 限流、降级阈值配置错误。

风险：

- 核心链路成功率下降。
- 错误价格或错误优惠。
- 大面积用户不可购买。

治理：

- 灰度发布。
- 配置审批和回滚。
- 发版 checklist。
- 大促变更冻结。
- 发布后观察业务成功率和错误码。

## 七、五层可观测体系

### 1. 业务监控：判断用户是否受影响

业务监控是第一层，也是最重要的一层。它回答的是：用户能不能完成购买。

| 链路阶段 | 核心指标 | 异常含义 |
| --- | --- | --- |
| 首页 | 首页请求量、成功率、P99、缓存命中率 | 用户入口异常、缓存或配置异常 |
| 搜索/详情 | 搜索成功率、详情成功率、零结果率、变价率 | 导购体验异常、索引或价格异常 |
| 加购/试算 | 试算成功率、价格计算失败率、优惠不可用率 | 计价、营销或库存预校验异常 |
| 下单 | 创单成功率、库存预占失败率、重复单比例 | 交易主链路异常 |
| 支付 | 支付发起成功率、支付成功率、回调延迟 | 支付渠道或回调链路异常 |
| 履约 | 履约成功率、履约 pending 数、供应商错误率 | 供应商、Hub 或补偿链路异常 |
| 退款 | 退款申请成功率、退款成功率、退款 pending 数 | 售后或资金链路异常 |
| 一致性 | 对账差异数、补偿 backlog、DLQ 数量 | 异步链路未收敛 |

面试表达：

> 我们把业务监控放在第一层，因为它最接近用户感知。CPU 高不一定是事故，但下单成功率、支付成功率、履约成功率下降通常就是事故。

### 2. 接口和链路监控：定位卡在哪一步

业务指标异常后，需要下钻到具体接口和阶段。

重点指标：

- QPS。
- 成功率。
- 错误码分布。
- P50 / P95 / P99。
- 请求大小、响应大小。
- 下游调用次数。
- 超时率。
- 重试次数。

链路拆解示例：

```text
Checkout Init
  -> 商品校验
  -> 库存可售校验
  -> 计价试算
  -> 营销优惠
  -> 支付渠道
  -> 聚合返回

Order Create
  -> 参数校验
  -> 重复单校验
  -> 商品快照
  -> 价格校验
  -> 库存/资源锁定
  -> 订单写库
  -> 延时取消任务

Payment Callback
  -> 幂等校验
  -> 支付单状态迁移
  -> 订单状态迁移
  -> 履约事件触发
  -> 补偿任务兜底
```

### 3. 依赖监控：定位外部依赖是否退化

DP 的核心风险来自依赖很多，而且依赖的业务语义不同。

| 依赖 | 关注指标 | 风险 |
| --- | --- | --- |
| SPM 支付 | init 成功率、query 成功率、callback 延迟、渠道错误码 | 支付失败、支付状态卡住 |
| Hub 履约 | create order 成功率、query 成功率、callback 延迟 | 支付后无法履约 |
| 供应商 | 查价成功率、查库存成功率、出票成功率、退款成功率 | 单供应商或单品类异常 |
| 计价中心 | 计算成功率、P99、差异率、异常金额 | 金额错误或用户投诉 |
| 营销系统 | 活动查询、券锁定、券释放、额度扣减 | 优惠不可用或资源未释放 |
| 库存系统 | 预占成功率、释放成功率、对账差异 | 超卖、资源占用 |

依赖监控要分桶：

- 市场。
- 品类。
- 供应商。
- 支付渠道。
- 错误码。
- 接口类型。

### 4. 中间件监控：定位基础组件是否成为瓶颈

| 中间件 | 核心指标 | DP 关注点 |
| --- | --- | --- |
| MySQL | 慢 SQL、连接池等待、行锁、主从延迟 | 创单、支付、退款状态更新 |
| Redis | 命中率、P99、hot key、big key、内存淘汰 | 首页、详情、库存、限流 |
| Kafka | produce 成功率、consumer lag、DLQ、分区倾斜 | 履约事件、补偿、异步同步 |
| Elasticsearch | 查询耗时、rejected、索引延迟、shard 状态 | 搜索、酒店检索、商品索引 |
| 对象存储/CDN | 可用性、下载成功率、缓存命中 | 首页配置兜底、静态资源 |

### 5. 系统监控：定位资源是否饱和

系统监控包括：

- CPU。
- 内存。
- Goroutine / 线程数。
- GC。
- FD 数。
- 容器重启。
- 网络。
- 磁盘。
- Pod 数和扩容状态。

系统监控不是第一告警源，但它是排障依据。业务指标告诉我们“用户受影响了”，系统指标帮助判断“是不是资源耗尽导致的”。

## 八、告警体系：NOC / Critical / 业务三级

### 1. 为什么要分级

如果所有异常都直接进最高优先级告警，值班同学会被噪音淹没。告警必须按用户影响和处理紧急度分级。

| 等级 | 定义 | 示例 | 响应方式 |
| --- | --- | --- | --- |
| NOC | 核心业务结果异常，用户明显受影响 | 下单成功率下降、支付成功率下降、履约成功率大幅下降 | 立即响应，事故群，按 SOP 止血 |
| Critical | 核心服务或关键依赖明显退化，有扩大风险 | SPM 超时率升高、Hub 履约错误率升高、Kafka lag 快速增长 | 值班响应，确认是否升级 |
| 业务告警 | 局部业务、单品类、单供应商、单任务异常 | 某供应商查价失败、某市场变价率升高、某补偿任务失败 | Owner 跟进，必要时升级 |

面试表达：

> 告警分级的核心不是技术组件，而是业务影响。下单成功率下降比某个 Pod CPU 高更适合作为 NOC 告警；CPU 高更适合作为排障信号或 Critical 前置风险。

### 2. 告警设计原则

好的告警要满足：

- 有明确业务含义。
- 有 owner。
- 有看板链接。
- 有 SOP。
- 有恢复条件。
- 能按市场、品类、渠道下钻。
- 能去重和收敛。

避免的告警：

- 只报 CPU 高，但不知道用户是否受影响。
- 一个根因触发几十条重复消息。
- 没有 owner 的历史告警。
- 没有处理动作的观察型告警。
- 夜间唤醒非核心低影响问题。

### 3. 告警噪音治理

治理动作：

- 将只影响局部或非核心链路的告警降级。
- 把同根因多条告警聚合到一个事故入口。
- 按市场、品类、渠道增加分桶，避免全局平均掩盖局部问题。
- 对毛刺指标增加持续时间和多窗口判断。
- 删除无 owner、无人处理、长期误报的告警。
- 高优告警必须绑定 runbook 和看板。

结果表达：

> 通过告警分级、聚合、阈值调整和 owner 治理，告警噪音减少约 70%。更重要的是，值班同学能优先看到用户感知异常，而不是被组件级毛刺淹没。

## 九、止血能力：限流、熔断、降级、开关

### 1. 入口限流

入口限流保护平台不被突发流量打垮。

维度：

- API。
- 用户。
- 设备。
- IP。
- 市场。
- 品类。
- 活动。
- 商品或 SKU。

典型场景：

- 首页或详情流量突增。
- 热门商品被大量刷。
- 大促活动集中下单。
- 异常客户端重试。

面试表达：

> 入口限流的目标不是简单拒绝用户，而是保证核心链路在极端流量下仍然有服务能力。读链路可以更激进保护，支付回调和补偿链路不能和普通用户流量使用同一套粗暴限流策略。

### 2. 出口限流

出口限流保护下游和外部供应商。

维度：

- 供应商。
- 支付渠道。
- Hub 接口。
- SPM 接口。
- 品类。
- 市场。

典型场景：

- 某供应商接口限流或变慢。
- 支付渠道异常。
- Hub 履约接口错误率升高。
- 实时查价查库存流量过大。

策略：

- 限制并发。
- 控制 QPS。
- 按供应商隔离资源。
- 对慢供应商熔断。
- 对非核心品类暂停销售或降级。

### 3. 熔断

熔断用于阻止慢依赖拖垮上游。

适合场景：

- 供应商超时率持续升高。
- Hub 或 SPM 错误率升高。
- 某个依赖 P99 明显退化。

不适合场景：

- 正常库存不足。
- 用户参数错误。
- 业务规则拒绝。

关键点：

- 熔断按依赖和接口维度配置。
- 熔断后必须有降级或失败语义。
- 半开探测要控制流量。

### 4. 降级

降级必须区分业务场景。

| 场景 | 可以降级 | 不能降级 |
| --- | --- | --- |
| 首页 | 使用 CDN 快照、隐藏非核心运营位 | 展示明显错误入口 |
| 搜索 | 默认排序、隐藏个性化标签 | 展示不可售商品为可购买 |
| 详情 | 缓存基础信息、隐藏非核心推荐 | 用过期库存允许下单 |
| 结算 | 非关键优惠提示降级 | 跳过价格和库存校验 |
| 支付 | 支付渠道切换、挂起等待对账 | 未确认支付却标记成功 |
| 履约 | 重试、挂起、人工介入 | 静默丢失履约失败 |

### 5. 功能开关

高风险能力需要可控开关：

- 新品类入口。
- 供应商销售。
- 支付渠道。
- 营销活动。
- 价格规则。
- 降级策略。
- 补偿任务。

开关要求：

- 有 owner。
- 有审批。
- 有审计。
- 有回滚。
- 有生效范围。
- 有监控联动。

## 十、恢复能力：补偿、对账、巡检

### 1. 支付补偿

问题：

- 支付 init 成功但本地状态更新失败。
- 支付回调超时。
- 支付回调重复。
- 支付状态 pending 长时间不收敛。

方案：

- 支付单状态机。
- 支付渠道流水幂等。
- 支付结果查询任务。
- Pending 单扫表补偿。
- 支付成功后触发履约事件。
- 支付失败或过期后释放库存、券、积分等资源。

监控：

- 支付 pending 数量。
- pending 超时数量。
- 支付查询补偿成功率。
- 回调处理延迟。
- 支付成功但订单未推进数量。

### 2. 履约补偿

问题：

- 支付成功但履约事件丢失。
- 调 Hub 或供应商履约失败。
- 供应商回调延迟、重复、乱序。
- 履约失败需要退款或人工处理。

方案：

- 履约状态机。
- 履约请求幂等。
- F0/F1 状态重试。
- 供应商订单查询补偿。
- 失败后触发退款或人工处理。

监控：

- 履约成功率。
- 履约 pending 数。
- 履约重试次数。
- 供应商错误率。
- 履约失败退款触发率。

### 3. 退款补偿

问题：

- 用户发起退款后审批、退货、退款链路多阶段。
- SPM 退款回调失败。
- Hub return order 状态不一致。
- 部分退款和多商品订单更复杂。

方案：

- 退款单状态机。
- 退款请求幂等。
- SPM refund query 补偿。
- Hub return order query 补偿。
- 退款回调幂等处理。

监控：

- 退款 pending 数。
- 退款失败率。
- 退款超时数量。
- 退款对账差异。

### 4. 自动对账

对账目标：

- 发现支付系统和订单系统状态差异。
- 发现订单和履约状态差异。
- 发现退款系统和支付系统差异。
- 发现价格、库存、券、积分等资金或资源差异。

对账层级：

- 实时旁路校验。
- 小时级对账。
- 天级汇总对账。
- 巡检报表。

面试表达：

> 对账不是为了替代主流程正确性，而是承认分布式系统一定会有部分失败。主流程通过状态机和幂等降低错误概率，对账和补偿负责把少数异常持续收敛。

## 十一、大促和故障预案

### 1. 为什么大促预案重要

大促和平时最大的区别不是 QPS 变高，而是所有风险会同时放大：

- 首页入口流量更集中。
- 搜索、详情、下单链路峰值更陡。
- Redis 热 Key 更明显。
- 供应商和支付渠道压力更大。
- 用户重试和客户端重试更多。
- 人工处理窗口更短。

### 2. 大促前准备

检查项：

- 核心链路容量评估。
- 首页、详情、下单、支付、履约压测。
- Redis 热 Key 和 big value 检查。
- MySQL 慢 SQL 和连接池水位。
- Kafka lag 和消费能力。
- 支付渠道容量和降级方案。
- 供应商容量和限流阈值。
- 功能开关和降级开关验证。
- NOC 看板和告警演练。
- 变更冻结和回滚预案。

### 3. 典型预案

| 场景 | 止血动作 | 恢复动作 |
| --- | --- | --- |
| 首页接口异常 | CDN 快照兜底、关闭非核心运营位 | 恢复 Redis/配置服务，验证缓存命中 |
| 搜索/详情 P99 升高 | 降级个性化、开启本地缓存、限制低优流量 | 定位 ES、Redis、供应商或聚合依赖 |
| 创单成功率下降 | 限制异常品类、关闭高风险活动、保护 DB | 排查库存、计价、营销、订单写库 |
| 支付成功率下降 | 切换支付渠道、挂起订单、启动支付查询 | 对账确认支付结果，补偿推进状态 |
| 履约失败率升高 | 暂停异常供应商销售、履约重试限速 | 查询供应商状态，失败订单退款或人工处理 |
| 价格异常 | 关闭活动或计价策略、拦截异常金额 | 空跑比对、价格对账、恢复规则 |
| Kafka 堆积 | 扩消费者、暂停低优生产、隔离 topic | 消费追平，处理 DLQ |

### 4. 故障处理顺序

事故处理中要避免一开始就追根因。推荐顺序：

1. 判断用户影响和范围。
2. 确认是否需要升级 NOC。
3. 先止血：限流、降级、回滚、关开关、切渠道。
4. 保留证据：指标、日志、trace、变更记录。
5. 定位根因。
6. 恢复数据：补偿、对账、人工处理。
7. 复盘并沉淀 SOP。

面试表达：

> 事故现场最重要的是先控制用户影响和资损风险，而不是马上证明根因。根因可以慢一点，但错误价格、重复扣款、履约失败堆积必须先止血。

## 十二、发布与配置治理

稳定性事故很多不是机器故障，而是变更引起的。

### 1. 代码发布

治理动作：

- 灰度发布。
- 观察错误率、P99 和业务成功率。
- 异常自动或人工回滚。
- 发布后检查核心看板。

重点观察：

- 下单成功率。
- 支付成功率。
- 履约成功率。
- 错误码分布。
- 依赖耗时。
- 补偿任务 backlog。

### 2. 配置变更

高风险配置：

- 限流阈值。
- 降级开关。
- 支付渠道。
- 供应商开关。
- 价格规则。
- 营销规则。
- 库存策略。

治理动作：

- 配置审批。
- 灰度生效。
- 版本记录。
- 一键回滚。
- 审计日志。
- 变更事件进入监控。

### 3. 大促变更冻结

大促期间限制非紧急变更，原因是：

- 流量高峰下小变更也可能放大成大事故。
- 回滚窗口更短。
- 事故定位会被活动流量干扰。
- 多团队同时变更容易互相影响。

## 十三、收益怎么讲

### 1. 对业务的收益

- 用户购买链路异常能更早发现。
- 单市场、单品类、单供应商异常能被局部识别。
- 支付、履约、退款问题不再只依赖用户投诉暴露。
- 大促期间有明确预案和负责人。

### 2. 对工程的收益

- 从“服务监控”升级为“业务旅程监控”。
- 从“告警很多”升级为“高优告警可行动”。
- 从“事故中临时查”升级为“看板、SOP、补偿、对账闭环”。
- 从“人工发现一致性问题”升级为“补偿 backlog 和对账差异可观测”。

### 3. 对资损防控的收益

- 价格异常有变价率和金额差异监控。
- 支付状态异常有查询补偿。
- 履约失败有重试和退款兜底。
- 退款异常有状态回查和对账。
- 高风险活动、供应商、支付渠道有开关和降级能力。

### 4. 简历结果表达

可以这样讲：

> 这套治理体系覆盖了从首页到退款的关键用户旅程，让团队能用业务指标发现问题，用五层监控定位问题，用分级告警减少噪音，用预案和补偿恢复问题。落地后告警噪音减少约 70%，核心链路的线上问题定位和恢复效率明显提升，也降低了价格、支付、履约、退款相关的资损风险。

## 十四、面试官可能追问

### 1. 你说做了五层监控，具体是哪五层？

回答：

> 第一层是业务监控，比如下单成功率、支付成功率、履约成功率、变价率、补偿 backlog。第二层是接口和链路监控，比如核心 API 的 QPS、P99、错误码和阶段耗时。第三层是依赖监控，比如 SPM、Hub、供应商、计价、营销、库存。第四层是中间件监控，比如 MySQL、Redis、Kafka、ES。第五层是系统监控，比如 CPU、内存、容器重启、GC。业务监控判断是否影响用户，下面几层帮助定位原因。

### 2. 为什么 CPU 高不一定是 P0，但下单成功率下降是 P0？

回答：

> CPU 高只是原因候选，不一定影响用户，也可能是正常流量或自动扩容前的短暂状态。下单成功率下降是用户结果异常，说明核心交易链路已经受影响。告警应该优先基于用户症状，再用资源指标定位根因。

### 3. 告警噪音减少 70% 是怎么做到的？

回答：

> 主要不是简单调高阈值，而是做了四件事。第一，把告警按 NOC、Critical、业务三级重新分级，只有用户感知的核心链路异常进入最高优先级。第二，把同根因告警聚合，避免一个依赖异常触发几十条消息。第三，删除无 owner、无人处理、无明确动作的历史告警。第四，对毛刺类指标增加持续时间、分桶和多窗口判断。这样减少噪音的同时保留真正需要响应的告警。

### 4. 业务监控如何避免被全局平均掩盖？

回答：

> 关键指标必须分桶。比如支付成功率要按市场、支付渠道、端、品类看；履约成功率要按供应商、品类、市场看；变价率要按品类、供应商、场景看。全站平均值可能只下降 0.2%，但某个市场某个渠道可能已经下降 20%。这种局部异常如果不分桶，很容易漏掉。

### 5. 供应商异常时怎么处理？

回答：

> 先看影响范围，是单供应商、单品类还是全链路。止血上可以对这个供应商做出口限流、熔断，必要时暂停相关商品销售或降级展示。对已经进入交易链路的订单，依赖 booking/fulfillment 状态机和补偿任务做查询、重试、释放资源或退款。恢复后通过供应商错误率、履约 pending、补偿 backlog 和对账差异确认是否收敛。

### 6. 支付回调异常怎么发现和恢复？

回答：

> 发现上看支付成功率、回调延迟、pending 支付单数量、支付成功但订单未推进数量。恢复上不能依赖回调一次成功，必须有支付查询补偿任务，定期查询 SPM 支付结果并推进本地支付单和订单状态。处理时还要保证幂等，避免重复回调导致重复状态推进或重复履约。

### 7. 履约失败会不会造成资损？

回答：

> 有风险，所以要做状态机、补偿和对账。支付成功后履约失败时，订单不能静默卡住。系统需要记录履约状态，支持重试和供应商查询；如果确认失败，需要触发退款或人工处理。监控上要看履约成功率、履约 pending、失败补偿数量、退款触发情况和供应商错误率。

### 8. 大促前你会重点检查什么？

回答：

> 先按用户旅程检查容量和风险：首页、搜索、详情、下单、支付、履约。技术上看 Redis 热 Key、MySQL 慢 SQL、Kafka lag、ES 查询、支付渠道容量、供应商限流。治理上验证功能开关、降级开关、限流阈值、NOC 看板、告警和 SOP。大促不是只看峰值 QPS，而是要验证高峰下保护机制和恢复机制是否有效。

### 9. 如何设计下单链路 SLI？

回答：

> 我会至少设计四类。第一是成功率，比如创单成功率、库存预占成功率。第二是延迟，比如创单 P99、库存预占 P99、计价 P99。第三是正确性，比如重复单拦截、非法状态迁移、价格差异。第四是收敛性，比如创单失败后资源释放成功率、补偿任务 backlog。这些指标共同描述下单链路是否可靠。

### 10. 监控、日志、链路追踪分别解决什么问题？

回答：

> 监控回答“有没有问题、影响多大”，日志回答“某一次请求发生了什么”，链路追踪回答“请求慢在哪一跳、依赖关系是什么”。事故处理中一般先看业务监控确认影响，再用链路监控定位阶段，然后用 trace 和日志定位具体请求、错误码和依赖。

表达重点：

> 我不是按服务堆监控，而是先按用户旅程定义业务结果，再把每个业务结果映射到底层服务、依赖、中间件和资源。这样事故发生时可以从“用户受影响了吗”一路下钻到“哪个依赖、哪个市场、哪个品类、哪个状态卡住了”。

## 十六、容易被追问的薄弱点

### 1. “你做的是监控还是稳定性？”

不要只回答监控。要强调闭环：

```text
指标定义 -> 告警分级 -> 看板定位 -> SOP 止血 -> 补偿对账 -> 复盘改进
```

### 2. “你怎么证明有效？”

可以从三个维度讲：

- 发现：业务指标能更早发现核心链路异常。
- 定位：五层看板能更快定位故障域。
- 恢复：预案、开关、补偿、对账能把异常订单收敛回来。

量化上可以讲：

- 告警噪音减少约 70%。
- 覆盖首页、详情、下单、支付、履约、退款核心链路。
- 沉淀 20+ 大促/故障预案。

### 3. “业务告警和接口告警怎么配合？”

回答：

> 业务告警负责拉响事故，接口告警负责定位原因。比如支付成功率下降是业务告警，接着看 payment init、callback、SPM query、支付渠道错误码、DB 状态更新这些接口和依赖指标。不能反过来让 CPU 或接口毛刺直接定义业务事故。

### 4. “如果供应商慢但还没失败，怎么处理？”

回答：

> 慢依赖比错误更危险，因为它会占住线程和连接。处理上要设置超时预算、出口并发限制和熔断阈值。导购场景可以降级或使用缓存，创单场景必须明确失败或进入可补偿状态，不能无限等待。

### 5. “为什么补偿任务也要监控？”

回答：

> 补偿任务是最终一致性的恢复链路。如果补偿任务长期堆积，说明系统表面可用但数据没有收敛。比如支付 pending、履约 pending、退款 pending 都会直接影响用户和资损风险，所以 backlog、失败率、最老任务年龄都必须监控。

## 十八、最终建议讲法

面试时建议按这个顺序讲：

1. **先讲背景**：DP 是多市场、多品类交易平台，核心问题是用户购买闭环是否可靠。
2. **再讲问题**：接口和机器监控不足，告警噪音高，外部依赖多，支付/履约/退款最终一致性风险高。
3. **讲方法**：按用户旅程定义业务 SLI，建设五层监控、三级告警、SOP、限流熔断降级、补偿对账。
4. **讲收益**：告警噪音减少 70%，问题发现和定位更快，大促有预案，资损风险通过监控、补偿和对账持续收敛。
5. **准备追问**：供应商异常、支付回调异常、履约失败、补偿 backlog、局部故障、告警治理。

不要把重点放在“用了 Grafana / Prometheus / 日志平台”。工具不是重点。重点是你如何定义业务可靠性、如何识别故障模型、如何设计保护和恢复闭环。
