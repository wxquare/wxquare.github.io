## 二、项目一：统一价格计算引擎（推荐主讲）

> 适合：偏架构 / DDD / 系统设计的岗位

### 5 分钟结构化介绍（STAR）

**【背景 30s】**
我们是一个数字电商平台，支持 10+ 品类。价格计算逻辑原来分散在 5 个以上服务中 — 前端、商品服务、营销服务、订单服务、支付服务都有各自的计价逻辑。这导致三个严重问题：新增一个变价因素需要改 5 个服务、花 2-3 周上线；价格术语不统一，团队沟通经常出现偏差；年均发生 3-5 次资损事故，单次损失超过 10 万。

**【目标 15s】**
我的任务是从零构建一个统一的价格计算中心，将所有计价逻辑收敛到一个服务，同时要保证安全迁移 10+ 个品类，0 资损。

**【方案 2min】**
我的核心方案有四个关键设计：

**第一，四层计价模型 + 场景驱动**。我把价格计算拆分为四层：基础价格 → 营销促销 → 费用附加 → 优惠券/积分。不同场景可以灵活跳层 — 比如 PDP 只走前两层（高缓存、P99 < 100ms），CreateOrder 跳过优惠券层（准确计算、不缓存），Payment 直接用快照验证（零计算）。

**第二，DDD 领域驱动设计**。我建立了统一的价格术语体系 — Price 聚合根封装四层计算逻辑，PriceBreakdown 值对象记录每一步计算明细，PriceComponent 表示价格组成元素。这样做的好处是：团队沟通有了统一语言，需求理解偏差从 20% 降到不到 5%。

**第三，双快照机制防资损**。订单创建时生成订单快照（30 分钟有效），收银台生成支付快照（15 分钟有效），支付时只做快照验证不重新计算。这样彻底杜绝了前端展示价格和实际扣款不一致的问题。

**第四，空跑比对 + 三阶段灰度**。迁移期间新老逻辑并行运行，自动比对差异并上报。灰度分三步：空跑 2 周 → 按品类灰度放量（1%→10%→50%→100%）→ 观察稳定后下线老逻辑。空跑期间发现并修复了 15 个隐藏问题。

**【成果 30s】**
- 日均调用量 5000 万+，P99 从 500ms 降到 200ms 以内
- 资损事故从年均 3-5 次降至 0 次
- 新增变价因素上线从 2-3 周缩短到 3-5 天
- 缓存命中率 92%，10+ 品类安全迁移

**【反思 15s】**
如果重新设计，我会更早引入性能基准测试，每个 Sprint 做一次；另外控制调用链层级在 3-4 层，当前 6-7 层有些深了。

### 高频追问

**Q：为什么选 DDD？**
A：价格计算业务逻辑复杂，传统贫血模型导致逻辑分散、术语混乱。DDD 的核心价值是：1）Price 聚合根让业务逻辑内聚，修改影响范围可控；2）统一语言解决了团队沟通问题；3）PriceBreakdown 让每笔计算可追溯，客诉时 5 分钟定位问题。

**Q：四层模型的设计依据？**
A：来自业务分析 — 价格变动因素本质上就四类：商家定的基础价、平台做的促销活动、各种附加费用、用户持有的优惠券。分层后职责单一，新增变价因素只需在对应层添加，不影响其他层。

**Q：如何保证灰度安全？**
A：三层保护：1）空跑比对（新老并行，差异自动上报）；2）灰度放量（按品类+地区+用户维度，基于 UserID 哈希保证同一用户稳定）；3）实时监控（差异率 > 0.01% 立即告警）。实际迁移 10+ 品类，0 资损。

**Q：性能如何优化？**
A：分场景差异化：PDP 用两级缓存（本地 5min + Redis 30min），命中率 92%；Checkout 场景不缓存，但并发调用 4 个依赖服务（商品/营销/优惠券/积分），延迟降低 60%；Payment 零计算，直接验证快照。

**Q：如何处理品类差异？**
A：策略模式。通用逻辑在基类（四层计算链），品类特殊逻辑注册 Calculator 接口。比如话费用面值计算器，酒店用价格日历计算器。新品类只需实现接口，从 2 个月缩短到 1 周。

**Q：精度如何处理？**
A：所有金额用分（int64）存储和计算，避免浮点误差。多币种按币种精度表对齐（泰铢 2 位小数、越南盾 0 位），银行家舍入。分摊场景用"余额递减法"——前 N-1 项按权重向下取整，最后一项承担尾差。

## 三、项目二：统一商品运营管理平台（偏业务架构）

> 适合：偏业务理解 / 团队管理 / 平台化的岗位

### 5 分钟结构化介绍（STAR）

**【背景 30s】**
我们平台有 10+ 品类的虚拟商品，每个品类原来有独立的上架系统和运营后台。核心痛点：代码复用率不到 10%，新品类接入需要 2 周从头开发；运营需要切换多个后台，批量操作（万级 SKU 调价、10 万券码导入）要么不支持要么 OOM 崩溃。

**【目标 15s】**
建设统一的商品上架和运营管理平台，让所有品类复用同一套流程，批量操作从不可用变为分钟级完成。

**【方案 2min】**
核心设计有四个方面：

**第一，统一状态机 + 策略模式**。所有品类共享一套状态流转：DRAFT → Pending Audit → Approved → Online。品类差异通过策略模式解耦 — ValidationRule 接口处理不同品类的校验规则（酒店校验价格日历、电影票校验场次时间），PublishStrategy 接口处理发布差异（酒店创建价格日历表、电子券关联券码池）。审核策略按数据来源动态路由：供应商走快速通道、运营免审核、商家需人工审核。

**第二，多源异步上架引擎**。6 类数据源（商家/运营/供应商 Push/Pull/Excel/API）统一接入后，通过 Kafka + 5 类 Worker 异步处理。商品发布使用 Saga 分布式事务保证多表写入一致性（创建 item → 创建 sku → 创建属性 → 更新状态 → 发送事件），失败自动逆序回滚。

**第三，统一批量操作框架**。这是架构层面的创新 — 设计了一套泛化的批量操作模型（operation_batch_task + operation_batch_item），通过模板方法模式抽象通用流程（流式解析 → 分批查询 → Worker Pool 并发 → 结果文件生成），新增一种批量操作从 550 行代码/2 周缩短到 50 行代码/2 天。底层三大技术支撑：流式 Excel 解析（内存恒定 < 200MB）、批量 SQL（TPS 500 → 50000+）、20 goroutine Worker Pool（CPU 利用率 12% → 85%）。

**第四，三层审计体系**。状态快照表记录每次流转、审计日志表记录审核详情、业务变更表记录价格/库存的 before/after 值，审计覆盖率从 33% 提升到 100%。

**【成果 30s】**
- 代码复用率从 < 10% → 90%+，新品类接入 2 周 → 2 天
- 万级 SKU 批量操作从数小时 → 分钟级（100 倍+）
- 上架成功率 85% → 99%+
- 新增批量操作开发效率提升 7 倍
- 运营人力成本降低 60%

### 高频追问

**Q：Saga 事务如何实现？失败如何回滚？**
A：编排模式。定义步骤链（CreateItem → CreateSKU → CreateAttributes → UpdateStatus → PublishEvent），每个步骤实现 Execute 和 Compensate 接口。执行失败时逆序调用已完成步骤的 Compensate 方法。补偿失败则记录告警，人工介入。Saga 状态持久化到 listing_saga_log_tab，支持断点恢复。

**Q：批量操作如何解决 OOM？**
A：三个技术点：1）流式解析 — 用 excelize.NewReader() 逐行读取，不全量加载，10 万行 Excel 内存恒定 < 200MB；2）分批处理 — 1000 条/批做批量 SQL，避免单条 INSERT 的网络往返开销；3）切片复用 — `codes = codes[:0]` 复用底层数组，减少 GC 压力。

**Q：Worker Pool 并发度怎么确定？**
A：经验值 + 压测调优。20 是因为：我们的操作是 IO 密集型（查 DB + 写 DB + 清缓存），单个任务约 35ms，20 并发时 CPU 利用率 85%，继续增加到 50 并发提升不明显反而增加数据库压力。批次大小 1000 也是类似逻辑 — 压测发现超过 2000 后 SQL 解析变慢。

**Q：如何保证并发安全？**
A：乐观锁。所有状态变更通过 version 字段控制：`UPDATE ... SET status=?, version=version+1 WHERE id=? AND version=?`。RowsAffected == 0 表示并发冲突，自动重试。配合 task_code 唯一索引保证幂等，重复创建直接返回已存在记录。

**Q：统一批量框架的扩展性怎么体现？**
A：新增批量操作只需三步：1）实现 BatchOperationWorker 接口的 Process 方法（约 50 行业务逻辑）；2）注册到 WorkerManager；3）API 层调用统一的 BatchService.Create()。底层的流式解析、分批处理、进度跟踪、结果文件生成、审计记录全部复用。

## 四、项目三：多品类统一库存系统（偏高并发/性能）

> 适合：偏高并发 / 性能优化 / 分布式系统的岗位

### 5 分钟结构化介绍（STAR）

**【背景 30s】**
我们平台的不同品类库存特性差异极大 — 电子券是券码制（每个券码唯一）、虚拟服务是数量制、酒店是时间维度按日期管理、话费充值是无限库存。原来每个品类独立设计库存逻辑，代码无法复用，Redis 和 MySQL 数据经常不一致，超卖和库存差异难以发现。

**【目标 15s】**
建设统一库存管理系统，支持万级 QPS 秒杀场景，新品类接入无需修改核心代码。

**【方案 2min】**
核心设计有三个关键点：

**第一，二维分类统一模型**。我把所有品类抽象为两个正交维度：谁管库存（自管理/供应商管理/无限库存）× 库存长什么样（券码制/数量制/时间维度/组合型）。任何新品类接入时，只需确定它属于哪个 (ManagementType, UnitType) 组合，在 inventory_config 表插入一行配置即可复用全部库存策略。比如"演唱会门票"= 供应商管理 + 时间维度，直接复用供应商策略和时间维度库存逻辑。

**第二，高并发库存引擎**。通过策略模式实现不同的库存操作：
- 券码制：Redis LIST 存储券码池，Lua 脚本原子出货（LRANGE + LTRIM），库存不足时从 MySQL 懒加载补货（分布式锁 + 游标分页），补货量 3000 张/次
- 数量制：Redis HASH 存储（available/booking/sold/locked），Lua 脚本原子扣减，支持营销活动独立库存字段，优先扣营销库存不足时扣普通库存
- 供应商品类：异步 Booking 轮询状态机（PENDING → CONFIRMED/FAILED），每 2 秒轮询一次，30 秒超时

**第三，数据一致性保障**。核心原则是"Redis 是热路径，MySQL 是权威数据源"。预订/售出走 Redis 同步扣减 + Kafka 异步落库 MySQL；库存恒等式（total = available + booking + locked + sold）+ 每小时自动对账；差异超过 100 或 10% 触发告警。券码池分 100 张表（按 item_id 取模），支撑亿级券码管理。Redis 宕机时降级到 MySQL 直接操作（SELECT FOR UPDATE），性能下降但业务不中断。

**【成果 30s】**
- 秒杀峰值 2 万 QPS，日均 200 万订单
- 数量制预订 P99 < 30ms，券码制 P99 < 50ms
- 超卖次数：0
- 对账差异发现率 4%（87 次 / 2160 次），均自动修复
- 新品类库存接入：配置一行即可，无需改代码

### 高频追问

**Q：Redis 和 MySQL 双写如何保证一致性？**
A：分场景处理。高频操作（预订/售出）：Redis 同步扣减（Lua 原子操作），通过 Kafka 异步更新 MySQL，接受最终一致。低频高准确操作（营销锁定）：Redis + MySQL 同步双写，走 DB 事务保证强一致。兜底机制：每小时对账，以 MySQL 为准修复 Redis。

**Q：券码补货如何防止并发问题？**
A：分布式锁（10s 超时）+ 游标分页。补货时先获取锁，读取游标（上次补到哪个 codeID），从 MySQL 查 3000 张可用券码（WHERE status=1 AND id > cursor），批量 RPUSH 到 Redis LIST。实际遇到过锁超时导致重复补货的问题，根因是 MySQL 慢查询（千万级表全表扫描），解决方案：加复合索引（item_id, status, id）+ 锁续期守护线程 + 低库存时提前异步补货。

**Q：秒杀场景如何处理热点 Key？**
A：三层保护：1）本地缓存拦截 80% 读请求；2）Key 分散 — 将热点 Key 复制 10 份（stock:item:0 ~ stock:item:9），读随机路由，写同步更新；3）网关限流 — 单商品最大 2.5 万 QPS。秒杀前 1 小时预热券码到 Redis，避免运行时 DB 补货。

**Q：供应商 Booking 超时怎么处理？**
A：状态机 + 降级。正常路径：2s 轮询，30s 超时。超时后：1）不立即关闭订单，降低轮询频率（1min/次，最长 24h）继续后台查询；2）通知用户"预订处理中"；3）企业微信告警，人工介入。供应商确认后设置 15 分钟支付超时，未支付则调供应商取消接口。

**Q：Kafka 消费积压怎么办？**
A：实际遇到过 — 秒杀后积压 50 万条消息。瓶颈是 Consumer 逐条更新 MySQL（TPS < 5000）。解决方案：1）攒批 100 条做批量 INSERT（TPS 提升到 8 万）；2）日志表改为异步从库写入，主库只更新核心库存表；3）Producer 端 linger.ms=100ms 攒批发送。

## 五、通用追问应对

### 技术选型类

**Q：为什么用 Go 不用 Java？**
A：团队技术栈统一用 Go。Go 的 goroutine 天然适合高并发场景（Worker Pool 20 goroutine 轻量级），编译部署快，内存占用小。我们的批量操作框架大量用 goroutine + channel，用 Java 需要线程池管理更复杂。

**Q：为什么用 Kafka 不用 RabbitMQ？**
A：吞吐量需求大（秒杀峰值 2 万 TPS），Kafka 分区模型天然支持水平扩展和消费并行。另外我们需要消息持久化和回溯能力（对账），Kafka 的日志存储模型比 RabbitMQ 更适合。

**Q：为什么用 Redis + MySQL 双写？**
A：纯 MySQL 扛不住秒杀 2 万 QPS；纯 Redis 不够可靠（数据可能丢失）。双写取长补短：Redis 保证低延迟高并发，MySQL 保证数据可靠，Kafka 做异步同步桥梁，定时对账兜底。

### 系统设计类

**Q：如何做容量评估？**
A：自下而上。日均 200 万订单 / 86400s ≈ 23 TPS；秒杀峰值 2 万 QPS = 日均 870 倍。Redis 3 主 3 从 32GB，实际核心数据 < 200MB，绰绰有余。MySQL 1 主 2 从 64GB，券码池分 100 张表（单表 500MB）。应用 10 台 4C8G Pod，单实例 2000 QPS。

**Q：如果流量增长 10 倍怎么办？**
A：分层扩展。Redis 加分片（当前 3 主 → 10 主）；MySQL 券码池从 100 张表扩到 1000 张；应用水平扩容（10 台 → 30 台）；Kafka 分区从 6 扩到 20；热点 Key 分散副本从 10 扩到 100。核心架构不需要改。

**Q：如果让你重新设计，会改什么？**
A：三点：1）更早引入性能基准测试，避免后期优化被动；2）计价引擎的调用链控制在 3-4 层（当前 6-7 层太深）；3）库存系统可以引入 CDC（Change Data Capture）替代定时对账，实现准实时一致性。

### 软技能类

**Q：你在项目中的角色？**
A：主架构师 + 核心开发。架构设计我主导（状态机、策略模式、四层计价模型、二维库存分类），核心代码我写了约 70%。同时负责 Code Review 和技术方案评审。团队 8 人（2 后端、2 前端、1 测试、1 DBA、1 运维、1 产品），我主要和 PM 对接需求、和前端对齐接口、指导初级开发写策略实现。

**Q：遇到过最棘手的问题？**
A：券码补货锁超时导致重复出货。根因是 MySQL 千万级表慢查询（12s），分布式锁 10s 过期，另一个进程拿到锁重复补货。解决：1）加复合索引（查询从 12s → 50ms）；2）守护线程每 5s 续期锁；3）检测低库存时提前异步补货。事后复盘增加了对账告警。

**Q：如何衡量项目成功？**
A：看三个维度：1）业务指标 — 资损降至 0、上架成功率 99%+、运营效率提升 60%；2）技术指标 — P99 < 200ms、缓存命中率 92%、超卖 0 次；3）效率指标 — 新品类接入 2 天、新批量操作 2 天、变价因素上线 3-5 天。
