# 交易系统稳定性治理面试话术

## 1. 项目背景

我负责过一个多市场、多品类、多供应商的数字商品交易系统稳定性治理。这个系统入口侧整体有 1000+ API，线上峰值约 5K+ QPS，背后对接 200+ 异构供应商，核心链路覆盖首页、商品查询、下单、支付、履约、退款、订单查询等场景。

这个项目的复杂点不只是 QPS，而是接口数量多、流量差异大、业务价值差异大，并且很多核心交易接口强依赖外部供应商。供应商的稳定性、响应时延、错误码规范、限流策略都不可控，所以单纯扩容或者加机器解决不了主要问题。

我当时的治理目标是把系统从“出了问题靠人肉排查”治理成一套可观测、可分级、可隔离、可降级、可预案化的稳定性体系。

## 2. 核心问题拆解

第一类问题是接口流量差异非常大。

高 QPS 接口通常集中在首页、搜索、列表、推荐、订单查询等读多写少场景。这类接口的特点是流量大、波动明显、用户感知强，但部分场景可以缓存或降级展示。

低 QPS 接口不一定不重要。比如下单、支付、履约、退款、补单、回调这类接口，流量可能没有查询接口高，但一旦失败会直接影响交易闭环，严重时还可能产生资损或订单状态不一致。

第二类问题是供应商依赖不稳定。

系统后面有 200+ 供应商，不同供应商在超时、限流、错误码、履约成功率、接口 SLA 上差异很大。某个供应商抖动时，如果内部没有做好超时、熔断、隔离和降级，就可能把局部供应商问题放大成入口接口超时、线程池耗尽、订单失败率升高，最终变成核心链路故障。

第三类问题是原有告警噪音大、定位慢。

早期监控更多是单点指标，比如某个 API 5xx 升高、某台机器 RT 升高、某个任务失败。但缺少从入口 API 到内部服务，再到供应商调用的链路视角。结果就是告警很多，但定位时还要人工查日志、问业务、查供应商，感知和定位都慢。

## 3. 总体治理思路

我的思路不是把 1000+ API 都套同一套告警规则，也不是先从 QPS 或错误率开始分级，而是先理解业务，按用户旅程盘点接口资产，再做分层、链路化监控和供应商依赖治理。

可以概括成四件事：

1. 业务链路盘点：先把接口归到 Homepage、Search / List、PDP、Cart / Checkout、Order、Payment、Fulfillment、Refund、Order Query、Callback / Compensation 等业务域。
2. 接口分级：在每个业务域内按用户影响、链路位置、资损风险、流量特征、供应商依赖和可降级性给接口分层。
3. 链路可观测：把入口 API、内部服务、市场、品类、供应商、错误码、订单影响串起来。
4. 故障隔离和降级：针对供应商抖动做超时、重试、熔断、限流、资源隔离、分流和预案。

## 4. 接口分级怎么做

我没有一开始就按 QPS 分级，因为交易系统里“高流量”和“高价值”不是一回事。第一步是先和业务、产品、运营一起把接口按用户旅程归类，明确每个接口服务哪个业务动作、依赖哪些内部服务和供应商、失败后对用户和订单有什么影响。

典型分类包括：

| 业务域 | 典型接口 | 稳定性关注点 |
| --- | --- | --- |
| Homepage | 首页入口、频道、资源位、活动配置 | 高 QPS、潮汐流量、缓存命中率、配置降级 |
| Search / List | 搜索、列表、筛选、排序、推荐 | 查询延迟、零结果率、缓存、ES / 推荐依赖 |
| PDP | 商品详情、价格库存查询、供应商实时校验 | 详情成功率、价格/库存新鲜度、供应商慢依赖 |
| Cart / Checkout | 加购、购物车、结算页初始化、价格试算 | 库存校验、计价一致性、非核心权益降级 |
| Order | 创单、订单确认、订单状态流转 | 创单成功率、价格/库存快照、幂等、半成功风险 |
| Payment | 支付发起、支付回调、支付查询 | 回调延迟、重复回调、支付成功但订单未更新 |
| Fulfillment | 履约请求、供应商下单、履约回调 | 供应商成功率、超时率、履约 pending、补偿队列 |
| Refund | 退款申请、退款执行、退款回查 | 重复退款、退款状态不一致、资金风险 |
| Order Query | 订单列表、订单详情、状态查询 | 高 QPS、读模型延迟、状态展示一致性 |
| Callback / Compensation | 供应商回调、支付回查、补偿任务、对账 | 幂等、乱序、DLQ、补偿 backlog、状态收敛 |

我会把接口分成三层。

NOC 层关注全局用户影响，比如整体成功率、整体 QPS、核心交易漏斗、下单成功率、支付成功率、履约成功率、退款成功率、整体 P95/P99 延迟。这一层主要回答：现在是不是有全局故障，用户面有没有明显受损。

Critical 层关注核心交易链路，比如下单、支付、履约、退款、订单状态流转、供应商回调、补偿任务。这一层的特点是 QPS 可能不高，但业务影响很重，所以要重点监控失败数、失败率、订单影响量、金额影响、状态不一致数量和供应商维度异常。

业务层关注局部业务问题，比如某个市场、某个品类、某个供应商、某个活动页、某个推荐模块异常。这一层主要是避免局部问题直接打到全局告警，同时也能让业务 owner 精准收到自己负责范围内的问题。

分级时我重点看几个维度：

- 用户感知：失败是否直接阻断用户完成购买。
- 交易闭环：是否影响下单、支付、履约、退款。
- 资损风险：是否可能导致重复扣款、重复履约、漏发、错发、订单状态不一致。
- 流量特征：高 QPS、低 QPS、周期性流量、活动峰值流量。
- 依赖复杂度：是否强依赖供应商、支付渠道、库存、价格、风控等外部或跨团队系统。
- 可降级性：异常时是可以展示兜底，还是必须阻断交易。

这个分级做完之后，告警优先级就不再是简单按错误率排序，而是按用户影响和交易风险排序。

## 5. NOC 和 Critical 应该包括哪些指标

NOC 和 Critical 的边界可以这样理解：NOC 看全局用户面有没有出事，Critical 看核心交易链路有没有断、有没有资损风险。

NOC 层不应该过细，否则会把每个普通接口、每个供应商的小抖动都打到一线值班，最后造成告警疲劳。NOC 更适合放少量但关键的大盘指标，目标是快速判断故障面、影响范围和故障等级。

NOC 指标可以包括：

| 指标类型 | 具体指标 | 作用 |
| --- | --- | --- |
| 流量 | 总 QPS、各市场 QPS、核心入口 QPS、QPS 突增/突降 | 判断是否流量异常、入口异常、大盘掉量 |
| 可用性 | 全局成功率、5xx 错误率、核心 API 成功率 | 判断用户请求是否大面积失败 |
| 延迟 | 全局 P95/P99、核心 API P95/P99 | 判断是否出现整体慢请求 |
| 交易漏斗 | 下单成功率、支付成功率、履约成功率、退款成功率 | 判断交易闭环是否异常 |
| 市场维度 | SG/ID/TH/TW/VN/PH/MY 等市场成功率、QPS、错误率 | 判断是否单市场故障 |
| 基础依赖 | DB、Redis、MQ、RPC、配置中心、网关可用性 | 判断是否公共依赖异常 |
| 订单大盘 | 订单创建量、支付单量、履约单量、订单状态堆积 | 判断业务结果是否偏离正常基线 |
| 告警聚合 | P0/P1 事故告警、跨市场异常、跨品类异常 | 方便值班快速判断故障等级 |

Critical 层关注的是下单、支付、履约、退款、订单状态一致性这些核心交易路径。它的特点是不一定流量最大，但业务后果最重，所以不能只看技术错误，还必须看订单状态、金额、补偿和供应商维度。

Critical 指标可以包括：

| 链路 | 关键指标 |
| --- | --- |
| 下单 | 下单 QPS、下单成功率、创建订单失败数、预占库存失败率、价格校验失败率、供应商 create-order 超时率 |
| 支付 | 支付发起成功率、支付回调成功率、支付成功但订单未更新数量、支付失败率、支付渠道异常率 |
| 履约 | 履约请求成功率、供应商履约成功率、履约超时率、履约失败订单数、履约补偿队列堆积 |
| 退款 | 退款申请成功率、退款执行成功率、退款失败数、退款状态不一致数、重复退款风险 |
| 订单状态 | 卡状态订单数、状态机非法流转数、支付/订单/供应商状态不一致数量 |
| 供应商 | 核心供应商成功率、P95/P99、超时率、限流率、业务错误码、熔断状态 |
| 资损 | 重复扣款、重复履约、支付成功未履约、履约成功未记账、退款重复执行、金额不一致 |
| 补偿 | 补偿任务积压量、补偿失败率、补偿延迟、DLQ 堆积 |
| 消息链路 | 订单事件 MQ 堆积、消费失败率、消费延迟、重复消费异常 |
| 幂等 | 幂等冲突数、重复请求数、重复供应商订单数 |

面试里可以这样总结：

```text
NOC 层我主要看全局用户面和交易大盘，比如整体 QPS、成功率、P95/P99、5xx、
核心交易漏斗、市场级成功率、公共依赖健康度。它回答的是“系统有没有大面积故障”。

Critical 层我主要看核心交易链路，比如下单、支付、履约、退款、订单状态一致性、
供应商成功率、补偿队列和资损风险。它回答的是“交易闭环有没有断、有没有订单卡住、
有没有资损风险”。
```

## 6. 监控指标怎么设计

对于高流量接口，我主要看：

- QPS 突增和突降。
- 5xx 错误率。
- 业务失败率。
- P95/P99 延迟。
- 超时率。
- 错误码分布。
- 同比/环比异常波动。

高流量接口适合用错误率和延迟分位数，因为样本足够大，指标比较稳定。

对于低流量但核心的接口，不能只看错误率。比如一分钟 1 个请求失败，错误率就是 100%，这类告警很容易误报。但如果是支付回调、履约确认、退款执行这种接口，即使失败数不大，也可能影响交易闭环。

所以低流量核心接口会结合这些指标：

- 失败绝对数量。
- 连续失败次数。
- 影响订单数。
- 影响金额。
- 订单卡状态数量。
- 补偿任务堆积量。
- 供应商错误码类型。
- 是否命中核心市场、核心品类或大促活动。

举个例子，普通查询接口短时间 3 个失败可能只是业务告警；但履约接口连续失败 3 次，而且都来自同一个供应商，同时有订单进入待补偿状态，就应该升级成 Critical 告警。

## 7. 动态阈值怎么做

固定阈值在这种系统里很容易失效，因为不同接口的流量规模差异太大，同一个接口在白天、凌晨、大促期间的基线也不一样。

我的做法是把阈值分成三类。

第一类是固定硬阈值，用在明确不能突破的指标上，比如核心接口 5xx、支付失败、履约失败、订单状态不一致、资损相关指标。这类指标宁可敏感一点，因为一旦出问题影响比较重。

第二类是动态基线，用在流量和延迟类指标上。比如根据过去 7 天同时间段的 QPS、错误率、P95/P99 做基线，当前值偏离基线一定比例才告警。这样可以减少周期性流量带来的误报。

第三类是组合阈值，用在低流量核心接口上。比如错误率 + 失败绝对数 + 连续时间 + 订单影响量同时满足，才触发高优告警。这样既避免单个失败造成误报，也避免关键链路的小流量故障被漏掉。

核心原则是：高流量接口看比例和趋势，低流量核心接口看绝对失败和业务影响，交易闭环接口额外看订单状态和资损风险。

## 8. 链路可观测怎么落地

我重点补了几个维度的可观测性。

第一是入口 API 维度。

每个 API 要能看到 QPS、成功率、错误率、P95/P99、5xx、业务错误码、市场、平台、版本等基础指标。

第二是供应商维度。

每个供应商要能看到调用量、成功率、超时率、P95/P99、业务失败率、错误码分布、履约成功率、限流次数等指标。

第三是业务链路维度。

把用户请求从入口 API 到内部服务，再到供应商接口串起来。比如下单链路会经过商品校验、价格计算、库存/供应商预占、订单创建、支付前置校验等步骤。监控里要能看出失败发生在哪一步。

第四是订单影响维度。

交易系统不能只看技术错误，还要看订单状态。比如订单创建成功但履约失败、支付成功但订单未更新、供应商成功但内部状态未落库，这些都不是普通 5xx 能完整表达的。

最终我希望告警不只是说：

```text
/order/create 错误率升高
```

而是能定位到：

```text
ID 市场某品类下单链路异常，supplier A 的 create-order 接口 P99 延迟升高且超时率上升，
导致入口 /order/create 超时增加，当前影响订单 N 单，暂未产生支付后资损。
```

这个定位粒度对值班和大促保障非常关键。

## 9. 供应商不稳定怎么解决

供应商不稳定不能只靠重试，重试只是其中一个手段。如果供应商已经慢了，盲目重试会进一步放大流量，把供应商和自己都打崩。

我会从六个方向治理。

### 9.1 超时预算

先确定入口接口的整体 SLA，再倒推每一层的 timeout。

比如入口 API 需要 2 秒内返回，那么内部业务逻辑、RPC、供应商 HTTP 调用都必须有明确的超时预算。供应商不能无限等待，否则连接池、goroutine、线程池、请求队列都会被拖住。

对强依赖供应商的接口，我会要求供应商调用有独立 timeout，并且失败要快速返回到业务层，让业务层决定是降级、切换供应商，还是阻断交易。

### 9.2 重试治理和幂等

重试只适合网络抖动、临时超时、连接重置这类可能瞬时恢复的问题。

对于明确的业务失败，比如库存不足、商品不可售、参数错误、余额不足，就不能重试。

交易链路里重试必须配合幂等，尤其是下单、支付、履约、退款这些动作。否则可能出现重复下单、重复扣款、重复发货、重复退款。

我会把错误类型拆成三类：

- 可重试错误：网络抖动、短暂超时、临时 5xx。
- 不可重试错误：参数错误、库存不足、商品不可售、业务校验失败。
- 需要人工或补偿错误：状态未知、供应商返回不确定、支付成功但内部状态失败。

### 9.3 熔断

熔断要按供应商维度、接口维度、市场维度细分，不能只做全局熔断。

比如 supplier A 的某个履约接口异常，不应该影响 supplier B，也不应该影响 supplier A 的其他正常接口。

熔断触发条件一般会结合：

- 超时率。
- 5xx 错误率。
- 业务失败率。
- 连续失败次数。
- P99 延迟。
- 订单失败影响量。

熔断后可以进入半开状态，少量探测供应商是否恢复，恢复后再逐步放量，而不是马上全量切回来。

### 9.4 资源隔离

供应商故障最怕拖垮公共资源。

所以需要在连接池、并发数、队列、限流 key 上做隔离。比如每个供应商有自己的最大并发、最大连接数、请求队列长度，避免一个供应商慢请求占满所有资源。

如果条件允许，还可以按核心链路和非核心链路隔离资源。下单、支付、履约的资源优先级要高于普通查询和推荐类请求。

### 9.5 降级和分流

不同类型接口降级方式不一样。

查询类接口可以返回缓存、展示兜底文案、隐藏异常商品、关闭非核心模块。

推荐和活动类接口可以降级为默认配置或静态配置。

下单和支付前置链路必须更谨慎。如果供应商不可用，宁可阻断交易，也不能让用户支付后无法履约。

履约链路如果供应商短暂异常，可以进入补偿队列；如果供应商长时间异常，需要切换备用供应商或暂停对应商品售卖。

对于有多个供应商可选的品类，可以根据供应商健康度做流量降权、切流或灰度恢复。

### 9.6 供应商健康度体系

我会为供应商建立实时健康度评分，指标包括：

- 调用成功率。
- 超时率。
- P95/P99 延迟。
- 业务失败率。
- 履约成功率。
- 订单状态异常数。
- 限流次数。
- 最近一段时间是否发生连续故障。

健康度可以直接驱动分流策略。比如供应商健康度下降时，自动降权；达到熔断阈值时停止新单；恢复后先小流量探测，再逐步恢复。

## 10. 资损风险怎么兜住

交易系统稳定性不能只讲可用率，还要讲资损控制。

我会重点关注这些风险：

- 用户支付成功，但订单没有成功创建或状态没有更新。
- 订单创建成功，但供应商履约失败。
- 供应商履约成功，但内部状态未更新。
- 回调重复导致重复发货或重复退款。
- 重试导致重复创建供应商订单。
- 供应商返回状态未知，内部贸然推进订单状态。

治理手段包括：

- 下单、支付、履约、退款链路都要有幂等 key。
- 状态机流转必须有明确的合法状态迁移。
- 对状态未知的订单进入待确认或待补偿状态，而不是直接成功或失败。
- 对账任务定期校验内部订单、支付单、供应商订单状态。
- 补偿任务处理支付成功但履约失败、供应商成功但内部失败等不一致场景。
- 大促期间对高风险供应商设置更保守的下单策略。

面试里可以强调一句：交易系统里有些场景不能为了可用率强行成功，遇到供应商不可控异常时，阻断交易比产生资损更重要。

## 11. 告警降噪怎么做

告警降噪不是简单关掉告警，而是让告警更接近真实用户影响。

我主要做了几类优化。

第一是分层告警。

全局问题打到 NOC，核心交易链路问题打到 Critical，局部供应商或品类问题打到业务 owner。不同层级的响应人、SLA、升级机制不一样。

第二是告警聚合。

如果同一个供应商异常导致多个入口 API 报错，不应该产生几十条独立告警，而应该聚合成一个供应商维度的根因告警，并展示影响哪些 API 和市场。

第三是告警抑制。

如果上游供应商已经触发 Critical 告警，下游由它引发的业务失败可以降级为关联告警，避免重复轰炸。

第四是动态阈值。

对周期性波动明显的接口采用历史基线，对低流量核心接口采用组合阈值。

第五是告警内容结构化。

告警里直接带上市场、接口、供应商、错误码、影响订单数、最近变更、建议预案链接，让值班同学拿到告警后能直接行动。

最终结果是告警噪音下降约 70%，关键问题能在分钟级完成感知和初步定位。

## 12. 大促保障怎么做

大促预案不是单纯写几个降级开关，而是一套围绕流量洪峰、供应商不稳定、交易资损和故障快速恢复的作战体系。

我会把大促保障拆成战前、战中、战后三个阶段。

战前做容量评估、核心链路压测、供应商健康检查、预案演练和开关确认。

战中围绕 NOC 大盘、Critical 链路、供应商健康度、订单状态和资损指标实时监控，一旦触发阈值，按预案执行限流、熔断、降级、切流或暂停售卖。

战后基于故障、告警和订单数据复盘，反向优化监控阈值、供应商策略和预案 SOP。

### 12.1 容量预案

容量预案的目标是确认系统能不能扛住大促流量，以及瓶颈会先出现在哪一层。

重点评估：

- `dp-api` 入口 QPS 峰值。
- 核心接口 QPS，比如 `/digital-product/api/order/create`。
- DB 连接数、慢查询、CPU、主从延迟。
- Redis QPS、命中率、热点 key。
- MQ 堆积、消费延迟、消费失败率。
- RPC / HTTP 调用耗时和错误率。
- 供应商接口容量、限流策略和大促期间 SLA。

面试可以这样讲：

```text
大促前我会基于历史峰值和活动预估做容量评估，不只看入口 QPS，
也会看下单、支付、履约这些核心链路的瓶颈，包括 DB、缓存、MQ、RPC
和供应商接口容量。如果某个环节水位过高，会提前扩容、限流或者准备降级方案。
```

### 12.2 核心链路预案

大促期间最重要的是交易闭环不能断。核心链路一般包括首页、商品列表、商品详情、下单、支付、履约、订单查询、退款和补偿任务。

每条链路都要有明确的红线指标和对应动作。

以下单链路为例，重点观察：

- 下单成功率是否低于阈值。
- `/digital-product/api/order/create` P99 是否明显升高。
- 供应商 create-order 超时率是否升高。
- 订单创建失败数是否持续上升。
- 支付前订单状态是否异常。

对应动作包括：

- 限制异常品类下单。
- 熔断异常供应商。
- 切换备用供应商。
- 关闭非核心校验或非核心推荐逻辑。
- 必要时暂停售卖，避免支付后无法履约。

### 12.3 供应商预案

在 200+ 供应商的系统里，供应商预案是大促保障里最重要的一块。因为大促期间外部供应商的容量、限流、响应耗时和错误码都不完全可控，局部供应商抖动很容易放大成入口接口超时和交易失败。

供应商预案要按供应商维度准备：

- 供应商健康度监控。
- 超时率阈值。
- 成功率阈值。
- P95 / P99 阈值。
- 限流错误码识别。
- 是否有备用供应商。
- 是否支持切流。
- 是否允许暂停售卖。
- 故障联系人和升级路径。

典型场景可以这样描述：

```text
如果某个供应商接口 P99 从 500ms 升到 5s，入口下单接口开始超时，
继续放量会拖垮连接池和请求队列。这个时候不能盲目重试，而是先降低 timeout，
触发供应商维度熔断，停止新单流量；如果有备用供应商，就按健康度切流；
如果没有备用供应商，就下架或隐藏对应商品，避免用户支付后无法履约。
```

### 12.4 降级、熔断、切流预案

大促预案一定要把触发条件和执行动作写清楚，否则战中很容易变成临场讨论。

| 场景 | 触发条件 | 操作 |
| --- | --- | --- |
| 查询接口变慢 | P99 持续升高 | 返回缓存、隐藏非核心模块 |
| 推荐接口异常 | 推荐服务超时或错误率升高 | 使用默认配置或静态兜底 |
| 供应商超时 | 超时率持续升高 | 熔断供应商、停止新单 |
| 下单失败升高 | 下单成功率下降 | 降级非核心校验、切供应商、限制异常品类 |
| MQ 堆积 | 消费延迟持续升高 | 扩容 consumer、暂停非核心消息 |
| DB 水位高 | 慢查询/连接数升高 | 限流、关闭非核心写入、降级查询 |
| 支付后履约失败 | 支付成功但履约失败升高 | 进入补偿队列、暂停对应商品售卖 |

这里的核心原则是：查询类接口优先保可用，可以缓存和兜底；交易类接口优先保一致性，必要时阻断交易，避免资损。

### 12.5 资损防控预案

交易系统大促最怕的不是单纯失败，而是状态不一致和资损。

需要重点关注：

- 支付成功但订单未更新。
- 支付成功但履约失败。
- 供应商履约成功但内部状态失败。
- 重复下单。
- 重复扣款。
- 重复履约。
- 重复退款。
- 回调重复或乱序。
- 补偿任务失败。

对应预案包括：

- 下单、支付、履约、退款全链路幂等。
- 状态机控制合法流转。
- 状态未知订单进入待确认状态。
- 支付单、订单单、供应商单定期对账。
- 补偿任务兜底处理状态不一致。
- 高风险供应商大促期间降低权重或暂停新单。

面试里可以强调：

```text
大促期间我不会为了追求表面可用率而强行放行交易。
对于供应商状态未知、履约不可控、支付后可能无法交付的场景，
宁可阻断交易，也不能产生资损。
```

我们最终沉淀了 20+ 套核心预案，包括降级、熔断、分流、供应商切换、支付异常处理、履约补偿、资损防控、订单状态修复等。

大促期间核心链路可用率做到 99.99%，没有 P0/P1 重大故障，资损 0 发生。

## 13. 两分钟面试话术

我负责过一个数字商品交易系统的稳定性治理。这个系统整体有 1000+ API，峰值约 5K+ QPS，后面接了 200+ 异构供应商。它的难点不是单纯 QPS，而是接口流量差异非常大，而且核心交易链路强依赖供应商，供应商的稳定性、超时、限流和错误码都不可控。

我当时第一步不是直接按 QPS 分级，而是先梳理业务链路，把 1000+ API 归到首页、搜索/列表、PDP、结算、创单、支付、履约、退款、订单查询、回调和补偿等业务域。之后再按用户影响、交易闭环、资损风险、流量特征和供应商依赖分级。比如首页、列表、搜索是高流量接口，主要看错误率、P95/P99、QPS 波动；创单、支付、履约、退款这些接口 QPS 不一定最高，但直接影响交易闭环，所以放到 Critical 链路重点监控。

第二步是重构监控和告警。高流量接口用错误率、延迟分位数和动态基线；低流量核心接口不能只看错误率，而是结合失败绝对数、连续失败、订单影响量、金额影响和供应商错误码。我们把告警从单点 API 告警升级成链路告警，能看到是哪个市场、哪个品类、哪个供应商、哪个错误码导致入口 API 异常。

第三步是做供应商故障隔离。核心手段包括超时预算、有限重试、幂等控制、供应商维度熔断、连接池和并发隔离、按健康度分流，以及查询类接口降级展示、交易类接口必要时阻断交易。因为交易系统不能为了可用率强行成功，遇到供应商不可控异常时，避免资损比强行放行更重要。

最后我们沉淀了 20+ 套大促预案，包括降级、熔断、分流、供应商切换、履约补偿和资损防控。结果是告警噪音下降约 70%，核心问题能分钟级感知和定位；双 11 / 大促期间核心链路可用率达到 99.99%，没有 P0/P1 重大故障，也没有资损。

## 14. 面试官追问回答

### 14.1 为什么不能只按 QPS 分接口优先级？

因为交易系统里 QPS 和业务价值不完全一致。首页、搜索、列表可能 QPS 很高，但部分场景可以缓存或降级；下单、支付、履约、退款接口 QPS 可能没有那么高，但失败会直接影响交易闭环，甚至产生资损。所以我会结合用户影响、交易链路位置、资损风险、是否可降级、供应商依赖来分级。

### 14.2 低流量接口怎么避免误报？

低流量接口不能只看错误率，因为样本太小，一个失败就可能是 100%。我会用组合条件，比如失败绝对数、连续失败时间、影响订单数、金额影响、订单状态异常、供应商错误码一起判断。对于普通低流量接口降低告警优先级，对于低流量但核心交易接口保留 Critical 监控。

### 14.3 供应商接口超时你怎么处理？

先做超时预算，避免供应商拖住入口请求；然后按错误类型决定是否重试，可重试错误才做有限重试，并且必须有幂等；如果超时率持续升高，就按供应商和接口维度熔断；同时用连接池、并发数、队列做资源隔离，避免单个供应商拖垮整体服务。有备用供应商的场景做分流切换，没有备用的交易场景宁可暂停售卖或阻断交易，避免支付后无法履约。

### 14.4 怎么判断是供应商问题还是内部问题？

链路监控里会同时看入口 API、内部服务、供应商调用和订单状态。如果入口 API 错误升高，同时某个供应商接口 P99、超时率、错误码异常升高，而内部 DB、缓存、服务自身指标正常，就可以基本定位为供应商侧。如果多个供应商同时异常，则要反查内部公共依赖，比如网络、DNS、连接池、数据库、缓存或共享中间件。

### 14.5 熔断后怎么恢复？

熔断后不会马上全量恢复，而是进入半开状态，用小流量探测供应商是否恢复。探测成功后逐步增加流量，观察成功率、P99、超时率和业务失败率。如果恢复过程中指标再次恶化，就重新熔断。这样可以避免供应商刚恢复就被全量流量再次打垮。

### 14.6 如何证明告警降噪不是漏报？

降噪不是关告警，而是聚合、分层、抑制和动态阈值。验证方式是看降噪后是否仍能覆盖 P0/P1/P2 事故，是否能在分钟级发现核心链路异常，是否减少无用户影响的重复告警。同时会复盘历史故障，看新的告警规则能不能命中历史问题。

### 14.7 为什么说这个项目有深度？

因为它解决的不是单点性能问题，而是复杂交易系统的不确定性问题。系统里有 1000+ API、200+ 供应商、不同市场和品类、差异化流量、强外部依赖和资损风险。治理重点是把复杂链路拆成可观测、可分级、可隔离、可降级、可预案化的体系，降低故障扩散半径，提高定位和恢复效率。

## 16. NOC 术语与替代表达

NOC 全称是 Network Operations Center，中文一般叫网络运营中心或运维监控中心。

在这个项目语境里，NOC 不一定指一个具体组织，而是指一线全局监控和事故响应视角，主要关注系统有没有大面积故障、用户面是否受影响、是否需要升级成事故响应。

很多公司把 NOC 告警放在最高级别，是因为它不是普通技术指标告警，而是全局用户面和事故响应入口。它通常关注大面积可用性、核心交易漏斗、跨市场异常和公共依赖故障。一旦触发，往往需要立即升级、跨团队协同和事故指挥，所以必须高优先级、低噪音、高可信。

如果不想使用 NOC 这个词，可以替换成更通用的表达：

| 表达 | 适合场景 |
| --- | --- |
| P0/P1 事故级告警 | 最通用，适合简历和面试 |
| 全局事故级告警 | 强调影响范围大 |
| 用户面故障告警 | 强调用户体验受影响 |
| 核心链路事故告警 | 强调交易链路 |
| 全局可用性告警 | 强调整体系统健康 |
| 事故响应级告警 | 强调会启动 incident response |

简历里更推荐写：

```text
主导重构 P0/P1 事故级、Critical 核心链路级、业务场景级三级告警体系。
```

面试里可以这样解释：

```text
我这里不强调 NOC 这个组织名，而是按告警影响面分层。
最高层是 P0/P1 事故级告警，关注全局用户面和生产事故风险；
第二层是 Critical 核心链路告警，关注下单、支付、履约、退款等交易闭环；
第三层是业务场景告警，关注具体市场、品类、供应商或活动模块。
```
