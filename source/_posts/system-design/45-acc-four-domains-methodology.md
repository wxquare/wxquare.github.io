## 引言

本文面向**单一产品或单条主业务线**语境下的技术负责人与核心架构参与者：我们关心的是在一轮迭代里如何把方案说清楚、把跨团队依赖对齐，并沉淀为可评审、可落地的工件。这与覆盖投资组合、标准体系与全公司治理节奏的**企业级 EA（Enterprise Architecture）**不是同一尺度——若组织已有强 EA，可把下文当作**交付团队侧**的对齐与收口工具，与上层治理边界分明。

为便于分工与复盘，后文将架构相关工作拆成四个**工程域**（指讨论与产物的划分，而非公司部门）：

- **业务域**：关注**能力与场景**——组织对外提供哪些业务能力、在何种情境下被触发、怎样定义成功与验收口径。
- **应用域**：关注**边界与契约**——服务或模块的职责切分、对外 API / 事件等契约形态，以及演进与兼容策略。
- **数据域**：关注**所有权与一致性**——主数据归属、读写路径、一致性与补偿的底线，以及审计与可追溯要求。
- **技术域**：关注**约束与横切能力**——容量与可用、安全与合规、可观测性、成本与平台化能力等跨切面设计与门禁。

本仓库「架构与整洁代码」系列前四篇侧重**建模、分层、CQRS 与评审**等具体能力：[架构与整洁代码（一）](/2026/04/01/system-design/41-acc-clean-arch-ddd-cqrs/) 从 Clean Architecture、DDD 与 CQRS 讲清分层与读写路径；[架构与整洁代码（二）](/2026/04/02/system-design/42-acc-clean-code/) 讨论复杂业务中的 Clean Code 战术；[架构与整洁代码（三）](/2026/04/03/system-design/43-acc-ddd-notes/) 补足战略 / 战术 DDD 与通用语言；[架构与整洁代码（四）](/2026/04/04/system-design/44-acc-code-review/) 给出分阶段 Code Review 清单。**本文（45）**则聚焦**方案成形与跨域对齐**：把上述知识收束成一条可重复走的流程，并配套下半篇的分域清单，便于在立项、详设与评审关口统一语言。

文中的叙述与例子默认取材于笔者约 **5 年**的电商相关工程实践，场景锚定为典型的 **B2C / 平台型**交易与履约链路。**跨境合规、全球库存与多租户强隔离**等假设显著更重，**超出本文默认范围**；若读者处于此类环境，请在数据域与技术域上自行加厚约束与清单条目。

读完引言后，可将全文视为一次「架构工作坊」的操作手册：**上半篇**将给出贯穿四域的**六步流程**（每步交代输入、活动与关键产出）；**下半篇**提供按四域组织的**审查清单**，供评审会与上线前逐项核对。

## 流程总览

下表按时间顺序列出六步：每步要回答的主要问题、建议沉淀的工件，以及讨论重心落在哪一域（后文分节展开）。

| 步骤 | 主要问题 | 关键产出（工件） | 主要落在哪一域 |
|------|----------|------------------|----------------|
| 1. 问题与成功标准 | 做什么、不做什么、如何验收 | 范围与目标说明、验收口径、NFR 清单 | 业务（目标）+ 技术（NFR）为主 |
| 2. 业务架构（轻量） | 谁、在什么场景、达成什么结果 | 业务能力与场景、角色与权限、业务结果定义 | 业务 |
| 3. 应用架构 | 系统如何切分、如何协作 | 限界上下文/服务边界、协作关系、API 与事件契约 | 应用（+ 契约影响数据） |
| 4. 数据架构 | 谁拥有真相、如何一致、如何演进 | 数据所有权与模型、一致性策略、演进与迁移方案 | 数据 |
| 5. 技术架构 | 如何部署、观测、安全、降级与成本 | 部署与容量方案、观测与安全设计、降级与成本估算 | 技术 |
| 6. 验证与固化 | 假设、风险、决策与评审关口 | 假设与风险登记、ADR、评审结论与行动项 | 四域交叉 + ADR |

## 步骤一：问题与成功标准

### 输入

业务痛点与**不做清单**、上线窗口与预算、法务红线；若有，事故 / 客诉 / 对账摘要。

### 活动

写清**范围**与**不在范围**；把目标落成**可验收**表述；拉 NFR 初稿并与优先级对齐。

### 输出

《范围与目标说明》；《验收口径》（可测 / 可观测）；《NFR 清单》初版（尽量带 SLO、阈值或抽检门禁）。其中验收条目建议统一写成「**前置条件 → 期望结果 → 如何度量 / 谁验收**」，与下节电商要点同一粒度，避免只堆名词而无判定标准。

### 电商示例

用**验收条**对齐产品、研发、风控与财务：

- **峰值下单**：约定窗口内目标 **QPS / 并发** 下错误率、超时率上限；若启用排队 / 限购等降级，须写明**用户可见行为**与可接受业务损失。
- **支付成功率**：指定渠道下成功率（或「可重试后成功」）≥ 约定值；区分用户取消与系统 / 渠道失败，后者纳入改进目标。
- **库存准确性**：选**超卖零容忍**或可接受短时不一致（如预售、在途）：零容忍则验收含占库与可售**对账**或等价证明；否则写明**最大偏差、恢复时限、补偿上限**（砍单 / 赔付）。
- **价格展示与结算一致性**：商详 / 购物车与下单页、支付页、**订单快照**在允许规则内一致；运费、券后实付等与预估不符时，须有**确认节点**或重算 / 免责规则并写入场景。
- **合规边界**：收单是否在**持牌与合同**范围内；个人信息须**最小化**、用途告知、留存与导出 / 删除时限——验收落到字段清单、处理可追溯、第三方共享边界书面化，而非「要合规」四字。

### 常见反例

只有愿景无**可测指标**；可用性不写峰值与降级口径；验收不提库存与价格，纠纷全由研发兜底。

## 步骤二：业务架构（轻量）

### 输入

步骤一产出；可选用户旅程或高频工单类型。

### 活动

用**业务能力**（动词 + 业务结果）归纳；为关键路径写**场景**（触发者、前置、主路径、异常与业务期望）；粗粒度**角色与权限**矩阵。

### 输出

能力清单（轻量）与依赖；优先场景「故事」列表；业务名词与状态初版（不求技术实现）。

### 电商示例

- **能力**：**下单**、**算价**、**占库**、**收款**、**发货**、**售后**——列表或简表标先后与依赖，全程业务用语。
- **场景**：**大促** + **券** + **会员价** + **平台补贴** 叠加时，写清顺序、出资方、页面承诺方；会员价与某类券是否可叠、补贴是否影响商家结算等，产出**业务规则**与**争议裁决原则**，非技术方案。
- **本步不写**：长篇**技术选型**与部署拓扑（留给步骤三及以后）。

### 常见反例

把本节写成微服务 / 中间件清单；只写主成功路径；能力与模块过早一一绑定，锁死后续限界讨论。

## 步骤三：应用架构

### 输入

步骤二的能力与场景；步骤一的 NFR 与合规边界；现有系统拓扑与对外依赖清单（若有）。

### 活动

划定**限界上下文**或服务边界；画协作图（谁调用谁、谁发布谁订阅）；为每条跨边界交互选定**契约形态**并约定兼容与弃用策略；与产品 / 业务方过一遍「边界是否听得懂、争议落在谁家」。

### 输出

《上下文 / 服务边界说明》；《协作关系图》（可手绘或 C4 容器级）；**契约包**：同步接口说明（路径、错误码、超时与重试约定）与**事件目录**（主题、载荷字段、投递语义、消费者责任）。

### 电商示例

用**上下文**串起链路：**订单**负责下单与状态推进；**库存**负责占库、释放与可售视图；**商品与价格**提供 SKU、类目与算价输入；**营销**输出券 / 活动资格与补贴口径；**支付**对接渠道与支付单；**物流**承接出库与运单回写。协作上可并存多种形态：例如对库存服务发起 **REST** `POST /reservations`（或等价的 **gRPC** `ReserveStock`）完成同步占库；支付成功后由订单上下文发布领域事件 **`OrderPaid` v1**（JSON Schema 或 Protobuf 定义版本号与必填字段），库存上下文订阅后生成**出库指令**下发仓储——**契约层**须写明：对外 HTTP 是否统一 **REST**、内部高性能链路是否 **gRPC**、事件是否带 **schema 版本**与向后兼容规则（新增可选字段、禁止随意改语义）。

### 常见反例

只有框图无契约；事件当「广播日志」无 schema 与幂等消费约定；把应用边界画成组织架构图而非业务能力边界。

## 步骤四：数据架构

### 输入

应用边界与契约草案；主数据与报表诉求；合规与审计要求（留存、脱敏、可解释）。

### 活动

标定**数据所有者**与**系统记录（System of Record）**；为每条关键实体选一致性模型（强一致、最终一致、补偿）；设计演进路径（双写、回填、切换窗口）；标出报表与风控只读副本的**延迟与权威来源**一句承诺。

### 输出

《数据所有权与 SoR 表》；核心实体关系与状态说明；《一致性策略说明》（含失败与乱序处理原则）；《迁移 / 演进计划》摘要。

### 电商示例

- **订单状态机与支付回调乱序**：产出**状态转移表**（合法边、触发方、幂等键）+ **支付回调处理说明**（乱序、重复、迟到时如何落库与重放）；工件可读作一页表 + 若干序列要点，而非长篇状态机教程。
- **库存流水与可售库存**：**流水表**（变动类型、关联单号、数量、时间）与**可售余额视图**（或汇总规则）对照说明；验收侧写清「流水可重建可售」或等价对账口径。
- **价格快照与订单行**：订单行上固化**价格快照**（原价、活动价、券分摊、币种）字段清单与来源追溯字段；与算价服务的「当时有效规则」引用方式一句话约定。
- **Outbox、幂等、对账**：**Outbox 表**示意（事件 ID、聚合 ID、载荷版本、发布状态）；**幂等键登记**或请求指纹表结构一行级描述；**对账批次工件**（日期、渠道、差异类型、处理 SLA）——各控制在短段，点明「表里有什么列、跑批产出什么报告」，避免展开实现代码级教程。

### 常见反例

全库一张 ER 图无所有权；把「最终一致」当万能答案不写补偿与对账；订单行只存商品 ID 不存成交价，纠纷时无法复盘。

## 步骤五：技术架构

### 输入

步骤三、四已定稿的边界与数据策略；步骤一的 NFR 与验收口径；现网容量基线、预算与合规红线（等保 / PCI 范围、密钥托管方式等）；运维与 SRE 对告警、值班与变更窗口的约定。

### 活动

把「能跑」推进到「在故障与攻击面下仍可预期地跑」：确定部署拓扑与**多可用区（Multi-AZ）**策略、弹性与限流、缓存与异步消息的职责切分、批处理与离线任务窗口、密钥与凭据生命周期、可观测性与降级路径，并对**成本敏感路径**做粗算或分档（算价、搜索、推荐、大促热点 SKU 等）。

### 输出

《部署与容量方案》（AZ / 区域、副本、扩缩容触发条件）；《观测与安全设计》（指标 / 日志 / 追踪、告警分级、接入与流量防护要点）；《降级与韧性说明》（超时、熔断、排队、人工兜底入口）；《成本与配额说明》（按链路或店铺的区间表）。工件不必细到采购单，但要经得起评审追问「峰值谁扛、故障谁先感知、预算主要烧在哪」。

### 电商示例

- **多 AZ**：交易、支付回调、占库等关键链路写明**同区域多 AZ** 或跨区容灾取舍；数据库主从 / 多活遇**脑裂**时的业务裁决（只读、暂停下单、切换 **RTO / RPO**）用短文写清，避免「容器编排上了就等于高可用」。
- **缓存与击穿**：热点商品、活动页、券批次等读多写少处上缓存时，同步写**穿透 / 击穿 / 雪崩**防护（空值占位、互斥重建、**TTL** 分层、热点发现）；订单与支付状态等强一致读勿把「最终一致缓存」当唯一真相。
- **消息堆积**：订单事件、支付结果、出库指令等 **topic** 约定**堆积阈值**、消费扩容与限流；配 **DLQ（死信队列）** 与人工 **replay** 规则写入运维手册，防止大促后出现拖垮履约的「消息黑洞」。
- **对账批处理**：渠道清算、补贴结算、库存流水与总账等**日切 / 小时批**的窗口、幂等与重跑在技术侧落表（此处不重复业务规则本身）；失败批次须告警到责任人并写补账 **SLA**。
- **密钥与卡数据**：渠道密钥、商户私钥、**Tokenization** 配置走 **KMS / HSM** 或平台托管；坚持**卡数据不进业务库**（**PAN**、**CVV** 等），业务侧仅存渠道 **token** 与脱敏展示字段；**PCI** 范围与日志脱敏一并写入工件。
- **成本敏感路径**：搜索、推荐、大模型导购、实时算价等链路标注单位成本或高 / 中 / 低档位，并与限流、降级、异步化、批处理挂钩，避免体验无限堆叠而账单无人复盘。

### 常见反例

只画理想拓扑不谈 **AZ** 与故障演练；缓存无击穿与回源预案；消息无 **DLQ** 与堆积处置；对账失败靠微信群；密钥进配置仓库或日志明文；默认「云厂商全包」却无成本与配额上限。

## 步骤六：验证与固化

### 输入

前五步全部工件；评审会上仍未闭合的争议点；压测、灰度或渗透测试的初步结论与排期（若有）。

### 活动

显式列出**假设**（业务、第三方、数据、容量）与对应**风险**、触发条件与缓释；对不可逆或高代价权衡撰写 **ADR（Architecture Decision Record）**；组织跨职能评审，沉淀结论、行动项与「上线后观测清单」，开放项进入 **backlog**。

### 输出

《假设与风险登记》（建议列：**假设陈述 | 若不成立的影响 | 监测信号 | 负责人 | 缓释 / 回滚**）；《**ADR** 索引》与单篇 **ADR**（背景、选项、决策、后果）；《评审纪要》（通过 / 附条件通过 / 延期）与行动项 **owner**、截止时间。至此上半篇流程闭环；上线前后勾选与复盘交给下半篇清单。

### 电商示例

- **风险与假设列表模板**：例如「假设大促峰值支付渠道成功率不低于 **x%**」「假设券批次缓存 **miss** 时数据库可承受 **y QPS**」「假设物流接口超时时页面可用默认妥投日兜底」——每条都能落到告警阈值、降级策略或客服话术，而不是停在「应该没问题」。
- **ADR 示例主题**：（1）**库存先扣后付** vs **先付后扣**：对超卖、体验、对账复杂度、渠道失败回滚的影响；（2）**拆单与部分退款**：父子单状态、补贴与运费分摊、逆向资金流与客服可操作边界——**ADR** 写清取舍与边界即可，细则链到订单 / 支付域设计文。
- **评审关口与下半篇预告**：架构评审与上线评审固定核对：假设与风险是否登记、关键 **ADR** 是否已批、观测与降级是否可演示。更细的**业务 / 应用 / 数据 / 技术**分域勾选项（完整性、一致性、可演进、可运维含安全）在本文**下半篇分域审查清单**展开，便于打印走查。

### 常见反例

纪要只写「同意」无行动项；风险表抄模板却无监测信号；争议口头拍板不写 **ADR**，数月后无人记得取舍原因；上线前不预演降级与对账失败；没有下半篇清单导致流程与日常核对脱节。

## 分域审查清单

下半篇将上文流程收成可打印、可会议走查的勾选表：按**业务 / 应用 / 数据 / 技术**四域展开，每域下再按**完整性、一致性、可演进、可运维（含安全）**四维组织。若要把清单落到 **Pull Request** 的逐条核对习惯上，建议对照 [架构与整洁代码（四）](/2026/04/04/system-design/44-acc-code-review/) 中的分阶段 Code Review 维度，以免业务域清单与技术实现脱节。数据域中涉及迁移、双写与幂等改动的专项走查，`44` 文在相近时间窗口提供了更细的评审切片，详见 [架构与整洁代码（四）](/2026/04/04/system-design/44-acc-code-review/)。技术域的 **SLO**、告警与灰度发布如需与代码门禁逐条对齐，同样可在 [架构与整洁代码（四）](/2026/04/04/system-design/44-acc-code-review/) 中找到并列的审查角度。下列条目使用 `- [ ]` 语法，便于在 **Markdown** 预览或纸质打印稿上逐项勾选。

### 业务域

#### 完整性

- [ ] 峰值与大促场景下的用户旅程（浏览、算价、下单、支付、履约、售后）是否写全主路径与关键分支。
- [ ] 异常与逆向流程（支付失败、占库失败、超时、部分发货、拒收、部分退款）是否各有业务期望与客服可操作边界。
- [ ] 第三方依赖（支付、物流、营销、风控）在业务侧的失败表现与用户可见文案是否已列清单。
- [ ] 合规与数据主体权利（同意、导出、删除、营销退订）是否在业务能力层有对应场景条目。

#### 一致性

- [ ] 页面展示价、购物车预估、下单快照与结算实付之间的允许差异与确认节点是否书面一致。
- [ ] 大促叠加（券、会员价、平台补贴、店铺活动）的出资方与结算口径是否在业务规则中无歧义。
- [ ] 库存「可售 / 占用 / 在途 / 预售」等业务语义是否与前台承诺用语一致，避免跨团队各说各话。
- [ ] 订单、子单、售后单之间的状态命名与可见状态机是否在业务文档与产品稿中对齐。

#### 可演进

- [ ] 新业务活动（新券种、新玩法、新渠道）接入时，是否预留规则扩展位而非硬编码一次性文案。
- [ ] 业务名词表（**SKU**、履约单、补贴批次等）是否有维护责任人与变更通知机制。
- [ ] 关键业务指标（支付成功率、超卖率、履约时效）的定义是否版本化，避免口径悄悄漂移。

#### 可运维（含安全）

- [ ] 大促与故障时的业务降级策略（限购、排队、关券、延迟发货）是否写明触发条件与恢复责任方。
- [ ] 客诉与工单高频类型是否映射回具体能力与场景，便于事后复盘与清单迭代。
- [ ] 涉及个人信息、支付敏感字段的业务流程是否标明最小收集原则与对外共享边界。
- [ ] 财务对账、补贴结算、渠道清算相关的业务 **SLA**（日切、补账时限）是否在业务侧有明确负责人。

### 应用域

#### 完整性

- [ ] 限界上下文或服务边界是否覆盖订单、库存、商品与价格、营销、支付、物流等电商主干协作。
- [ ] 同步 **API**（**REST** / **gRPC**）与异步事件是否均有契约说明（字段、错误码、超时、重试语义）。
- [ ] 跨边界调用链是否标注第三方超时、熔断与用户可见降级路径。
- [ ] **Webhook**、回调与批处理入口是否列入契约目录并标明幂等与重放约定。

#### 一致性

- [ ] **API** 版本、弃用窗口与兼容策略（新增可选字段、禁止静默改语义）是否对消费方公示。
- [ ] 领域事件 **schema**（主题、版本、必填字段）是否与订单状态、支付结果等权威语义一致。
- [ ] 同步写与异步通知对同一业务事实的表述是否一致（例如「已支付」与 **OrderPaid** 载荷对齐）。
- [ ] 多语言、多币种场景下契约中的金额、币种与舍入规则是否显式约定。

#### 可演进

- [ ] 新消费者订阅既有事件时，是否评估回放、快照与顺序要求，避免隐性耦合。
- [ ] 防腐层与适配器是否隔离第三方 **DTO** 与内部领域模型，便于替换渠道或升级接口。
- [ ] 契约变更是否配套 **changelog** 与联调窗口，支持灰度消费方逐步升级。

#### 可运维（含安全）

- [ ] 对外接口的认证、鉴权、**rate limit** 与防重放是否在契约层有统一要求。
- [ ] 敏感字段在日志、**trace** 与调试接口中的脱敏规则是否写入集成说明。
- [ ] 第三方调用失败时的补偿入口（人工重试、对账任务触发）是否在应用协作图中标出。
- [ ] **DLQ**、**replay** 与运营后台操作对应的应用服务责任是否明确到团队。

### 数据域

#### 完整性

- [ ] 核心实体（订单、订单行、支付单、库存流水、券核销记录）的 **SoR** 与只读副本是否逐一标明。
- [ ] 订单行价格快照、活动与券分摊字段是否齐全，足以支撑售后与审计复盘。
- [ ] **Outbox**、幂等键表、对账批次表等是否在设计中有列级或行级示意，并与业务 **SLA** 挂钩。
- [ ] 审计字段（创建 / 更新人时、操作来源、请求 **ID**）是否在关键表上统一要求。

#### 一致性

- [ ] 支付回调乱序、重复与迟到时，状态落库与业务幂等键策略是否成文。
- [ ] 库存流水与可售余额的重建或对账口径是否写明，并能覆盖大促高峰写入模式。
- [ ] 最终一致路径上的补偿任务（补发事件、回补状态）是否有触发条件与幂等设计。
- [ ] 跨库或跨服务事务边界是否与聚合不变式一致，避免隐式双写无登记。

#### 可演进

- [ ] 双写、回填、切换窗口与回滚预案是否在数据迁移计划中分阶段列出。
- [ ] **schema** 变更是否评估大表锁、在线 **DDL** 与业务低峰窗口。
- [ ] 历史订单与归档策略是否影响报表与合规留存，并标注延迟可读承诺。

#### 可运维（含安全）

- [ ] 快照、流水与审计日志的保留周期、脱敏与访问审批是否符合合规分类。
- [ ] 对账失败、批次重跑与人工订正是否有数据层操作规范与双人复核要求。
- [ ] 备份、**PITR** 与关键表恢复演练是否覆盖支付与订单核心路径。
- [ ] 生产数据导出、**BI** 同步与测试环境造数是否遵循最小权限与令牌轮换。

### 技术域

#### 完整性

- [ ] 容量与弹性是否覆盖大促峰值、热点 **SKU** 与突发营销活动的预估与压测结论摘要。
- [ ] **Multi-AZ**、主从切换、**RTO** / **RPO** 与脑裂时的业务裁决是否写入部署方案。
- [ ] 缓存（穿透、击穿、雪崩）、消息堆积阈值与扩容 playbook 是否成文。
- [ ] **SLO**、**SLI** 与错误预算是否在核心链路上有定义，并与产品体验挂钩。

#### 一致性

- [ ] 限流、熔断、超时默认值是否与业务可接受的失败语义一致（含用户重试提示）。
- [ ] 灰度发布、金丝雀与配置开关的默认值是否在多环境间一致且有回归检查。
- [ ] 时钟、时区与日切批处理在分布式节点上是否一致，避免对账边界错位。

#### 可演进

- [ ] 特性开关与技术债条目是否登记，避免临时方案永久化。
- [ ] 基础设施即代码或部署流水线是否支持安全补丁与镜像升级的滚动策略。
- [ ] 成本敏感链路（算价、搜索、推荐）的配额与预算告警是否与扩容决策联动。

#### 可运维（含安全）

- [ ] 告警分级、值班路由、**on-call** 升级路径是否与 **SLO** 违约场景一一映射。
- [ ] 灰度与回滚（含配置回滚、流量切换）是否在上线检查单中可演练。
- [ ] **KMS** / **HSM**、密钥轮换与**卡数据不进业务库**（**PCI** 范围）是否在技术方案中勾选确认。
- [ ] 关键链路仪表盘、日志采样与分布式追踪是否足以支撑大促与第三方超时排障。

## 复盘框架

走完一轮六步与分域清单后，建议用下面三个问题收束讨论；不必追求「正确答案」，重点是留下可检索的决策痕迹，供下一轮迭代对照。

### 本轮最省的决策

某匿名团队在「会员日」改版里选择**沿用既有订单状态机**，只扩展两条业务边与一份 **ADR**，而不是趁机重写编排服务。省下的三周工程窗口被挪去做支付回调幂等与对账批次的联调，上线后首周客诉里与状态相关的工单明显下降。事后复盘认为，**省**的不是文档页数，而是避免了在峰值前同时动两条高风险链路。

### 本轮最险的决策

另一合成例子里，为赶窗口，团队在**库存最终一致**路径上临时放宽了「可售视图延迟上限」，并在营销侧同步上了更高力度的券批次缓存。大促当晚未超卖，但次日对账出现系统性偏差，不得不人工冻结部分补贴结算。最险之处在于：**风险被业务上可接受**的口头共识掩盖，却没有写进假设表与监测阈值，导致信号迟到。

### 下一轮要提前问业务的一个问题

建议在下一轮立项之初就问清：**当第三方物流接口持续超时时，用户侧默认展示的妥投承诺与客服赔付上限，以哪一份业务规则为准、由谁签字背书？** 若答案依赖「大促时再定」，则步骤二的能力与场景清单应显式标为开放项，并挂 **owner** 与决策截止日，避免步骤三、四的契约与数据设计在真空中定稿。

## 总结

本文的成功标准，是读者在**一次迭代**内能同时带走三样东西：**可重复执行的六步流程**、按四域组织的**分域审查清单**，以及至少**一类**被写清取舍的架构权衡（通常落在 **ADR** 或风险登记里）。流程回答「先做什么、产出什么」；清单回答「评审与上线前漏掉了什么」；权衡则把口头共识变成可追溯的文字，避免数月后无人记得为何如此设计。三者叠加，才能把架构讨论从偶发工作坊变成团队习惯。与 `41`–`44` 系列分工明确：前四篇深化建模、代码与评审能力，本文把对齐节奏与工件形态讲透；电商系列长文则提供领域细节，便于对照清单中的条目落地。若你只做其中一步——例如只写流程不写清单——短期省事，长期往往在重复扯皮与返工上付出代价。建议把复盘框架里的三个问题固定进迭代尾声，使每一轮都能压缩一点不确定性。最后，清单与流程都应视 living document 持续裁剪：新业务形态、新合规要求或新第三方一出现，就回到步骤一至六与分域表上增补，而不是另起一套方言。
