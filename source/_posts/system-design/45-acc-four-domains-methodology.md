## 引言

本文面向**单一产品或单条主业务线**语境下的技术负责人与核心架构参与者：我们关心的是在一轮迭代里如何把方案说清楚、把跨团队依赖对齐，并沉淀为可评审、可落地的工件。这与覆盖投资组合、标准体系与全公司治理节奏的**企业级 EA（Enterprise Architecture）**不是同一尺度——若组织已有强 EA，可把下文当作**交付团队侧**的对齐与收口工具，与上层治理边界分明。

为便于分工与复盘，后文将架构相关工作拆成四个**工程域**（指讨论与产物的划分，而非公司部门）：

- **业务域**：关注**能力与场景**——组织对外提供哪些业务能力、在何种情境下被触发、怎样定义成功与验收口径。
- **应用域**：关注**边界与契约**——服务或模块的职责切分、对外 API / 事件等契约形态，以及演进与兼容策略。
- **数据域**：关注**所有权与一致性**——主数据归属、读写路径、一致性与补偿的底线，以及审计与可追溯要求。
- **技术域**：关注**约束与横切能力**——容量与可用、安全与合规、可观测性、成本与平台化能力等跨切面设计与门禁。

本仓库「架构与整洁代码」系列前四篇侧重**建模、分层、CQRS 与评审**等具体能力：[架构与整洁代码（一）](/2026/04/01/41-acc-clean-arch-ddd-cqrs/) 从 Clean Architecture、DDD 与 CQRS 讲清分层与读写路径；[架构与整洁代码（二）](/2026/04/02/42-acc-clean-code/) 讨论复杂业务中的 Clean Code 战术；[架构与整洁代码（三）](/2026/04/03/43-acc-ddd-notes/) 补足战略 / 战术 DDD 与通用语言；[架构与整洁代码（四）](/2026/04/04/44-acc-code-review/) 给出分阶段 Code Review 清单。**本文（45）**则聚焦**方案成形与跨域对齐**：把上述知识收束成一条可重复走的流程，并配套下半篇的分域清单，便于在立项、详设与评审关口统一语言。

文中的叙述与例子默认取材于笔者约 **5 年**的电商相关工程实践，场景锚定为典型的 **B2C / 平台型**交易与履约链路。**跨境合规、全球库存与多租户强隔离**等假设显著更重，**超出本文默认范围**；若读者处于此类环境，请在数据域与技术域上自行加厚约束与清单条目。

读完引言后，可将全文视为一次「架构工作坊」的操作手册：**上半篇**将给出贯穿四域的**六步流程**（每步交代输入、活动与关键产出）；**下半篇**提供按四域组织的**审查清单**，供评审会与上线前逐项核对。

## 流程总览

下表按时间顺序列出六步：每步要回答的主要问题、建议沉淀的工件，以及讨论重心落在哪一域（后文分节展开）。

| 步骤 | 主要问题 | 关键产出（工件） | 主要落在哪一域 |
|------|----------|------------------|----------------|
| 1. 问题与成功标准 | 做什么、不做什么、如何验收 | 范围与目标说明、验收口径、NFR 清单 | 业务（目标）+ 技术（NFR）为主 |
| 2. 业务架构（轻量） | 谁、在什么场景、达成什么结果 | 业务能力与场景、角色与权限、业务结果定义 | 业务 |
| 3. 应用架构 | 系统如何切分、如何协作 | 限界上下文/服务边界、协作关系、API 与事件契约 | 应用（+ 契约影响数据） |
| 4. 数据架构 | 谁拥有真相、如何一致、如何演进 | 数据所有权与模型、一致性策略、演进与迁移方案 | 数据 |
| 5. 技术架构 | 如何部署、观测、安全、降级与成本 | 部署与容量方案、观测与安全设计、降级与成本估算 | 技术 |
| 6. 验证与固化 | 假设、风险、决策与评审关口 | 假设与风险登记、ADR、评审结论与行动项 | 四域交叉 + ADR |
