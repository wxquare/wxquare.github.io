系统交互边界设计：三种方案详细对比

已确认：营销库存与商品库存独立设计

方案 A：库存系统主导（Smart Service）

架构示意
┌─────────────────────────────────────────────────────────────┐
│                      订单系统                                │
│  - 订单创建                                                  │
│  - 支付回调处理                                              │
│  - 取消/退款                                                 │
└────────┬────────────────────────────────────────────────────┘
         │ 调用高级 API
         ▼
┌─────────────────────────────────────────────────────────────┐
│                    库存系统 (Facade)                        │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  CheckAndReserveInventory(orderItems, timeout)       │  │
│  │  • 校验商品库存                                       │  │
│  │  • 校验营销库存                                       │  │
│  │  • 原子预扣（带超时）                                 │  │
│  │  • 内部重试逻辑                                       │  │
│  │  • 返回 reservation_id                               │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  ConfirmReservation(reservation_id)                  │  │
│  │  • 确认扣减                                           │  │
│  │  • 事务保证                                           │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  CancelReservation(reservation_id, reason)           │  │
│  │  • 超时自动取消                                       │  │
│  │  • 手动取消支持                                       │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌─────────────────┐  ┌──────────────────┐                │
│  │ 商品库存服务    │  │  营销库存服务    │                │
│  │ Redis + MySQL   │  │  Redis + MySQL   │                │
│  └─────────────────┘  └──────────────────┘                │
└─────────────────────────────────────────────────────────────┘

优势

  • 单一职责清晰：订单系统专注订单逻辑，库存复杂度封装在库存侧
  • 原子性保证：CheckAndReserve 一次调用完成商品+营销双重校验和预扣
  • 超时管理内聚：库存系统统一管理预扣超时（如 15 分钟），订单侧无需关心
  • 重试逻辑封装：库存侧内部处理瞬时失败（如 Redis 抖动）
  • 易于测试：库存系统可独立测试各种边界条件

劣势

  • 库存系统变重：需要维护 reservation 状态机
  • 跨服务事务：如果营销库存在独立服务，需要分布式事务协调
  • API 设计复杂：需要支持各种业务场景（秒杀、拼团、预售等）
  • 单点瓶颈：库存系统成为核心依赖，需要高可用保障
适用场景
  • 库存规则复杂（如：多仓协调、分级预扣）
  • 需要统一的库存管控策略
  • 订单系统需要保持轻量

方案 B：订单系统编排（Orchestration）

架构示意
┌─────────────────────────────────────────────────────────────┐
│                    订单系统 (Saga Orchestrator)              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  CreateOrder Saga:                                   │  │
│  │    1. CheckProductStock(sku, qty) ────┐             │  │
│  │    2. CheckCampaignStock(activity_id)  │             │  │
│  │    3. ReserveProductStock(sku, qty) ◄──┘             │  │
│  │    4. ReserveCampaignStock(activity_id)              │  │
│  │    5. CreateOrderRecord()                            │  │
│  │    6. 等待支付...                                     │  │
│  │                                                       │  │
│  │  Compensate:                                         │  │
│  │    - ReleaseProductStock()                           │  │
│  │    - ReleaseCampaignStock()                          │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  • 管理预扣超时（定时任务扫描）                             │
│  • 协调多个库存服务调用                                     │
│  • 实现补偿逻辑                                             │
└────┬────────────────────────────────────────────┬───────────┘
     │                                            │
     ▼                                            ▼
┌─────────────────┐                    ┌──────────────────┐
│ 商品库存服务    │                    │  营销库存服务    │
│                 │                    │                  │
│ API:            │                    │ API:             │
│ • CheckStock    │                    │ • CheckQuota     │
│ • ReserveStock  │                    │ • ReserveQuota   │
│ • ReleaseStock  │                    │ • ReleaseQuota   │
│ • SellStock     │                    │ • ConsumeQuota   │
│                 │                    │                  │
│ (纯数据操作)   │                    │ (纯数据操作)    │
└─────────────────┘                    └──────────────────┘

优势

  • 库存服务纯粹：只提供原子操作，无业务逻辑
  • 业务流程可见：Saga 编排在订单侧，易于理解和调试
  • 灵活性高：订单侧可根据业务场景灵活调整扣减顺序和策略
  • 独立扩展：商品库存和营销库存完全解耦，各自演进
  • 测试友好：可以 mock 库存服务，测试订单 Saga

劣势

  • 订单系统变重：需要实现 Saga 引擎、超时管理、补偿逻辑
  • 网络开销：多次 RPC 调用（Check + Reserve * 2）
  • 一致性保证难：需要仔细处理各种失败场景（如：预扣商品成功，营销失败）
  • 重复代码：多个订单场景（秒杀、拼团）可能重复实现类似 Saga
适用场景
  • 库存逻辑简单，主要是数值增减
  • 订单系统需要强控制力（如：特殊扣减顺序）
  • 团队希望订单系统作为核心编排层

方案 C：事件驱动协作（Choreography）

架构示意
                        ┌──────────────────┐
                        │      Kafka       │
                        └────────┬─────────┘
                                 │
        ┌────────────────────────┼────────────────────────┐
        │                        │                        │
        ▼                        ▼                        ▼
┌───────────────┐      ┌─────────────────┐      ┌──────────────────┐
│  订单系统     │      │  商品库存服务   │      │   营销库存服务   │
└───────┬───────┘      └────────┬────────┘      └────────┬─────────┘
        │                       │                         │
        │ Publish:              │ Subscribe:              │ Subscribe:
        │ • OrderCreated        │ • OrderCreated          │ • OrderCreated
        │                       │   ├→ ReserveStock       │   ├→ ReserveQuota
        │                       │   └→ Publish:           │   └→ Publish:
        │                       │      StockReserved      │      QuotaReserved
        │                       │                         │
        │ Subscribe:            │ • PaymentCompleted      │ • PaymentCompleted
        │ • StockReserved       │   └→ ConfirmStock       │   └→ ConsumeQuota
        │ • QuotaReserved       │                         │
        │ • StockReserveFailed  │ • OrderCancelled        │ • OrderCancelled
        │   └→ Compensate       │   └→ ReleaseStock       │   └→ ReleaseQuota
        │                       │                         │
        │ Publish:              │                         │
        │ • PaymentCompleted    │                         │
        │ • OrderCancelled      │                         │
        │                       │                         │

事件流示例（下单成功）：
1. OrderCreated (order_id, items) → 广播
2. StockReserved (order_id, reservation_id) ← 商品库存
3. QuotaReserved (order_id, quota_id) ← 营销库存
4. 订单系统收集到两个 Reserved 事件 → 更新订单状态为"待支付"
5. PaymentCompleted (order_id) → 广播
6. 各库存服务监听并确认扣减

失败场景（商品库存预扣失败）：
1. OrderCreated → 广播
2. StockReserveFailed (order_id, reason) ← 商品库存
3. 订单系统监听 → 发布 OrderCancelled
4. 营销库存监听 OrderCancelled → ReleaseQuota（如果已预扣）

优势

  • 完全解耦：各服务独立演进，无直接依赖
  • 易于扩展：新增服务（如积分系统）只需订阅事件
  • 天然异步：不阻塞订单创建主流程
  • 容错性好：某个服务宕机不影响其他服务
  • 事件溯源：完整的事件日志，便于排查和审计

劣势

  • 最终一致性：无法即时知道库存是否充足（需要等待事件）
  • 调试困难：分布式追踪复杂，需要完善的 tracing 体系
  • 消息顺序问题：需要处理乱序、重复消息
  • 复杂度高：需要设计完善的事件模型、幂等性保障
  • 用户体验：下单后需要"等待库存确认"，可能体验较差
适用场景
  • 大型分布式系统，服务众多
  • 可接受最终一致性
  • 需要高度可扩展性
  • 有成熟的事件溯源基础设施

三种方案核心对比

维度 库存主导 订单编排 事件驱动
复杂度集中 库存系统 订单系统 分散
网络调用 少（1-2 次） 多（4-6 次） 异步（MQ）
一致性 强一致（内部） 强一致（Saga） 最终一致
响应时间 快（50-100ms） 中（100-200ms） 慢（秒级）
调试难度 中 低 高
扩展性 中 中 高
团队要求 库存团队强 订单团队强 全员高水平

推荐选择

基于电商实践和您的架构

推荐

混合方案：A + B（分场景选择）

  • 常规下单：使用方案 B（订单编排），保持灵活性
  • 秒杀/大促：使用方案 A（库存主导），在库存侧做流控和限流
  • 异步场景：局部使用方案 C（事件驱动），如：库存变更通知
理由：
  • 常规场景订单编排更清晰，易于理解和维护
  • 高并发场景库存主导可以集中防护（限流、降级）
  • 事件驱动用于非关键路径，降低复杂度
