商品库存 vs 营销库存：核心差异

商品库存 (Product Inventory)
数据维度
  • 主键: SKU + 仓库
  • 数量管理: 可售、已售、预扣、在途
  • 批次管理: 批次号、过期日期
业务特点
  • 库存有限，需要采购补货
  • 扣减不可逆（售出即消耗）
  • 需要多仓协调（距离优先、成本优先）
  • 支持锁定库存（预售、拼团）
Redis 数据结构
# 数量型
inventory:sku:{sku_id}:warehouse:{wh}
  → Hash {available, reserved, sold}

# 码池型（优惠码、激活码）
inventory:pool:{pool_id}
  → List [code1, code2, ...]
典型操作流程
1. CheckStock(sku, qty)
   → 查询 available >= qty
2. ReserveStock(order_id, sku, qty, ttl)
   → HINCRBY available -qty
   → HINCRBY reserved +qty
   → SETEX reserve:{order_id} ttl
3. ConfirmStock(order_id)
   → HINCRBY reserved -qty
   → HINCRBY sold +qty
   → DEL reserve:{order_id}
4. ReleaseStock(order_id)
   → HINCRBY reserved -qty
   → HINCRBY available +qty
营销库存 (Campaign Inventory)
数据维度
  • 主键: 活动 ID + SKU + 用户维度
  • 配额管理: 总配额、已用配额、单用户限额
  • 时间约束: 时段配额（如每小时 1000 件）
业务特点
  • 虚拟库存，预算驱动
  • 可动态调整配额
  • 用户维度限制（防刷）
  • 时段配额（错峰）
  • 支持多种补充策略（定时、按比例）
Redis 数据结构
# 活动总配额
campaign:quota:{activity_id}:{sku}
  → Hash {total, used, reserved}

# 用户已用次数
campaign:user:{activity_id}:{user_id}
  → Hash {count, last_time}

# 时段配额（可选）
campaign:quota:{activity_id}:hour:{YYYYMMDDHH}
  → String (可用数量)
典型操作流程
1. CheckQuota(activity_id, user_id, sku)
   → 检查总配额、用户限额、时段限额
   → 布隆过滤器防刷（可选）
2. ReserveQuota(order_id, activity, user, sku)
   → HINCRBY quota used +1
   → HINCRBY user count +1
   → SETEX reserve:{order_id} ttl
3. ConsumeQuota(order_id)
   → HINCRBY reserved -1
   → 更新 MySQL 日志
4. ReleaseQuota(order_id)
   → HINCRBY used -1
   → HINCRBY user count -1
