# Hexo 博客写作规范

## 通用文章规范（所有目录适用）

### 文章命名
- 重要技术文章使用数字前缀：`数字-文章标题.md`（如 16-leetcode.md）
- 技术笔记可使用描述性名称（如 OpenClaw深度调研.md）
- 非技术类文章使用中文或英文描述性名称

### 文章结构
- 必须包含 front-matter：
  ```yaml
  ---
  title: 文章标题
  date: YYYY-MM-DD HH:MM:SS
  categories: [分类1, 分类2]
  tags: [标签1, 标签2]
  ---
  ```
- 开篇简要说明文章目的和受众
- 复杂内容使用目录导航

### 代码规范
- 所有代码块指定语言：` ```语言名 `
- 提供可运行的完整代码示例
- 复杂算法标注时间复杂度和空间复杂度
- 关键代码行添加注释说明

## 目录特定规范

### AI 目录 (`source/_posts/AI/`)

#### 子目录分类
- `computer-vision/` - 计算机视觉相关
- `tensorflow/` - TensorFlow 框架相关
- `tvm/` - TVM 编译器相关

#### 文章位置规则
- 系统设计类 AI 文章 → AI 根目录
- Agent 设计类文章 → AI 根目录
- 深度学习框架/工具文章 → 对应子目录

#### 术语规范
- 使用中文全称，不用缩写：
  - 深度学习（不用 DL）
  - 神经网络（不用 NN）
  - 机器学习（不用 ML）
  - 卷积神经网络（不用 CNN）
  - 循环神经网络（不用 RNN）
- 英文术语首次出现时加中文注释

### System Design 目录 (`source/_posts/system-design/`)

#### 文章命名
- 系统设计类使用数字前缀：`数字-系统名-design.md`
- 架构实践类：`数字-ddd/微服务/事件驱动-实践.md`
- 性能优化类：`数字-优化目标-optimization.md`

#### 内容结构
- 先描述业务场景和约束条件
- 使用 CAP 定理分析架构选择
- 包含性能指标和扩展性考虑
- 复杂架构图使用分层说明

#### 技术术语
- 使用业界标准术语
- 架构模式名称保留英文（如 Event Sourcing、CQRS）
- 数据库和中间件使用官方名称

### Other 目录 (`source/_posts/other/`)

#### 内容类型
- 职业发展思考
- 投资框架和经验
- 工具使用教程（如 Hexo 博客搭建）
- 读书笔记和随笔

#### 风格要求
- 允许更灵活的写作风格
- 个人经验分享需有具体案例
- 教程类文章需有步骤编号

## 引用与图表规范（全站通用）

### 论文引用
- 格式：`[作者, 年份]`
- 提供论文链接或 DOI

### 代码引用
- 提供 GitHub 链接或代码仓库链接
- 引用特定代码行时使用永久链接（包含 commit hash）

### 图片与图表
- 优先使用 Excalidraw（架构图）或 Mermaid（流程图/数据流图）
- 所有图表放在 `source/diagrams/`
- 文件命名要有意义，避免使用默认名称（如 `Untitled-1.png`）
- 引用图片时使用相对路径：`/diagrams/有意义的文件名.png`

### 外部资源
- 确保链接有效
- 添加访问日期（格式：`访问于 YYYY-MM-DD`）
- 重要资源提供备用链接

## 写作风格指南

### 语言规范
- 技术文章使用专业术语，保持准确性
- 避免口语化表达（随笔类文章除外）
- 中英文混排时，英文前后加空格

### 段落结构
- 每段聚焦一个主题
- 使用列表和表格增强可读性
- 关键结论使用加粗或引用块突出

### 中英文格式
- 英文单词前后留空格
- 专有名词保持首字母大写
- 代码中的命名保持原样

## Hexo 特殊注意事项

### 避免发布到博客的文件
- 配置文件、规范文件 → 放在 `.cursor/rules/` 或 `_private/` 目录
- 草稿文章 → 使用 `_drafts/` 目录
- 临时笔记 → 文件名前缀加下划线 `_`

### 文章元数据
- 使用 `categories` 建立层级结构
- `tags` 用于横向关联
- 系列文章使用相同前缀编号（如 22-, 23-）
