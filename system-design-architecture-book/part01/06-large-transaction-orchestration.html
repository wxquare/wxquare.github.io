系统设计与架构实战：原理、工程与电商案例

第 5 章 大事务处理方法论：流程编排、Saga 与工作流系统

从电商创单出发，理解为什么跨系统业务流程会天然演化为大事务问题，以及如何在同步 RPC、Saga、状态机和工作流引擎之间做出可恢复、可审计、可补偿的设计选择。

很多工程师第一次听到“大事务”时，会自然联想到数据库事务、分布式事务，甚至 2PC / XA。这些概念当然重要，但如果把它们直接套到互联网业务流程上，往往会把问题理解偏。

互联网系统里的大事务，真正要解决的不是“如何让多个数据库像一个数据库那样一起提交”，而是：

当一个业务目标必须跨库存、价格、订单、支付、履约、通知等多个系统推进时，系统应该如何在失败必然发生的现实里，仍然把业务结果收敛到可解释、可恢复、可补偿的状态。

这也是为什么本章会从电商创单切入。创单不是唯一的大事务场景，但它最能把大事务设计里的关键矛盾一次性暴露出来：多步骤、多系统、长等待、补偿、超时、审计、人工接管、状态收敛。

5.1 大事务到底在解决什么问题

先把三个经常混在一起的概念拆开：

概念解决的问题典型手段
数据库事务单库内的原子性和一致性本地事务、锁、隔离级别
分布式事务多资源之间的提交一致性2PC、XA、TCC
业务大事务跨系统业务目标如何推进和收敛Saga、状态机、Workflow、补偿

数据库事务关心的是“同一个数据库里的几条 SQL 要不要一起成功或失败”。
分布式事务关心的是“多个资源管理器之间能不能形成统一提交语义”。
业务大事务关心的则是“一个业务流程跨多个系统推进时，如何保证结果最终正确且过程可治理”。

这三者并不是谁替代谁的关系，而是处理层次不同的问题。

在互联网系统里，很多流程天生就不适合直接上 2PC / XA：

  • 支付网关、短信平台、物流系统、供应商接口都不可能加入你的本地事务。
  • 用户支付不是毫秒级动作，可能会等待几十秒、几分钟，甚至跨天。
  • 高并发下长时间持锁会迅速击穿吞吐和可用性。
  • 很多步骤天然只能做“确认 / 取消 / 补偿”，而不是“全局原子提交 / 回滚”。

所以大事务设计的重点很少是“全局一次提交”，更常见的是：

  1. 找到权威状态源。
  2. 把业务流程拆成多个可提交、可恢复、可补偿的阶段。
  3. 为每个阶段定义状态、超时、补偿和审计规则。
  4. 确保流程在局部失败后仍然能够最终收敛。

这也意味着，大事务系统的设计对象不是“接口调用顺序”，而是“状态推进机制”。

5.2 为什么创单流程天然是大事务

电商创单看起来只是用户点了一次“提交订单”，但系统内部通常至少涉及下面这些动作：

  1. 校验商品是否仍然可售。
  2. 校验价格、优惠、税费、运费是否仍然有效。
  3. 预占库存或其他资源。
  4. 创建订单主记录和订单行。
  5. 生成支付单或支付上下文。
  6. 等待支付结果回写。
  7. 支付失败时释放库存、优惠和其他预占资源。
  8. 支付成功后推进履约、通知、积分等后续流程。

这条链路的真正难点，不在于步骤多，而在于每一步都可能拥有独立的本地事务和独立的失败语义：

  • 库存预占成功，但订单写库失败。
  • 订单创建成功，但支付发起失败。
  • 支付网关返回超时，但用户实际上已经支付成功。
  • 支付成功回调丢失，订单状态没有推进。
  • 订单超时取消后，库存释放和优惠释放只完成了一部分。

所以创单流程天然带有四个大事务特征：

  • 跨系统：库存、价格、订单、支付、履约通常不是同一个服务。
  • 跨阶段：下单前确认、下单时落单、支付后推进不是一个瞬时动作。
  • 可失败：任何一步都可能独立失败，而且失败之后未必能简单回滚。
  • 需解释：用户、客服、运营都必须知道当前到底卡在哪一步。

用一句话概括：创单不是“一个接口调多个下游”，而是一条需要持续推进和持续收敛的业务流程。

这也是为什么它特别适合作为大事务处理方法论的主案例。

5.3 如何判断一个流程该不该进入编排层

不是所有跨系统调用都值得上 Saga 或工作流引擎。真正需要进入编排层的流程，通常至少满足下面几个判断维度：

判断维度典型问题
跨系统数是否依赖 3 个以上独立系统
步骤数是否已经不止 2-3 个固定步骤
补偿需求任一步失败后是否需要显式回收已有副作用
长等待是否存在支付回调、人工审核、异步确认等等待点
审计要求是否必须知道“当前做到哪一步、谁做过什么动作”
人工接管出现异常后是否允许或需要人工介入恢复

可以先记一个很实用的经验判断：

  • 同步 RPC 短流程：步骤少、依赖少、失败后整体返回即可，不需要长期保存中间状态。
  • 应用层 Saga / 状态机：步骤较多、需要补偿、需要阶段推进，但复杂度还没高到引入重型工作流引擎。
  • Workflow / 编排引擎：流程长、等待点多、状态复杂、审计和人工介入强，靠普通应用代码已经难以维护。

对创单来说，如果只是“查库存 -> 算价 -> 落单”三步同步确认，应用层短流程可能够用；但一旦把支付回写、超时取消、预占释放、人工补偿、履约触发都纳入统一治理，流程就已经明显进入编排层问题域了。

5.4 四种主流方案的能力边界

围绕创单场景，最常见的四种实现路线是：

5.4.1 同步 RPC 短流程

做法是由一个入口服务串行调用库存、价格、订单等下游，所有动作尽量在一次同步请求里完成。

优点：

  • 实现简单，链路直观。
  • 对短流程最友好。
  • 不需要额外引入 Orchestrator 或工作流引擎。

缺点：

  • 一旦步骤变多，超时、重试、局部失败会迅速复杂化。
  • 中间状态不清晰，补偿和人工接管困难。
  • 很难优雅覆盖支付回写、超时取消这种长等待场景。

它适合“同步闭环”的短交易前确认，不适合承载完整创单生命周期。

5.4.2 协同式 Saga

做法是各系统通过事件彼此驱动：订单创建后发事件，库存系统消费后再发事件，支付系统收到事件后继续推进，最后由更多消费者完成履约和通知。

优点：

  • 松耦合，系统自治性强。
  • 对异步链路友好。
  • 能顺着领域边界自然扩展。

缺点：

  • 全局流程不容易一眼看清。
  • 补偿逻辑分散在多个系统里。
  • 审计和排障成本偏高。

它适合“多系统异步协作、主控方不那么强”的场景，但对交易主链路来说，可见性往往不够。

5.4.3 编排式 Saga

做法是引入一个 Orchestrator，由它显式推进流程：先预占库存，再创建订单，再发起支付，失败时按策略触发补偿。

优点：

  • 流程可见性强。
  • 补偿逻辑集中，便于治理。
  • 很适合创单、退款、审核发布这类有明确主控方的业务流程。

缺点：

  • 编排器本身会成为复杂系统。
  • 需要认真设计状态持久化、重试、幂等和高可用。

它是很多互联网主交易链路最常见的折中方案。

5.4.4 Workflow / 状态机引擎

当流程已经拥有大量等待点、人工介入、条件分支、重试与回退策略时，可以进一步交给工作流引擎承载。

优点：

  • 状态持久化、超时、重试、补偿、回放能力更系统。
  • 更容易实现可视化、可审计、可人工接管。
  • 对长期运行和复杂分支流程更友好。

缺点：

  • 技术和组织成本更高。
  • 如果流程并不复杂，引入引擎会过度设计。

它更适合“复杂长流程”，而不是每一个普通同步接口。

5.4.5 一张对比表

方案实现复杂度状态可见性补偿能力审计能力适用场景
同步 RPC低低低低步骤少、同步闭环
协同式 Saga中中偏低中中偏低多系统异步协作
编排式 Saga中偏高高高高主链路交易、退款、审核
Workflow 引擎高很高很高很高长流程、强治理、强审计

5.5 大事务系统的核心不是“调接口”，而是“状态推进”

很多失败的大事务系统，都犯了同一个错误：把大事务理解成“把多个下游接口按顺序调完”。这会导致系统看似有流程，实则没有治理能力。

真正可生产的大事务系统，设计对象应该是状态，而不是调用顺序。

以创单为例，比“库存接口调没调成功”更重要的问题是：

  • 当前订单处在“待创建”“待支付”“支付中”“待确认”“待补偿”还是“已关闭”？
  • 当前库存预占是“未预占”“已预占”“待释放”还是“已确认”？
  • 支付状态是“未发起”“发起中”“待回写”“成功”“失败”还是“未知待查单”？

一旦把系统设计成状态推进问题，很多治理动作才有落点：

  • 超时：不是简单重试接口，而是把状态推进到“待补偿 / 待人工确认”。
  • 补偿：不是“反向再调一遍接口”，而是把业务状态回收到允许的上一层状态。
  • 幂等：不是只看请求 ID，而是同一状态迁移不能重复制造副作用。
  • 人工接管：不是 DBA 直接改库，而是通过受控动作推进状态。

所以大事务系统设计时，必须优先回答：

  1. 状态机怎么定义？
  2. 每个状态的进入条件和退出条件是什么？
  3. 超时如何转移？
  4. 失败后由谁补偿？
  5. 什么状态允许人工接管？

这也是为什么很多成熟系统最后都会自然走向“显式状态机 + 编排器 / 工作流”的设计。

5.6 状态机、Saga 与工作流引擎怎么选

这里不要从“喜欢哪个框架”开始，而要从复杂度来源开始。

5.6.1 数据库状态机足够的情况

如果流程满足下面这些特点，数据库状态机通常就够：

  • 步骤有限。
  • 主控服务明确。
  • 等待点不多。
  • 人工介入少。
  • 编排逻辑仍然能被一个服务清晰维护。

典型例子：简单创单、简单退款、短审核流。

5.6.2 应用层 Orchestrator 更合适的情况

当流程已经明显跨系统、需要显式补偿和统一审计，但还没复杂到要引入完整工作流平台时，可以选择编排式 Saga。

典型例子：交易主链路、售后退款主链路、库存预占与回收主链路。

5.6.3 Workflow 引擎值得引入的情况

如果流程有如下特征，引入 Temporal、Camunda、Zeebe 或同类能力会更合理：

  • 长时间等待。
  • 分支复杂。
  • 审批或人工确认节点多。
  • 需要可视化运行视图。
  • 恢复、回放、审计要求高。
  • 普通应用代码已经难以维护流程复杂度。

5.6.4 一个务实的升级路径

建议遵循下面的演进顺序：

  1. 先用同步短流程解决最简单问题。
  2. 补偿和状态边界开始复杂时，升级到状态机 / Saga。
  3. 长等待、人工接管、流程可视化成为常态时，再考虑工作流引擎。

这条路径通常比一开始就引入 BPMN 或复杂工作流平台更符合互联网系统的实际演进。

5.7 从创单扩展到其他业务流程

创单并不是例外，它只是最典型的大事务样本。用同样的视角看，很多流程都属于同一类问题。

5.7.1 用户注册 / 认证流程

注册往往不只是“写一条用户记录”，还会涉及账号创建、风控校验、验证码验证、实名状态更新、欢迎权益发放。只要这些动作跨多个系统推进，它就已经具备大事务特征。

5.7.2 退款 / 售后流程

退款天然比创单更像编排问题：售后申请、审核、退款发起、支付回写、库存 / 权益回补、订单关闭都发生在不同阶段，而且强依赖补偿和审计。

5.7.3 审核 / 发布流程

商品审核、内容审核、供应商同步发布这类流程，虽然不是资金交易，但同样具备多步骤、跨系统、长等待、人工接管的典型特征，往往非常适合编排式 Saga 或工作流引擎。

5.7.4 一个统一视角

这些流程表面业务不同，但底层共性高度一致：

  • 都不是单事务问题，而是阶段推进问题。
  • 都需要显式状态。
  • 都需要超时、补偿和审计。
  • 都需要在简单方案和重型编排器之间做选型。

5.8 方法论沉淀：如何在真实系统中做出选择

到这里可以把整章压缩成一组可复用判断。

5.8.1 五个判断句

  1. 只要一个业务目标跨多个系统和多个本地事务推进，它就可能是大事务问题。
  2. 只要失败后不能简单整体回滚，就必须设计补偿和状态收敛。
  3. 只要流程存在长等待和人工介入，就不要只靠同步 RPC 思维。
  4. 真正决定系统能否生产可用的，不是接口链路，而是状态推进模型。
  5. 选型顺序永远应该是：先解决问题，再决定是否值得引入更重的编排基础设施。

5.8.2 一张务实选型表

场景特征推荐方案
步骤少、同步闭环、失败整体返回即可同步 RPC
跨系统、多步骤、需要补偿，但等待点不多应用层状态机 / 编排式 Saga
多系统自治、事件天然存在、主控方不强协同式 Saga
长等待、强审计、人工接管多、流程复杂Workflow 引擎

5.8.3 面试与评审中的一句话表达

如果要把这一章压缩成一句可直接复用的话，可以这样讲：

互联网系统里的大事务，本质上不是把多个数据库事务绑成一个全局事务，而是把跨系统业务流程建模成可推进、可补偿、可审计的状态机；同步 RPC、Saga 和工作流引擎只是不同复杂度下的实现手段。
