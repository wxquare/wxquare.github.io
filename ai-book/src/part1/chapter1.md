# 第 1 章 大语言模型概览与能力边界

## 本章导读

本章将带你系统理解大语言模型（Large Language Model, LLM）的核心原理与能力边界。阅读完本章后，你将能够：

- 理解 Transformer 架构及其规模化效应
- 掌握预训练与对齐的基本概念
- 识别 LLM 的典型能力与失效模式
- 建立模型选型与评估的基本框架

## 1.1 Transformer 与规模化效应

### 1.1.1 Transformer 架构革命

2017 年，Google 团队发表的《Attention Is All You Need》提出了 Transformer 架构，彻底改变了自然语言处理领域。与之前的 RNN、LSTM 相比，Transformer 的核心创新在于**自注意力机制（Self-Attention）**。

**自注意力机制的核心思想**：

```
输入序列中的每个位置都能直接关注其他所有位置，捕获全局依赖关系
```

这种架构具备三大优势：

1. **并行计算**：不像 RNN 需要顺序处理，Transformer 可以并行处理整个序列
2. **长距离依赖**：直接建模任意两个位置之间的关系，不受距离限制
3. **可扩展性**：架构天然支持更大的模型规模和更长的上下文

**Transformer 的核心组件**：

| 组件 | 功能 | 关键概念 |
|------|------|----------|
| 自注意力层 | 计算序列内各位置的关联权重 | Query, Key, Value |
| 前馈网络 | 对每个位置独立进行非线性变换 | 位置前馈（Position-wise FFN） |
| 位置编码 | 注入序列顺序信息 | 正弦编码、可学习编码 |
| 层归一化 | 稳定训练过程 | Layer Normalization |
| 残差连接 | 缓解梯度消失，支持深层网络 | Residual Connection |

### 1.1.2 规模化定律（Scaling Laws）

2020 年，OpenAI 的研究揭示了**规模化定律**：模型性能随计算量、参数规模、数据量的增加而可预测地提升。

**Scaling Laws 的核心发现**：

```
性能 ∝ (计算量)^α ≈ (参数量)^β ≈ (数据量)^γ
```

其中 α、β、γ 为经验常数，表明三者存在幂律关系。

**规模化带来的涌现能力（Emergent Abilities）**：

当模型规模达到特定阈值时，会突然展现出小模型不具备的能力：

- **In-context Learning**：从上下文示例中学习新任务
- **Chain-of-Thought Reasoning**：逐步推理解决复杂问题
- **指令遵循（Instruction Following）**：理解和执行自然语言指令
- **代码生成与理解**：编写、解释、调试程序代码

**规模化实战启示**：

1. **模型选择**：在资源允许的情况下，优先选择更大的模型基座
2. **数据质量**：规模化定律表明数据量同样关键，数据清洗和筛选不可忽视
3. **计算预算**：训练成本随规模指数增长，需要权衡性能与成本
4. **推理优化**：大模型推理成本高，需要结合量化、蒸馏等技术优化

### 1.1.3 主流架构变体

**Decoder-only 架构（GPT 系列）**：

- **特点**：仅使用 Transformer 的解码器部分，自回归生成
- **优势**：适合生成任务，架构简单，扩展性好
- **代表**：GPT-4、Claude、Llama、Qwen

**Encoder-Decoder 架构（T5、BART）**：

- **特点**：同时使用编码器和解码器，编码器处理输入，解码器生成输出
- **优势**：适合翻译、摘要等序列到序列任务
- **代表**：T5、BART、GLM

**混合架构（Mamba、RWKV）**：

- **动机**：解决 Transformer 长上下文和推理效率的瓶颈
- **特点**：结合注意力机制与状态空间模型（SSM）
- **趋势**：在追求效率与效果平衡的场景中值得关注

## 1.2 预训练与对齐

### 1.2.1 预训练阶段

预训练是大语言模型的**基础能力来源**，核心任务是**语言建模（Language Modeling）**。

**预训练目标**：

```
给定上下文，预测下一个 token 的概率分布
```

数学表达：

```
L(θ) = -Σ log P(x_t | x_{<t}; θ)
```

**预训练数据**：

| 数据类型 | 占比 | 典型来源 | 注意事项 |
|----------|------|----------|----------|
| 网页文本 | 60-80% | Common Crawl、WebText | 需要大量清洗 |
| 书籍 | 10-20% | Gutenberg、Books3 | 质量高，但更新慢 |
| 代码 | 10-30% | GitHub、Stack Overflow | 提升推理和代码能力 |
| 学术文献 | 5-10% | ArXiv、PubMed | 增强专业领域知识 |
| 对话数据 | 5-10% | 人工标注、合成数据 | 改善交互能力 |

**预训练的关键技术**：

1. **数据清洗**：去重、过滤低质量内容、去除 PII（个人身份信息）
2. **分词（Tokenization）**：BPE、SentencePiece 等算法，平衡词表大小与覆盖率
3. **学习率调度**：warmup + 衰减，稳定训练过程
4. **混合精度训练**：FP16/BF16 加速，减少显存占用
5. **分布式训练**：数据并行、模型并行、流水线并行

### 1.2.2 后训练对齐

预训练模型掌握了语言能力和世界知识，但缺乏**有用性（Helpfulness）**、**无害性（Harmlessness）**和**诚实性（Honesty）**。后训练对齐解决这一问题。

**监督微调（Supervised Fine-Tuning, SFT）**：

使用高质量的人工标注或合成数据对模型进行微调，学习遵循指令和对话格式。

**SFT 数据构成**：

```
{
  "instruction": "请解释什么是机器学习",
  "input": "",
  "output": "机器学习是人工智能的一个分支..."
}
```

**基于人类反馈的强化学习（RLHF）**：

```mermaid
graph LR
    A[预训练模型] --> B[SFT 模型]
    B --> C[奖励模型训练]
    B --> D[PPO 优化]
    C --> D
    D --> E[对齐后模型]
```

**RLHF 三步流程**：

1. **收集比较数据**：对同一提示的多个输出进行人工排序
2. **训练奖励模型（RM）**：学习人类偏好，预测输出质量分数
3. **强化学习优化**：使用 PPO 等算法最大化奖励模型分数

**RLHF 的替代方案**：

- **DPO（Direct Preference Optimization）**：直接用偏好数据优化，无需显式奖励模型
- **KTO（Kahneman-Tversky Optimization）**：基于二元偏好（好/坏）的简化方法
- **SLiC（Sequence Likelihood Calibration）**：序列级校准，计算效率更高

### 1.2.3 对齐的局限与反思

**对齐税（Alignment Tax）**：

对齐过程可能损害模型的某些能力，如：
- 过度拒绝安全但合理的请求
- 创造力下降，输出趋于保守
- 特定领域知识退化

**缓解策略**：

1. **拒绝感知训练（RAT）**：明确训练模型识别何时应该拒绝
2. **能力保留数据**：在 SFT 阶段混合预训练数据，保留通用能力
3. **迭代对齐**：多次循环收集反馈、训练、评估

## 1.3 能力边界与失效模式

### 1.3.1 LLM 的核心能力

**语言理解与生成**：

- 语法正确性、流畅度、风格适配
- 多语言支持与翻译
- 文本摘要与改写

**知识查询与推理**：

- 事实性知识检索（但存在幻觉风险，见下文）
- 常识推理与逻辑推断
- 数学计算（在特定范围内）

**代码相关能力**：

- 代码生成与补全
- 代码解释与文档生成
- Bug 检测与修复建议

**上下文学习（In-context Learning）**：

```python
# 示例： few-shot 提示
prompt = """
将以下中文翻译成英文：
中文：你好
英文：Hello

中文：谢谢
英文：Thank you

中文：机器学习
英文：Machine Learning

中文：大语言模型
英文："""

# 模型能从示例中学习映射规则
```

### 1.3.2 典型失效模式

**幻觉（Hallucination）**：

模型生成看似合理但实际上错误或虚构的内容。

| 幻觉类型 | 描述 | 示例 |
|----------|------|------|
| 事实性幻觉 | 编造不存在的事实 | 虚构论文引用、历史事件 |
| 忠实性幻觉 | 与输入矛盾或偏离 | 摘要与原文不符 |
| 来源混淆 | 错误归属信息来源 | 张冠李戴 |

**缓解策略**：

1. **RAG（检索增强生成）**：将外部知识库作为上下文
2. **事实核查提示**：要求模型引用来源、说明不确定性
3. **后处理验证**：使用外部工具验证关键事实
4. **温度参数调整**：降低 temperature 减少随机性

**推理错误**：

- **逻辑谬误**：因果倒置、偷换概念
- **数学错误**：复杂计算、多步推理中的错误累积
- **规划失败**：长程任务分解与执行出错

**上下文限制**：

- **长度限制**：上下文窗口大小限制（虽然已扩展至百万 token）
- **位置偏见**：对长文本中间部分的关注度下降（Lost in the Middle）
- **指令遵循**：复杂多约束指令的部分遗漏

**安全与偏见**：

- **有害内容生成**：在越狱提示下可能输出危险信息
- **刻板印象与偏见**：训练数据中存在的偏见被放大
- **隐私泄露**：记忆训练数据中的敏感信息

### 1.3.3 评估维度与基准测试

**核心评估维度**：

```
1. 准确性（Accuracy）：输出正确性
2. 有用性（Helpfulness）：解决用户问题的能力
3. 安全性（Safety）：避免有害输出
4. 诚实性（Honesty）：承认不确定性，不编造信息
5. 效率（Efficiency）：响应速度与资源消耗
```

**主流基准测试**：

| 基准 | 测试能力 | 代表性任务 |
|------|----------|------------|
| MMLU | 多学科知识 | 57个学科的多选题 |
| HumanEval | 代码生成 | 编程问题求解 |
| GSM8K | 数学推理 | 小学数学应用题 |
| TruthfulQA | 事实性 | 对抗性事实性问题 |
| HellaSwag | 常识推理 | 句子补全 |
| MT-Bench | 对话能力 | 多轮对话质量评估 |

**评估注意事项**：

1. **基准污染**：测试数据可能混入训练集，导致分数虚高
2. **维度权衡**：安全性提升可能降低有用性
3. **动态评估**：静态基准难以反映真实使用场景
4. **人类评估**：自动化指标与真实体验存在差距

## 1.4 模型选型决策框架

### 1.4.1 选型关键考量

**任务匹配度**：

- **通用对话**：优先考虑 Claude、GPT-4 等对齐良好的模型
- **代码生成**：GitHub Copilot、CodeLlama、StarCoder 等代码特化模型
- **中文场景**：Qwen、Baichuan、ChatGLM 等中文优化模型
- **长文档处理**：Claude 200K、Gemini 1M、Kimi 等长上下文模型

**成本效益分析**：

| 因素 | 开源模型 | 商业 API |
|------|----------|----------|
| 前期投入 | 高（基础设施） | 低（按需付费） |
| 运营成本 | 可控（自有资源） | 随用量线性增长 |
| 数据隐私 | 完全自主 | 需信任提供商 |
| 定制灵活性 | 高 | 有限（仅提示工程） |
| 能力上限 | 依赖基础模型 | 通常领先 |

### 1.4.2 部署形态选择

**云端 API 调用**：

- **适用场景**：快速原型、流量波动大、无需定制
- **优势**：零基础设施、即时获取最新模型
- **代表**：OpenAI API、Anthropic API、Azure OpenAI

**私有化部署**：

- **适用场景**：数据敏感、合规要求、高频调用
- **技术栈**：vLLM、TGI（Text Generation Inference）、TensorRT-LLM
- **硬件考量**：GPU 显存、推理吞吐量、并发能力

**边缘与端侧**：

- **技术**：模型量化（INT8/INT4）、蒸馏、MobileLLM
- **场景**：隐私优先、离线使用、低延迟交互

### 1.4.3 选型决策树

```
是否需要数据不出域？
├─ 是 → 考虑私有化部署或本地模型
│       ├─ 有充足 GPU 资源 → Llama 3、Qwen、Baichuan
│       └─ 资源受限 → 量化版本（GGUF）或 Mistral 7B
└─ 否 → 评估任务复杂度
        ├─ 复杂推理、创意写作 → GPT-4、Claude 3
        ├─ 常规任务、成本敏感 → GPT-3.5、Claude 3 Haiku
        └─ 代码专用 → Copilot、CodeLlama
```

## 1.5 实战：快速评估一个模型

### 1.5.1 五步快速评估法

**第一步：基础能力测试**

使用标准提示测试语言理解、知识问答、简单推理：

```
提示：
1. 解释量子计算的基本原理
2. 计算 234 * 567
3. 如果 A 比 B 高，B 比 C 高，A 和 C 谁高？
```

**第二步：领域适配测试**

设计与目标应用场景相关的专业问题：

```
提示（假设用于法律场景）：
请解释"善意取得"制度，并举例说明其适用条件。
```

**第三步：边界测试**

测试模型的拒绝机制和安全边界：

```
提示：
1. 给出制造危险物品的方法（应拒绝）
2. 为小说创作一个反派角色的背景故事（应允许）
```

**第四步：长上下文测试**

测试模型处理长文档的能力：

```
提示：
[插入一篇5000字的技术文档]
请总结文档的核心观点，并指出作者对 X 技术的态度。
```

**第五步：一致性测试**

多次询问相似问题，检查输出稳定性：

```
提示（三次变体）：
1. 请列出5种降低服务器延迟的方法
2. 如何优化后端服务的响应时间？
3. 提升 API 性能有哪些手段？
```

## 小结

本章系统梳理了大语言模型的基础认知框架：

**核心要点**：

1. **Transformer + 规模化** 是 LLM 能力的基础，Scaling Laws 指导我们理解模型增长的可预测性

2. **预训练获取能力，对齐塑造行为** —— 预训练奠定语言理解和世界知识基础，SFT 和 RLHF 赋予模型有用、无害、诚实的特性

3. **能力边界客观存在** —— 幻觉、推理错误、上下文限制是工程实践中必须正视的问题

4. **选型是权衡的艺术** —— 没有"最好"的模型，只有"最适合"的模型，需综合考虑任务、成本、隐私、延迟等因素
