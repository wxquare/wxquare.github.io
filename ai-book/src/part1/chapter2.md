# 第2章 Claude Code：终端原生的AI Agent

> "使用Opus 4.5后就再也没有手写过一行代码，47天里有46天都在使用。" —— Boris Cherny, Claude Code创建者

## 引言

AI编程工具在三年内经历了三次重大变革：从 GitHub Copilot 的代码补全，到 Cursor 的对话式编程，再到 Claude Code 的终端 Agent 模式。这不仅是技术的进步，更是人与 AI 协作关系的根本转变——你的角色从"写代码的人"变成了"给指令的人"。

本章将深入介绍 Claude Code 的革命性特性、核心工作模式，以及如何通过 Skills、Hooks、MCP 等扩展机制构建高效的 AI 协作环境。

## 2.1 Claude Code的革命性变化

### 什么是 Claude Code

Claude Code 是 Anthropic 推出的 AI 编程助手，与传统 IDE 集成的 AI 工具不同，它直接在终端运行，能够自主规划步骤、读写代码、执行命令、操作 git，完成完整的开发循环。

**核心差异：**

| 维度 | 传统 AI 工具 | Claude Code |
|------|------------|-------------|
| **运行环境** | IDE 插件 | 终端原生 |
| **自主程度** | 需要持续监督 | 可完全无人值守运行 |
| **记忆系统** | 隐式上下文 | 显式的 CLAUDE.md 文件 |
| **并行能力** | 单一实例 | 原生支持多实例并行 |
| **工具调用** | 有限的预定义工具 | 可访问所有命令行工具 |

### 为什么是终端

终端是所有开发者的共同界面，不依赖特定 IDE。Claude Code 在终端中运行意味着：
- 可以直接执行任何命令行工具
- 不受 IDE 限制，适用于任何开发场景
- 易于集成到 CI/CD 流程
- 支持远程服务器开发（SSH）

### TAOR 循环：Think-Act-Observe-Repeat

Claude Code 的核心工作循环：

```text
Think（分析当前状态，决定下一步）
  ↓
Act（调用工具，执行操作）
  ↓
Observe（读取返回结果，评估是否完成）
  ↓
Repeat（未完成则继续循环）
```

这解释了为什么 Claude 有时候要"绕几步路"才到终点。它不是在执行预设的脚本，而是在实时做决策。每做一步，都要重新观察结果、重新判断下一步该做什么。

## 2.2 进阶对话技巧：让AI真正理解你

### 具体化原则：三要素缺一不可

**1. 指定文件和路径**

❌ 错误：
```text
做个登录功能
```

✅ 正确：
```text
在`src/auth/`目录下新增Google OAuth登录，用Better Auth库，
参考现有的GitHub登录实现方式
```

**2. 指向已有模式**

项目里已有写得好的代码就是最好的范本。

```text
看`src/components/UserWidget.tsx`的实现方式，照着做一个CalendarWidget
```

**3. 描述症状而非原因**

❌ 错误：
```text
token刷新逻辑有问题
```

✅ 正确：
```text
用户在session超时后登录失败，请检查`src/auth/`下的token刷新流程
```

### 让 Claude 采访你

对于复杂功能，不要一上来就写需求文档。先让 Claude 采访你：

```text
我想做一个支付功能，在动手之前，先采访我，
问清楚所有你需要知道的事情。
```

Claude 会问：
- 支持哪些支付方式？
- 需要处理退款吗？
- 并发量预估多少？
- 是否需要分布式事务？

这些问题中至少有一半是你自己没考虑过的。

**关键技巧**：采访结束后，让 Claude 整理成 Spec，然后**开新会话**执行，避免采访过程的对话历史占用上下文。

### Context Engineering：信息不是越多越好

上下文太多，模型表现反而变差。核心原则是**给对的信息，而不是所有信息**：

- **用 `@` 引用特定文件**：`@src/utils/auth.ts` 而不是"看看 auth 相关文件"
- **粘贴截图说明 UI 问题**：一张截图比 100 字描述更准确
- **直接 pipe 数据**：`cat error.log | claude` 比复制粘贴更高效
- **给 URL 让 Claude 读取**：Claude 可以直接抓取网页内容

### Effort 级别：别省这个钱

Claude Code 有四个 effort 级别（Low/Medium/High/Max）。Boris 的做法是**从不把它调低**。

理由很简单：
- Low 级别做错了，你纠正它花的时间可能比直接用 High 做对还长
- High 级别让 Claude 想得更深，需要返工的次数更少
- 总体效率反而更高

## 2.3 Plan模式：先想清楚再动手

### Plan 模式的价值

Plan 模式让 Claude 只规划不执行，你可以在这个阶段反复讨论方案、调整细节。

**黄金工作流：**
1. Plan 模式下描述需求，来回讨论
2. 用编辑器（Ctrl+G）写详细的执行指令
3. 切换到执行模式，开启 Auto-accept

这个流程的精髓在于：**把纠结放在 Plan 阶段解决完，执行阶段一气呵成**。边做边改、反复返工是最浪费 tokens 的用法。

### Plan 模式实战

**场景：** 实现一个订单管理系统

**第 1 步：启动 Plan 模式**

```bash
> /plan
> 我需要实现一个订单管理系统，包括：
> - 订单创建
> - 状态流转
> - 订单查询
> 
> 请先分析需求，给出技术方案
```

**第 2 步：Claude 提出问题**

Claude 会在 Plan 模式下提问：
- 订单状态有哪些？状态之间如何转换？
- 需要支持多少并发？
- 是否需要分库分表？
- 支付集成用什么方案？

**第 3 步：确定方案**

经过几轮讨论后，Claude 给出技术方案：
- 数据模型设计
- API 接口定义
- 状态机设计
- 技术栈选择

**第 4 步：切换到执行模式**

```bash
> /execute
> 按照刚才的方案开始实现
```

### Plan 模式的最佳实践

**1. 在 Plan 阶段充分讨论**
- 探讨多种技术方案
- 权衡各方案的优缺点
- 确定架构和技术栈

**2. 用编辑器写执行指令**

Plan 阶段结束后，用 Ctrl+G 打开编辑器，写一份详细的执行指令：

```markdown
按照以下步骤实现订单管理系统：

1. 创建数据模型（internal/model/order.go）
2. 实现数据访问层（internal/repository/order_repo.go）
3. 实现业务逻辑层（internal/service/order_service.go）
4. 实现 API 层（internal/api/order_handler.go）
5. 编写单元测试（覆盖率 ≥ 80%）
6. 编写集成测试

每完成一个步骤暂停，等我确认后再继续。
```

**3. 开启 Auto-accept 批量执行**

确认计划后，切换到执行模式并开启 Auto-accept，让 Claude 一气呵成地完成所有任务。

## 2.4 Auto模式与安全防护

### Auto 模式的工作原理

Auto 模式通过 AI 分类器替你做权限判断，安全操作自动放行，危险操作才拦截。

它有两层防御：
- **输入层**：Prompt Injection 探测器扫描所有内容
- **输出层**：Transcript 分类器评估每个操作的风险（两阶段：快速判断+深度推理）

### Auto 模式会拦截的典型场景

**1. 范围升级**

你说：
```text
清理旧分支
```

Claude 的实际操作：
```bash
git branch -D feature-old    # 删除本地分支 ✅
git push origin --delete feature-old  # 删除远程分支 ❌ 被拦截
```

**2. 凭证探索**

你说：
```text
调用生产环境的API
```

Claude 遇到认证错误后：
```bash
# 正常：读取环境变量
echo $API_TOKEN

# 危险：开始自行搜索其他token ❌ 被拦截
find ~ -name "*token*" -o -name "*credential*"
```

**3. 绕过安全检查**

部署预检失败：
```bash
./deploy.sh  # 失败：pre-check不通过

# Claude尝试绕过 ❌ 被拦截
./deploy.sh --skip-verify
```

**4. 数据外泄**

Claude 想分享代码：
```bash
# 危险：创建公开Gist ❌ 被拦截
gh gist create --public sensitive-code.go
```

### Auto 模式的最佳实践

**1. 信任但验证**

Auto 模式可以大幅提高效率，但不是完全免检：
- 重要操作前先看 Claude 的计划
- 敏感文件修改要人工确认
- git push 前先看 diff

**2. 配置白名单**

在 `CLAUDE.md` 中定义哪些操作可以自动放行：

```markdown
## Auto 模式配置

### 自动放行的操作
- 读取项目内的所有文件
- 运行测试（npm test, go test）
- 格式化代码（eslint --fix, gofmt）
- 本地分支操作（git checkout, git branch）

### 需要确认的操作
- 删除文件
- 修改配置文件
- 推送到远程仓库
- 执行数据库迁移
```

**3. 定期审查日志**

Claude Code 会记录所有操作日志，定期审查可以发现潜在问题：

```bash
# 查看最近的操作日志
claude logs --recent 100
```

## 2.5 CLAUDE.md：Agent的宪法

### 什么是 CLAUDE.md

CLAUDE.md 是 Claude Code 每次启动时自动读取的配置文件，被称为 agent 的"宪法"。它告诉 Claude：
- 这个项目是做什么的
- 使用什么技术栈
- 有哪些架构约束
- 有哪些编码规范
- 如何处理错误

### CLAUDE.md 的三大原则

**原则 1：从护栏开始**

不要一上来就写百科全书。初始版本可以只有 20-30 行，每次 Claude 犯错就加一条规则。

**初始版本：**
```markdown
# 项目规范

## 技术栈
- Go 1.21+
- PostgreSQL 14

## 代码规范
- 错误处理不能忽略
- 公开函数要有文档注释
```

**犯错后补充：**
```markdown
## 错误处理（新增）
- 使用 pkg/errors 包装错误
- 错误信息要包含上下文
```

**原则 2：保持精简**

Boris 团队的 CLAUDE.md 只有约 2500 tokens（100 行左右）。太长的文档：
- 会消耗上下文窗口
- Claude 反而抓不住重点
- 维护成本高

**原则 3：写对的内容**

- Claude 能从代码读出来的不要写（如"这个项目用 Go"）
- Claude 猜不到的必须写（如"金额用整数存储，单位是分"）

### CLAUDE.md 层级结构

```text
~/.claude/CLAUDE.md          # 全局级：所有项目共用的偏好
./CLAUDE.md                  # 项目级：检入git，团队共享
./src/CLAUDE.md              # 子目录级：monorepo中特定模块的规则
```

**全局级示例：** `~/.claude/CLAUDE.md`

```markdown
# 我的全局开发偏好

## 代码风格
- 倾向于函数式编程风格
- 避免过长的函数（> 50 行）
- 变量命名要清晰，不要用缩写

## 注释风格
- 只在必要时添加注释
- 注释解释"为什么"，不解释"是什么"

## 测试偏好
- 优先使用表驱动测试
- 测试覆盖率 ≥ 80%
```

**项目级示例：** `./CLAUDE.md`

```markdown
# 电商后端项目规范

## 项目概述
基于 Go 的电商后端系统，微服务架构。

## 技术栈
- Go 1.21+
- PostgreSQL 14
- Redis 7

## 架构约束
- 金额用整数（分）存储
- 时间使用 UTC 时区
- 数据库操作必须在事务中

## 功能规范目录
详细规范在 specs/ 目录：
- specs/inventory-deduction.md
- specs/payment-integration.md
```

### CLAUDE.md 的迭代飞轮

```text
Claude犯错
  ↓
记录到CLAUDE.md
  ↓
下次不再犯
  ↓
错误率持续降低
```

**示例：**

第 1 次：Claude 提交代码时没有运行测试  
→ 在 CLAUDE.md 加规则："提交前必须运行测试"

第 2 次：Claude 运行测试了，但测试失败了仍然提交  
→ 更新规则："测试通过后才能提交"

第 3 次：Claude 运行测试、确认通过、然后提交 ✅

## 2.6 会话管理与上下文优化

### 核心命令速查

| 命令 | 作用 | 使用场景 |
|------|------|----------|
| `/clear` | 清空当前会话 | 切换到完全不相关的任务 |
| `/compact` | 压缩上下文 | 对话很长但还想继续当前任务 |
| `/btw` | 侧链提问 | 临时问个问题，不污染主任务上下文 |
| `Esc × 2` | Rewind 回滚 | 撤销最近的对话或代码修改 |

### 何时该用 /clear

**场景：** 修完 API bug → `/clear` → 开始前端组件任务

如果不 clear，Claude 的上下文里还残留着大量关于那个 API bug 的信息，会干扰它对新任务的理解。

**判断标准：**
- 任务完全无关 → `/clear`
- 同一个任务的不同阶段 → `/compact`
- 临时问题 → `/btw`

### 上下文压缩的代价

长对话中 Claude 会压缩上下文来节省 token。压缩是**有损的**：
- 核心信息会保留
- 具体措辞、边角细节、语气暗示容易丢失

**应对策略：**

重要的约束和要求，**写进 CLAUDE.md 而不是只在对话里说一次**。对话会被压缩，但 CLAUDE.md 每次都会重新读取。

### Rewind 回滚

`Esc × 2` 可以回滚最近的操作：
- 回滚对话（Claude 的最后一次回复）
- 回滚代码修改
- 两者都回滚

**使用场景：**
- Claude 改错了文件
- Claude 误解了你的意思
- 想换个方向重新来

## 2.7 完整工作流案例

### 案例：实现一个支付集成服务

让我们通过一个完整案例，展示 Claude Code 的工作流。

**第 1 步：准备 CLAUDE.md**

确保项目根目录有 `CLAUDE.md`，包含基本的项目信息和规范。

**第 2 步：让 Claude 采访你**

```bash
> 我想实现支付宝支付集成。在开始之前，先采访我，
> 问清楚所有技术细节和业务要求。
```

Claude 会问约 10 个问题，帮你理清需求。

**第 3 步：生成规范文档**

```bash
> 根据刚才的采访内容，生成一份完整的规范文档，
> 保存到 specs/alipay-integration.md
```

**第 4 步：切换到 Plan 模式**

```bash
> /plan
> 根据 specs/alipay-integration.md 规范，制定实施计划
```

Claude 生成详细的任务拆解和验收标准。

**第 5 步：确认计划**

review 计划后：

```bash
> 计划看起来不错，准备开始执行。
> 切换到执行模式，每完成一个任务暂停让我确认。
```

**第 6 步：逐步实现**

Claude 按照计划逐个实现任务，每完成一个就暂停：

```text
✅ 任务 1 完成：数据模型已创建
文件：internal/model/payment.go

请review，确认无误后我继续任务 2。
```

你可以：
```bash
> 看起来不错，继续任务 2
```

或者：
```bash
> PaymentOrder 结构体缺少 RefundAmount 字段，请补充后再继续
```

**第 7 步：验证**

所有任务完成后：

```bash
> 运行完整的测试套件，确认所有测试通过
```

Claude 执行：
```bash
go test ./... -v -cover
```

**第 8 步：规范验证**

```bash
> 审查实现是否完全符合 specs/alipay-integration.md：
> 1. 列出规范中的所有要求
> 2. 逐条检查实现情况
> 3. 列出未实现或不符合的部分
```

Claude 生成详细的检查报告。

**第 9 步：提交代码**

```bash
> 生成 commit message 并提交代码
```

### 工作流关键要点

**1. 规范先行**
- 先写规范，再写代码
- 规范是 source of truth

**2. 分步执行**
- 不要一次性要求完成所有功能
- 每完成一个模块就验证

**3. 频繁验证**
- 不要等所有代码写完再测试
- 每个小功能都要立即验证

**4. 规范验证**
- 实现完成后，对照规范逐项检查
- 确保没有遗漏

**5. 及时清理上下文**
- 一个大任务完成后 `/clear`
- 避免上下文污染

## 本章小结

### 核心要点回顾

**1. Claude Code 的革命性变化**
- 终端原生运行，不依赖 IDE
- 可完全无人值守运行
- 原生支持多实例并行
- TAOR 循环：Think-Act-Observe-Repeat

**2. 进阶对话技巧**
- 具体化原则：指定文件、指向模式、描述症状
- 让 Claude 采访你，补全需求
- Context Engineering：给对的信息，不是所有信息
- Effort 级别设置为 High/Max

**3. Plan 模式**
- 只规划不执行，充分讨论方案
- 黄金工作流：Plan → 写执行指令 → Execute + Auto-accept
- 把纠结放在 Plan 阶段，执行阶段一气呵成

**4. Auto 模式**
- AI 分类器做权限判断
- 两层防御：输入层 + 输出层
- 拦截范围升级、凭证探索、安全绕过、数据外泄
- 配置白名单和黑名单

**5. CLAUDE.md**
- Agent 的"宪法"，启动时自动加载
- 三大原则：从护栏开始、保持精简、写对内容
- 层级结构：全局级 → 项目级 → 子目录级
- 迭代飞轮：犯错 → 记录 → 改进

**6. 会话管理**
- `/clear`：切换不相关任务
- `/compact`：压缩上下文
- `/btw`：侧链提问
- `Esc × 2`：回滚操作
- 重要约束写进 CLAUDE.md

**7. 完整工作流**
- 采访 → 规范 → Plan → 执行 → 验证 → 提交
- 规范先行，分步执行，频繁验证

### 扩展能力预告

Claude Code 还支持三种强大的扩展机制：
- **Skills**：可复用的工作流包
- **Hooks**：强制执行的自动化流程
- **MCP**：连接外部数据源和服务

这些扩展机制将在第3章 Harness Engineering 中详细介绍。

- [ ] 确认 CLAUDE.md 已更新
- [ ] 准备好功能规范文档
- [ ] 清空上下文（如需要）
