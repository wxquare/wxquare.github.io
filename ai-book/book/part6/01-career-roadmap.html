AI工程实践：从编程到Agent的完整指南

第22章 Agent 应用工程师能力地图与 30 天冲刺计划

面试不是把所有概念背下来，而是证明你能把模型、工具、数据、评估和业务闭环组合成一个可靠系统。

引言

前面四部分已经覆盖了 AI 编程、Agent 架构、完整案例和基础理论。本章开始进入面试实战：如果目标是在一个月后面试 Agent 应用工程师，你需要把这些内容重新组织成一张能力地图，并把每天的复习动作落实到可展示的项目和可讲清楚的设计决策上。

Agent 应用工程师通常不是被期待训练基础模型的人，而是被期待把 LLM 变成真实业务能力的人。面试重点会落在：需求是否判断准确、架构是否可控、工具是否安全、RAG 是否可评估、上线后是否能观测和迭代。

12.1 岗位能力地图

Agent 应用工程师的能力可以拆成八个层次：

业务问题理解
    ↓
LLM 能力边界
    ↓
RAG 与上下文工程
    ↓
工具调用与 MCP
    ↓
Agent 工作流设计
    ↓
Guardrails 与安全
    ↓
Evals 与可观测性
    ↓
生产部署与持续优化

1. 业务问题理解

面试官首先关心的不是你会不会用某个框架，而是你能不能判断问题是否适合 Agent。

你需要能回答：

  • 这个问题为什么不能只用传统后端、规则引擎或普通 RAG？
  • 哪些步骤需要推理，哪些步骤应该保持确定性？
  • 业务成功指标是什么？
  • 哪些错误可以容忍，哪些错误必须拦截？

2. LLM 能力边界

你需要理解模型适合做什么、不适合做什么：

  • 适合：意图理解、摘要、分类、生成、推理、代码辅助；
  • 不适合：精确计算、实时事实、强一致决策、无工具的权限判断；
  • 必须补齐：检索、工具、验证、约束、人工确认。

面试表达模板：

我不会让模型直接做最终事实判断。
我会让模型负责理解和规划，把事实查询、计算、权限校验交给工具，
最后通过评估和 guardrails 控制输出质量。

3. RAG 与上下文工程

RAG 不是“把文档塞给模型”，而是一个检索、压缩、排序和引用的系统。

你需要掌握：

  • 文档切分策略；
  • embedding 与混合检索；
  • metadata filter；
  • rerank；
  • query expansion；
  • 上下文预算；
  • citation 与可追溯性；
  • RAG eval 指标。

4. 工具调用与 MCP

Agent 的能力边界由工具决定。工具设计不好，模型再强也会不稳定。

你需要能解释：

  • function calling 的 schema 如何设计；
  • 工具参数如何校验；
  • 工具错误如何结构化返回；
  • 工具如何做幂等、重试、超时和审计；
  • MCP 如何把外部系统统一接入；
  • 哪些工具必须加人工确认。

5. Agent 工作流设计

面试常见误区是把所有系统都设计成“一个大 Agent 自由循环”。更成熟的做法是把不确定性限制在必要区域。

常见模式：

模式适用场景风险
Chain固定步骤任务灵活性低
Router多意图分流路由错误
ReACT需要边查边推理循环和工具误用
Plan-and-Execute复杂任务拆解计划漂移
State Machine + Agent生产流程设计成本更高
Multi-Agent角色清晰的复杂协作协调成本高

6. Guardrails 与安全

Agent 上线后最危险的部分不是“回答错”，而是“做错事”。

你需要能覆盖：

  • prompt injection；
  • 越权工具调用；
  • 敏感信息泄露；
  • 高风险操作审批；
  • 输出合规；
  • 工具白名单；
  • 审计日志；
  • 回滚能力。

7. Evals 与可观测性

Agent 系统不能只靠主观体验判断好坏。你需要用数据证明系统有效。

核心问题：

  • 离线怎么测？
  • 上线后怎么测？
  • 出错后怎么复盘？
  • 新版本是否比旧版本更好？
  • 成本、延迟和质量如何权衡？

8. 生产部署与持续优化

面试高级信号来自生产化意识：

  • 异步任务与队列；
  • 限流、熔断、降级；
  • trace 与 replay；
  • 模型版本管理；
  • prompt 版本管理；
  • eval dataset 版本管理；
  • 成本预算和报警。

12.2 面试中的四类问题

概念题

典型问题：

  • Agent 和 workflow 有什么区别？
  • ReACT 和 Plan-and-Execute 怎么选？
  • RAG 为什么会幻觉？
  • MCP 解决了什么问题？

回答方法：先定义，再讲边界，最后讲工程化处理。

系统设计题

典型问题：

  • 设计一个企业知识库问答 Agent；
  • 设计一个客服工单处理 Agent；
  • 设计一个代码审查 Agent；
  • 设计一个生产告警诊断 Agent。

回答方法：需求澄清 → 选型判断 → 架构 → 工具/RAG → 安全 → 评估 → 观测 → 迭代。

项目深挖题

典型问题：

  • 你做过的 Agent 项目里最难的问题是什么？
  • 准确率怎么评估？
  • 上线后出现过什么失败？
  • 成本怎么优化？

回答方法：不要只讲“用了什么技术”，要讲“为什么这么设计”和“如何证明有效”。

Coding / 实现题

典型问题：

  • 写一个 function calling tool；
  • 实现一个简单 ReACT loop；
  • 设计一个 eval runner；
  • 给工具调用加权限校验。

回答方法：代码可以简单，但边界要完整：输入、输出、错误、超时、日志、测试。

12.3 30天冲刺计划

第1周：补齐基础与主线

目标：能清楚解释 Agent 应用工程的全貌。

  • 第 1 天：复习第 10 章，整理 LLM 能力边界；
  • 第 2 天：复习第 11 章，画出 RAG 流程图；
  • 第 3 天：复习第 4 章，背熟 Agent 选型决策树；
  • 第 4 天：复习第 5 章，整理工具设计原则；
  • 第 5 天：复习第 7 章，整理可观测性指标；
  • 第 6 天：读第 8 章，把 DoD Agent 讲成 5 分钟项目；
  • 第 7 天：做一次完整 mock，总结卡壳问题。

第2周：做一个可讲的 Mini Agent

目标：有一个能展示架构和代码的项目。

  • 搭建 API 层、Agent runner、Tool registry；
  • 接入一个简单 RAG；
  • 增加 trace log；
  • 增加 10 条 eval case；
  • 增加输入/输出 guardrail；
  • 写 README 和架构图。

第3周：系统设计专项

目标：能稳定回答 Agent 系统设计题。

每天练 1 个题目，固定使用这个框架：

1. 业务目标
2. 成功指标
3. 为什么需要 Agent
4. 核心架构
5. RAG 设计
6. 工具设计
7. Guardrails
8. Evals
9. Observability
10. 失败模式

第4周：项目表达与查漏补缺

目标：把知识转成面试表达。

  • 准备 3 分钟项目介绍；
  • 准备 10 分钟系统设计讲解；
  • 准备 5 个失败案例；
  • 准备 5 个指标数据；
  • 准备 3 个技术取舍；
  • 每天做一次录音复盘。

12.4 面试回答的标准结构

回答 Agent 问题时，可以使用“6 层回答法”：

业务目标 → 架构模式 → 数据与工具 → 安全边界 → 评估指标 → 迭代闭环

示例问题：如何设计一个客服 Agent？

业务目标：
提升客服自动解决率，同时控制错误回答和越权操作风险。

架构模式：
我会采用 Router + RAG + Tool Calling + Human-in-the-loop。
常见问题走 RAG，订单/退款类问题走工具查询，高风险退款操作需要人工确认。

数据与工具：
知识库包括 FAQ、政策文档和历史工单；工具包括订单查询、退款状态查询、工单创建。

安全边界：
输入侧做越权和 prompt injection 检测；工具侧按用户身份校验；输出侧检查是否引用政策来源。

评估指标：
看自动解决率、人工升级率、答案引用准确率、用户满意度、平均处理时长。

迭代闭环：
把失败 trace 转成 eval case，每周回归测试 prompt、retriever 和工具 schema。

12.5 JD导向的准备清单：AI Agent 后端开发

如果目标岗位写着“AI Agent 后端开发”，面试重点通常不只在 LLM 概念，而是看你能否把 Agent 系统放进真实后端工程里运行。可以把 JD 拆成六类能力来准备：

JD关键词面试官真正关心你要准备的证据
全生命周期开发是否做过从架构到上线的闭环需求澄清、架构图、灰度方案、上线监控
LLM / RAG / MCP是否理解核心模块边界RAG 流程、工具协议、权限校验、失败模式
GO / Python是否能落到工程实现后端服务、异步任务、工具封装、eval runner
LangChain / LangGraph是否会用框架也能跳出框架为什么用框架、哪些模块自研、如何避免框架绑架
多 Agent 协作是否能处理复杂任务编排Router、状态机、Coordinator、任务队列、结果合并
监控与评估体系是否有生产化意识trace schema、离线 eval、线上指标、回归集

1. 项目表达要从“用了 Agent”升级为“系统如何闭环”

不要只说：

我做了一个 RAG + Agent 系统。

更好的表达是：

我把系统拆成 API 层、Agent Runner、RAG Retriever、Tool Registry、Trace 和 Eval 六个模块。
模型负责意图理解和步骤规划，事实查询通过工具完成，高风险动作进入人工确认。
上线后通过 trace 采样和 eval 回归持续优化 prompt、检索和工具 schema。

2. 技术栈回答要体现取舍

如果被问“你熟悉 LangChain / LangGraph 吗”，不要只回答“用过”。可以这样答：

我会把 LangChain 看作工具调用、prompt template 和 RAG pipeline 的工程脚手架。
如果流程需要显式状态、分支和可恢复执行，我会更倾向 LangGraph 或自研状态机。
但生产系统里不会把业务状态只放在框架内存里，关键状态需要落库，方便重试、审计和恢复。

3. 后端能力要主动露出来

AI Agent 岗位容易让候选人陷入“模型话术”，但后端岗位会继续追问：

  • 工具调用超时怎么办？
  • Agent 任务如何异步化？
  • 多轮会话状态放哪里？
  • RAG 索引如何增量更新？
  • 工具调用如何做幂等？
  • trace 数据量很大怎么存？
  • eval 用例如何版本化？

这些问题的答案要回到传统工程能力：队列、状态机、数据库事务、幂等键、限流、熔断、审计、灰度和回滚。

12.6 自测清单

面试前，你应该能不用查资料回答以下问题：

  • 什么场景不应该用 Agent？
  • 一个生产级 Agent 至少需要哪些模块？
  • RAG 的召回率和回答忠实度分别怎么测？
  • 工具调用失败时 Agent 应该怎么处理？
  • 如何防止 Agent 泄露敏感信息？
  • 多 Agent 一定比单 Agent 好吗？
  • 如何设计一个 Agent 的 tracing schema？
  • 如何把生产失败样本变成 eval dataset？
  • 如何向业务方解释 Agent 的 ROI？

本章小结

Agent 应用工程师的核心竞争力，不是记住多少框架名，而是能把开放式模型能力收敛进可控的工程系统。

一个月冲刺的重点应该放在三件事上：

  1. 建立完整能力地图；
  2. 做出一个可讲、可演示、可评估的 Mini Agent；
  3. 用系统设计题反复训练表达结构。
