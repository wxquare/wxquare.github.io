```markdown
#### 📊 题目1：如何设计一个中大型电商平台的服务拆分边界？

**问题描述**：
假设你接手一个日订单50万的电商平台，目前是单体应用，团队规模100人。现在需要进行微服务化改造。请设计服务拆分方案。

**答案**：

**问题分析**：
服务拆分的核心挑战在于：
1. 既要保证领域边界清晰（DDD视角）
2. 又要考虑团队规模和协作效率
3. 还要兼顾性能和一致性要求
4. 需要平衡拆分粒度和运维复杂度

**方案一：按业务能力垂直拆分**

核心服务划分：
- **核心交易域**：订单、支付、结算、购物车（4个服务）
- **商品供给域**：商品中心、库存、计价、营销（4个服务）
- **用户导购域**：搜索、推荐、用户中心（3个服务）
- **运营支撑域**：商品上架、B端运营、数据分析（3个服务）

拆分原则：
1. 每个服务对应一个限界上下文（Bounded Context）
2. 服务间通过稳定的API契约通信
3. 核心链路服务优先拆分，支撑域可暂时保留

集成模式：
- 同步调用：购物车→计价、订单→库存
- 异步事件：订单创建→履约、商品变更→搜索索引
- 读写分离：搜索独立构建读模型

优点：
- 团队自治性强，可并行开发
- 业务边界清晰，易于理解和维护
- 符合DDD最佳实践
- 故障隔离效果好

缺点：
- 需要处理分布式事务
- 服务间调用增加网络开销
- 初期实施复杂度较高
- 需要完善的基础设施支持

**方案二：按技术特征水平拆分**

划分：
- **高并发读服务**：商品详情、搜索、列表（使用缓存和ES）
- **强一致写服务**：订单、支付、库存扣减（使用分布式事务）
- **异步处理服务**：消息通知、数据同步、报表生成

优点：
- 技术栈统一，便于基础设施复用
- 性能优化方向明确
- 团队技能要求更聚焦

缺点：
- 业务边界模糊，团队协作困难
- 不符合微服务自治原则
- 业务变更可能需要跨多个服务修改
- 数据所有权不清晰

**方案三：绞杀者模式渐进拆分**

实施步骤：
1. **第一阶段**：拆出搜索（读多写少，影响面小，独立技术栈）
2. **第二阶段**：拆商品中心（依赖少，数据模型清晰）
3. **第三阶段**：拆订单链路（核心但复杂，需要Saga编排）
4. **第四阶段**：处理遗留单体（逐步清理剩余功能）

每个阶段：
- 使用特性开关（Feature Toggle）控制流量切换
- 双写模式保证数据一致性
- 保留回滚能力
- 完整的监控和告警

优点：
- 风险可控，可持续交付
- 团队学习曲线平缓
- 每个阶段都有明确产出
- 可以根据业务优先级调整顺序

缺点：
- 迁移周期较长（可能需要6-12个月）
- 中间状态维护成本高
- 需要维护双写逻辑
- 技术债累积风险

**方案对比**：

| 维度 | 方案一（垂直） | 方案二（水平） | 方案三（渐进） |
|------|---------------|---------------|---------------|
| 团队自治 | ★★★★★ | ★★☆☆☆ | ★★★☆☆ |
| 技术复杂度 | ★★★☆☆ | ★★★★☆ | ★★☆☆☆ |
| 交付速度 | ★★★☆☆ | ★★★★☆ | ★★★★☆ |
| 长期维护 | ★★★★★ | ★★☆☆☆ | ★★★★☆ |
| 业务隔离 | ★★★★★ | ★★☆☆☆ | ★★★☆☆ |
| 实施风险 | ★★★☆☆ | ★★☆☆☆ | ★★★★☆ |

**推荐方案**：
采用**方案一+方案三的结合**：按领域垂直划分目标架构，采用绞杀者模式渐进实施。

实施要点：
1. **前期准备**：梳理系统依赖图，识别核心路径和边界
2. **基础设施先行**：建立服务网格、监控、链路追踪、配置中心
3. **服务契约规范**：定义API标准、版本管理、错误码体系
4. **数据迁移策略**：双写+对账+延迟删除，确保数据完整性
5. **灰度发布机制**：按用户维度或地域逐步切流量
6. **应急预案**：每个服务都有降级开关和回滚方案

分批次拆分建议：
- **第一批**（1-2个月）：搜索服务（读多写少，技术栈独立）
- **第二批**（2-3个月）：商品中心+库存（供给侧核心）
- **第三批**（3-4个月）：订单+支付（交易核心，需要Saga）
- **第四批**（2-3个月）：营销+计价+购物车（支撑服务）

**延伸思考**：
1. **如何处理拆分过程中的数据迁移？**
   - 采用双写模式（同时写老库和新库）
   - 建立数据对账机制
   - 逐步切读流量
   - 保留一定周期后清理老数据

2. **分布式事务如何保证？**
   - 核心链路采用Saga模式（订单创建、支付、库存扣减）
   - 非核心场景采用最终一致性+补偿
   - 建立完善的监控和对账机制

3. **服务间调用的超时和重试策略如何设计？**
   - 根据SLA设置合理超时（如订单创建3s，查询500ms）
   - 幂等性保证：使用业务唯一键去重
   - 重试策略：指数退避，最多重试3次
   - 熔断降级：连续失败达阈值后熔断

#### 🔧 题目2：如何设计电商系统的数据一致性方案？

**问题描述**：
电商系统中，订单创建涉及库存扣减、优惠券核销、积分扣除等多个操作，这些操作分散在不同的微服务中。如何保证数据一致性？

**答案**：

**问题分析**：
数据一致性的核心挑战：
1. 多个微服务涉及写操作，无法使用传统的数据库事务
2. 部分操作可能失败，需要补偿机制
3. 性能要求高，不能因为一致性牺牲太多性能
4. 需要考虑系统可用性，不能因为一个服务故障导致整体不可用

**方案一：Saga模式（编排式）**

设计思路：
由订单服务作为编排器（Orchestrator），协调各个服务的操作。

流程：
```
1. 订单服务：创建订单（状态：PENDING）
2. 调用库存服务：预占库存
   - 成功：继续
   - 失败：取消订单，结束
3. 调用营销服务：锁定优惠券
   - 成功：继续
   - 失败：释放库存，取消订单，结束
4. 调用积分服务：扣除积分
   - 成功：继续
   - 失败：释放库存+券，取消订单，结束
5. 更新订单状态：CONFIRMED
```

补偿逻辑：
- 每个服务都提供补偿接口（释放库存、释放优惠券、返还积分）
- 订单服务维护补偿表，记录需要补偿的操作
- 失败时按相反顺序执行补偿

优点：
- 流程清晰，易于理解和调试
- 中心化控制，便于监控和排查问题
- 补偿逻辑集中管理

缺点：
- 订单服务成为单点，压力较大
- 流程变更需要修改编排器
- 服务间耦合度较高

**方案二：Saga模式（事件编排式）**

设计思路：
通过事件总线（Kafka）进行编排，各服务监听事件并发布新事件。

流程：
```
1. 订单服务：创建订单 → 发布 OrderCreated 事件
2. 库存服务：监听 OrderCreated → 预占库存 → 发布 InventoryReserved 事件
3. 营销服务：监听 InventoryReserved → 锁定优惠券 → 发布 CouponLocked 事件
4. 积分服务：监听 CouponLocked → 扣除积分 → 发布 PointsDeducted 事件
5. 订单服务：监听 PointsDeducted → 更新订单状态为 CONFIRMED
```

补偿流程：
```
如果某个服务失败（如积分不足）：
1. 发布失败事件（如 PointsDeductionFailed）
2. 各服务监听失败事件，执行补偿操作
3. 订单服务最终更新状态为 CANCELLED
```

优点：
- 服务解耦，各服务独立演进
- 无中心化瓶颈，扩展性好
- 天然支持异步，性能更好

缺点：
- 流程分散，难以全局把控
- 调试困难，需要完善的链路追踪
- 事件顺序和幂等性要求高

**方案三：TCC（Try-Confirm-Cancel）**

设计思路：
分为三个阶段：Try（预留）、Confirm（确认）、Cancel（取消）。

各服务接口设计：
```
库存服务：
- tryReserve(orderId, skuId, quantity): 预占库存
- confirmReserve(orderId): 确认扣减
- cancelReserve(orderId): 释放库存

营销服务：
- tryLockCoupon(orderId, couponId): 锁定优惠券
- confirmLockCoupon(orderId): 确认核销
- cancelLockCoupon(orderId): 释放优惠券
```

流程：
```
第一阶段（Try）：
1. 订单服务：tryCreateOrder
2. 库存服务：tryReserve
3. 营销服务：tryLockCoupon
4. 积分服务：tryDeductPoints

全部成功 → 进入第二阶段（Confirm）
任一失败 → 进入第三阶段（Cancel）

第二阶段（Confirm）：
1. 库存服务：confirmReserve
2. 营销服务：confirmLockCoupon
3. 积分服务：confirmDeductPoints
4. 订单服务：confirmCreateOrder

第三阶段（Cancel）：
按相反顺序取消已完成的Try操作
```

优点：
- 强一致性，业务语义清晰
- 资源锁定明确，不会出现超卖
- 适合对一致性要求极高的场景

缺点：
- 实现复杂，需要每个服务提供三个接口
- 性能开销大（锁定资源时间长）
- Try阶段占用资源，影响并发度

**方案对比**：

| 维度 | Saga-编排 | Saga-事件 | TCC |
|------|-----------|-----------|-----|
| 实现复杂度 | ★★★☆☆ | ★★★★☆ | ★★★★★ |
| 性能 | ★★★☆☆ | ★★★★☆ | ★★☆☆☆ |
| 一致性强度 | 最终一致 | 最终一致 | 强一致 |
| 可观测性 | ★★★★☆ | ★★★☆☆ | ★★★★☆ |
| 扩展性 | ★★★☆☆ | ★★★★★ | ★★☆☆☆ |
| 适用场景 | 中等复杂度 | 高并发、松耦合 | 强一致性要求 |

**推荐方案**：
对于电商系统，推荐**Saga-编排模式**作为主方案，辅以事件通知。

实施要点：

1. **编排器设计**
   - 订单服务作为编排器
   - 维护状态机记录每步执行状态
   - 使用数据库事务保证编排状态的一致性

2. **幂等性保证**
   - 每个操作携带唯一业务ID（orderId + operationType）
   - 服务侧记录已处理的操作，防止重复执行
   - 补偿操作也要幂等

3. **超时处理**
   - 每个步骤设置合理超时（如库存3s，优惠券2s）
   - 超时后执行补偿
   - 定时任务扫描长时间未完成的订单进行兜底

4. **补偿表设计**
```
order_compensation:
  - order_id
  - service_name (inventory/coupon/points)
  - operation_type (reserve/lock/deduct)
  - compensation_status (pending/success/failed)
  - retry_count
  - created_at
```

5. **可观测性**
   - 每个步骤记录日志，包含traceId
   - 监控每个步骤的成功率和耗时
   - 补偿操作单独统计和告警

6. **降级策略**
   - 积分扣除失败：允许后续异步补扣
   - 优惠券失败：取消优惠但继续创单
   - 库存失败：直接拒绝订单（不可降级）

**延伸思考**：

1. **如何处理补偿失败的情况？**
   - 补偿操作也可能失败（如网络问题、服务宕机）
   - 需要重试机制：指数退避，最多重试5次
   - 最终兜底：人工介入+对账系统

2. **最终一致性的"最终"是多久？**
   - 正常情况：秒级（1-3秒内所有操作完成）
   - 异常情况：分钟级（补偿+重试）
   - 极端情况：小时级（对账+人工介入）
   - 需要根据业务SLA设置合理的超时和告警

3. **如何进行一致性验证和对账？**
   - 实时对账：订单状态与库存/券/积分状态比对
   - 定时对账：每日全量对账，发现差异自动补偿
```

```markdown
#### 🎯 案例1：双11大促全链路备战方案

**场景描述**：
公司准备参加双11大促，预估流量是平时的50倍，订单量达到平时的100倍。作为架构师，请设计完整的备战方案。

关键数据：
- 平时：日均订单50万，峰值QPS 5000
- 大促预估：日订单5000万，峰值QPS 50万
- 团队规模：100人
- 准备时间：3个月

**答案**：

**问题分析**：
大促备战的核心挑战：
1. **容量规划**：如何评估各系统需要的资源？
2. **限流降级**：如何保证核心链路的可用性？
3. **数据库扩容**：如何应对写入和查询压力？
4. **缓存预热**：如何避免冷启动导致的雪崩？
5. **监控告警**：如何实时掌握系统健康度？
6. **应急预案**：如何快速响应突发问题？

**方案一：垂直扩容为主**

核心思路：
通过提升单机性能和适度扩容来应对流量，保持架构简单。

技术方案：
1. **数据库层**
   - MySQL：升级为更高配置（32核128G → 64核256G）
   - 读写分离：增加只读从库（1主4从 → 1主10从）
   - 连接池优化：增加最大连接数

2. **应用层**
   - 服务器扩容：从50台扩到200台
   - JVM调优：调整堆内存和GC参数
   - 线程池调优：增加核心线程数

3. **缓存层**
   - Redis扩容：从3节点扩到12节点
   - 缓存预热：提前加载热点数据
   - 本地缓存：增加进程内缓存减少Redis压力

优点：
- 改动小，风险可控
- 不需要大规模架构调整
- 回退方便

缺点：
- 成本高（需要高配机器）
- 扩展性有限（垂直扩展有瓶颈）
- 单点风险依然存在

**方案二：水平扩展+架构优化**

核心思路：
通过分库分表、读写分离、异步化等手段，从架构层面提升并发能力。

技术方案：
1. **数据库层**
   - 分库分表：订单表按user_id分32库128表
   - 读写分离：每个分片1主5从
   - 分布式事务：采用Saga模式

2. **应用层**
   - 服务拆分：进一步拆分大服务（如订单服务拆成创单/查询/状态管理）
   - 异步化：非核心流程异步处理（如消息通知、数据同步）
   - 限流降级：多级限流（网关、服务、接口）

3. **缓存层**
   - 多级缓存：L1本地缓存 + L2 Redis + L3 CDN
   - 缓存预热：分批预热，避免雪崩
   - 降级数据：准备降级缓存数据

4. **中间件层**
   - 消息队列扩容：Kafka分区数翻倍
   - 服务网格：引入Istio做流量治理
   - 分布式追踪：完善链路追踪

优点：
- 扩展性强，可支持更大规模
- 性能提升明显
- 降低单点风险

缺点：
- 改动大，风险较高
- 分布式事务复杂
- 运维成本增加

**方案三：混合方案（推荐）**

核心思路：
短期垂直扩容+重点优化，长期水平扩展+架构改造。

分阶段实施：

**第一阶段（T-3个月）：架构评估与重点优化**
- 压测识别瓶颈：全链路压测找出薄弱环节
- 数据库优化：慢查询优化、索引优化、分库分表（热点表先拆）
- 缓存优化：增加缓存命中率、引入多级缓存
- 代码优化：消除N+1查询、批量化处理

**第二阶段（T-2个月）：容量扩充与基础设施**
- 服务器扩容：按压测结果扩容（预留30%buffer）
- 数据库扩容：读库扩容 + 分片扩容
- 缓存扩容：Redis集群扩容
- 消息队列扩容：增加分区和副本
- CDN扩容：增加带宽和节点

**第三阶段（T-1个月）：限流降级与预案**
- 限流策略：
  - 网关层：总QPS限流（50万）
  - 服务层：单服务限流（如订单创建10万）
  - 接口层：单接口限流（如查询详情5万）
- 降级策略：
  - 非核心功能降级（推荐、评论、收藏）
  - 读服务降级（返回缓存数据）
  - 写服务限流（拒绝部分请求）
- 应急预案：
  - 数据库应急：主从切换、临时扩容
  - 缓存应急：Redis雪崩处理、降级数据
  - 服务应急：快速扩容、快速回滚

**第四阶段（T-2周）：全链路压测与演练**
- 压测目标：峰值QPS × 1.5倍
- 压测内容：
  - 单接口压测：查商品、加购、下单、支付
  - 全链路压测：模拟真实用户行为
  - 容灾演练：模拟数据库宕机、Redis宕机、服务宕机
- 压测输出：
  - 性能报告：各接口P99、P999
  - 瓶颈清单：需要优化的点
  - 容量规划：资源是否够用

**第五阶段（T-1周）：缓存预热与监控**
- 缓存预热：
  - 商品缓存：提前加载热门商品
  - 用户缓存：分批加载活跃用户数据
  - 配置缓存：预加载营销规则、计价规则
- 监控准备：
  - 大盘监控：QPS、成功率、响应时间
  - 业务监控：订单量、支付成功率、库存水位
  - 基础监控：CPU、内存、磁盘、网络
  - 告警配置：分级告警（P0/P1/P2）

**延伸思考**：

1. **如何评估各系统需要的容量？**
   - 基于历史数据：去年大促数据 × 增长率 × 安全系数（1.5）
   - 压测验证：实际压测得出各系统瓶颈
   - 分层评估：接入层、应用层、数据层、缓存层分别评估
   - 监控验证：观察平时的资源使用率（CPU、内存、网络）

2. **大促期间出现故障如何应急？**
   - **数据库慢查询**：
     - 立即：Kill慢查询
     - 短期：增加索引、优化SQL
     - 长期：分库分表
   - **缓存雪崩**：
     - 立即：启用降级数据
     - 短期：恢复Redis集群
     - 长期：多级缓存、缓存预热
   - **服务雪崩**：
     - 立即：限流、熔断
     - 短期：快速扩容
     - 长期：服务拆分、异步化

3. **大促后如何复盘和优化？**
   - **数据复盘**：
     - 流量峰值：实际 vs 预估
     - 成功率：订单成功率、支付成功率
     - 性能指标：P99、P999响应时间
     - 资源使用：CPU、内存、网络峰值
   - **问题复盘**：
     - 故障清单：出现的问题和解决时间
     - 预案验证：预案是否有效
     - 监控盲区：哪些问题没监控到
   - **优化方向**：
     - 架构优化：哪些地方需要重构
     - 容量规划：明年需要多少资源
     - 流程优化：备战流程哪里可以改进
```
