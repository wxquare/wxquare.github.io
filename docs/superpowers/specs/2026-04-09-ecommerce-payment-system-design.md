### 第 4 章：状态机设计（2000字）

#### 设计思路
状态机是支付系统面试的高频考点，需要清晰展示状态转换规则和实现方式。

#### 4.1 支付单状态机（800字）

**状态定义**
```
PENDING（待支付）
    ↓
PAYING（支付中）
    ↓ ↙ ↘
SUCCESS  FAILED  CANCELED
    ↓
REFUNDING（退款中）
    ↓
REFUNDED（已退款）
```

**状态转换表**

| 当前状态 | 允许的下一状态 | 触发条件 |
|---------|--------------|---------|
| PENDING | PAYING | 用户发起支付 |
| PAYING | SUCCESS | 第三方回调成功 |
| PAYING | FAILED | 第三方回调失败 |
| PAYING | CANCELED | 用户取消支付 |
| SUCCESS | REFUNDING | 用户申请退款 |
| REFUNDING | REFUNDED | 退款成功 |

**状态机图**（Mermaid 状态图）

#### 4.2 退款单状态机（500字）

**状态定义**
```
PENDING（待退款）
    ↓
PROCESSING（退款中）
    ↓ ↙ ↘
SUCCESS  FAILED
```

**与支付单状态联动**
- 支付单必须是 SUCCESS 状态才能发起退款
- 退款成功后，支付单状态变为 REFUNDED

#### 4.3 状态机实现（700字）

**核心要点**
- **状态转换校验**: 不允许非法状态跳转
- **乐观锁控制**: version 字段防止并发冲突
- **状态变更事件**: 发布到 Kafka，订单服务订阅

**精简代码示例**（Go 伪代码，30 行）
```go
func TransitState(paymentID int64, targetState string, version int) error {
    // 1. 校验状态转换是否合法
    if !isValidTransition(currentState, targetState) {
        return errors.New("invalid state transition")
    }
    
    // 2. 乐观锁更新
    result := db.Exec(`
        UPDATE payment_order 
        SET payment_status = ?, version = version + 1
        WHERE payment_id = ? AND version = ?
    `, targetState, paymentID, version)
    
    if result.RowsAffected == 0 {
        return errors.New("concurrent update conflict")
    }
    
    // 3. 发布状态变更事件
    publishStateChangeEvent(paymentID, targetState)
    
    return nil
}
```

- 1 个状态转换表（清晰列出合法转换）

### 第 5 章：高频考点深入（4000字）

#### 设计思路
将面试中最常被问到的三大技术问题单独成章，深入讲解。

#### 5.1 分布式事务（1500字）

**问题场景**
支付成功后，如何保证订单状态同步更新？

**Saga 模式**（支付流程）
- **正向流程**: 
  1. 订单服务创建订单
  2. 支付服务支付成功
  3. 订单服务更新状态为"已支付"
  
- **补偿流程**: 
  1. 支付失败
  2. 订单服务取消订单

- **实现方式**: 本地消息表 + 定时任务扫描
  1. 支付服务在本地消息表插入"订单支付成功"事件
  2. 定时任务扫描未发送消息，发送到 Kafka
  3. 订单服务消费消息，更新订单状态

**精简代码示例**（30 行）
```go
// 本地消息表插入
func InsertLocalMessage(tx *sql.Tx, orderID int64, event string) error {
    _, err := tx.Exec(`
        INSERT INTO local_message (order_id, event, status, retry_count)
        VALUES (?, ?, 'PENDING', 0)
    `, orderID, event)
    return err
}

// 定时任务扫描
func ScanAndSendMessages() {
    messages := queryPendingMessages()
    for _, msg := range messages {
        if err := kafka.Send(msg); err == nil {
            updateMessageStatus(msg.ID, "SENT")
        } else {
            incrementRetryCount(msg.ID)
        }
    }
}
```

**TCC 模式**（退款流程）
- **Try**: 冻结账户余额
- **Confirm**: 退款成功，解冻并扣减
- **Cancel**: 退款失败，直接解冻

**对比表格**: Saga vs TCC

| 维度 | Saga | TCC |
|-----|------|-----|
| 一致性 | 最终一致性 | 强一致性 |
| 实现复杂度 | 低 | 高 |
| 适用场景 | 支付流程 | 退款流程 |
| 性能 | 高 | 中 |

#### 5.2 幂等性设计（1500字）

**为什么需要幂等**
- 第三方回调可能重复
- 用户重复点击
- 网络抖动导致重试

**三个关键场景**

**场景 1: 支付单创建幂等**
- **幂等键**: order_id + user_id + amount
- **实现方式**: 数据库唯一索引 + Redis 分布式锁

**场景 2: 支付回调幂等**
- **幂等键**: 第三方交易号（channel_trade_no）
- **实现方式**: 数据库唯一索引

**场景 3: 退款幂等**
- **幂等键**: 退款单号（refund_id）
- **实现方式**: 数据库唯一索引

**精简代码示例**（40 行）
```go
func CreatePaymentOrder(req *CreatePaymentRequest) (*PaymentOrder, error) {
    // 1. 生成幂等键
    idempotencyKey := fmt.Sprintf("%d_%d_%s", req.OrderID, req.UserID, req.Amount)
    
    // 2. Redis 分布式锁（防并发）
    lock := redis.Lock(idempotencyKey, 10*time.Second)
    if !lock.TryLock() {
        return nil, errors.New("concurrent request")
    }
    defer lock.Unlock()
    
    // 3. 检查是否已存在
    existing := queryPaymentOrderByIdempotencyKey(idempotencyKey)
    if existing != nil {
        return existing, nil  // 幂等返回
    }
    
    // 4. 创建支付单
    order := &PaymentOrder{
        PaymentID:      snowflake.Generate(),
        OrderID:        req.OrderID,
        IdempotencyKey: idempotencyKey,
        // ...
    }
    
    // 5. 插入数据库（唯一索引保证幂等）
    if err := db.Insert(order); err != nil {
        if isDuplicateKeyError(err) {
            return queryPaymentOrderByIdempotencyKey(idempotencyKey), nil
        }
        return nil, err
    }
    
    return order, nil
}
```

**实现手段总结**
- **Redis 分布式锁**: 防止并发请求
- **数据库唯一索引**: 防止重复插入
- **乐观锁**: 防止并发更新

#### 5.3 一致性保证（1000字）

**账户余额一致性**
- **问题**: Redis 缓存和 MySQL 数据不一致
- **方案**: 
  1. Redis + MySQL 双写
  2. Lua 脚本保证原子性
  3. 定时对账任务（每日凌晨）
  
**示例**:
```lua
-- Redis Lua 脚本（原子操作）
local balance = redis.call('GET', KEYS[1])
if tonumber(balance) >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1
else
    return 0
end
```

**支付流水一致性**
- **问题**: 支付单金额与流水表汇总金额不一致
- **方案**:
  1. 每笔支付/退款都记录流水
  2. 流水表不可更新，只能插入
  3. 定时任务校验: 支付单金额 = 流水表汇总金额

- 1 个 Saga 流程图

- 1 个对比表格（Saga vs TCC）
- 2 个代码片段（本地消息表、分布式锁幂等）

### 第 6 章：扩展模块（2000字）

#### 设计思路
清结算、对账、风控在面试中通常不是重点，用概念介绍的方式快速过一遍。

#### 6.1 清结算系统（700字）

**核心概念**
将支付金额按规则分配给平台、商家、营销补贴。

**分账规则示例**
- 平台佣金: 5%
- 商家收益: 95%
- 营销补贴: 平台承担

**结算周期**
- T+1: 次日结算（快速到账）
- T+7: 7 日后结算（标准）

**提现管理**
- 单笔限额: ≤ 50,000 元
- 每日限额: ≤ 200,000 元
- 风控校验: 异常交易暂停提现

**简单流程图**
```
定时任务 → 计算分账金额 → 生成结算单 → 商家确认 → 提现申请 → 打款
```

#### 6.2 对账系统（700字）

**为什么需要对账**
系统与第三方的交易数据可能不一致。

**对账维度**
- **交易对账**: 订单号、金额、状态是否一致
- **资金对账**: 入账金额、手续费是否一致

**对账流程**
1. 每日凌晨拉取第三方对账文件
2. 与本地流水对比
3. 生成差错报告
   - 长款: 第三方有，本地无
   - 短款: 本地有，第三方无
   - 金额不符
4. 人工复核处理

**简单示意图**
```
对账任务 → 拉取对账文件 → 数据对比 → 差错分类 → 人工处理
```

#### 6.3 风控系统（600字）

**事前风控**
- 支付密码验证
- 指纹/人脸验证

**事中风控**
- 异常交易监控（短时间大额、异地登录）
- 限额控制（单笔、每日、每月）

**事后风控**
- 对账差错分析
- 资金流向追踪

**常见风控规则**
- 单笔支付 > 10,000 元: 需要二次验证
- 单日支付超过 5 次: 触发风控审核
- 异地登录支付: 短信验证码

#### 呈现方式
- 3 个简化流程图（清结算、对账、风控）
- 概念清晰，不深入技术细节

**Q1: 如何设计一个支付系统？**

回答框架（3 分钟讲清楚）:
1. **分层架构**: 接入层 → 应用层 → 核心层 → 基础设施层 → 第三方层

**Q2: 支付系统如何保证高可用？**

回答要点:
- 多机房部署（异地容灾）
- Redis 主从 + 哨兵（缓存高可用）
- Kafka 集群（消息高可用）
- 限流降级（大促保护）
- 熔断机制（第三方支付故障）

详见: 第 2 章

#### 7.2 技术方案类（600字）

**Q3: 支付成功后如何保证订单状态同步更新？**

回答要点:
1. **Saga 模式**: 本地消息表 + 最终一致性
2. **实现步骤**:
   - 支付服务插入本地消息表
   - 定时任务扫描并发送到 Kafka
   - 订单服务消费消息，更新状态
3. **补偿机制**: 支付失败，订单服务取消订单

详见: 第 5.1 章

**Q4: 如何保证支付幂等性？**

回答要点（三个场景）:
1. **支付单创建幂等**: order_id + user_id 唯一键
2. **回调幂等**: channel_trade_no 去重
3. **退款幂等**: refund_id 去重

实现手段:
- Redis 分布式锁（防并发）
- 数据库唯一索引（防重复）
- 乐观锁（防并发更新）

详见: 第 5.2 章

**Q5: 第三方支付回调失败怎么办？**

回答要点:
1. **重试机制**: 最多 3 次，指数退避（1s, 2s, 4s）
2. **定时任务补偿**: 每分钟扫描超时支付单，主动查询第三方状态
3. **人工介入**: 超过重试次数，人工复核

详见: 第 3.3 章

**Q6: 如何处理部分退款？**

回答要点:
1. **可退金额计算**: 实付金额 - 已退金额
2. **多次退款支持**: 累计退款金额 ≤ 实付金额
3. **营销优惠处理**: 按比例退款

示例:
- 实付 100 元（原价 120，优惠 20）
- 退款 50 元 → 退实付 50，营销优惠退 10

详见: 第 3.2 章

#### 7.3 场景题类（400字）

**Q7: 大促期间支付峰值如何应对？**

回答要点（4 个维度）:
1. **缓存**: Redis 缓存热点数据（账户余额）
2. **限流**: 令牌桶限流（单用户 QPS）
3. **降级**: 关闭非核心功能（账单查询）
4. **异步化**: 支付回调异步处理

详见: 第 1.3 章

**Q8: 如何防止恶意刷单？**

回答要点（风控规则）:
1. **短时间大额**: 单用户 1 小时内支付 > 10 笔，触发风控
2. **异地登录**: IP 地址异常，短信验证码
3. **限额控制**: 单笔 ≤ 50,000，每日 ≤ 200,000

详见: 第 6.3 章
