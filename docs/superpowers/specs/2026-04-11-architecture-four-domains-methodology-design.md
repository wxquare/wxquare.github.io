## 一、背景与目标

### 1.1 动机

用一篇可复用的文章，把 **业务架构、应用架构、数据架构、技术架构** 从「概念并列」收敛为 **单产品 / 技术负责人** 视角下的 **时间线 + 工件**，并配套 **自检与评审清单**。读者能按流程推进设计，也能用清单做架构评审前的预检。

### 1.2 经验叙事与场景锚点

- **主场景**: 约 **5 年电商系统** 的设计与实现经验（交易、商品、库存、价格与营销、支付与结算、履约与售后等典型域）。
- **写法**: 流程各步的 **输入 / 活动 / 输出** 均绑定 **可辨认的电商子域示例**；反例与权衡优先选自电商高频坑（库存超卖、价格不一致、促销叠加、支付与订单状态机、拆单合单等）。
- **边界**: 本文不展开完整领域模型教学；**战略 / 战术 DDD、Clean Architecture、CQRS、Code Review** 以 **指向现有 ACC 系列（41–44）** 为主，正文只保留 **对照表 + 一两句衔接**，避免与长文重复。

### 1.3 目标读者

- 负责 **0→1 或 1→N** 产品线 / 核心链路的技术负责人、骨干工程师。  
- 需要 **与业务、测试、运维对齐** 架构方案、并留下 **可审计工件** 的读者。

### 1.4 成功标准（文章层面）

读者在 **一次设计迭代** 内能够：

1. 按流程产出 **最小但完整** 的四域工件集合（见第三节「每步输出映射」）。  
2. 用下半篇清单完成 **一轮自检**，并明确 **待决议项** 是否已进入 ADR 或评审议程。  
3. 能说清 **电商场景下** 至少一类典型权衡（例如 **一致性 vs 延迟**、**库存预留 vs 实扣**）在本方案中的取舍。

## 三、上半篇：可重复流程（流程驱动 + 四域产出）

### 3.1 引言（短）

- **四域在工程向的定义**: 业务（能力与场景）、应用（边界与契约）、数据（所有权与一致性）、技术（约束与横切能力）。  
- **与 ACC 系列关系**: 本文覆盖 **方案成形与对齐**；`41`–`44` 覆盖 **建模、分层、读写分离与评审**。  
- **电商锚点声明**: 下文示例默认 **B2C / 平台型电商** 常见子域，特例（跨境、多租户、线下门店）在脚注或「扩展阅读」一句带过。

### 3.2 总览表（文中一张表）

列建议：**步骤 | 主要问题 | 关键产出（工件）| 主要落在哪一域**。

### 3.3 分步说明（每步固定小节结构）

每步包含：**输入**、**活动**、**输出**、**电商示例（简短案例句或 bullet）**、**常见反例（可选）**。

| 步骤 | 名称 | 核心问题 | 四域落点（主） |
|------|------|----------|----------------|
| 1 | 问题与成功标准 | 做什么、不做什么、如何验收 | 业务（目标）+ 技术（NFR） |
| 2 | 业务架构（轻量） | 谁、在什么场景、达成什么结果 | 业务 |
| 3 | 应用架构 | 系统如何切分、如何协作 | 应用（+ 契约影响数据） |
| 4 | 数据架构 | 谁拥有真相、如何一致、如何演进 | 数据 |
| 5 | 技术架构 | 如何部署、观测、安全、降级与成本 | 技术 |
| 6 | 验证与固化 | 假设、风险、决策与评审关口 | 四域交叉 + ADR |

**电商绑定示例（写入规格级要点，正文可展开）**

- **步骤 1**: 峰值下单 QPS、支付成功率、库存准确性（可接受超卖率或零容忍）、价格展示与结算一致性 SLA、合规（支付牌照边界、个人信息）。  
- **步骤 2**: 能力示例 — 「下单」「算价」「占库」「收款」「发货」「售后」；场景 — 「大促叠加券 + 会员价 + 平台补贴」。  
- **步骤 3**: 典型上下文 — 订单、库存、商品与价格、营销、支付、物流；同步 API vs 领域事件（订单已支付 → 出库指令）。  
- **步骤 4**: 订单状态机与支付回调乱序；库存流水与可售库存；价格快照与订单行；Outbox / 幂等与对账。  
- **步骤 5**: 多 AZ、缓存与击穿、消息堆积、对账任务、密钥与 PCI 边界（卡数据不进业务库等原则级表述）。  
- **步骤 6**: ADR 示例主题 — 「库存先扣后付 vs 先付后扣」「拆单策略与部分退款」。

## 四、下半篇：分域审查清单

### 4.1 组织方式

- 四大块：**业务 / 应用 / 数据 / 技术**。  
- 每块下按四个维度设勾选项：**完整性、一致性、可演进、可运维（含安全与合规子项）**。  
- **密度目标**: 全清单可控制在 **约 40–70 条** 勾选级条目（过细项用「详见 44」互链，避免单篇失控）。

### 4.2 电商向清单主题（必须覆盖）

**业务**: 场景边界、角色权限、大促与异常流程（缺货、拆单、部分发货）、与财务/客服的衔接描述是否清晰。

**应用**: 限界上下文是否与服务边界基本一致；事件名与版本；同步链路过长识别；第三方（支付、物流）超时与补偿是否在设计中体现。

**数据**: 单一写者原则；订单 / 支付 / 库存 **幂等键**；快照与审计字段；冷热数据与归档是否考虑。

**技术**: SLO、告警、容量与降级；密钥与配置；灰度与回滚；成本敏感路径（例如无谓的全表扫描、大 key）。
