# 统一执行模型设计：从传统长短任务到 Agent 与 Multi-Agent 协作

## 1. 背景

传统后端系统和当前热门的 Agent 系统，经常被当成两套完全不同的世界来讨论。前者强调确定性流程、批处理、状态机和治理；后者强调自然语言理解、动态规划、工具调用和自主协作。

但在真实工程里，这两类系统面对的核心问题高度相似：

- 任务到底应该同步完成还是异步推进
- 什么时候需要显式状态和 checkpoint
- 什么时候需要重试、补偿、审计和人工介入
- 什么时候单执行单元已经不够，必须拆分职责

这说明问题的本质不是“是否使用 LLM”，而是“任务执行跨度、认知复杂度和协作复杂度分别处在什么区间”。因此，需要一套能同时覆盖传统任务系统、Agent Runtime 和 Multi-Agent 的统一执行模型。

## 2. 设计目标

本文档希望建立一套统一方法论，用来回答以下问题：

1. 如何定义短任务与长任务，而不是仅按耗时做粗糙区分。
2. 如何用同一套抽象描述同步任务、异步任务、Batch、Workflow、单 Agent、Agent Runtime 和 Multi-Agent。
3. 不同执行方案分别适合什么场景，各自的优缺点和升级条件是什么。
4. 为什么 Agent 在处理长链路任务时，会重新遇到传统长任务系统早已解决过的状态、恢复和治理问题。
5. 当任务复杂度继续上升时，如何用渐进式方式从简单执行形态演进到受控协作系统。

## 3. 核心论点

本文档的核心判断有三条：

1. 短任务和长任务的区别，本质上不在耗时，而在于是否需要跨请求、跨时间、跨步骤、跨系统或跨角色持续推进。
2. Agent 和传统任务系统的区别，不在于是否有状态和治理，而在于执行单元是否具备动态规划、语义理解和自主决策能力。
3. 同步函数、异步任务、Batch、阶段化 Workflow、单 Agent、Agent Runtime 和 Multi-Agent，本质上是统一执行模型在不同复杂度区间下的不同实现。

可以把整条演进路径概括为：

```text
同步函数
  → 异步任务
  → Batch / 分布式 Job
  → 阶段化 Workflow
  → 单 Agent
  → Stateful Agent Runtime
  → Multi-Agent 协作系统
```

## 4. 全文结构

本文档按以下结构展开：

1. 统一执行模型的抽象和术语
2. 执行方案谱系、优缺点对比与适用场景
3. 短任务与长任务的判定标准
4. 传统工程里的执行谱系
5. 为什么 Agent 会重走长任务系统的路
6. Agent 长短任务统一框架
7. Multi-Agent 协作模型
8. 统一治理机制
9. 落地建议与演进路线

## 5. 执行方案谱系：从同步任务到 Multi-Agent 的选型矩阵

### 5.1 统一对比维度

为了避免只按“先进不先进”比较方案，统一从以下维度评估：

| 维度 | 关注的问题 |
|------|------------|
| `执行时长` | 是否必须在一次请求或一次会话内完成 |
| `状态持久化` | 中途失败后是否能从中间状态恢复 |
| `流程灵活性` | 执行路径是固定的还是动态规划的 |
| `恢复与补偿` | 是否支持局部重试、跳过、回放和人工介入 |
| `治理要求` | 是否需要审计、权限、版本保护、人工审批 |
| `交互性` | 是否需要等待用户反馈或外部事件 |
| `认知复杂度` | 是否需要推理、规划、工具选择和结果审查 |
| `协作复杂度` | 是否需要多个角色或执行单元协作 |
| `工程成本` | 搭建、维护、观测和排障的整体复杂度 |
| `适配规模` | 更适合单对象、小批量还是大批量、长链路任务 |

### 5.2 方案总表

| 方案 | 状态持久化 | 恢复能力 | 流程灵活性 | 治理能力 | 工程成本 | 适合场景 |
|------|------------|----------|------------|----------|----------|----------|
| `同步短任务` | 弱 | 弱 | 低 | 低 | 低 | 查询、单次写入、简单工具调用 |
| `异步任务 / MQ Consumer` | 弱到中 | 中 | 低 | 低到中 | 低到中 | 通知、缓存刷新、轻量后台处理 |
| `Batch / 分布式 Job` | 中到强 | 强 | 中 | 中 | 中 | 批处理、全量同步、离线计算 |
| `阶段化 Workflow` | 强 | 强 | 中 | 强 | 中到高 | 跨系统长任务、强治理流水线 |
| `单 Agent` | 弱到中 | 弱 | 强 | 弱 | 低到中 | 问答、分析、轻量多步任务 |
| `Agent Runtime` | 强 | 强 | 强 | 中到强 | 高 | 长链路智能执行、托管 Agent |
| `Multi-Agent` | 强 | 中到强 | 很强 | 中到强 | 很高 | 复杂分工、规划执行审查闭环 |

### 5.3 典型方案、优缺点与适用场景

#### 方案 A：同步短任务

特点：单请求闭环、低延迟、实现简单。

优点：

- 开发成本低
- 调用链直
- 调试和回归成本小

缺点：

- 无法跨进程恢复
- 无法承载长链路状态
- 对副作用治理能力弱

适用场景：

- 查询接口
- 单次规则计算
- 单次工具调用
- 简单 Agent 问答

#### 方案 B：异步任务 / MQ Consumer

特点：请求入队，后台异步执行单个任务。

优点：

- 削峰填谷
- 解耦调用方
- 改善主链路响应时间

缺点：

- 任务状态通常较粗
- 复杂流程容易演化成大 Consumer 泥球
- 不天然具备长任务治理能力

适用场景：

- 发通知
- 刷新缓存
- 导出报表
- 轻量后台处理

#### 方案 C：Batch / 分布式 Job

特点：面向大批量对象处理，通常按批次、分片和 checkpoint 推进。

优点：

- 适合大数据量
- 可恢复
- 易于做并发和分片

缺点：

- 交互性较弱
- 流程变化时不够灵活
- 阶段治理能力取决于额外建模

适用场景：

- 全量同步
- 索引重建
- 离线对账
- 周期性批处理

#### 方案 D：阶段化 Workflow / 长任务流水线

特点：把任务拆成多个显式阶段，阶段间用状态边界连接。

优点：

- 职责清晰
- 易治理
- 易补偿
- 适合强审计场景

缺点：

- 建模成本更高
- 前期设计要求高
- 不适合变化过快且边界尚不稳定的场景

适用场景：

- 供应商供给同步
- 订单履约编排
- 复杂审批流程
- 跨系统数据发布

#### 方案 E：单 Agent

特点：一个智能执行单元完成理解、规划、工具调用和结果汇总。

优点：

- 灵活
- 开发快
- 适合认知型短任务

缺点：

- 上下文容易膨胀
- 长任务恢复性差
- 结果治理和副作用治理较弱

适用场景：

- 问答
- 轻量诊断
- 查询分析
- 一次性辅助操作

#### 方案 F：Stateful Agent Runtime

特点：Agent 执行被 Runtime 托管，支持状态持久化、恢复和人工介入。

优点：

- 同时保留 Agent 的灵活性和长任务治理能力
- 适合多步动态执行
- 更容易引入 checkpoint、review 和 policy

缺点：

- 系统复杂度明显上升
- 对 Runtime 设计要求高
- 观测与审计成本更大

适用场景：

- 代码代理
- 长链路运维诊断
- 调研任务
- 多步业务助手

#### 方案 G：Multi-Agent

特点：多个角色化 Agent 协作完成复杂目标。

优点：

- 适合复杂任务分治
- 能隔离规划、执行、审查职责冲突
- 便于引入专职 reviewer 和 specialist

缺点：

- 通信成本高
- 容易重复工作
- 治理难度显著上升
- 延迟和成本更高

适用场景：

- 复杂研发任务
- 深度调研
- 规划-执行-审查闭环
- 跨领域决策支持

### 5.4 推荐选型原则

1. 默认从最小可行执行单元开始，能用同步短任务解决就不要先上 Workflow，能用单 Agent 解决就不要默认上 Multi-Agent。
2. 一旦出现跨请求、跨时间、跨步骤持续推进，就要按长任务设计，而不是继续在短任务上打补丁。
3. 认知复杂度和执行复杂度要分开判断。需要推理不等于需要长任务，需要长任务也不等于必须引入 LLM。
4. 治理要求越高，越要前置状态边界、审查边界和权限边界。
5. Multi-Agent 只在单执行单元无法同时承担规划、执行、审查三类职责时才值得引入。

### 5.5 一条简单的选型路径

```text
先判断是否需要持久化状态
  → 再判断是否需要动态规划
  → 最后判断是否需要多角色协作
```

对应的推荐路径是：

1. 不需要持久化状态：优先 `同步短任务` 或 `单 Agent`
2. 需要异步解耦但流程简单：选 `异步任务 / MQ Consumer`
3. 需要长链路恢复和批量推进：选 `Batch` 或 `阶段化 Workflow`
4. 需要动态规划和长任务治理：选 `Agent Runtime`
5. 需要角色分工、审查闭环和并行协作：再升级到 `Multi-Agent`

### 5.6 当前典型的长任务场景

长任务并不是少数特殊系统才会遇到的边缘问题。在现代工程体系里，很多核心业务和基础设施任务天然就是长任务。它们虽然技术栈不同，但共同特征都很明显：执行时间长、状态跨度大、通常需要异步推进、失败恢复和治理闭环。

#### 场景 A：供应商数据同步与主数据对接

典型例子：

- 酒店、机票、商品、库存、价格等供应商供给同步
- 外部 ERP / WMS / OMS / PIM 数据接入
- 多租户主数据对账与回灌

为什么是长任务：

- 数据量大，通常需要分页、分片或游标推进
- 外部系统慢且不稳定
- 中途失败后不能从头重跑
- 涉及正式业务数据发布，必须有版本保护和审查机制

这类任务通常对应 `Batch + Checkpoint + 阶段化 Workflow`，是最典型的工业级长任务场景。

#### 场景 B：机器学习模型训练与模型交付

典型例子：

- 大规模模型训练
- 特征工程与数据预处理流水线
- 超参数搜索
- 训练后评估、模型注册和部署

为什么是长任务：

- 训练时间长，可能从数小时到数天
- 任务通常分成数据准备、训练、评估、导出、发布多个阶段
- 需要记录实验参数、版本、指标和产物
- 失败后往往需要从某个阶段继续，而不是全部重来

这类任务通常对应 `Job Orchestrator + Artifact Store + Pipeline State`，本质上是典型的状态化长任务。

#### 场景 C：Spark 大数据离线计算任务

典型例子：

- ETL 清洗任务
- 离线数仓加工
- 用户画像计算
- 大规模日志聚合与指标生成

为什么是长任务：

- 处理数据规模大，通常跨多个 Stage 和 Shuffle
- 执行资源昂贵，需要调度、排队和失败恢复
- 经常需要依赖上游数据就绪，再触发下游任务
- 结果可能进入报表、特征平台、推荐系统等关键链路

这类任务通常更接近 `Batch / DAG Workflow`，重点在批次组织、资源调度和阶段恢复。

#### 场景 D：Flink / Streaming 流处理任务

典型例子：

- 实时风控
- 实时指标计算
- 实时推荐特征更新
- 实时日志处理和告警检测

为什么它也是长任务：

- 虽然单条事件处理很短，但作业本身是持续运行的
- 需要长期维护状态、Checkpoint 和 Exactly-once / At-least-once 语义
- 作业升级、故障恢复、状态回放都需要系统级治理
- 常常涉及水位线、窗口、状态后端和反压处理

这类任务的特别之处在于：它不是“长时间才结束”，而是“持续运行并持续维护状态”的长任务系统。

#### 场景 E：Agent 长任务处理

典型例子：

- 代码 Agent 执行多文件修改、运行测试、失败后调整策略
- 调研 Agent 分阶段检索、阅读、总结、交叉验证
- 运维 Agent 分析告警、查日志、等待人工确认、继续执行
- 业务 Agent 处理多轮审批、多工具调用和跨系统操作

为什么是长任务：

- 任务包含多步推理和多步执行
- 需要等待用户反馈、系统回调或人工审批
- 需要把中间计划、工具结果和审查结果外置持久化
- 失败后不能简单从头再问一遍

这类任务通常需要 `Stateful Agent Runtime`，而不是只靠单轮对话或单 Agent prompt。

#### 场景 F：Multi-Agent 协作任务

典型例子：

- 复杂研发任务中的规划、编码、测试、审查分工
- 深度研究任务中的并行检索、事实核查和总结整合
- 跨领域业务分析中的多角色协作决策

为什么是长任务：

- 存在多个角色和多个子任务并行推进
- 需要共享状态、角色通信和结果汇总
- 需要显式终止条件、返工机制和升级路径
- 成本、权限、审计和冲突控制要求高

这类任务本质上是“协作型长任务”，重点不只是执行时间，而是协作链路和治理复杂度。

#### 场景 G：复杂业务流程与审批链路

典型例子：

- 订单履约编排
- 售后退款与逆向流程
- 资金结算与对账
- 内容审核与发布
- 风险审批与人工复核

为什么是长任务：

- 流程跨多个系统和多个组织角色
- 中途可能等待人工操作或外部结果
- 每一步都可能带来正式业务副作用
- 必须具备审计、补偿、回滚和权限控制

这类任务通常属于最典型的 `阶段化 Workflow + 审批治理` 场景。

#### 场景 H：基础设施与平台运维任务

典型例子：

- 大规模数据迁移
- 集群升级与灰度发布
- 索引重建
- 批量回刷缓存
- 全链路压测和结果收集

为什么是长任务：

- 任务影响面大，副作用强
- 需要分批推进、实时监控、暂停恢复和熔断
- 失败后通常需要回滚、补偿或人工接管

这类任务说明，长任务并不只存在于业务系统，也广泛存在于平台和基础设施层。

### 5.7 一个统一归纳

从今天主流系统的现实来看，长任务大致可以归纳为五大类：

1. `大批量数据处理类`：如供应商同步、Spark ETL、索引重建
2. `持续运行状态类`：如 Flink 流任务、实时风控作业
3. `高认知多步骤类`：如 Agent 调研、代码 Agent、运维 Agent
4. `多角色协作类`：如 Multi-Agent 规划-执行-审查闭环
5. `高风险业务流程类`：如审批、发布、结算、履约编排

这五类场景虽然实现方式不同，但都共享同一组核心约束：

- 需要状态
- 需要恢复
- 需要治理
- 需要审计
- 往往需要分阶段推进

## 6. 统一执行模型的核心概念与术语表

统一执行模型的核心观点是：

```text
任何任务系统，本质上都是在约束条件下，
让一组执行单元按照某种状态推进规则，
逐步逼近目标。
```

### 6.1 Task

Task 是系统试图完成的最小业务目标单元，回答“到底要完成什么”。

典型包含：

- `Goal`：目标
- `Input`：输入
- `Constraint`：约束
- `Output`：预期结果
- `Policy`：执行规则

### 6.2 Step

Step 是任务执行过程中的一个可观测、可推进、可失败的动作单元，回答“下一步做什么”。

一个好的 Step 应该具备：

- 明确输入和输出
- 有状态边界
- 定义清楚副作用
- 失败时知道如何重试、跳过或补偿

### 6.3 State

State 是任务在生命周期中的位置描述，回答“任务现在进行到哪里了”。

建议至少区分三层：

- `Task State`
- `Step State`
- `Object State`

### 6.4 Checkpoint

Checkpoint 是任务推进到某个可恢复边界时留下的恢复点，回答“中断后从哪里继续”。

典型例子：

- 页码
- 游标
- 最后处理 ID
- 已完成计划步骤
- 待唤醒等待点

### 6.5 Scheduler

Scheduler 决定任务何时开始、何时重试、何时暂停、何时继续，回答“什么时候由谁执行”。

### 6.6 Executor

Executor 是真正做事的执行单元，回答“这一步由谁完成”。

可能是：

- Worker
- Consumer
- Processor
- 单 Agent
- 专职角色 Agent

### 6.7 Reviewer

Reviewer 是结果评估和风险守门的单元，回答“这一步结果能不能被接受”。

可能是：

- 质量校验器
- 规则引擎
- 人工审批节点
- Critic Agent
- Guardrail

### 6.8 Memory

Memory 是任务在执行过程中可被后续步骤继续使用的上下文存储，回答“系统还记得什么”。

它可以包括：

- 批次上下文
- Raw Snapshot
- 标准化结果
- 会话历史
- Scratchpad
- 外部持久化知识

### 6.9 Policy

Policy 是约束执行边界的规则集合，回答“允许怎么做，不允许怎么做”。

典型包括：

- 最大重试次数
- 并发限制
- 风险等级
- 权限边界
- 成本预算
- 是否必须人工确认

### 6.10 统一关系图

```text
Task
  → 被 Scheduler 创建和推进
  → 拆成一组 Step
  → 由 Executor 执行
  → 中间状态写入 State 和 Memory
  → 在关键边界形成 Checkpoint
  → 结果经过 Reviewer 审核
  → 全程受 Policy 约束
```

### 6.11 统一例子：Agoda 酒店全量同步

在传统长任务系统里：

- `Task`：一次全量同步批次
- `Scheduler`：定时任务或人工触发器
- `Step`：Sharder / Fetcher / Transformer / Publisher
- `Executor`：各阶段 Worker
- `State`：Batch、Shard、Object Ledger 状态
- `Checkpoint`：城市 + 页码 / Cursor
- `Reviewer`：质量校验、版本保护、熔断规则
- `Memory`：Raw Snapshot、Normalized Model、Mapping
- `Policy`：限流、重试、发布权限、下线阈值

在 Agent Runtime 语境里：

- `Task`：完成供应商同步目标
- `Scheduler`：Runtime Loop 或 Orchestration Layer
- `Step`：计划、抓取、映射、校验、发布、复核
- `Executor`：单 Agent 或多个专职 Agent
- `State`：任务阶段、等待状态、工具执行状态
- `Checkpoint`：已完成子任务、待处理 Shard、已审查结果
- `Reviewer`：规则引擎、Critic Agent、人工审批
- `Memory`：上下文窗口 + 外部持久化 Memory
- `Policy`：工具权限、自动化级别、风险升级条件

## 7. 短任务与长任务的判定标准

短任务和长任务的区别，不在于耗时长短，而在于任务是否需要跨边界持续推进。

这里的边界通常包括：

- 跨请求边界
- 跨进程边界
- 跨时间边界
- 跨步骤边界
- 跨角色边界
- 跨系统边界

### 7.1 短任务定义

短任务通常在单次请求或单次会话中闭环完成，状态主要存在执行现场，失败后通常整体重试或直接返回错误。

### 7.2 长任务定义

长任务需要脱离单次请求持续推进，状态必须外置持久化，失败后需要局部恢复、补偿、审计或人工介入。

### 7.3 五个高价值判定信号

#### 信号 1：是否需要持久化状态

如果任务中断后必须从中间状态恢复，而不是从头再来，那么它已经具备长任务属性。

#### 信号 2：是否存在显式阶段边界

如果任务天然分成抓取、转换、校验、发布、复核等多个阶段，并且阶段之间要独立追踪和治理，那么应该按长任务建模。

#### 信号 3：是否需要等待外部事件或人工反馈

只要任务中途会等待人工审批、外部回调或异步信号，就应高度警惕它已经脱离短任务范畴。

#### 信号 4：是否需要失败补偿和局部重试

如果系统要求某一步失败只重试该步，或某一批对象失败后进入补偿队列，就说明它不能靠简单的“整体重试”解决。

#### 信号 5：是否需要强治理

当任务涉及高风险副作用，且需要写前校验、人工审批、版本保护、熔断或审计时，即使耗时不长，也应按长任务或受控 Workflow 设计。

### 7.4 一个很实用的判断口诀

```text
看状态
看阶段
看等待
看补偿
看治理
```

这五项中只要命中两到三项以上，就不建议继续按短任务设计。

### 7.5 典型误判场景

#### 误判 1：执行时间长，就一定是长任务

不一定。一个慢 SQL 查询如果没有状态恢复和阶段治理需求，本质上仍然更像短任务，只是性能差。

#### 误判 2：执行时间短，就一定是短任务

不一定。只要任务跨时间、跨角色并且需要审批或回调，即使入口动作很短，本质上也已是长任务。

#### 误判 3：放进 MQ 里，就是长任务系统

不对。MQ 解决的是异步解耦，不自动解决状态建模、补偿和审计。

#### 误判 4：用了 Agent，就天然适合长任务

不对。单 Agent 更像智能版短任务执行器，长任务仍然需要 Runtime 托管。

### 7.6 面向 Agent 的额外判定标准

还需要再看两个维度：

- `认知跨度`：是否需要多轮理解、规划、反思和改写策略
- `协作跨度`：是否需要多个角色分工完成

## 8. 传统工程里的短任务与长任务实现谱系

### 8.1 四层执行谱系

#### Level 1：同步短任务

典型特点：

- 请求内闭环
- 路径固定
- 状态在内存和调用栈里
- 失败后整体返回错误

#### Level 2：异步任务与 Consumer

典型做法：

- 请求写 DB 或发 MQ
- 后台 Worker / Consumer 执行
- 任务通常相对独立

这一层解决的是异步和解耦，不是复杂长任务治理。

#### Level 3：Batch / 分布式 Job

典型特点：

- 任务按批次组织
- 数据按分片、分页、游标或时间窗口推进
- 有 Checkpoint
- 能重试、续跑和补偿

#### Level 4：阶段化 Workflow / 长任务流水线

典型特点：

- 任务被拆成多个阶段
- 每个阶段有明确输入输出和职责边界
- 阶段之间用状态机和持久化状态连接
- 支持局部失败、补偿、人工介入和审计

### 8.2 从同步函数到阶段化流水线的升级驱动力

传统系统不断升级执行模型，通常由以下因素驱动：

- `规模上升`
- `耗时上升`
- `副作用上升`
- `治理要求上升`
- `失败代价上升`

### 8.3 一个典型演进路径

```text
同步接口
  → 异步消费者
  → 批处理任务
  → 分片批处理
  → 阶段化流水线
  → 带治理能力的 Workflow 平台
```

### 8.4 四阶段模型在传统工程中的位置

供应商同步四阶段模型属于典型的 `Level 4：阶段化 Workflow / 长任务流水线`：

- `Phase 0: Sharder`
  回答同步边界和拆分粒度问题，负责把大任务变成可调度、可追踪的 Shard。
- `Phase 1: Fetcher`
  忠实拉取外部数据，推进 Checkpoint，沉淀 Raw Snapshot，但不做业务裁决。
- `Phase 2: Transformer`
  完成外部语义到平台语义的映射、标准化和基础风险标注。
- `Phase 3: Publisher`
  在写前保护、版本校验和状态机约束下，把标准结果安全发布到主域。

它的核心价值不是“分四步”，而是：

- 把慢源不稳定性和业务发布风险隔离开
- 把技术问题和业务裁决问题隔离开
- 把原始数据、标准数据和正式发布状态隔离开
- 为重试、回放、补偿和熔断提供自然边界

### 8.5 传统工程里的常见反模式

#### 反模式 1：大函数串到底

一个函数里把抓取、转换、写库、发消息全部做完，出错后边界模糊、恢复困难。

#### 反模式 2：大 Consumer 泥球

把复杂长任务都塞进一个 Consumer，靠 if-else 维护多个阶段，短期快，长期难治理。

#### 反模式 3：只有任务级状态，没有对象级状态

批次看似成功，但对象级失败、跳过或错误覆盖难以追溯。

#### 反模式 4：把重试当成恢复

简单重复执行不等于恢复。真正恢复要求清楚失败点、状态边界、幂等条件和补偿策略。

## 9. 为什么 Agent 会重走传统长任务系统的路

Agent 很新，但它在长任务上的工程约束并不新。

### 9.1 单 Agent 的起点，本质上像智能短任务执行器

很多 Agent 系统的起点是：

```text
用户输入
  → LLM 理解意图
  → 选择工具
  → 调用工具
  → 整理结果
  → 返回答案
```

这依然是请求内闭环，只是执行单元更智能。

### 9.2 当 Agent 遇到长任务，五类问题会立刻出现

#### 问题 1：上下文窗口不是状态系统

Conversation History 只能暂存信息，不能替代正式状态管理。

#### 问题 2：工具调用不等于流程编排

Agent 可以决定调用哪个工具，但不天然等于拥有完整的 Orchestration 能力。

#### 问题 3：推理能力不等于恢复能力

重新想一遍不等于从正确状态恢复。

#### 问题 4：自主性提升后，治理压力同步上升

越能自主调用工具，越需要权限、审批、风险控制和审计。

#### 问题 5：复杂任务天然逼近角色分工

规划、执行、验证和总结职责集中在一个 Agent 身上时，冲突会越来越明显。

### 9.3 Agent 世界正在重新发明的，实际上是传统长任务能力

| Agent 领域常见概念 | 本质上对应的传统能力 |
|--------------------|----------------------|
| `Agent Runtime` | 长任务执行框架 |
| `Orchestrator` | 调度器 / Workflow Engine |
| `Scratchpad / Memory` | 执行上下文 / 状态存储 |
| `Plan` | 任务分解 / 执行计划 |
| `Tool Call` | 步骤执行单元 |
| `Reflection / Critic` | 审核器 / 质量校验器 |
| `Human-in-the-loop` | 人工审批节点 |
| `Checkpoint / Resume` | 断点续跑 |
| `Guardrail` | 策略控制 / 风险防护 |
| `Supervisor Agent` | 协调器 / 管控器 |

### 9.4 一个典型演进路径

```text
单轮问答 Agent
  → 带工具调用的单 Agent
  → 多步 Plan-and-Execute Agent
  → 支持状态持久化与恢复的 Agent Runtime
  → 带审查与人工介入的受控 Runtime
  → 按角色拆分的 Multi-Agent 协作系统
```

### 9.5 一个统一例子：代码 Agent 的演进

如果任务只是解释代码、找小 Bug、改小函数，单 Agent 往往够用。

但当任务升级为理解需求、扫描代码库、制定计划、修改多个文件、运行测试、等待确认再继续时，系统很快就会遇到：

- 需要知道当前做到哪一步
- 失败后不能从头来
- 某些动作需要审查后才能继续
- 工具权限不能无限放开
- 上下文太长时必须把状态外移
- 处理多个子任务时需要角色分工

这时，单 Agent 已不足以支撑完整执行生命周期。

## 10. Agent 长短任务统一框架

### 10.1 Agent 不是一个点，而是一条执行形态谱系

可以把 Agent 执行形态分成四层：

1. `Prompted Tool Use`
2. `Plan-and-Execute Agent`
3. `Stateful Agent Runtime`
4. `Role-Based Multi-Agent System`

这四层不是互斥方案，而是自然升级路径。

### 10.2 短任务 Agent

短任务 Agent 的本质是：

```text
用 LLM 弥补规则系统在理解、判断、归纳上的不足，
但执行仍然在单次交互里闭环。
```

适合：

- 问答
- 资料总结
- 一次性诊断
- 单次查询分析
- 小范围代码修改建议

边界：

- 状态依赖上下文窗口
- 恢复能力弱
- 工具副作用治理弱
- 不适合长时间等待和跨会话推进

### 10.3 长任务 Agent

长任务 Agent 必须把 Runtime 显式引入系统：

```text
Goal
  → Planner
  → Step Queue
  → Executor
  → State Store
  → Reviewer / Guardrail
  → Scheduler / Resume Loop
```

至少要补齐六类能力：

1. 任务状态持久化
2. Checkpoint 与 Resume
3. 异步等待
4. 工具执行隔离
5. 结果审查
6. 运行策略控制

### 10.4 Agent 世界里的统一映射

| 统一执行模型 | Agent 语境中的对应物 |
|--------------|----------------------|
| `Task` | 用户目标、任务指令、目标对象 |
| `Step` | 计划步骤、工具调用、子任务 |
| `State` | 任务状态、步骤状态、等待状态 |
| `Checkpoint` | 已完成计划、已确认结果、恢复点 |
| `Scheduler` | Runtime Loop、Orchestrator、Event Dispatcher |
| `Executor` | 单 Agent、工具执行器、专职角色 Agent |
| `Reviewer` | Evaluator、Critic、Guardrail、Human Approval |
| `Memory` | 会话上下文、Scratchpad、长期 Memory、外部存储 |
| `Policy` | Tool Policy、Risk Policy、Budget Policy、Approval Rule |

### 10.5 一个推荐的 Agent Runtime 分层

建议分成六层：

1. `Intent Layer`
2. `Planning Layer`
3. `Execution Layer`
4. `State & Memory Layer`
5. `Review & Policy Layer`
6. `Orchestration Layer`

这套分层的核心价值是把“会思考”和“能托管”分开。

### 10.6 单 Agent 什么时候够，什么时候不够

#### 单 Agent 通常够用的场景

- 输入目标清晰
- 工具调用数量有限
- 不需要跨会话持久化
- 不需要复杂审批或审查
- 出错后可以整体重来

#### 单 Agent 明显不够的信号

- 任务持续很久
- 中途等待用户或系统反馈
- 工具副作用高，不能随便重试
- 需要分阶段推进
- 需要局部恢复而不是整体重跑
- 需要规划、执行、审查分离
- 需要并行处理多个子问题

## 11. Multi-Agent 协作模型

### 11.1 核心观点

Multi-Agent 不是更高级的默认架构，而是单执行单元无法同时承担规划、执行、审查和协调职责时的一种受控分工机制。

### 11.2 为什么会走向 Multi-Agent

通常由以下结构性特征触发：

- `目标复杂，天然可分治`
- `职责之间存在冲突`
- `上下文无法被单一执行者有效持有`
- `需要显式审查闭环`

### 11.3 推荐的最小 Multi-Agent 结构

最稳妥的结构通常是三角模型：

- `Planner`
- `Worker`
- `Reviewer`

它们分别对应统一执行模型中的规划、执行和裁决三类职责。

### 11.4 可扩展角色，但不要过度设计

可按需要扩展：

- `Router`
- `Supervisor`
- `Memory Manager`
- `Specialist Agents`

但新增一个角色，就等于新增一个边界、一条通信链路和一份治理成本。

### 11.5 几种典型协作方案

#### 方案 A：Supervisor + Workers

优点：

- 结构直观
- 易于统一控制
- 适合并行子任务

缺点：

- Supervisor 容易成为瓶颈
- 复杂度集中在主控

适用场景：

- 多子任务并行调研
- 多文件代码修改
- 分片数据分析

#### 方案 B：Planner -> Executor -> Reviewer 串行闭环

优点：

- 质量门清晰
- 责任边界明确
- 适合高风险输出

缺点：

- 吞吐较低
- 反馈链路较长

适用场景：

- 代码修改与测试
- 文档生成与审校
- 高风险自动化操作

#### 方案 C：Router + Specialist Agents

优点：

- 专业分工明显
- 易扩展不同能力域

缺点：

- 路由错误会直接影响结果
- 角色能力边界可能重叠

适用场景：

- 企业知识助手
- 多领域运维助手
- 混合型业务助手平台

#### 方案 D：Peer Collaboration

优点：

- 发散探索能力强
- 适合开放式研究任务

缺点：

- 最难治理
- 容易重复劳动和空转
- 责任边界模糊

适用场景：

- 深度研究
- 创意探索
- 头脑风暴

工程系统里默认不推荐从 `Peer Collaboration` 起步。

### 11.6 Multi-Agent 的主要收益

- 分治复杂问题
- 隔离职责冲突
- 提升结果质量
- 匹配不同能力域
- 更适合复杂长任务

### 11.7 Multi-Agent 的主要代价

- 通信成本
- 重复劳动
- 责任稀释
- 延迟和成本上升
- 治理难度上升
- 失控风险

### 11.8 Multi-Agent 的治理重点

需要重点控制：

1. `角色边界`
2. `通信协议`
3. `终止条件`
4. `共享状态`
5. `权限控制`
6. `审计与观测`

### 11.9 一个渐进式升级建议

```text
单 Agent
  → 单 Agent + Reviewer
  → Planner + Worker + Reviewer
  → Supervisor + 多 Worker
  → 按领域扩展 Specialist Agents
```

## 12. 统一治理机制

治理层的核心问题不是“能不能跑起来”，而是“能不能在失败、异常、变更和风险面前稳住”。

### 12.1 治理层目标

治理层至少要回答六个问题：

1. 任务是否可追踪
2. 任务是否可恢复
3. 任务是否可限制
4. 任务是否可纠错
5. 任务是否可审计
6. 任务是否可演进

### 12.2 状态机：治理的骨架

建议区分三层状态：

#### 任务级状态

例如：

```text
PENDING → RUNNING → WAITING → SUCCESS / FAILED / CANCELLED
```

#### 步骤级状态

例如：

```text
READY → PROCESSING → RETRYING → SKIPPED → DONE
```

#### 对象级状态

例如：

```text
INIT → FETCHED → NORMALIZED → REVIEWED → PUBLISHED
```

对象级状态往往决定了问题定位和补偿精度。

### 12.3 幂等、重试与恢复：不要混为一谈

#### 幂等

解决同一个动作重复执行会不会造成重复副作用。

#### 重试

解决临时失败后，是否可以再次尝试同一个动作。

#### 恢复

解决任务中断后，如何从正确状态继续，而不是整任务重做。

三者关系：

- 没有幂等，重试会放大风险
- 没有状态边界，恢复只能退化成整任务重跑
- 没有 Checkpoint，恢复就只是猜测从哪继续

### 12.4 补偿与人工介入

任何真正的长任务系统都不能假设“所有步骤最终都会自动成功”。

因此必须内建：

- `补偿`
  - DLQ
  - 补偿任务
  - 局部重放
  - 回滚或反向动作
- `人工介入`
  - 审批
  - 冲突仲裁
  - 高风险结果复核

### 12.5 Policy Layer：把边界显式写出来

至少应包括：

- `权限策略`
- `预算策略`
- `并发策略`
- `风险策略`
- `终止策略`

在 Agent 系统里，这些策略还会体现在：

- Tool Policy
- Model Policy
- Guardrail
- Approval Gate
- Budget Controller

### 12.6 观测与审计

治理层至少需要三类观测：

#### 执行观测

- 当前状态
- 当前步骤
- 心跳
- Lease / Owner
- Checkpoint 位置

#### 质量观测

- Success Rate
- Retry Rate
- Skipped Rate
- 审查通过率
- 幻觉率或误判率

#### 成本观测

- 单任务耗时
- 外部调用次数
- Token 消耗
- 工具调用成本
- 人工介入比例

同时要保留审计记录，至少回答：

- 谁触发了任务
- 哪个执行者做了动作
- 使用了什么输入和工具
- 生成了什么结果
- 为什么通过或被拒绝
- 最终影响了哪些正式对象

### 12.7 版本与演进

长任务系统会持续演进，建议版本化以下对象：

- `task definition version`
- `workflow version`
- `sharder strategy version`
- `prompt / policy version`
- `tool contract version`
- `review rule version`

### 12.8 传统系统与 Agent 系统的治理对照

| 治理问题 | 传统系统做法 | Agent / Multi-Agent 做法 |
|----------|--------------|--------------------------|
| 状态推进 | 状态机 / Batch 状态表 | Runtime State / Session State |
| 恢复点 | Page / Cursor / Shard Checkpoint | Plan Checkpoint / Step Resume Point |
| 审批 | 审批节点 / 发布门 | Human-in-the-loop / Approval Gate |
| 风险控制 | 规则引擎 / 熔断 / 限流 | Guardrail / Tool Policy / Risk Router |
| 结果审查 | 校验器 / Reviewer | Critic / Evaluator / Reviewer Agent |
| 补偿 | DLQ / Replay / Rollback | Retry Queue / Task Replay / Human Repair |
| 审计 | 操作日志 / Ledger | Trace / Reasoning Log / Tool Call Log |

## 13. 落地建议与演进路线

### 13.1 默认设计原则

执行系统的默认起点，不应该是最复杂的方案，而应该是最小可行、边界清晰、可以继续演进的方案。

推荐判断顺序：

1. 先判断任务是不是请求内可闭环
2. 再判断是否需要异步解耦
3. 再判断是否需要显式状态与 Checkpoint
4. 再判断是否需要动态规划
5. 最后才判断是否需要角色化协作

本质上是：

```text
先解决执行跨度
  → 再解决认知复杂度
  → 最后解决协作复杂度
```

### 13.2 一条推荐的技术选型路径

#### 阶段 1：同步短任务

如果任务具备以下特征，优先选择同步接口、普通函数编排或轻量单 Agent：

- 单次交互内可完成
- 没有跨步骤恢复需求
- 没有显式等待
- 副作用小
- 失败可整体重试

#### 阶段 2：异步任务

如果主要问题是主链路不适合同步阻塞，但流程仍然简单，升级到 MQ Consumer、异步 Worker 或后台任务队列。

#### 阶段 3：Batch / Workflow

如果任务出现批量对象处理、长时间运行、Checkpoint、阶段边界、补偿和审计要求，就应升级到 Batch、分布式 Job 或阶段化 Workflow。

#### 阶段 4：Agent Runtime

如果任务在长任务基础上进一步出现输入不确定、步骤需要动态规划、工具路径不能完全预定义、需要根据中间结果改计划，就应进入 Stateful Agent Runtime 阶段。

#### 阶段 5：Multi-Agent

只有当规划、执行、审查职责冲突明显，任务天然可并行分治，且需要不同工具权限和专业角色时，才建议升级到 Multi-Agent。

### 13.3 三条典型演进路线

#### 路线 A：传统数据任务路线

```text
同步脚本
  → 异步任务
  → Batch
  → 分片批处理
  → 阶段化 Workflow
  → 强治理流水线
```

适用场景：

- 数据同步
- 对账
- 批量修复
- 内容发布
- 供应商集成

#### 路线 B：Agent 助手路线

```text
单轮问答
  → 工具调用 Agent
  → Plan-and-Execute
  → Stateful Agent Runtime
  → 带 Reviewer 的受控 Runtime
  → 按需升级为 Multi-Agent
```

适用场景：

- 智能助手
- 运维诊断
- 研发助手
- 调研助手
- 业务分析助手

#### 路线 C：复杂协作系统路线

```text
单执行者
  → 单执行者 + 审查者
  → Planner + Worker + Reviewer
  → Supervisor + 多 Worker
  → 领域化 Specialist Agents
```

适用场景：

- 复杂研发任务
- 多阶段知识工作
- 高风险自动化
- 跨领域决策支持

### 13.4 什么时候不要继续升级

#### 不要急着上 Workflow

- 任务仍然很短
- 没有恢复和补偿需求
- 阶段边界还不稳定

#### 不要急着上 Agent Runtime

- 规则和流程已经高度确定
- 不需要动态规划
- 当前问题本质上只是普通长任务

#### 不要急着上 Multi-Agent

- 单 Agent 已经足够
- 当前瓶颈在工具质量或状态托管
- 还没有清晰角色边界
- 缺的其实是 Reviewer、State Machine 或 Policy

### 13.5 一套可执行的设计检查清单

设计一个任务系统时，至少问以下十个问题：

1. 这个任务是请求内闭环，还是跨边界推进？
2. 失败后是整体重试，还是必须局部恢复？
3. 有没有明确的阶段边界和对象级状态？
4. 是否需要 Checkpoint？
5. 是否需要等待人工或外部事件？
6. 是否存在高风险副作用，需要审查或审批？
7. 任务路径是固定的，还是需要动态规划？
8. 单个执行单元是否已经承担了冲突职责？
9. 当前问题缺的是认知能力，还是治理能力？
10. 如果复杂度继续上升，当前方案是否能平滑演进？

### 13.6 全文结论

短任务、长任务、单 Agent 和 Multi-Agent 并不是四个割裂概念，而是统一执行模型在不同复杂度区间下的不同实现。

成熟的执行系统设计，核心不是盲目追求更智能或更多 Agent，而是在合适的复杂度区间选择合适的执行形态，并始终把以下能力放在核心位置：

- `状态`
- `恢复`
- `治理`
- `审计`

最终可以收束成三条原则：

1. `先看执行跨度`：是否需要跨边界持久推进
2. `再看认知复杂度`：是否需要动态规划和工具选择
3. `最后看协作复杂度`：是否需要角色分工与审查闭环

## 14. 附：本设计与供应商同步四阶段模型的关系

本文档中的统一执行模型，与供应商同步四阶段模型并不冲突，反而能解释它为什么工程上成立。

四阶段模型的价值在于，它把一个典型的高复杂度长任务拆成了可治理的职责边界：

- `Sharder`：决定切分策略和调度单位
- `Fetcher`：解决慢源拉取、分页推进和 Raw Snapshot 沉淀
- `Transformer`：解决语义翻译、标准化和基础风险标注
- `Publisher`：解决写前保护、版本控制、状态迁移和正式发布

这说明一个成熟的长任务系统设计，不是把所有逻辑塞进一个 Worker，而是主动把切分、执行、裁决和发布隔离开来。这个原则同样适用于 Stateful Agent Runtime 和 Multi-Agent 协作系统。
