## 1. 背景与目标

当前书稿已经形成 5 个部分、18 章、3 个附录的完整 mdBook 结构。书籍主线定位为：以 AI 工程实践为核心，覆盖 AI 编程、Agent 系统设计、完整案例、核心能力深化，并将面试部分作为实践体系的输出场景。

本次 review 的重点不是重写内容，而是解决目录和章节身份的问题：

1. 部分“专题”承担了主线章节功能，但在目录中显得像可选补充。
2. 第四部分名为“基础理论补充”，但其中 Prompt、Memory、检索系统都是 Agent 工程核心能力。
3. 面试表达分散在主线章节中，削弱了工程实践书的定位。
4. RAG 基础章节和向量检索专题存在重叠，需要重新划分边界。

优化目标：

- 保持“工程实践”为主线，不让面试内容反客为主。
- 将承担主线职责的专题升格为正式章节。
- 将第四部分从“理论补充”调整为“工程核心能力深化”。
- 将面试表达集中到第五部分，让它成为对前文体系的再组织。
- 尽量通过目录、标题、导读和少量内容迁移完成优化，避免大规模重写。

## 2. 推荐方案

采用“专题升格为章节，第四部分改名为工程深化”的方案。

### 2.1 为什么推荐该方案

该方案改动适中，但能明显改善读者体验：

- 生命周期专题内容完整，适合作为第二部分收束章节。
- Prompt、Memory、检索系统不再被读者误认为可跳过材料。
- 第五部分继续保留面试价值，但角色变为“实践体系的表达与输出”。
- 保留现有书稿资产，主要做结构和定位调整。

### 2.2 不采用的方案

保守方案：只把“专题”改成“深化专题”。  
缺点是目录层级仍然暗示这些内容是旁支，不能彻底解决主线断裂感。

激进方案：拆成工程主书和面试手册两本书。  
缺点是改动太大，也会削弱当前书稿“一本读完工程体系和面试表达”的完整价值。

## 5. 面试内容迁移原则

主线章节中不应频繁出现“面试官问”“面试表达”这类措辞。它会让读者误以为主书定位从工程实践切换成面试资料。

处理规则：

1. 第一至第四部分保留工程设计、检查清单、决策模板。
2. 直接面试话术迁移到第五部分对应章节。
3. 如果迁移成本较高，先将小标题从“面试表达”改为“工程表达”或“设计说明模板”。
4. 第五部分第 16 章增加一段说明：本部分会把前四部分的工程能力重新组织为面试表达、作品集和系统设计题。

示例：

```markdown
## 面试表达
```

可在主线章节中改为：

```markdown
## 设计表达模板
```

或者迁移到第五部分：

```markdown
### Prompt 工程如何讲清楚
```

## 6. RAG、Memory、检索章节边界

为了减少重复，第四部分内部需要明确边界。

### 第 13 章：RAG 与上下文工程基础

负责基础链路：

- RAG 的定义和适用场景。
- 文档分块、Embedding、向量库的入门实现。
- 基础检索、上下文构建、端到端 RAG 示例。
- 初级评估与优化。

不深入展开：

- 企业级 metadata 体系。
- 权限过滤细节。
- 增量索引更新。
- hybrid search / rerank 的生产权衡。

这些留给第 15 章。

### 第 14 章：Agent Memory 与状态管理

负责 Agent 的持续性：

- working memory、conversation memory、long-term memory、episodic memory。
- Task Store、Session Store、长期记忆的边界。
- 记忆写入、压缩、遗忘、污染防护。
- Memory eval。

不重复讲 RAG 的文档检索链路，只说明 episodic memory 通常可通过 RAG 实现。

### 第 15 章：检索系统工程

负责生产检索系统：

- Parser、Chunker、Metadata Enricher、Embedding、Vector Index。
- Hybrid Retrieval、Rerank、Context Builder。
- 权限过滤、增量更新、检索层 eval。
- 检索系统从 demo 到生产的工程闭环。

不重复 RAG 入门代码，只保留必要的系统设计说明。

## 9. 风险与注意事项

主要风险：

- 章节编号顺延会影响正文中的交叉引用。
- 已生成的 `ai-book/book/` 构建产物可能已有未提交变更，实施时应避免误覆盖用户改动。
- 第五部分章节编号变化较多，需要统一调整标题、目录和“下一章预告”。
- 只改标题不改导语，会导致“名义升格、语气仍像专题”的割裂感。

实施建议：

- 先改 `SUMMARY.md` 和 README，确认新目录。
- 再逐个处理标题和章节衔接。
- 最后做全文搜索，检查“专题：”“理论专题：”“面试表达”“下一章”等关键词。
- 构建前注意当前工作区已有用户改动，避免误提交无关文件。
