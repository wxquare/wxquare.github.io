### 第一部分：订单系统基础（40%）

#### 1. 系统概览

**目标**：建立订单系统的整体认知，理解业务场景和技术挑战

**内容**：
1.1 业务场景
- 订单在电商系统中的核心地位
- 订单系统的职责边界
- 与其他系统的交互（商品、库存、支付、物流、营销）

1.2 核心挑战
- 高并发：大促期间百万级订单/秒
- 强一致性：涉及库存、优惠、积分等多系统协调
- 状态复杂：订单生命周期涉及多个状态和转换
- 类型多样：物理订单、虚拟订单、O2O订单、预售订单等
- 幂等性：支付回调、物流回调等外部系统重试
- 可追溯：订单快照、状态变更历史

1.3 系统架构
- 整体架构图（外部绘制）：订单系统在电商平台中的位置
- 模块划分：
  - Order Service：订单核心服务
  - Payment Service：支付服务
  - Fulfillment Service：履约服务
  - After-sale Service：售后服务
- 技术栈：
  - 存储：MySQL（订单主数据）、Redis（缓存）、Elasticsearch（搜索）
  - 消息队列：Kafka（事件驱动）
  - 分布式事务：TCC/Saga框架

1.4 数据模型设计
- 订单主表（order）：订单基本信息
- 订单子表（order_item）：订单明细
- 订单快照（order_snapshot）：商品快照
- 订单状态变更历史（order_state_log）：状态变更记录
- 幂等表（idempotent_record）：幂等性控制
- ER图（Mermaid）

#### 5. 幂等性与去重

**目标**：深入讲解订单系统中的幂等性设计原则和实现方案

**内容**：
5.1 幂等性设计原则
- 什么是幂等性
- 为什么需要幂等性（外部重试、网络闪断、用户重复点击）
- 幂等性的边界：业务幂等 vs 技术幂等

5.2 幂等性实现方案
- Token机制（前端生成）：
  - 前端生成唯一Token
  - 后端校验Token并存储
  - 适用场景：用户主动操作
- 业务唯一键（后端生成）：
  - 基于业务参数生成唯一键
  - 数据库唯一索引约束
  - 适用场景：外部回调
- 分布式锁（Redis）：
  - Redis SETNX实现
  - 锁的过期时间设置
  - 适用场景：高并发防重
- 状态机防重：
  - 状态转换的幂等性
  - 状态机天然防重
  - 适用场景：状态驱动的操作

5.3 各场景幂等实现
- 订单创建幂等性：
  - 基于请求ID的Token机制
  - 防止用户重复下单
- 支付回调幂等性：
  - 基于支付平台订单号的业务唯一键
  - 防止重复扣款
- 履约操作幂等性：
  - 基于履约单号的业务唯一键
  - 防止重复发货
- 售后操作幂等性：
  - 基于售后单号的业务唯一键
  - 防止重复退款
- 营销扣减幂等性：
  - 基于请求ID的分布式锁
  - 防止重复扣减优惠券/积分

5.4 幂等性的监控和告警
- 幂等命中率监控
- 重复请求来源分析
- 异常幂等请求告警

##### 6.1 虚拟订单（Virtual Order）

**业务场景**：
- 数字商品：电子书、音乐、视频会员、游戏点卡
- 虚拟服务：在线课程、软件订阅、云服务

**特点和挑战**：
- 无物流：不需要发货和物流跟踪
- 即时履约：支付成功后立即完成履约
- 自动发货：系统自动发放虚拟商品（如激活码、权益）
- 幂等性更强：防止重复发货导致重复权益

**与通用流程的差异**：
- 订单创建：相同
- 订单支付：相同
- 订单履约：简化
  - 无"待发货"、"运输中"状态
  - 支付成功后直接变为"已完成"
  - 自动发放虚拟商品
- 订单售后：相同（但可能不支持退货，仅退款）

**状态机对比**：
- 通用订单：待支付 → 已支付 → 待发货 → 已发货 → 运输中 → 已送达 → 已完成
- 虚拟订单：待支付 → 已支付 → 已完成

**技术要点**：
- 虚拟商品发放：调用虚拟商品服务发放权益
- 发放失败补偿：重试 + 人工介入
- 幂等性：防止重复发放

##### 6.2 O2O订单（Online-to-Offline Order）

**业务场景**：
- 外卖订单
- 酒店预订
- 电影票
- 充值订单
- 礼品卡

**特点和挑战**：
- LBS定位：基于用户位置匹配商家/服务
- 即时配送：骑手实时配送，配送时效要求高
- 超时取消：超时未接单/配送自动取消
- 状态复杂：增加"商家接单"、"骑手接单"、"配送中"等状态

**与通用流程的差异**：
- 订单创建：增加LBS定位和商家匹配
- 订单支付：相同
- 订单履约：复杂化
  - 增加"待接单"、"商家已接单"、"骑手已接单"、"配送中"等状态
  - 超时取消：商家超时未接单自动取消
  - 实时追踪：骑手位置实时上报
- 订单售后：相同

**状态机对比**：
- 通用订单：待支付 → 已支付 → 待发货 → 已发货 → 运输中 → 已送达 → 已完成
- O2O订单：待支付 → 已支付 → 待接单 → 商家已接单 → 骑手已接单 → 配送中 → 已送达 → 已完成

**技术要点**：
- LBS定位：基于用户经纬度匹配商家
- 骑手调度：调用配送系统分配骑手
- 超时取消：定时任务扫描超时订单
- 实时追踪：WebSocket推送骑手位置

##### 6.3 预售订单（Pre-sale Order）

**业务场景**：
- 新品预售
- 大促预热
- 众筹商品

**特点和挑战**：
- 定金尾款：分两次支付（定金 + 尾款）
- 延迟履约：尾款支付后才发货
- 库存预留：定金支付后锁定库存
- 超时取消：尾款未支付自动取消并释放库存

**与通用流程的差异**：
- 订单创建：相同
- 订单支付：复杂化
  - 增加"定金已支付"、"待支付尾款"、"尾款已支付"状态
  - 定金支付后锁定库存
  - 尾款支付后才进入履约
- 订单履约：延迟
  - 尾款支付后才开始履约
- 订单售后：复杂化
  - 定金支付后取消：退定金
  - 尾款支付后取消：退全款

**状态机对比**：
- 通用订单：待支付 → 已支付 → 待发货 → ...
- 预售订单：待支付定金 → 定金已支付 → 待支付尾款 → 尾款已支付 → 待发货 → ...

**技术要点**：
- 定金尾款拆分：两个支付单
- 库存预留：定金支付后冻结库存
- 尾款超时：定时任务扫描超时未支付尾款订单
- 预留释放：超时后释放预留库存

**图表**：

### 第四部分：工程实践（10%）

#### 8. 实践要点

**目标**：提供订单系统工程实践的关键要点和最佳实践

**内容**：
8.1 订单ID生成
- Snowflake算法：
  - 时间戳（41位）+ 机器ID（10位）+ 序列号（12位）
  - 优点：全局唯一、趋势递增、高性能
  - 缺点：依赖时钟、需要机器ID分配
- UUID：
  - 优点：简单、无依赖
  - 缺点：无序、存储空间大
- 数据库自增ID：
  - 优点：简单
  - 缺点：性能瓶颈、分库分表困难
- 选型建议：推荐Snowflake

8.2 异步处理和削峰
- Kafka事件驱动：
  - 订单创建事件（OrderCreatedEvent）
  - 订单支付事件（OrderPaidEvent）
  - 订单发货事件（OrderShippedEvent）
  - 订单完成事件（OrderCompletedEvent）
- Worker池批量处理：
  - 消费者组并发消费
  - 批量处理提高吞吐
  - 失败重试和死信队列
- 削峰填谷：
  - 消息队列缓冲
  - 限流和熔断

8.3 监控告警体系
- 业务指标：
  - 订单创建量、支付成功率、履约完成率、售后率
  - 各状态订单数量分布
  - 订单超时率
- 应用指标：
  - 接口响应时间、QPS、错误率
  - 事务成功率、补偿执行率
  - 幂等命中率
- 依赖监控：
  - 下游服务可用性（库存、支付、物流）
  - 消息队列延迟
  - 数据库慢查询
- 系统指标：
  - CPU、内存、磁盘、网络
- 告警策略：
  - 分级告警（P0/P1/P2）
  - 告警收敛
  - 告警升级

8.4 性能优化
- 数据库优化：
  - 索引优化：订单号、用户ID、状态、创建时间
  - 分库分表：按用户ID或订单ID哈希
  - 读写分离：主从同步
- 缓存优化：
  - Redis缓存订单详情
  - 缓存失效策略：主动更新 + TTL
  - 缓存雪崩、穿透、击穿防护
- 接口优化：
  - 批量接口：批量查询订单
  - 异步接口：订单创建改为异步
  - 限流和熔断：保护核心接口

8.5 故障处理
- 常见故障：
  - 库存不足：提示用户并引导到其他商品
  - 支付超时：自动取消并释放库存
  - 履约失败：重试 + 人工介入
  - 消息丢失：本地消息表保证可靠性
- 故障预案：
  - 数据库主从切换
  - 缓存集群故障降级
  - 消息队列故障降级
  - 下游服务故障熔断
- 灾备演练：
  - 定期演练故障场景
  - 验证故障预案有效性
