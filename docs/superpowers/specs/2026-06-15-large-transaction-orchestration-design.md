# 大事务处理方法论章节设计

- 日期：2026-06-15
- 目标书籍：`books/system-design-architecture-book`
- 目标位置：第一部分《系统设计方法论》
- 设计主题：新增“大事务处理方法论”章节，以电商创单为主案例，抽象流程编排 / Saga / Workflow 方法论

## 1. 背景与目标

当前第一部分已经覆盖了：

- 系统设计与技术方案方法论
- 架构边界与结构设计
- 编码原则与设计模式
- 可靠性、恢复与资损防控
- 任务处理方法论

但仍然缺少一块关键内容：**跨系统业务流程的大事务处理方法论**。

这类问题在互联网系统里极其常见，例如：

- 电商下单：库存预占 -> 价格确认 -> 创建订单 -> 发起支付 -> 支付回写 -> 失败释放
- 用户注册 / 认证：账号创建 -> 风控校验 -> 短信 / 邮件验证 -> 用户状态激活
- 退款 / 售后：售后申请 -> 审核 -> 退款发起 -> 库存 / 权益回补 -> 状态关闭
- 审核 / 发布流程：草稿提交 -> 质检 / 审批 -> 发布生效 -> 下游投影同步

这些场景都不是单机事务，也不适合用 2PC 强行覆盖。它们真正需要的是：

- 多步骤协调
- 跨系统状态推进
- 补偿与超时控制
- 可审计、可恢复、可人工接管
- 适当的编排模型与工作流基础设施

本章节的目标不是介绍某个特定产品，而是建立一套稳定的方法论，帮助读者回答：

1. 什么问题本质上是“大事务 / 流程编排”问题。
2. 什么场景可以继续用同步 RPC，什么场景必须引入 Saga / Workflow。
3. 编排式 Saga、协同式 Saga、状态机、工作流引擎分别解决什么问题。
4. 如何用电商创单这一主案例，抽象出注册、退款、售后、审核流等通用模式。

## 2. 章节定位

本章放在第一部分《系统设计方法论》中，定位为**方法论章节**，而不是电商垂直业务章节。

写法采用：

- **问题驱动为主**
- **电商创单作为主案例锚点**

这意味着章节不会写成“订单系统实现细节”，而会始终围绕以下抽象问题展开：

- 大事务到底是什么
- 为什么它不能等价成数据库事务
- 为什么创单天然会演化为跨系统流程编排问题
- 编排层的职责边界是什么
- 如何做状态推进、补偿、超时、审计和人工接管

## 4. 建议大纲

### 6.1 大事务到底在解决什么问题

目标：

- 区分数据库事务、分布式事务、业务大事务
- 明确这里讨论的是“一个业务目标跨多个本地事务和多个系统完成”的问题
- 说明为什么不能把这类问题简单压缩成 2PC

### 6.2 为什么创单流程天然是大事务

目标：

- 拆解创单主链路：库存预占、价格确认、订单创建、支付发起、支付回写、失败释放
- 说明这些动作分别属于不同系统、不同本地事务、不同失败语义
- 让读者先看到问题全貌，再进入方法论

### 6.3 如何判断一个流程该不该进入编排层

建议判断维度：

- 跨系统数
- 步骤数
- 是否需要补偿
- 是否存在长等待
- 是否存在人工审批 / 接管
- 是否必须保留审计轨迹
- 是否需要显式状态持久化

输出形式：

- 一张判断清单
- 一棵决策树

### 6.4 四种主流方案的能力边界

对同一个创单场景做横向比较：

1. 同步 RPC 短流程
2. 协同式 Saga
3. 编排式 Saga
4. Workflow / 状态机引擎

建议比较维度：

- 实现复杂度
- 一致性控制力
- 补偿能力
- 状态可见性
- 审计能力
- 适合流程长度
- 团队维护成本

### 6.5 大事务系统的核心不是“调接口”，而是“状态推进”

这是全章方法论中心。

需要明确讲清：

- 状态才是最小治理单元，不是接口调用
- 步骤边界应该围绕“可提交 / 可恢复 / 可补偿”来划分
- 补偿不是“反向调一次接口”，而是业务状态回收
- 超时、重试、人工接管都应该作用在状态推进模型上

### 6.6 状态机、Saga 与工作流引擎怎么选

方法论重点：

- 什么时候数据库状态机足够
- 什么时候需要应用层 Orchestrator
- 什么时候需要独立 Workflow Engine
- 什么时候 BPMN 值得引入

技术映射：

- Temporal
- Camunda
- Zeebe
- 自研状态机 / 自研编排器

章节重点是选型边界，不是工具教程。

### 6.7 从创单扩展到其他业务流程

用简短映射展示方法论普适性：

- 用户注册 / 认证
- 退款 / 售后
- 审核 / 发布
- 供应商同步发布

重点强调：创单不是特例，而是流程编排问题的代表。

### 6.8 方法论沉淀：如何在真实系统中做出选择

最后收束为可直接复用的表达：

- 一张选型矩阵
- 一套判断句
- 一段面试 / 评审话术

## 6. 与现有章节的衔接

### 与第 4 章《生产级系统保障》的关系

第 4 章更偏“系统如何稳”，本章更偏“跨系统业务流程如何编排成可恢复事务”。两章关系是：

- 第 4 章强调可靠性、恢复、补偿、告警、资损治理
- 第 6 章强调这些能力在跨系统业务流程里如何落地为状态推进与编排机制

### 与第 7 章《任务处理方法论》的关系

本章和第 7 章不是重复，而是递进：

- 第 6 章重点是业务流程编排 / Saga / 工作流 / 大事务
- 第 7 章重点是更一般的短任务、长任务、Batch、Workflow、Stateful Agent、Multi-Agent

本章解决的是“跨系统交易 / 审核 / 退款 / 发布”类流程的治理；
第 7 章解决的是“更广义任务系统”的治理。

## 7. 写作边界

本章明确不做：

- 不展开讲 2PC / XA 的完整协议细节
- 不展开讲 Temporal / Camunda / Zeebe 的 API 教程
- 不把章节写成某个单一订单系统实现说明
- 不深入数据库表结构和具体部署脚本

本章重点是：

- 建立判断框架
- 用创单案例压实抽象
- 给出方案取舍
- 服务于方案设计、架构评审和面试表达

## 8. 推荐写作节奏

建议比例：

- 前 30%：定义问题
- 中间 50%：对比方案
- 后 20%：沉淀方法论

建议叙事节奏：

1. 先让读者感到“创单为什么天然麻烦”
2. 再解释“为什么不是所有麻烦都值得上工作流引擎”
3. 最后给出一套能落地的选型与治理方法
