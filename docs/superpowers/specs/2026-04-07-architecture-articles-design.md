# 架构方法论文章体系设计规格

## 概述

- **目标**：将博客中的架构与编码实践内容形成"三篇联动"体系
- **三个产物**：
  1. `15-clean-code.md` — 补齐空壳章节（十~十三章）
  2. `26-clean-architecture-ddd-cqrs.md` — 深化扩展
  3. `27-architecture-checklist.md` — 新建，按审查阶段组织的 Code Review Checklist
- **关系**：27 是可执行的检查清单，15 和 26 是检查项背后的"为什么"和"怎么做"

## 三篇文章定位

| 文章 | 定位 | 读者场景 |

| 15-clean-code | "怎么写" — 编码层面的模式与反模式 | 写代码时查阅、重构时参考 |

### 文章结构

#### 引言（约 100 行）

- 为什么需要系统化的检查规约
- 本清单的使用方法（按阶段走，每个阶段过一遍对应的检查项）
- 与 15、26 号文章的关系（本文是 what to check，那两篇是 why 和 how）

#### 一、架构评审阶段 —— 设计期（约 200 行）

在项目启动或新模块设计时使用。

检查项：
1. **分层结构**：是否定义了清晰的层级（Domain / Application / Adapter / Infra）？依赖方向是否向内？
2. **Bounded Context 划分**：是否识别了核心域、支撑域、通用域？上下文之间的通信方式是否明确？
3. **聚合边界**：每个聚合的一致性边界是否合理？是否存在跨聚合的事务？
4. **读写路径评估**：读写比是否悬殊？是否需要 CQRS？是否需要独立的读模型？
5. **技术选型**：存储选型是否与读写特征匹配？是否需要 Event Bus？
6. **过度设计检查**：是否有 YAGNI 违规？当前复杂度是否真的需要 DDD/CQRS？

#### 二、设计评审阶段 —— 详设期（约 250 行）

在详细设计评审时使用。

检查项：
1. **聚合根识别**：聚合根是否是外部访问聚合的唯一入口？
2. **实体 vs 值对象**：是否正确区分了实体和值对象？值对象是否不可变？
3. **Repository 接口**：是否定义在领域层？是否只暴露业务需要的方法（非 generic CRUD）？
4. **Command 设计**：Command 是否表达业务意图（PlaceOrder vs CreateOrder）？是否只返回执行状态？
5. **Query 设计**：Query 是否直接返回 DTO？是否绕过领域模型？
6. **领域事件**：关键业务状态变更是否发布领域事件？事件命名是否用过去式（OrderPlaced）？
7. **模式选型**：
   - 多步骤流程 → 是否考虑 Pipeline？
   - 多分支逻辑 → 是否考虑策略模式？
   - 动态规则 → 是否考虑规则引擎？

#### 三、代码评审阶段 —— PR 期（约 400 行）

日常 Code Review 时使用，检查项最多、最细。

**3.1 SOLID 原则**
1. **SRP**：每个 struct/函数是否只有一个变化原因？
2. **OCP**：新增业务逻辑是否不需要修改已有代码？
3. **LSP**：接口的实现是否可以互换而不影响调用方？
4. **ISP**：接口是否足够小？是否存在"胖接口"？
5. **DIP**：高层模块是否依赖抽象而非具体实现？

**3.2 函数质量**
1. 函数长度是否 < 80 行？
2. 圈复杂度是否 < 10？
3. 嵌套深度是否 < 3 层？
4. 参数个数是否 < 5 个？超过时是否用 Options Pattern 或 Context？

**3.3 命名与通用语言**
1. 变量/函数/类型名是否反映业务术语？
2. 是否与团队的 Ubiquitous Language 一致？
3. 是否存在技术术语代替业务术语的情况（如 SetStatus(1) vs Place()）？

**3.4 错误处理**
1. 是否所有错误都被处理（不存在 _ = xxx）？
2. 错误是否携带足够上下文（wrap error）？
3. 是否区分了业务错误和系统错误？

**3.5 依赖方向**
1. domain 包是否有任何 import 指向 adapter/infra？
2. application 包是否只依赖 domain？
3. 是否存在循环依赖？

**3.6 DDD 战术模式**
1. 聚合根的方法是否保护了业务不变量？
2. 值对象是否不可变（没有 setter）？
3. 是否在聚合外部直接修改聚合内部的实体？
4. 一个事务是否只修改一个聚合？

#### 四、上线前检查 —— 合并期（约 150 行）

Merge 前的最终检查。

检查项：
1. **性能**：关键路径是否有 benchmark？是否有内存泄漏风险（goroutine、channel）？
2. **并发安全**：共享状态是否有锁保护或使用了并发安全的数据结构？
3. **可观测性**：关键操作是否有 metrics 埋点？是否接入 trace？日志是否结构化？
4. **测试覆盖**：核心业务逻辑测试覆盖率 > 80%？是否有集成测试？
5. **回滚方案**：是否支持 feature flag？数据库变更是否可回滚？
6. **文档更新**：架构变更是否更新了相关文档？

#### 附录：快速参考卡片（约 100 行）

一页纸的精简版 Checklist（每阶段 5 条最关键的检查项），适合打印或钉在屏幕旁边。
