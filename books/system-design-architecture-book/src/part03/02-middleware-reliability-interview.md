# 第 32 章 中间件与可靠性高频追问

系统设计面试中的中间件问题，很少只考概念定义。面试官更关心你能否把 MySQL、Redis、Kafka、Elasticsearch 和可靠性手段放进具体业务链路里，解释为什么选、怎么用、失败后怎么办。

## MySQL 高频追问

**为什么核心交易状态通常放在 MySQL？**

因为订单、支付、库存流水这类数据需要事务、约束、审计和可恢复性。MySQL 不一定承担所有读请求，但适合作为权威数据源。回答时要强调：缓存和搜索可以加速读，消息可以异步扩散，但最终状态要有一个可信来源。

**库存扣减如何防止超卖？**

常见回答是用条件更新、乐观锁、Redis 原子扣减或库存预占。更完整的回答要补充：Redis 适合高并发预扣，MySQL 库存流水和对账负责兜底；支付失败、订单取消、超时未支付都要释放预占；重复请求要通过幂等键和状态机控制。

## Redis 高频追问

**缓存和数据库不一致怎么办？**

先判断业务能接受多长时间的不一致。常见策略包括缓存过期、更新后删除缓存、延迟双删、消息驱动刷新和读写都走权威存储。不要承诺绝对一致，除非业务链路真的能承担同步写入成本。

**分布式锁能解决所有并发问题吗？**

不能。分布式锁只是在特定临界区内减少并发冲突，仍然要考虑锁超时、误释放、续租、主从切换和业务幂等。涉及资金、库存、订单状态时，数据库唯一约束、条件更新和状态机通常比单纯依赖锁更可靠。

## Kafka 高频追问

**Kafka 如何保证消息不丢？**

要分段回答：Producer 到 Broker 依赖 `acks`、重试和幂等 Producer；Broker 内部依赖副本和 ISR；Consumer 侧依赖处理成功后再提交 offset。最后要补一句：端到端可靠性仍然需要业务幂等、Outbox、补偿和监控。

**消息重复怎么办？**

默认按至少一次投递设计。消费者使用业务唯一键、消息 ID 去重表、状态机条件更新或幂等写入来保证重复消费结果一致。不要把 Kafka 的 Producer 幂等误认为业务幂等。

## Elasticsearch 高频追问

**为什么搜索不用 MySQL 直接查？**

如果只是主键查询或简单条件查询，MySQL 足够。只有当业务需要全文检索、多字段筛选、相关性排序、聚合和深分页治理时，Elasticsearch 才更合适。引入 ES 后要接受索引延迟，并设计重建索引、补偿同步和降级查询。

## 可靠性高频追问

**超时、重试和幂等是什么关系？**

超时让请求不要无限等待，重试提高瞬时失败下的成功率，幂等保证重试不会造成重复副作用。没有幂等的重试可能把支付、库存和消息消费变成事故放大器。

**什么时候用熔断，什么时候用降级？**

熔断主要保护调用方不被下游拖垮，降级主要保护核心业务不被非核心能力拖垮。例如推荐系统异常时商品详情页可以降级推荐模块；支付渠道异常时不能简单忽略支付结果，而要切渠道、挂起状态或进入人工处理。

## 回答模板

遇到中间件追问，可以按下面的结构回答：

1. 先说明这个组件解决的业务问题。
2. 再说明它不承担什么职责。
3. 然后讲正常链路如何工作。
4. 再讲失败、重复、延迟、积压等异常处理。
5. 最后补充监控、对账、补偿和演进空间。

这个顺序能让回答从“背组件”变成“做系统设计”。
