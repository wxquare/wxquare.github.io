# 第 36 章 白板答辩与容量估算表达

系统设计面试的白板表达，本质上是在有限时间内把问题拆开、把取舍讲清楚、把风险兜住。好的白板不是图画得复杂，而是让听的人知道每条线、每个组件、每个数据流为什么存在。

## 白板答辩顺序

建议按下面的顺序组织：

1. **确认目标**：业务目标、核心指标、非目标。
2. **估算规模**：用户量、QPS、数据量、峰值、增长。
3. **画主链路**：入口、服务、存储、缓存、消息和下游。
4. **拆读写路径**：读路径如何加速，写路径如何保证正确性。
5. **处理异常**：超时、重试、幂等、补偿、降级。
6. **说明演进**：从简单方案到复杂方案如何平滑升级。

先报结构再展开，会让你的回答更像技术方案评审，而不是临场背诵。

## 容量估算口径

容量估算不要求精确，但要有口径。以电商下单链路为例，可以这样估：

```text
DAU = 1000 万
高峰小时活跃 = DAU * 10% = 100 万
下单转化 = 高峰活跃 * 5% = 5 万单/小时
平均下单 QPS = 50000 / 3600 ≈ 14 QPS
峰值放大 = 10 倍到 50 倍
峰值下单 QPS ≈ 140 到 700 QPS
```

如果是大促秒杀，不能用均匀分布估算，要按活动瞬时峰值估算。例如 10 万人同时抢 1 万件商品，入口请求、库存扣减和排队系统的设计会完全不同。

## 常见估算维度

| 维度 | 估算内容 |
| --- | --- |
| 读请求 | 首页、详情页、搜索、购物车刷新 |
| 写请求 | 加购、结算、下单、支付回调、取消订单 |
| 数据增长 | 订单、库存流水、支付单、消息、日志 |
| 热点 | 秒杀商品、热门搜索词、大促活动 |
| 异步流量 | 消息重试、索引同步、对账、补偿 |
| 外部依赖 | 支付渠道、供应商接口、短信和通知 |

估算后要能落到设计选择：读请求高就考虑缓存和读模型，写请求高就考虑分片和削峰，热点集中就考虑预热、隔离和限流，异步流量大就关注积压和回压。

## 图应该画到什么粒度

白板图不需要一开始画出所有细节。第一张图应该是主链路：

```text
Client -> API Gateway -> Application Service -> DB
                         -> Cache
                         -> MQ -> Downstream Services
```

然后根据追问展开局部，例如库存扣减、支付回调、搜索索引同步。不要一开始画十几个服务，否则容易失去重点。

## 面试表达句式

可以使用下面这些句式让表达更稳定：

- “这里我先把权威数据和读模型分开。”
- “这条链路允许最终一致，但需要补偿和对账兜底。”
- “这个组件解决的是吞吐问题，不解决业务幂等问题。”
- “如果规模还没到这个量级，可以先用单库方案，保留分片键和归档策略。”
- “故障时优先保护下单和支付主链路，推荐、通知、非关键营销可以降级。”

这些句式的价值在于，它们把技术选择和业务取舍绑定在一起。

## 收尾方式

回答结束前，最好主动补充风险和演进：

1. 当前方案的瓶颈在哪里。
2. 哪些数据需要监控。
3. 哪些地方需要降级和补偿。
4. 规模扩大后如何演进。

一个完整的系统设计答案，不是把所有组件都说完，而是让面试官相信：你知道这个系统上线后会在哪里出问题，也知道如何提前布防。
