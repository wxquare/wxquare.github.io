# 第 17 章 对账、补偿、DLQ 与故障恢复

分布式系统无法保证每一次跨系统协作都在同一时间成功。订单创建、库存预占、支付回调、营销核销、搜索索引更新、供应商同步这些链路都会遇到超时、重复、乱序、部分成功和下游不可用。对账、补偿和 DLQ 的价值，就是承认失败必然发生，并把失败从“不可见的数据错乱”变成“可发现、可隔离、可修复的任务”。

## 对账解决什么问题

对账不是财务系统独有的概念。只要两个系统之间存在异步同步、最终一致或外部依赖，就需要对账。

| 场景 | 对账对象 | 典型异常 |
| --- | --- | --- |
| 支付回调 | 支付渠道订单与本地支付单 | 渠道成功、本地未更新 |
| 库存扣减 | 库存流水与可售库存快照 | 流水成功、快照未刷新 |
| 搜索索引 | 商品主数据与 ES 文档 | 商品已下架、搜索仍可见 |
| 供应商同步 | 供应商库存与平台库存 | 供应商变更未同步 |
| 消息投递 | Outbox 表与 Kafka 消费位点 | 事件已生成、下游未处理 |

对账的关键不是“定时扫一遍”，而是明确权威数据源、比对维度、差异容忍窗口和修复动作。没有修复动作的对账只能制造告警噪声。

## 补偿任务的设计

补偿任务应该具备三个特征：

1. **幂等**：重复执行不会造成二次扣款、重复发券或重复扣库存。
2. **可追踪**：每一次补偿都有任务 ID、业务 ID、执行状态、失败原因和操作人。
3. **有上限**：失败重试要有退避、次数限制和人工介入入口。

一个常见的补偿任务表可以包含：

| 字段 | 含义 |
| --- | --- |
| `task_id` | 补偿任务 ID |
| `biz_type` | 业务类型，例如 payment、inventory、search_index |
| `biz_id` | 业务对象 ID |
| `status` | pending、running、success、failed、manual_required |
| `retry_count` | 已重试次数 |
| `next_retry_at` | 下一次重试时间 |
| `error_code` | 失败分类 |
| `error_message` | 最近一次失败原因 |
| `created_at` / `updated_at` | 创建和更新时间 |

补偿任务不要直接替代主链路事务。主链路仍然要尽可能保证局部正确性，补偿负责处理跨系统不确定性。

## DLQ 的职责边界

DLQ（Dead Letter Queue）用于接住无法继续自动处理的消息或任务。进入 DLQ 的数据意味着系统已经判断“继续立即重试大概率没有意义”，需要隔离、观察或人工修复。

常见进入 DLQ 的原因包括：

- 消息格式不兼容，消费者无法解析。
- 业务对象不存在，短期重试无法恢复。
- 下游连续失败，超过最大重试次数。
- 数据违反约束，需要人工判断。
- 依赖系统返回不可重试错误。

DLQ 不能只是一个“失败垃圾桶”。它至少要支持查看、分类、重新投递、跳过、标记已处理和导出排查。否则失败只是换了一个地方堆积。

## 故障恢复的分层

故障恢复可以分为四层：

| 层次 | 目标 | 常见手段 |
| --- | --- | --- |
| 请求级恢复 | 单次请求失败后快速返回或重试 | 超时、重试、幂等键 |
| 任务级恢复 | 异步任务失败后继续推进 | 重试队列、补偿任务、DLQ |
| 数据级恢复 | 多系统数据重新收敛 | 对账、重放、人工修复 |
| 系统级恢复 | 整个服务从故障中恢复 | 降级、切流、回滚、灾备 |

设计时要避免把所有恢复责任都压到某一层。只靠请求重试会放大故障，只靠人工修复会拖垮运维，只靠对账会让用户体验长时间不一致。成熟的系统通常是多层共同承担。

## 电商链路中的落点

在电商系统中，对账、补偿和 DLQ 会贯穿全链路：

- 商品发布后，同步搜索索引失败，需要补偿重建索引。
- 库存预占成功但订单创建失败，需要释放预占。
- 支付渠道成功但本地回调处理失败，需要支付对账修正状态。
- 订单取消后优惠券释放失败，需要营销补偿。
- 供应商数据解析失败，需要进入任务明细失败表或 DLQ 等待人工处理。

这也是为什么后面的电商实战篇会反复出现幂等、状态机、Outbox、对账和补偿。它们不是附加设计，而是业务系统能长期运行的基础能力。
