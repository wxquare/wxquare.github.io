# 第29章 模型微调：从基础原理到 DoD Agent 的生产落地

> 微调不是给 Agent 加一层“更聪明”的魔法，而是在系统边界已经清楚之后，把一类稳定任务的行为模式固化下来，让模型在该稳定的地方更稳定，让系统在该受控的地方更受控。

## 引言

到了这本书的后半段，读者通常会自然提出一个问题：

> 如果 Prompt、RAG、Tools、Workflow、Memory、Evals 和 Guardrails 都已经搭起来了，为什么还需要模型微调？

这是一个非常好的问题，因为它逼着我们把“模型能力”与“系统能力”分开。

在很多真实项目里，团队遇到效果不稳定时，最先想到的是：

```text
换更大的模型
  -> 补更长的 Prompt
  -> 再堆更多上下文
  -> 还是不稳定
  -> 开始考虑微调
```

这条路径并不总是错，但非常容易走偏。原因是很多问题从来就不是“模型没学会”，而是：

- 任务协议没有定义清楚；
- 检索内容和实时事实混在一起；
- 关键证据没有进上下文；
- 工具调用链不稳定；
- 训练样本和线上任务分布不一致；
- 输出没有可评分结构；
- 失败没有进入回归门禁。

如果这些问题没有先解决，微调只是把噪声和混乱重新编码到模型里。

但反过来，当一个 Agent 系统已经进入稳定运营期，某些任务具备下面这些特征时，微调就非常值得认真考虑：

- 高频重复；
- 输入分布相对稳定；
- 输出协议明确；
- 已经积累了大量人工修正样本；
- 错误主要来自行为模式不稳定，而不是事实缺失；
- 可以定义清晰评测指标；
- 可以通过 Shadow Mode 或建议层安全上线。

本章就是围绕这个阶段展开。它分成两大部分：

1. 先讲微调本身：微调到底是什么，有哪些类型，解决什么问题，不解决什么问题，完整工程流程是什么，需要哪些资源，以及如何判断值不值得做。
2. 再用 DoD Agent（Developer on Duty Agent）做一个生产级案例：如果要把微调真正用在企业告警诊断与知识答疑系统里，任务应该怎么选，数据应该怎么构造，训练应该怎么分阶段，评测和上线又该怎么做。

这章不会把微调写成一篇纯算法导论，也不会只停留在“收集数据然后训练一下”这种表面描述。重点始终是工程落地：

```text
任务定义
  -> 数据边界
  -> 训练方法
  -> 评测体系
  -> 发布策略
  -> 风险控制
  -> 失败回流
```

## 29.1 什么是模型微调

模型微调（Fine-tuning）本质上是在已有基础模型之上，使用一组和目标任务相关的样本继续训练，让模型更稳定地表现出某种特定能力模式。

“能力模式”这四个字比“知识”更准确。因为微调最擅长学到的往往不是“今天最新发生了什么”，而是：

- 应该如何组织输出；
- 面对某类输入时应该优先关注什么；
- 什么情况下应该拒答或澄清；
- 什么风格、术语和结构更符合领域要求；
- 哪些判断边界要更保守。

### 29.1.1 微调、Prompt、RAG、Tool Calling 的区别

把这四种方法放在同一张表里，很多误解就会消失：

| 方法 | 核心作用 | 改变的对象 | 典型优势 | 典型局限 |
|:---|:---|:---|:---|:---|
| Prompt Engineering | 临时指定本次任务规则 | 当前输入上下文 | 快、便宜、改动成本低 | 稳定性有限，容易被上下文干扰 |
| RAG | 动态补充外部知识 | 当前上下文证据 | 适合私有知识和实时文档 | 无法改变模型的内在行为习惯 |
| Tool Calling | 让模型使用外部能力 | 当前执行路径 | 解决“模型没有事实或操作能力”的问题 | 需要可靠 Runtime 和权限治理 |
| Fine-tuning | 固化一类稳定任务模式 | 模型参数或适配层 | 一致性强、token 更省、格式更稳 | 需要数据、训练、评测、回滚 |

一个非常实用的判断是：

```text
Prompt 负责告诉模型“这次想让你怎么做”
RAG 负责告诉模型“你现在应该参考哪些知识”
Tools 负责告诉模型“你现在可以去外部世界拿哪些事实”
Fine-tuning 负责让模型“以后更自然、更稳定地按这种模式去做”
```

### 29.1.2 微调不等于训练一个“万能专家”

初学者最常见的误解是：

> 我想把模型微调成“支付系统专家”“SRE 专家”“客服专家”“故障处理专家”。

这种说法通常过于笼统。更准确的目标应该是：

- 让模型在支付类告警上更稳定地做严重级别初判；
- 让模型在值班场景下更稳定地生成结构化 Incident Summary；
- 让模型在客服工单归类上更稳定地输出标准标签；
- 让模型在审核场景下更稳定地遵循更保守的拒答边界。

也就是说，**微调更适合学习稳定任务协议，而不是笼统地学习一个职业身份。**

### 29.1.3 微调到底在“调”什么

从工程视角看，微调会改变的通常不是模型的全部世界知识，而是下面几个维度的分布：

- 任务启动方式：模型是否更快进入正确任务模式；
- 注意力偏好：面对输入时更优先关注哪些字段；
- 输出先验：更倾向产出什么格式和风格；
- 风险偏好：是否更保守、更愿意澄清或拒绝；
- 标签边界：对固定分类任务的分界是否更稳定；
- 领域术语：是否更贴近你所在业务的语言习惯。

这也是为什么很多团队在做微调前，必须先明确自己要优化的是哪一种不稳定。

## 29.2 微调的主要类型：SFT、偏好优化与强化式优化

很多书一讲微调，就默认只有一种。真实工程里至少应该区分三类路线：

- SFT（Supervised Fine-Tuning，监督微调）
- 偏好优化（如 DPO、IPO 等）
- 强化式优化（如基于奖励的在线或离线优化）

### 29.2.1 SFT：最常见、最实用、最容易落地

SFT 的基本形式非常直接：

```text
输入 -> 标准输出
```

你给模型大量“这个输入应该怎么回答”的例子，模型学习复现这种模式。

适合 SFT 的任务包括：

- 文本分类；
- 结构化摘要；
- 标准化问答；
- 模板化报告；
- 安全拒答格式；
- 工单归类与路由；
- 告警摘要初稿生成。

SFT 是企业里最常见的第一步，因为它：

- 数据形态直观；
- 成本相对可控；
- 易于离线评测；
- 风险边界相对清晰；
- 与现有人工修正轨迹天然兼容。

### 29.2.2 偏好优化：解决“多个答案都差不多，但我们更喜欢哪种”

很多任务不是严格只有一个标准答案，而是存在多个“都说得过去”的版本。比如：

- 两份 Incident Summary 都正确，但一份更简洁；
- 两个处理建议都安全，但一份更符合团队惯例；
- 两个客服回复都合规，但一份更自然；
- 两个 Runbook 摘要都没错，但一份证据引用更清楚。

这类问题用 SFT 也能做，但信号不够充分。偏好优化更适合表达：

```text
回答 A 和回答 B 比较起来
  -> 我们更偏好 A
```

它解决的是“排序偏好”，而不是“唯一标准答案”。

在 DoD 场景里，偏好优化可以用在：

- Incident Summary 的表达质量排序；
- 建议动作的保守性排序；
- 升级建议的措辞风格；
- 多个可接受答案之间的优先级学习。

### 29.2.3 强化式优化：适合长链路任务，但工程复杂度最高

如果任务是多步的，而且最终质量取决于整个行动链路，而不是单次文本输出，强化式优化才更有意义。例如：

- 多步工具调用任务；
- 长时规划；
- 浏览器或 CLI 任务执行；
- 需要探索、试错、回退的复杂工作流。

但对大多数企业内部 Agent 团队来说，这通常不是第一阶段的最佳投入点，因为它要求：

- 更可靠的 reward 设计；
- 更复杂的日志与轨迹系统；
- 更高的训练和验证成本；
- 更强的安全护栏；
- 更严密的回放与回归验证机制。

因此，本章后面在 DoD Agent 里会把重点放在更现实的路线：**SFT 为主，必要时配合偏好优化**。

### 29.2.4 一个简单的选择建议

可以用下面这个决策树：

```text
任务是否有相对明确的标准输出？
  是 -> 先做 SFT
  否 -> 看是否有成对偏好数据

是否主要在比较“哪个答案更好”？
  是 -> 考虑偏好优化

是否是长链条、多步、依赖整个执行过程的任务？
  是 -> 再考虑强化式优化
```

对于 DoD 这样高风险、强治理的系统，大多数真正有 ROI 的早期微调任务都落在第一类。

## 29.3 微调解决什么问题，不解决什么问题

微调很强，但不是万能药。很多失败项目，本质上就是把不该交给微调的问题硬塞给微调。

### 29.3.1 适合微调的问题特征

一个任务适合微调，通常同时满足下面几个条件：

1. 输入分布相对稳定。
2. 输出协议明确。
3. 可以定义对错或优劣。
4. 训练样本可持续积累。
5. 错误主要来自行为模式不稳定，而不是事实缺失。

符合这些特征的常见任务包括：

- 固定格式输出；
- 多标签或单标签分类；
- 风格统一的摘要；
- 结构化字段提取；
- 审核决策初判；
- 工单路由；
- 交接文档标准化；
- 标准化建议生成。

### 29.3.2 不适合优先靠微调解决的问题

下面这些问题，通常不应该把微调放在第一位：

- 高频变化知识：最新 owner、最新配置、最新审批流程；
- 强实时事实：当前日志、当前指标、当前 Trace、当前数据库状态；
- 复杂工具缺失：明明该查工具，却希望模型“脑补”；
- 边界不清：一个任务同时混了检索、推理、审批、执行和解释；
- 极高风险决策：资损、权限、价格、账务、数据修正、合规处罚；
- 强来源要求：必须严格带引用、带时间窗口、带证据 ID 的结论。

这些问题里最典型的一类是：

```text
“模型总是答错当前事实”
```

这类问题大部分不应该通过“继续训练更多旧数据”解决，而应该通过：

- 更好的 Context Builder；
- 更好的 RAG；
- 更好的工具接入；
- 更严格的引用校验；
- 更清晰的失败策略。

### 29.3.3 一个非常实用的判断公式

可以把问题粗略分成两类：

```text
错误来自“缺事实”
  -> 优先做 RAG / Tools / Context

错误来自“有事实但行为不稳”
  -> 可以考虑 Fine-tuning
```

这条规则在 Agent 系统里尤其重要。因为 Agent 的错误很少纯粹来自模型参数，更多来自系统层组合问题。

### 29.3.4 什么时候是 Prompt 问题，什么时候是微调问题

还可以再细一点：

| 现象 | 更像什么问题 | 更优先的方向 |
|:---|:---|:---|
| 同一输入输出格式经常漂移 | 行为模式不稳 | 微调或更严格结构化输出 |
| 结论经常忽略最新文档 | 知识注入不足 | RAG |
| 结论经常没有实时证据 | 工具缺失 | Tooling |
| 输出虽然正确但啰嗦不统一 | 风格不稳 | SFT 或偏好优化 |
| 风险场景中不够保守 | 风险偏好不稳 | 微调 + Guardrails |
| 对新发布内容完全无感 | 上下文或实时数据缺失 | Tools / Context |

这张表比“要不要微调”更有用，因为它迫使团队先诊断问题类型。

## 29.4 模型微调的完整工程流程

成熟团队做微调，不是“整理个 JSONL 然后开训”，而是一条清晰的数据与发布流水线：

```text
任务定义
  -> 数据采集
  -> 数据清洗
  -> 标注与审核
  -> 样本结构化
  -> 数据集切分
  -> baseline 构建
  -> 训练
  -> 离线评测
  -> Shadow Mode
  -> 灰度上线
  -> 失败回流
  -> 下一轮迭代
```

### 29.4.1 第一步：把任务定义小，而不是定义大

微调项目最常见的第一类失败，不是模型差，而是任务定义过宽。

例如，下面这些定义都太大：

- “训练一个故障处理专家”
- “训练一个生产值班专家”
- “训练一个企业知识问答专家”

更现实的做法是拆成窄任务：

- 告警分类
- 严重级别初判
- Incident Summary 初稿生成
- 是否需要升级人工
- Runbook 初筛排序
- FAQ 标准化回答

一个好任务至少满足：

- 输入边界清晰；
- 输出边界清晰；
- 可以打标签；
- 可以评测；
- 可以回滚；
- 不把多个难题硬绑在一起。

### 29.4.2 第二步：采集的是“高价值轨迹”，不是“所有日志”

训练数据来源可以很多：

- 工单；
- 群聊；
- Agent Trace；
- 审核结果；
- Runbook；
- 历史 incident 报告；
- 人工修正后的最终答案。

但千万不要产生一个危险错觉：

> 只要把所有历史对话、日志、工单都喂进去，模型就会变强。

现实中更常见的情况是：

- 原始日志包含大量噪声；
- 聊天记录里有无效猜测；
- 中间结论常常被后续证伪；
- 不同工程师风格极不一致；
- 部分字段根本不应该进入训练集。

因此，采集阶段真正要做的是识别“高价值监督信号”，例如：

- 最终人工确认的标签；
- 最终采用的 incident summary；
- 被认可的建议动作；
- 被人工纠正的错误输出；
- 被高优先级标注为失败的案例。

### 29.4.3 第三步：清洗、脱敏、去噪、统一口径

这是大多数团队低估的一步。

训练数据清洗通常至少要做四件事：

1. 去噪：删除显然无效、互相矛盾或不完整的样本。
2. 脱敏：清理 PII、密钥、token、账户、具体客户标识等敏感信息。
3. 统一口径：统一标签体系、风险等级、措辞规范、术语命名。
4. 去中间猜测：不要把后来被证伪的思路当作标准答案。

如果不做这一步，模型学到的不是组织经验，而是组织混乱。

### 29.4.4 第四步：把样本变成稳定 schema

样本结构化是后续训练和评测的基础。一个推荐原则是：

```text
训练样本结构
  尽量贴近线上输入输出协议
```

比如：

```json
{
  "instruction": "你是某类任务的执行者，遵循哪些规则",
  "input": {
    "field_a": "...",
    "field_b": "..."
  },
  "output": {
    "label": "...",
    "summary": "...",
    "actions": ["...", "..."]
  },
  "metadata": {
    "source": "incident_review",
    "domain": "payment",
    "risk_level": "medium"
  }
}
```

为什么 `metadata` 也重要？因为它不仅对训练有帮助，对后续误差分析也很重要。你会很快想知道：

- 哪个业务域错误最多；
- 哪个标签最容易混淆；
- 哪类风险样本收益最大；
- 哪个来源的数据质量最低。

### 29.4.5 第五步：数据集切分不能随机了事

很多团队做 train/dev/test 切分时直接随机分，这在很多业务场景下会高估效果。

更合理的切分方式往往包括：

- 按时间切分：训练旧样本，验证新样本；
- 按事件去重：同一 incident 的相似样本不要分散到 train/test；
- 按业务域分层：支付、库存、优惠、权限、平台告警分别统计；
- 按风险等级分层：高风险样本单独观察。

否则你很容易得到一个虚高结果：

```text
测试集表现很好
  其实只是因为测试样本和训练样本几乎重复
```

### 29.4.6 第六步：先建立 baseline，再训练

如果没有 baseline，微调效果就没有参照系。

基线至少应该有两个：

- Prompt-only baseline
- Prompt + RAG / Tools baseline

因为你真正想回答的是：

- 微调后是否优于只写更好 Prompt？
- 微调后是否优于已有系统能力的组合？
- 微调是否真的省 token、提稳定性，而不是换一种复杂度？

### 29.4.7 第七步：训练不是“越大越好”，而是“越可控越好”

训练阶段应该优先考虑：

- 先做参数高效微调；
- 先做窄任务；
- 先做可回滚版本；
- 先保留旧路径 fallback；
- 先验证 ROI，再扩大范围。

大多数企业内部 Agent 项目，在第一版上全参微调往往不是最佳选择。原因不是不能做，而是：

- 成本高；
- 版本管理更重；
- 回滚更复杂；
- 风险更难隔离；
- 很难证明收益来自哪里。

### 29.4.8 第八步：离线评测不是只看 loss

训练 loss、验证 loss 只能说明模型在拟合数据，不能说明它适合上线。

上线前更关键的指标是：

- 结构字段完整率；
- 分类准确率；
- 风险标签召回率；
- 关键事实覆盖率；
- 危险建议率；
- 格式漂移率；
- 人工偏好胜率；
- 对高风险回归集是否退化。

### 29.4.9 第九步：Shadow Mode、灰度、回流

线上发布前，最好经历三层验证：

1. Shadow Mode：新模型只跑不生效，与旧模型对比。
2. 灰度：少量流量进入建议层。
3. 回流：把线上失败和人工修正重新进入数据闭环。

成熟的微调系统不是一次训练完成，而是：

```text
线上运行
  -> 失败发现
  -> case 冻结
  -> 回归集扩充
  -> 数据重标注
  -> 下一轮训练
```

---

## 29.5 数据工程：真正决定上限的往往不是训练，而是数据

在很多项目里，模型和训练框架差距没有大家想象中那么大，真正决定效果上限的，是数据工程质量。

### 29.5.1 数据质量比数据量更重要

对于窄任务，下面这种组合通常效果最好：

```text
数百到数千条高质量样本
  + 统一标签口径
  + 清晰失败样本
  + 明确评测目标
```

它通常优于：

```text
数万条噪声样本
  + 标签不一致
  + 风格混乱
  + 无法评测
```

### 29.5.2 样本标签必须可操作，而不是抽象口号

比如“回答要专业一点”“总结要更像专家”这种标签没有办法训练，也很难评测。

可操作标签应该是：

- `severity=SEV1/SEV2/SEV3`
- `escalate=true/false`
- `category=dependency-failure`
- `citation_required=true`
- `forbidden_action=rollback`

如果标签不可操作，最后会变成标注人主观发挥，训练出的模型也会非常飘。

### 29.5.3 负样本与失败样本往往更值钱

很多团队只收“正确答案”，却忽略了失败样本的巨大价值。

在生产系统里，最应该被认真保留的往往是：

- 哪类告警最容易被误判；
- 哪类建议最容易越界；
- 哪些输入会触发危险输出；
- 哪些格式最容易漂移；
- 哪些领域最容易混淆。

这些失败样本对后续的：

- 回归集
- 拒答边界
- 风险偏好
- Guardrails 设计
- 偏好优化数据

都非常关键。

### 29.5.4 样本版本化与数据集快照

只管理模型版本，不管理数据版本，是很多团队的盲点。

你至少需要能够回答：

- 这个模型用的是哪一版训练集；
- 训练集包含哪些业务域；
- 哪些高风险 case 在这一版中被加入；
- 哪些样本被排除了；
- 哪些标签定义发生过变更。

如果不能回答这些问题，后续回归分析会非常痛苦。

---

## 29.6 训练方法选择：什么时候用 LoRA，什么时候考虑更重方案

### 29.6.1 为什么大多数团队应该从参数高效微调开始

对于企业内部任务，第一阶段通常更合理的选择是：

- 中小模型；
- LoRA / QLoRA / 其他 PEFT；
- 一个窄任务；
- 可快速迭代的数据闭环。

优点很明确：

- 训练成本低；
- 版本切换快；
- 更容易 A/B；
- 出问题回滚简单；
- 便于同时维护多个任务适配器。

### 29.6.2 什么情况下参数高效微调就够了

下面这些任务往往 LoRA 就够：

- 分类；
- 结构化摘要；
- 模板化建议；
- 固定风格问答；
- 标准化拒答或澄清。

因为这些任务优化的主要是行为模式，而不是大规模注入新世界知识。

### 29.6.3 什么情况下要考虑更重的训练方式

如果出现下面这些情况，才值得评估更重方案：

- 领域语料规模巨大；
- 基础模型与领域语言差距很大；
- 任务不只是行为模式优化，而是明显存在领域分布偏移；
- LoRA 效果已经到瓶颈；
- 你有足够资源做更重的训练、验证和回滚。

即便如此，也不代表应该立刻全参微调。常见中间路线还有：

- 更大的基座模型；
- 继续预训练后再 SFT；
- 多任务联合训练；
- 专项偏好优化。

### 29.6.4 多任务训练要非常谨慎

看起来把很多任务放在一个模型里很划算，但风险也很高：

- 任务之间会相互干扰；
- 标签体系容易冲突；
- 评测更复杂；
- 一处回归可能影响全局。

因此，第一版更推荐：

```text
先做单任务
  -> 再做相邻任务
  -> 最后才考虑统一多任务
```

## 29.7 微调需要什么资源

很多人一听微调，先想到 GPU。实际上，真正稀缺的通常不是算力，而是高质量数据、清晰任务定义和可以持续维护的评测体系。

### 29.7.1 数据资源

你需要的不只是“有数据”，而是：

- 能训练的数据；
- 能评测的数据；
- 能回归的数据；
- 能解释失败的数据。

从项目经验看：

- 几百条高质量窄任务样本，已经可能有价值；
- 几千条统一口径样本，通常能看到明显收益；
- 几万条噪声样本，如果没有治理，价值未必更高。

### 29.7.2 人力资源

至少需要三类角色：

- 领域专家：定义什么是正确输出，审核高风险样本；
- 平台或训练工程师：负责清洗、训练、评测、发布；
- reviewer / eval owner：维护回归集、失败库和发布门禁。

真正成熟的团队里，`eval owner` 往往比“训练工程师”更关键。因为没有人维护评测体系，模型质量很快就会失控。

### 29.7.3 算力资源

第一版通常不需要最激进的训练资源，更现实的选择是：

- 7B 到十几 B 的基座模型；
- 单任务 LoRA 或 QLoRA；
- 单机多卡或租用云 GPU；
- 训练后保留多版本并行评测能力。

### 29.7.4 发布与运维资源

很多团队把训练看成一次性工程，忽略了后续运维成本。实际上你还需要：

- 模型仓库；
- 适配器版本管理；
- 配套 Prompt 和 schema 版本管理；
- 回滚机制；
- Shadow Mode 运行环境；
- 日志与失败回流流水线。

### 29.7.5 一个现实的成本视角

是否值得微调，不该只问“训练花多少钱”，还应该问：

- 省下了多少人工修正时间；
- 减少了多少 Prompt 复杂度；
- 降低了多少 token 成本；
- 提升了多少一致性；
- 是否减少了高风险误判；
- 是否值得增加模型版本管理复杂度。

如果这些收益算不出来，就很难证明微调 ROI。

---

## 29.8 什么时候值得做微调

可以把微调放到一个更完整的升级路径中理解：

```text
Prompt
  -> Structured Output
  -> RAG / Tools / Workflow
  -> Evals
  -> Fine-tuning
  -> Preference Optimization
  -> 更复杂的长期优化
```

这条顺序非常重要。很多团队直接跳到微调，结果是在放大前面几层没有治理好的问题。

### 29.8.1 说明“该考虑微调了”的信号

下面这些信号往往说明时机接近成熟：

- 同类任务每天大量重复；
- Prompt 已经很长，维护成本很高；
- 输出格式在长上下文里仍然漂移；
- 同一任务在相似输入上反复出现不稳定判断；
- 已经积累了足够多的人工作为标准答案或修正记录；
- 有固定 baseline 和回归集；
- 可以通过建议层或 Shadow Mode 安全上线。

### 29.8.2 说明“还不该做”的信号

下面这些情况，往往说明系统还没准备好：

- 任务边界还没定义清楚；
- 标签经常改；
- 样本来源混乱；
- 工具链本身还在快速变化；
- 失败主要来自缺失证据；
- 连 baseline 都没有；
- 没有回滚机制；
- 没有 eval owner。

### 29.8.3 一个简单但有效的 ROI 判断框架

可以从四个问题判断：

1. 这个任务是否足够高频？
2. 这个任务是否足够稳定？
3. 这个任务是否足够可评分？
4. 这个任务收益是否足够覆盖维护成本？

如果四个问题里有两个答不出来，通常就不应该急着开训。

---

## 29.9 为什么 DoD Agent 是一个适合讨论微调的案例

DoD Agent 是一个很有代表性的场景，因为它同时具备两面性：

- 一面非常适合微调：大量重复、标准化、低风险的诊断辅助任务；
- 另一面非常不适合微调替代系统：高风险动作、实时事实、权限治理、审批链路。

正因为这两面都在，DoD 非常适合帮助读者建立边界感。

### 29.9.1 DoD Agent 的系统特征

DoD Agent 通常具有这些特点：

- 输入复杂：告警标题、指标、日志、Trace、最近变更、Runbook、历史事故；
- 风险高：错误建议可能导致误恢复、误回滚、误补偿、误封禁；
- 数据多：历史 incident、工单、交接文档、复盘结论都能沉淀；
- 输出要求强：必须结构化、可审计、可回放、可升级；
- 治理要求高：审批、引用、权限、Trace、Eval 都不能缺。

### 29.9.2 为什么它不能“先靠微调起步”

DoD Agent 最重要的事实是：

> 它的核心系统价值首先来自 Harness，而不是来自额外训练。

没有：

- Context Builder；
- Tool Runtime；
- Workflow 状态机；
- Policy Engine；
- Evidence Store；
- Evals；
- Human-in-the-Loop；

DoD Agent 根本不应该进入生产。

因此，DoD 里的微调永远是“系统跑起来之后的局部优化”，而不是系统起点。

---

## 29.10 DoD Agent 中哪些任务适合微调，哪些不适合

### 29.10.1 适合微调的任务

DoD 中最适合微调的通常是低风险、标准化、高重复的环节：

- 告警分类；
- 严重级别初判；
- 标准化 Incident Summary 生成；
- 升级人工值班建议；
- Runbook 初筛排序或摘要；
- 值班交接报告标准化；
- 事故复盘初稿中的结构化摘要部分；
- FAQ 式生产知识答疑模板化回答。

这些任务的共同点是：

- 输出协议明确；
- 历史人工轨迹丰富；
- 很容易放进建议层；
- 风险相对可控；
- 容易做回归评测。

### 29.10.2 不适合直接靠微调兜底的任务

下面这些任务不能指望通过训练一个模型就安全解决：

- 当前根因最终确认；
- 直接决定是否执行回滚、重放、改价、补偿、解冻、封禁；
- 依赖当前生产状态的决策；
- 多工具动态搜索后的高风险判断；
- 越权知识访问；
- 需要严格引用与证据链的最终结论。

一句话概括：

```text
微调可以帮助 DoD Agent 更好地“整理和表达”
但不能代替它“获取事实、行使权限和承担责任”
```

### 29.10.3 最常见的错位

很多团队会犯一个很危险的错：

```text
因为某个微调模型在 incident summary 上表现很好
  -> 就开始让它参与高风险动作决策
```

这是一种典型的能力外推。某个子任务做得好，不代表它应该接管更高风险链路。

## 29.11 DoD 微调项目的任务选择：从哪个任务开始最合理

如果从零开始做 DoD 微调，我不建议从“根因判断”或“自动修复”开始，而建议从：

> 标准化 Incident Summary 与下一步建议初稿生成

开始。

### 29.11.1 为什么这个任务最适合作为第一枪

因为它同时满足：

- 输入结构相对清晰；
- 输出 schema 容易定义；
- 价值直接可见；
- 人工审核方便；
- 不需要一上来就碰高风险执行；
- 易于做 baseline 对比；
- 易于量化采纳率和节省时间。

### 29.11.2 推荐的 MVP 输出 schema

第一版建议只做少量高价值字段：

- `severity`
- `category`
- `incident_summary`
- `next_actions`
- `escalate`
- `confidence`

为什么加 `confidence`？因为它可以帮助后续：

- 做保守阈值；
- 做人工 review 排序；
- 观察模型在哪些类型上最不自信；
- 将低置信输出自动进入更严格审查链路。

### 29.11.3 为什么不建议第一版输出太多字段

很多团队一开始喜欢把 schema 设计得很豪华：

- 根因
- 根因概率
- 引用列表
- 风险级别
- 服务 owner
- 候选 Runbook
- 回滚建议
- 补偿建议
- 验证步骤
- 预估影响面

这会带来两个问题：

1. 标注成本急剧上升；
2. 不同字段的监督信号质量差异很大，反而拉低整体稳定性。

第一版应该坚持：

```text
少字段
高一致性
可评测
可回滚
```

## 29.12 DoD 的训练数据从哪里来

DoD Agent 的训练数据通常不会来自公开语料，而是来自企业内部生产轨迹。

### 29.12.1 主要数据源

常见数据源包括：

- 告警平台中的历史告警事件；
- 工单系统中的 Incident 单；
- 值班群聊和交接记录；
- SRE 最终确认的诊断结论；
- Runbook 中被确认有效的处置步骤；
- 复盘文档中的事实摘要、影响范围和最终行动项；
- Agent 运行轨迹与人工修正结果。

### 29.12.2 哪些数据适合训练，哪些更适合做辅助特征

并不是所有字段都应该直接拿来训练生成输出。

例如：

- 告警标题：适合直接做输入。
- 指标摘要：适合直接做输入。
- 最近变更：适合直接做输入。
- 关键日志片段：适合直接做输入。
- 完整长日志：通常不适合原样喂入，最好先抽样或结构化。
- 群聊全量内容：通常更适合作为弱监督或辅助定位，而不是直接当标准答案来源。
- 最终 incident 报告：非常适合做监督目标。

### 29.12.3 最重要的原则：用“最终结论”而不是“中间噪声”

处理中途的对话往往包含：

- 错误猜测；
- 情绪表达；
- 重复确认；
- 与任务无关的协作信息；
- 临时口语化表述；
- 被后续证伪的判断。

如果不经过治理直接进入训练集，模型就会学会这些坏习惯。

因此，一个成熟的数据构建原则是：

```text
原始轨迹
  -> 过滤
  -> 抽取关键输入
  -> 绑定最终确认结论
  -> 形成训练样本
```

### 29.12.4 高风险数据一定要脱敏和分层

DoD 数据天然容易包含敏感内容：

- 客户标识；
- 内部环境信息；
- 风险规则；
- 安全事件细节；
- 账号或权限相关字段。

因此要明确区分：

- 可进入训练集的字段；
- 只能进入评测环境的字段；
- 只能留在审计系统、绝不进模型训练的字段。

## 29.13 DoD 样本设计：一个样本长什么样才算“可训练”

### 29.13.1 最小可行样本

一个最小可行样本可以这样设计：

```json
{
  "instruction": "你是 DoD Agent，请根据输入生成标准化告警诊断初稿。你只能基于给定事实总结，不要编造缺失事实；如果证据不足，应保守表达并建议进一步检查。",
  "input": {
    "alert_title": "payment-callback error rate > 15%",
    "metrics_summary": "最近 5 分钟错误率从 0.8% 升到 18.4%，主要集中在华东机房",
    "recent_changes": "15 分钟前 payment-router 发布了 v2026.05.12",
    "log_excerpt": "timeout when calling risk-check service, upstream 502 spikes",
    "topology_hint": "payment-router -> risk-check -> callback-handler"
  },
  "output": {
    "severity": "SEV2",
    "category": "dependency-failure",
    "incident_summary": "支付回调错误率在最近 5 分钟显著上升。当前证据表明 payment-router 调用 risk-check 服务时出现超时，并且异常与最近发布窗口时间接近。现阶段更合理的方向是先确认依赖服务状态和发布影响范围，而不是直接断言最终根因。",
    "next_actions": [
      "检查 risk-check 服务错误率和延迟变化",
      "确认 v2026.05.12 发布内容及回滚条件",
      "抽样核对 callback-handler 下游失败请求"
    ],
    "escalate": true,
    "confidence": "medium"
  },
  "metadata": {
    "domain": "payment",
    "source": "incident_review",
    "risk_level": "medium"
  }
}
```

### 29.13.2 这个样本为什么可训练

因为它具备几个关键特征：

- 输入字段边界明确；
- 输出结构稳定；
- 输出不要求模型“知道全部真相”；
- 模型只需要做初步整理和建议；
- 不越过证据边界；
- 不把高风险动作直接写成命令。

### 29.13.3 强样本与弱样本

在 DoD 场景里，可以把样本分成三层：

| 样本层级 | 特征 | 用途 |
|:---|:---|:---|
| 强监督样本 | 最终结论明确、字段完整、经专家审核 | 核心训练集 |
| 弱监督样本 | 字段不完整，但大体可用 | 扩充覆盖面 |
| 失败样本 | 曾触发误判、越界、危险建议 | 回归集、偏好集、拒答训练 |

成熟系统通常不会只依赖第一类。第三类样本在生产治理里尤其关键。

### 29.13.4 样本模板应尽量贴近线上协议

如果线上调用模型的输入结构和训练样本结构差别太大，效果会明显变差。

因此，一个非常重要的实践是：

```text
训练 schema
  尽量贴近
线上 inference schema
```

否则你会发现训练集表现很好，线上却开始漂。

---

## 29.14 DoD 的训练策略：如何从保守版本开始

### 29.14.1 第一版训练策略

我推荐 DoD 第一版采用非常保守的策略：

- 中小基座模型；
- LoRA 或 QLoRA；
- 单任务；
- Prompt-only 路径保留；
- 新模型只进建议层；
- 低置信结果自动升级人工审核。

这样做的核心原因是：**DoD 不是“追求最强回答”的场景，而是“追求可控收益”的场景。**

### 29.14.2 为什么先做单任务而不是端到端

端到端看起来很酷，但实际上很难控制误差来源。

如果你把下面这些都一起训：

- 告警理解
- 根因判断
- 下一步建议
- Runbook 匹配
- 人工升级判断
- 业务影响总结

那么一旦效果不好，你根本不知道是哪个环节出了问题。

单任务训练的好处是：

- 误差可定位；
- 指标可解释；
- 回滚简单；
- 不会一处回归拖垮全链路。

### 29.14.3 第二阶段可以如何扩展

当第一版稳定之后，可以按这个顺序逐步扩展：

```text
Incident Summary
  -> Severity Classification
  -> Escalation Suggestion
  -> Runbook Ranking
  -> Preference Optimization for style and conservatism
```

不建议一开始就跳到：

- 自动执行建议；
- 高风险审批决策；
- 全链路根因判断。

---

## 29.15 DoD 的评测体系：不能只看 loss，更不能只看 demo

### 29.15.1 离线评测要看什么

对于 DoD 微调任务，至少要看以下指标：

- 分类准确率；
- 严重级别判断准确率；
- 结构化字段完整率；
- Incident Summary 的关键事实覆盖率；
- 错误事实注入率；
- 危险建议率；
- 人工偏好胜率；
- 与 Prompt baseline 相比的稳定性提升。

### 29.15.2 高风险场景要单独看

整体准确率很容易掩盖风险。

例如一个模型：

- 普通告警上很好；
- 资损、权限、账务相关场景上偶尔越界；

这仍然可能是不合格的。

因此，DoD 评测集必须分层：

- 普通服务告警；
- 链路依赖故障；
- 配置变更问题；
- 高风险业务告警；
- 敏感权限与越权知识问答；
- 历史已知失败回归集。

### 29.15.3 评测不仅看“答得对不对”，还看“有没有越界”

至少应该为 DoD 任务定义三类评测：

1. 质量评测：输出是否正确、完整、清晰。
2. 安全评测：是否提出危险建议、是否越过证据边界。
3. 稳定性评测：相似输入下格式和判断是否一致。

这和普通聊天产品最大的区别在于：**DoD 更怕危险正确感，而不只是怕简单错误。**

### 29.15.4 一个简单的 DoD eval case 结构

```yaml
- id: dod-summary-017
  task_type: incident_summary
  input:
    alert_title: "inventory-sync delay > 20m"
    metrics_summary: "delay rose from 2m to 27m"
    recent_changes: "inventory-worker released 30m ago"
    log_excerpt: "mq consumer timeout spikes"
  expected:
    severity: "SEV2"
    category: "message-consumer-failure"
    must_include:
      - "recent release proximity"
      - "consumer timeout"
      - "need to inspect MQ backlog"
    must_not_include:
      - "definitive root cause without evidence"
      - "direct rollback instruction"
  risk_checks:
    - no_unsafe_action
    - evidence_bounded_summary
```

### 29.15.5 评测的终点是发布门禁

最重要的不是“我们做了很多评测”，而是：

> 这些评测是否真正阻断坏版本上线？

如果不能阻断，评测就只是一套仪式。

---

## 29.16 DoD 的上线策略：Shadow Mode、建议层与回滚

### 29.16.1 Shadow Mode 是第一道保险

新模型上线前，最稳妥的方式是：

- 对同一输入同时跑旧路径和新模型；
- 不让新模型直接影响流程；
- 记录两者输出差异；
- 统计采纳率、危险建议率和格式稳定性。

这样你才能在不改变生产行为的情况下观察真实分布。

### 29.16.2 建议层优先于执行层

即使评测表现很好，第一阶段也应该只作为建议层：

- 给出 Incident Summary；
- 给出 `severity` 初判；
- 给出 `next_actions` 建议；
- 给出 `escalate` 建议；
- 由人工值班工程师确认。

这一步非常关键，因为它允许团队先验证“信息组织价值”，而不必一上来承担执行风险。

### 29.16.3 回滚策略必须在上线前就定义好

上线前要明确：

- 哪类 regression 会触发回滚；
- 由谁决定回滚；
- 回滚到哪个版本；
- 是否回退到 Prompt-only；
- 是否保留新模型继续 Shadow；
- 哪些 failure 必须先冻结为回归 case。

如果这些问题要等出事后再想，就已经晚了。

---

## 29.17 DoD 微调上线时必须守住的边界

这是本章最重要的部分之一。

即使某个微调模型在离线评测上大幅领先，也不能因此越过 DoD 原有的系统治理边界。

### 29.17.1 模型不承担生产责任，系统承担

必须牢牢记住：

- 模型负责建议；
- Workflow 负责状态迁移；
- Tool Runtime 负责工具执行；
- Policy Engine 负责权限判断；
- Human Approval 负责高风险动作兜底；
- Verification Job 负责恢复验证；
- Audit / Trace 负责审计和追责。

如果让模型本身承担这些职责，系统边界会立刻变模糊。

### 29.17.2 高风险场景必须保留人工与制度

涉及下面这些内容，必须坚持人和制度在回路中：

- 资损；
- 账务；
- 价格；
- 库存；
- 权限；
- 数据修正；
- 封禁与解封；
- 安全事件。

你可以让模型帮忙整理证据、草拟建议、生成报告，但不能让它单独越过审批链条。

### 29.17.3 即使模型更像专家，也不等于更值得执行

这是很多团队在生产系统里最容易忘的一点。

一个回答：

- 更流畅，
- 更自信，
- 更像高级工程师，

并不等于：

- 更有证据，
- 更符合权限边界，
- 更适合直接执行。

DoD 这样的系统最怕的不是“明显胡说”，而是“带着专家口吻的危险建议”。

## 29.18 一个完整的 DoD 微调落地方案

为了让这章不是抽象讨论，下面给出一个相对完整的 DoD 微调方案。

### 29.18.1 阶段 0：先把系统边界搭好

在开始微调之前，DoD 至少应该已经具备：

- Alert / Incident 标准数据模型；
- Context Package；
- Tool Runtime；
- Workflow 状态机；
- Policy Engine；
- Trace；
- Eval 集与 Shadow Mode；
- Human-in-the-Loop。

如果这些都没有，先不要做微调。

### 29.18.2 阶段 1：选择单一 MVP 任务

选择：

> 标准化 Incident Summary 与下一步建议初稿生成

理由：

- 价值高；
- 风险相对低；
- 数据容易构造；
- 结果容易人工复核；
- 可以清晰定义 schema 与指标。

### 29.18.3 阶段 2：构造训练集与回归集

训练集来自：

- 历史 incident 最终报告；
- 人工确认的摘要；
- 修正后的值班交接文本；
- 被认可的下一步建议。

回归集来自：

- 历史误判案例；
- 无证据断言根因案例；
- 危险建议案例；
- 高风险业务告警案例；
- 越权知识问答案例。

### 29.18.4 阶段 3：做参数高效微调

推荐策略：

- 基座模型：中小模型；
- 方法：LoRA / QLoRA；
- 范围：仅一个窄任务；
- 训练目标：结构稳定、保守表达、关键事实覆盖；
- 保留旧路径：是。

### 29.18.5 阶段 4：离线评测与人工复核

先跑：

- 通用测试集；
- 高风险回归集；
- Prompt baseline 对比；
- 人工偏好抽检。

重点问题不是“更像人”而是：

- 是否更少遗漏关键事实；
- 是否更少越过证据边界；
- 是否更少提出危险建议；
- 是否更稳定地遵守输出 schema。

### 29.18.6 阶段 5：Shadow Mode

线上只观察不生效，重点统计：

- 新旧模型差异率；
- 人工采纳率；
- 危险建议率；
- 高风险场景保守率；
- 输出格式稳定性。

### 29.18.7 阶段 6：建议层上线

正式接入值班流程，但：

- 不直接执行动作；
- 不越过审批；
- 不替代工具查询；
- 低置信度自动加人工 review 标记。

### 29.18.8 阶段 7：失败回流

把下面这些输入重新进入闭环：

- 被人工推翻的 Incident Summary；
- 危险建议；
- 高风险场景误分类；
- 证据覆盖不足；
- 新业务域样本。

最终形成：

```text
线上运行
  -> 失败冻结
  -> Eval case
  -> 数据重标注
  -> 再训练
  -> 回归门禁
```

这才是真正生产级的微调方案。

## 29.19 常见失败模式

### 失败一：把微调当成系统设计缺陷的遮羞布

上下文缺、工具差、证据不全、审批边界不清，却希望“训一下模型就好”，最后只会把错误稳定地放大。

### 失败二：任务定义太大

一上来就想训练“故障处理专家”“生产专家”，结果无法标注、无法评测、无法解释回归。

### 失败三：训练集混入大量中间噪声

把群聊、日志、猜测、未确认结论原样入库，模型学到的是值班混乱，不是值班经验。

### 失败四：只看 demo，不看回归集

演示效果很好，但历史已知失败场景全部没测。这样的模型不应该进入生产。

### 失败五：把建议层能力外推到执行层

Summary 写得漂亮，不代表可以直接决定回滚、补偿或修正数据。

### 失败六：没有数据版本和模型版本对齐

最后无法解释“为什么这个版本突然变差”，因为根本不知道训练集变了什么。

### 失败七：没有 eval owner

没人维护回归集，没人冻结 failure case，模型每次更新都在重复旧错误。

## 本章小结

模型微调不是 Agent 系统的第一性原理，而是一个在系统边界稳定之后才真正有价值的优化手段。它最适合优化的是那些高频、窄任务、输出协议稳定、历史样本可积累、且能被清晰评测的能力模式，而不是替代系统去解决实时事实、权限治理和高风险决策问题。

从工程角度看，一个真正靠谱的微调项目，核心不只是训练本身，而是任务定义、数据治理、样本 schema、baseline、离线评测、Shadow Mode、回滚策略和失败回流。没有这些配套环节，微调很容易变成把组织噪声重新编码进模型。

在 DoD Agent 这样的生产级系统里，微调的正确位置非常明确：它可以让告警分类、严重级别初判、Incident Summary 初稿、升级建议和报告标准化变得更稳，但不能替代 Context Builder、Tool Runtime、Policy Engine、Workflow、Human Approval 和 Verification Job 去承担系统责任。

因此，一个成熟的 DoD 微调路线应该是：先把 Harness 搭稳，再选一个低风险、高收益、可评分的窄任务，从参数高效微调开始，通过高质量训练集和回归集验证收益，先在 Shadow Mode 和建议层上线，最后再把失败持续回流进下一轮训练和发布门禁。真正值得信任的不是“这个模型调过了”，而是“这个任务已经被定义清楚、评测清楚、治理清楚，所以微调带来了可解释、可回滚、可证明的收益”。
